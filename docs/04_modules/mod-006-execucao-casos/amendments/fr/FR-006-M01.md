### Alteracoes neste pilar

#### A1 — FR-009 (Listagem de Casos): adicionar `primary_assignee_name`

O item 6 do Done Funcional de FR-009 atualmente retorna:

```
codigo, cycle_name, current_stage_name, status, pending_gates_count
```

DEVE passar a retornar tambem:

```
primary_assignee_name, opened_at
```

**Regra de resolucao de `primary_assignee_name`:**
1. Buscar `case_assignments` WHERE `case_id = :id` AND `is_active = true` AND `stage_id = current_stage_id`
2. Se multiplas atribuicoes ativas, retornar a do papel com `is_primary = true` (ou o primeiro papel por ordem de `stage_role_links.sort_order`)
3. Incluir `user.name` via JOIN com tabela de usuarios
4. Se nenhuma atribuicao ativa: retornar `null`

**Impacto OpenAPI:** Adicionar campo `primary_assignee_name: string | null` e `opened_at: string (ISO 8601)` ao schema `CaseListItem` no contrato OpenAPI.

---

#### A2 — FR-001 (Abertura de Caso): aceitar `description`, `priority`, `notes`

O endpoint POST /api/v1/cases atualmente aceita `{ cycle_id, tenant_id }`.

DEVE passar a aceitar:

```json
{
  "cycle_id": "uuid",
  "tenant_id": "uuid",
  "description": "string (opcional, max 500 chars)",
  "priority": "NORMAL | HIGH | URGENT (default: NORMAL)",
  "notes": "string (opcional, max 1000 chars)"
}
```

**Done Funcional adicional:**
- Salvar `description` em `case_instances.description` (novo campo — verificar se DATA-006 ja possui ou se precisa adicionar)
- Salvar `priority` em `case_instances.priority` (novo campo — enum NORMAL/HIGH/URGENT, default NORMAL)
- Salvar `notes` como primeiro `case_event` com event_type=COMMENT (reutiliza FR-007)

**Nota DATA:** Os campos `description` e `priority` provavelmente requerem novas colunas em `case_instances`. Se DATA-006 nao possui esses campos, sera necessario um DATA-006-M01. Verificar schema antes do codegen.

**Impacto OpenAPI:** Adicionar campos `description`, `priority`, `notes` ao schema `CreateCaseRequest`. Adicionar `description`, `priority` ao schema `CaseInstance`.

---

#### A3 — FR-010 (Detalhes do Caso): incluir `description` e `priority`

O endpoint GET /api/v1/cases/:id DEVE retornar tambem os novos campos `description` e `priority` na response, pois o HeaderCard da CasePanelPage pode exibi-los.
