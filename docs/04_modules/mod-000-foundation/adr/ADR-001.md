# ADR-001 — Sessão Ancorada em Banco (Desvio de Stateless JWT)

- **Contexto:** O padrão JWT stateless é amplamente adotado, mas não permite revogação imediata de sessões. O Foundation necessita de Kill-Switch (revogar sessão a qualquer momento para um usuário comprometido ou bloqueado). Com JWT stateless puro, o token permanece válido até expirar, mesmo após logout ou bloqueio.

- **Decisão:** Adotar modelo híbrido — JWT carrega `sessionId` que é validado em banco a cada requisição via `verifyActiveSession()`. Sessão revogada (isRevoked=true) resulta em 401 imediato, mesmo com JWT ainda dentro do TTL.

- **Alternativas:**
  - JWT stateless puro com blacklist em Redis → complexidade de sincronização, TTL de blacklist
  - Tokens opacos (session-only, sem JWT) → perde benefícios de JWT (claims, federação)
  - JWT com TTL curto (5min) + refresh frequente → latência e UX degradada

- **Consequências:**
  - Toda requisição autenticada faz 1 query ao banco (user_sessions) → impacto de latência mitigado por índice `(user_id, is_revoked)`
  - Kill-Switch imediato e confiável (individual e global)
  - Overhead aceitável: query simples por PK, ~1-2ms com índice

- **Status:** accepted
- **Timebox:** N/A (decisão permanente para o Foundation)

# ADR-002 — Substituição Total de Escopos em Roles (PUT vs PATCH)

- **Contexto:** Na gestão de Roles/RBAC, a atualização de escopos poderia seguir dois modelos: substituição total (PUT) onde todos os escopos são reescritos, ou append/remove incremental (PATCH). O modelo incremental é mais complexo de implementar e pode levar a inconsistências quando múltiplos admins editam a mesma role simultaneamente.

- **Decisão:** PUT /roles/:id substitui completamente os escopos (DELETE all + INSERT new em transação). Não é append.

- **Alternativas:**
  - PATCH incremental (add/remove individual) → mais complexo, risco de inconsistência concorrente
  - Versionamento otimista com ETag → overhead desnecessário para escala atual

- **Consequências:**
  - Operação atômica e previsível
  - Frontend deve enviar lista completa de escopos (não apenas diff)
  - Cache Redis invalidado após cada operação
  - Simplicidade de implementação e debugging

- **Status:** accepted
- **Timebox:** N/A

# ADR-003 — Separação de Tabelas users / content_users

- **Contexto:** Dados de autenticação (email, passwordHash, status) e dados de exibição (nome, CPF, avatar) têm ciclos de vida e requisitos de segurança diferentes. Manter tudo em uma tabela expõe campos sensíveis em queries de listagem e dificulta a aplicação de mascaramento diferenciado.

- **Decisão:** Separar em `users` (identidade/autenticação) e `content_users` (dados de exibição), com relação 1:1 via FK `content_users.user_id → users.id`. Inserção e deleção atômicas via transação Drizzle.

- **Alternativas:**
  - Tabela única com views de projeção → mais simples, mas views não protegem contra `SELECT *`
  - Coluna JSONB para dados de perfil → dificulta queries de busca e índices

- **Consequências:**
  - Queries de listagem buscam apenas `content_users` (sem passwordHash)
  - Inserção/deleção exigem transação (2 tabelas)
  - Mascaramento natural: campos sensíveis isolados em tabela separada

- **Status:** accepted
- **Timebox:** N/A
