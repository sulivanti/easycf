# FR-003 — Fluxo de Convite e Ativação de Usuário

- **Descrição:** Tela de acompanhamento do status do convite de um usuário com card de status contextual por estado (PENDING/ACTIVE/BLOCKED/INACTIVE), botão de reenvio com cooldown anti-spam de 60s, alerta de token expirado e navegação de volta preservando filtros da listagem.
- **Prioridade:** Must
- **Perfis impactados:** Administrador (gestão), Administrador (completo)
- **Done funcional:**
  - [ ] Acesso a /usuarios/:id/convite requer scope `users:user:read` — redirect sem ele (BR-001)
  - [ ] Skeleton no card de status durante carregamento de `users_get`
  - [ ] Breadcrumb: "Usuários > Convite" (sem nome até resposta); com nome após load
  - [ ] Título da tela: "Convite — {nome}" — nome, nunca e-mail (BR-002)
  - [ ] Card status PENDING: badge âmbar "Aguardando ativação" + botão "Reenviar convite" habilitado
  - [ ] Card status PENDING + token expirado: alerta âmbar "O link de ativação expirou. Reenvie o convite."
  - [ ] Card status ACTIVE: badge verde "Ativo" + texto "O usuário ativou a conta." + botão NÃO renderizado
  - [ ] Card status BLOCKED: badge vermelho "Bloqueado" + botão desabilitado + tooltip explicativo
  - [ ] Card status INACTIVE: badge cinza "Inativo" + botão NÃO renderizado
  - [ ] 404 (usuário não encontrado): mensagem inline + sem botão de reenvio (BR-006)
  - [ ] Reenvio bem-sucedido (200): Toast "Convite reenviado com sucesso." sem e-mail (BR-002) + cooldown 60s (BR-004)
  - [ ] Cooldown: countdown em tempo real, botão desabilitado, reabilita após 60s (BR-004)
  - [ ] Erro no reenvio (500): Toast + correlationId, botão habilitado sem cooldown (BR-004)
  - [ ] Erro 422 (status mudou para ACTIVE): Toast genérico + reload da tela (BR-006)
  - [ ] Botão voltar / breadcrumb "Usuários": redirect para /usuarios preservando filtros (query params)
- **Efeito colateral?** [x] Sim — `users_invite_resend` (POST reenvia e-mail de ativação) *(Idempotência via Idempotency-Key com TTL 60s — BR-005)*
- **Idempotência:** required (Idempotency-Key UUID v4 com TTL 60s — BR-005)
- **Timeline/Notificações:** Sim — reenvio emite `user.invite_resent` (DATA-003, SEC-002)
- **Dependências (IDs):** BR-001, BR-002, BR-004, BR-005, BR-006, DATA-003, INT-001, SEC-001, UX-001, NFR-001

### operationIds Consumidos

| operationId | Ação | Método |
|---|---|---|
| `users_get` | Carregar dados e status do usuário | GET /api/v1/users/:id |
| `users_invite_resend` | Reenviar convite de ativação | POST /api/v1/users/:id/invite/resend |

### Amendment Obrigatório

> `users_invite_resend` requer amendment no MOD-000-F05 **antes** do scaffolding desta feature.
> Contrato mínimo documentado em US-MOD-002-F03 §1.

### Critérios de aceite (Gherkin)

```gherkin
Funcionalidade: Fluxo de Convite e Ativação — FR-003

  Cenário: Status PENDING com reenvio bem-sucedido
    Dado que GET /api/v1/users/:id retorna status=PENDING
    E o admin clica "Reenviar convite"

    Então o card exibe badge verde "Ativo"
    E o botão "Reenviar convite" NÃO é renderizado

  Cenário: Erro 500 não inicia cooldown
    Dado que POST /api/v1/users/:id/invite/resend retorna 500
    Então Toast de erro exibe mensagem + correlationId
    E o botão fica habilitado sem cooldown

  Cenário: Navegação preserva filtros da listagem
    Dado que o admin veio de /usuarios?status=PENDING
    Quando clica "Voltar" ou breadcrumb "Usuários"
    Então é redirecionado para /usuarios?status=PENDING
```
