# FR-002 — Requisitos Funcionais de Departamentos

---

## FR-007 — CRUD de Departamentos

- **Descrição:** O sistema DEVE prover API CRUD completa para departamentos (entidade flat por tenant), incluindo: criação com idempotência (`POST /departments`), listagem com cursor pagination e filtros (`GET /departments`), detalhe (`GET /departments/:id`), atualização parcial de nome/descrição/cor (`PATCH /departments/:id`), soft delete (`DELETE /departments/:id`), e restore (`PATCH /departments/:id/restore`).
- **Prioridade:** Must
- **Perfis impactados:** Administrador organizacional
- **Done funcional:**
  - POST /departments cria departamento vinculado ao tenant do usuário autenticado
  - GET /departments lista com cursor pagination, filtros por status e search (nome/codigo)
  - GET /departments/:id retorna departamento completo
  - PATCH /departments/:id atualiza nome, descricao, cor — codigo é imutável (BR-014)
  - DELETE /departments/:id executa soft delete (BR-015)
  - PATCH /departments/:id/restore restaura departamento (BR-016)

---

### Endpoints

#### POST /api/v1/departments — Criar Departamento

- **Scope:** `org:dept:write`
- **Headers:** `Idempotency-Key` (obrigatório — ADR-004), `Authorization: Bearer <token>`
- **Request Body:**

```json
{
  "codigo": "DEPT-ENG",
  "nome": "Engenharia Civil",
  "descricao": "Departamento de engenharia civil e infraestrutura",
  "cor": "#2E86C1"
}
```

| Campo | Tipo | Obrigatório | Validação |
|---|---|---|---|
| `codigo` | string | Sim | max 50 chars, pattern: `^[A-Za-z0-9_-]+$` |
| `nome` | string | Sim | max 200 chars, min 1 char |
| `descricao` | string | Não | max 2000 chars |
| `cor` | string/null | Não | pattern: `^#[0-9A-Fa-f]{6}$` ou null |

- **Response 201:**

```json
{
  "id": "uuid",
  "tenant_id": "uuid",
  "codigo": "DEPT-ENG",
  "nome": "Engenharia Civil",
  "descricao": "Departamento de engenharia civil e infraestrutura",
  "status": "ACTIVE",
  "cor": "#2E86C1",
  "created_by": "uuid",
  "created_at": "2026-03-30T12:00:00.000Z",
  "updated_at": "2026-03-30T12:00:00.000Z"
}
```

- **Erros:** 409 (codigo duplicado — BR-013), 422 (validação), 401 (não autenticado), 403 (sem scope)
- **Domain event:** `org.dept_created` (payload: snapshot mínimo)
- **Soft limit:** Se count > 80 (80% de 100), response inclui header `X-Limit-Warning` (BR-018)
- **tenant_id:** Derivado do JWT do usuário autenticado (não aceito no body)

#### GET /api/v1/departments — Listar Departamentos

- **Scope:** `org:dept:read`
- **Query params:**

| Param | Tipo | Default | Descrição |
|---|---|---|---|
| `cursor` | string | — | Cursor para paginação (base64 de id) |
| `limit` | integer | 20 | Itens por página (max 100) |
| `status` | string | `ACTIVE` | Filtro: `ACTIVE`, `INACTIVE`, `ALL` |
| `search` | string | — | Busca por nome ou codigo (ILIKE) |

- **Response 200:**

```json
{
  "data": [
    {
      "id": "uuid",
      "codigo": "DEPT-ENG",
      "nome": "Engenharia Civil",
      "status": "ACTIVE",
      "cor": "#2E86C1",
      "created_at": "2026-03-30T12:00:00.000Z"
    }
  ],
  "next_cursor": "base64-cursor-or-null",
  "has_more": true
}
```

- **Nota:** Listagem retorna subset de campos (sem descricao, sem created_by, sem updated_at, sem deleted_at). Filtro por tenant_id automático via JWT. Response em formato flat (`data` + `next_cursor` + `has_more`), conforme `paginatedResponse` do Foundation (DOC-ARC-001).

#### GET /api/v1/departments/:id — Detalhe do Departamento

- **Scope:** `org:dept:read`
- **Response 200:**

```json
{
  "id": "uuid",
  "tenant_id": "uuid",
  "codigo": "DEPT-ENG",
  "nome": "Engenharia Civil",
  "descricao": "Departamento de engenharia civil e infraestrutura",
  "status": "ACTIVE",
  "cor": "#2E86C1",
  "created_by": "uuid",
  "created_at": "2026-03-30T12:00:00.000Z",
  "updated_at": "2026-03-30T12:00:00.000Z"
}
```

- **Erros:** 404 (não encontrado ou outro tenant), 401, 403

#### PATCH /api/v1/departments/:id — Atualizar Departamento

- **Scope:** `org:dept:write`
- **Request Body (partial update):**

```json
{
  "nome": "Engenharia Civil Atualizada",
  "descricao": "Nova descrição",
  "cor": "#27AE60"
}
```

| Campo | Tipo | Obrigatório | Validação |
|---|---|---|---|
| `nome` | string | Não | max 200 chars, min 1 char |
| `descricao` | string/null | Não | max 2000 chars |
| `cor` | string/null | Não | pattern: `^#[0-9A-Fa-f]{6}$` ou null |

- **Campos proibidos:** `codigo` → 422 (BR-014), `tenant_id` → ignorado, `status` → ignorado (usar DELETE/restore)
- **Response 200:** Departamento completo atualizado (mesmo schema de GET :id)
- **Domain event:** `org.dept_updated` (payload: diff only)

#### DELETE /api/v1/departments/:id — Soft Delete

- **Scope:** `org:dept:delete`
- **Response 204:** No content
- **Erros:** 422 (já inativo — BR-015), 404, 401, 403
- **Domain event:** `org.dept_deleted` (payload: metadata — id, codigo, tenant_id)

#### PATCH /api/v1/departments/:id/restore — Restaurar Departamento

- **Scope:** `org:dept:write`
- **Response 200:** Departamento completo restaurado (mesmo schema de GET :id)
- **Erros:** 422 (já ativo — BR-016), 404, 401, 403
- **Domain event:** `org.dept_restored` (payload: metadata — id, codigo, tenant_id)

### Critérios de Aceitação (Gherkin)

```gherkin
Funcionalidade: CRUD de Departamentos

# --- CREATE ---

Cenário: Criar departamento com dados válidos
  Dado que o usuário tem scope "org:dept:write"
  Quando POST /api/v1/departments com { codigo: "DEPT-DIR", nome: "Diretoria", cor: "#2E86C1" } e Idempotency-Key válida
  Então deve retornar 201 com id, tenant_id, status="ACTIVE"
  E deve emitir domain event "org.dept_created"

Cenário: Criar departamento com codigo duplicado no tenant retorna 409
  Dado que existe departamento com codigo="DEPT-DIR" no tenant do usuário
  Quando POST /api/v1/departments com { codigo: "DEPT-DIR", nome: "Outro" }
  Então deve retornar 409 com type="/problems/conflict"

Cenário: Idempotency-Key previne duplicação em POST
  Dado que POST /departments foi chamado com Idempotency-Key "key-abc" e retornou 201
  Quando o mesmo POST é chamado novamente com Idempotency-Key "key-abc"
  Então deve retornar 201 com o mesmo recurso sem criar duplicado

Cenário: Soft limit warning ao ultrapassar 80%
  Dado que o tenant possui 81 departamentos ativos
  Quando POST /api/v1/departments com dados válidos
  Então deve retornar 201
  E o response header X-Limit-Warning deve conter "departments_count=82"

# --- READ ---

Cenário: Listar departamentos com filtro de status
  Dado que o usuário tem scope "org:dept:read"
  Quando GET /api/v1/departments?status=ACTIVE
  Então deve retornar 200 com array de departamentos ativos do tenant
  E cada item deve conter id, codigo, nome, status, cor, created_at

Cenário: Buscar departamento por nome
  Dado que existem departamentos "Diretoria" e "Engenharia"
  Quando GET /api/v1/departments?search=eng
  Então deve retornar 200 com array contendo "Engenharia"
  E array NÃO deve conter "Diretoria"

Cenário: Detalhe de departamento existente
  Dado que existe departamento com id=":id" no tenant do usuário
  Quando GET /api/v1/departments/:id
  Então deve retornar 200 com todos os campos incluindo descricao

# --- UPDATE ---

Cenário: Atualizar nome e cor do departamento
  Dado que existe departamento com id=":id"
  Quando PATCH /api/v1/departments/:id com { nome: "Novo Nome", cor: "#27AE60" }
  Então deve retornar 200 com nome="Novo Nome" e cor="#27AE60"
  E deve emitir domain event "org.dept_updated"

Cenário: Tentativa de alterar codigo retorna 422
  Dado que existe departamento com id=":id"
  Quando PATCH /api/v1/departments/:id com { codigo: "NOVO-COD" }
  Então deve retornar 422 com detail "O campo 'codigo' é imutável após criação."

Cenário: Cor inválida retorna 422
  Quando PATCH /api/v1/departments/:id com { cor: "#FFF" }
  Então deve retornar 422 com detail contendo "cor"

# --- DELETE / RESTORE ---

Cenário: Soft delete de departamento ativo
  Dado que existe departamento ativo com id=":id"
  Quando DELETE /api/v1/departments/:id
  Então deve retornar 204
  E deve emitir domain event "org.dept_deleted"

Cenário: Restaurar departamento inativo
  Dado que existe departamento inativo com id=":id"
  Quando PATCH /api/v1/departments/:id/restore
  Então deve retornar 200 com status="ACTIVE" e deleted_at=null
  E deve emitir domain event "org.dept_restored"
```

### Domain Events

| Evento | Quando | Payload policy | Scope de emissão |
|---|---|---|---|
| `org.dept_created` | POST 201 | snapshot mínimo (id, tenant_id, codigo, nome, cor, status, created_by) | `org:dept:write` |
| `org.dept_updated` | PATCH 200 | diff only (id, codigo, campos alterados) | `org:dept:write` |
| `org.dept_deleted` | DELETE 204 | metadata (id, codigo, tenant_id) | `org:dept:delete` |
| `org.dept_restored` | PATCH /restore 200 | metadata (id, codigo, tenant_id) | `org:dept:write` |
