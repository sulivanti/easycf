# ADR-003 — Idempotência Frontend via Idempotency-Key Header

## Contexto

O MOD-002 realiza 2 operações POST mutáveis: criação de usuário (`users_create`) e reenvio de convite (`users_invite_resend`). Em ambos os casos, double-click, retry automático do browser ou resubmissão acidental podem causar duplicação de registros ou reenvio excessivo de e-mails.

O backend (MOD-000-F05) suporta o header `Idempotency-Key` para deduplicação server-side. A questão é: **como o frontend deve gerar e gerenciar a chave de idempotência?**

## Decisão

**Gerar UUID v4 no mount do componente e propagá-lo via header `Idempotency-Key` em todas as requisições POST mutáveis.** O UUID é mantido até sucesso (2xx) e regenerado no próximo mount.

### Ciclo de vida da key

```text
1. Componente monta → UUID v4 gerado (síncrono, sem async)
2. Usuário submete → POST enviado com header Idempotency-Key: <uuid>
3. Double-click → mesma key → backend deduplica
4. Resposta 2xx → UUID descartado
5. Navegação/remount → novo UUID gerado
6. Resposta 4xx/5xx → mesmo UUID mantido para retry
```

### Endpoints cobertos

| Endpoint | Componente | Gerado em |
|---|---|---|
| `POST /api/v1/users` | `UserCreateForm` (UX-USR-002) | Mount do formulário |
| `POST /api/v1/users/:id/invite/resend` | `UserInviteScreen` (UX-USR-003) | Mount da tela |

### Complemento: disable de botão

O `Idempotency-Key` não substitui o disable imediato do botão submit (NFR-001 §3). Ambos são obrigatórios:

- **Disable:** UX — previne double-click visual
- **Idempotency-Key:** Infraestrutura — previne duplicação em casos de race condition, retry de rede ou resubmissão

## Alternativas Consideradas

### Alternativa A: Disable de botão apenas (sem header)

- Desabilitar botão após click e reabilitar após resposta.
- **Rejeitada:** Não protege contra retry de rede (browser retransmit), falhas intermitentes com resubmissão automática (service worker, interceptors) ou manipulação direta via DevTools. Insuficiente para garantia server-side.

### Alternativa B: Token de formulário (CSRF-like)

- Backend gera token único por sessão de formulário, frontend envia como campo hidden.
- **Rejeitada:** Requer endpoint extra para gerar token. Complexidade desnecessária quando `Idempotency-Key` no header resolve o problema sem round-trip adicional.

### Alternativa C: UUID gerado no mount + header (escolhida)

- UUID v4 gerado localmente, sem round-trip.
- **Vantagens:** Zero latência extra. Stateless no frontend (sem armazenamento persistente). Backend deduplica via `Idempotency-Key`. Padrão amplamente adotado (Stripe, AWS).
- **Tradeoffs:** Depende do backend implementar deduplicação por key. Validado: MOD-000-F05 suporta `Idempotency-Key`.

## Consequências

1. **Implementação:** Cada formulário mutável gera UUID v4 no `useEffect` de mount (ou equivalente). O UUID é passado ao mutation hook que injeta no header.
2. **Teste:** Cenários Gherkin de BR-005 cobrem double-click e regeneração pós-sucesso.
3. **Não usar localStorage:** A key vive apenas em memória React (state). Sem persistência entre sessões.
4. **Evolução futura:** Se importação em massa entrar no escopo, o pattern se estende naturalmente (1 key por batch request).
5. **Dependência:** MOD-000-F05 DEVE implementar deduplicação por `Idempotency-Key` com TTL adequado (recomendado: 24h).
