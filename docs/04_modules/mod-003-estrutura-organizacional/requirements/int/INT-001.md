## INT-001 — Consumo de Tenants (MOD-000-F07)

- **Sistema externo:** MOD-000 (Foundation API) — tabela `tenants`
- **Propósito:** Validar existência e status do tenant antes de criar vínculo N4-N5 via `org_unit_tenant_links`
- **Modo:** API (SQL direto via FK — mesmo banco, mesma transação)
- **Auth:** N/A (acesso interno via FK, não via HTTP)
- **Contrato:**
  - **Leitura:** `SELECT id, codigo, nome, status FROM tenants WHERE id = :tenant_id`
  - **Validação:** Tenant deve existir e estar acessível para vinculação
  - **Request example:** N/A (SQL direto, não HTTP)
  - **Response example:** `{ id: "uuid", codigo: "SP-01", nome: "Filial SP 01", status: "ACTIVE" }`
- **Comportamento de falha:**
  - **Timeout:** N/A (acesso interno, mesma transação DB)
  - **Retries:** N/A
  - **Backoff:** N/A
  - **DLQ:** Não
- **Idempotência:** UNIQUE (org_unit_id, tenant_id) garante idempotência natural (BR-012)
- **Fallback:** 404 se tenant não existir; 422 se tenant inativo
- **Observabilidade:** `["correlationId", "logs"]` — X-Correlation-ID propagado em domain_events

## INT-002 — Consumo de Catálogo de Escopos (MOD-000-F12)

- **Sistema externo:** MOD-000 (Foundation API) — catálogo de permissões (DOC-FND-000 §2.2)
- **Propósito:** Registrar os 3 escopos do MOD-003 no catálogo canônico de permissões do sistema
- **Modo:** Configuração (novos escopos adicionados ao catálogo via PR)
- **Auth:** N/A (configuração estática)
- **Contrato:**
  - Novos escopos registrados: `org:unit:read`, `org:unit:write`, `org:unit:delete`
  - Formato: `dominio:entidade:acao` (3 segmentos, conforme DOC-FND-000 §2.1)
  - Validação: Gate 3 (DOC-ARC-003B) — CI falha se scope não registrado
- **Comportamento de falha:** N/A (configuração estática, não runtime)
- **Observabilidade:** Escopos validados em runtime via RBAC middleware (`@RequireScope`) do MOD-000

## INT-003 — Consumo de Users (MOD-000)

- **Sistema externo:** MOD-000 (Foundation API) — tabela `users`
- **Propósito:** Referenciar o autor (`created_by`) de operações de escrita em `org_units` e `org_unit_tenant_links`
- **Modo:** API (SQL direto via FK — mesmo banco, mesma transação)
- **Auth:** N/A (acesso interno via FK, o `user_id` é extraído do token JWT pelo middleware de auth do MOD-000)
- **Contrato:**
  - FK: `org_units.created_by` → `users.id` (ON DELETE RESTRICT)
  - FK: `org_unit_tenant_links.created_by` → `users.id` (ON DELETE RESTRICT)
  - O `user_id` é injetado pelo middleware de autenticação, nunca aceito do body da requisição
- **Comportamento de falha:** N/A (FK constraint — se user não existir, a transação falha com erro de integridade)
- **Observabilidade:** `["correlationId", "logs"]` — `actor_id` gravado em domain_events via middleware

## INT-004 — Emissão de Domain Events (MOD-000 — domain_events)

- **Sistema externo:** MOD-000 (Foundation) — tabela `domain_events`
- **Propósito:** Registrar todas as operações de escrita do MOD-003 na tabela unificada de auditoria/timeline do Foundation
- **Modo:** API (SQL direto — insert na tabela `domain_events` dentro da mesma transação DB)
- **Auth:** N/A (acesso interno, mesma transação)
- **Contrato:**
  - **Escrita:** INSERT em `domain_events` com campos: `event_type`, `entity_type='org_unit'`, `entity_id`, `tenant_id`, `actor_id`, `correlation_id`, `payload`, `created_at`
  - **Eventos emitidos:** `org.unit_created`, `org.unit_updated`, `org.unit_deleted`, `org.unit_restored`, `org.tenant_linked`, `org.tenant_unlinked` (ver DATA-003)
- **Comportamento de falha:**
  - **Timeout:** N/A (mesma transação DB — se falhar, a operação principal também falha via rollback)
  - **Retries:** N/A
  - **DLQ:** Não (a consistência é garantida pela transação)
- **Idempotência:** Garantida pela transação atômica — evento só é persistido se a operação principal for commitada
- **Observabilidade:** `["correlationId", "logs"]` — cada evento inclui `correlation_id` para rastreio end-to-end

## INT-005 — Consumo de Idempotency Keys (MOD-000)

- **Sistema externo:** MOD-000 (Foundation) — mecanismo de idempotência
- **Propósito:** Garantir idempotência em endpoints de criação conforme BR-012 e DOC-ESC-001 §3.6 (Nível 1 MUST)
- **Modo:** API (middleware Fastify — intercepta header `Idempotency-Key` antes do handler)
- **Auth:** N/A (middleware interno)
- **Contrato:**
  - Header: `Idempotency-Key: <uuid-v4 ou string única>`
  - TTL: 60 segundos
  - Endpoints protegidos: `POST /api/v1/org-units`, `POST /api/v1/org-units/:id/tenants`
  - Comportamento: se chave já vista dentro do TTL, retorna response original (201) sem executar operação
  - Storage: tabela `idempotency_keys` do MOD-000 (não criar tabela paralela)
- **Comportamento de falha:**
  - Se mecanismo de idempotência falhar (ex: Redis/tabela indisponível): operação prossegue normalmente (fail-open)
  - Log de warning emitido para observabilidade
- **Observabilidade:** `["correlationId", "logs", "metrics"]` — métrica de cache hit/miss para idempotency

## INT-006 — Consumo de Error Handler RFC 9457 (MOD-000)

- **Sistema externo:** MOD-000 (Foundation) — middleware de tratamento de erros
- **Propósito:** Garantir que todos os erros HTTP do MOD-003 sigam o padrão RFC 9457 (DOC-FND-000 §5)
- **Modo:** Middleware (error handler global do Fastify)
- **Auth:** N/A (middleware interno)
- **Contrato:**
  - Erros mapeados: 400 (validação Zod), 404 (recurso não encontrado), 409 (duplicata código/vínculo), 422 (regras de negócio), 403 (scope insuficiente)
  - Response: `{ type, title, detail, status, correlationId }`
  - **409 Conflict:** catch PostgreSQL error code 23505 (unique_violation) → 409 com `detail` amigável (FR-001-C01)
- **Observabilidade:** `["correlationId"]` — correlationId copiado do `X-Correlation-ID`

## INT-007 — Consumo de Domain Events para Timeline/Histórico (MOD-000)

- **Sistema externo:** MOD-000 (Foundation API) — endpoint `GET /api/v1/domain-events`
- **Propósito:** Fornecer timeline/histórico de operações sobre unidades organizacionais na tela UX-ORG-001 (ação "Ver Histórico do Nó")
- **Modo:** API (HTTP GET — chamada do frontend diretamente ao endpoint do MOD-000)
- **Auth:** RBAC — scope `org:unit:read` via middleware `@RequireScope` do MOD-000
- **Contrato:**
  - **Endpoint:** `GET /api/v1/domain-events?entity_type=org_unit&entity_id=:id`
  - **Filtro:** `entity_type=org_unit` + `entity_id` do nó selecionado
  - **Segurança:** Protegido por RBAC (`org:unit:read`), **não** por `tenant_id` — `org_units` é cross-tenant (ADR-003)
  - **Response example:** `[{ event_type: "org.unit_created", entity_id: "uuid", actor_id: "uuid", payload: {...}, created_at: "ISO8601" }]`
- **Comportamento de falha:**
  - **Timeout:** Padrão HTTP do frontend (configurado em query client)
  - **Retries:** 1 retry automático via React Query (padrão)
  - **Fallback:** Exibir mensagem "Não foi possível carregar o histórico" na UI
- **Idempotência:** N/A (leitura)
- **Observabilidade:** `["correlationId"]` — `X-Correlation-ID` propagado na chamada HTTP
- **Decisão:** PENDENTE-002 (Opção A) — consumir diretamente do MOD-000 em vez de criar proxy no MOD-003

## 8. Convenção de Mapeamento Response (INT-001-C01)

> Todos os handlers de endpoints do MOD-003 MUST mapear campos camelCase do domínio para snake_case do DTO Zod antes de enviar a resposta via `reply.send()`. Enviar objetos de domínio diretamente causa 500 (schema validation failure).

**Endpoints com mapeamento obrigatório:**

| Endpoint | Campos críticos |
|----------|----------------|
| GET /api/v1/org-units/tree | `tenantId`→`tenant_id` (recursivo em cada nó da árvore + tenants) |
| GET /api/v1/org-units | `parentId`→`parent_id`, `createdAt`→`created_at` (Date→ISO) |
| GET /api/v1/org-units/:id | `parentId`→`parent_id`, `createdBy`→`created_by`, `razaoSocial`→`razao_social`, `emailContato`→`email_contato`, etc. |
| POST /api/v1/org-units | Mesmos campos que GET /:id |
| PATCH /api/v1/org-units/:id | Mesmos campos que GET /:id (subset) |
| POST /api/v1/org-units/:id/tenants | `orgUnitId`→`org_unit_id`, `tenantId`→`tenant_id`, `tenantCodigo`→`tenant_codigo` |

**Padrão de referência:** Os handlers já corrigidos (POST /, GET /, GET /:id, PATCH /:id, POST tenants) servem como modelo. Cada handler faz mapeamento explícito campo-a-campo.
