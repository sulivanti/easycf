# ADR-006 — Dry-Run via Flag no Request Body (Supressão de Domain Events)

## Contexto

O motor de avaliação (`POST /routine-engine/evaluate`) emite domain event `routine.applied` (DATA-003 EVT-012) para toda avaliação com efeito (BR-010). O UX de configuração de enquadradores e rotinas (F04, F05) precisa de um preview (dry-run) do motor para mostrar ao administrador o resultado da avaliação antes de publicar rotinas ou alterar regras de incidência. Sem mecanismo de dry-run, previews no UX poluiriam a timeline de auditoria com eventos "falsos", comprometendo a integridade do audit trail (SEC-007).

Três abordagens foram avaliadas (PEN-006):
1. Flag `dry_run: true` no request body do endpoint existente
2. Endpoint separado `POST /routine-engine/dry-run`
3. Persistir eventos com tag `is_dry_run: true`

## Decisão

**Flag `dry_run: boolean` (default: false) no request body de `POST /routine-engine/evaluate`.** Quando `dry_run: true`:

1. O motor executa os 6 passos normalmente (always fresh — ADR-005)
2. O response é idêntico ao modo normal (mesma estrutura, mesmos dados)
3. O domain event `routine.applied` (EVT-012) **NÃO** é persistido
4. O response inclui campo `dry_run: true` para transparência
5. Nenhum side-effect ocorre (sem escrita em banco, sem emit para outbox)

O scope `param:engine:evaluate` é suficiente para dry-run — não é necessário scope adicional.

## Alternativas Consideradas

1. **Endpoint separado `POST /routine-engine/dry-run`:** Mais explícito no OpenAPI, mas duplica a lógica do motor (ou exige refactor para fator comum). Adiciona 1 endpoint ao schema (26 total). Curl/SDK precisam de dois métodos para mesma lógica.
2. **Persistir com tag `is_dry_run: true`:** Auditoria completa de dry-runs, mas polui a timeline. Toda query de timeline precisa filtrar `is_dry_run = false`. Risco de contabilizar dry-runs em métricas de uso real.
3. **Preview estático (sem chamar o motor):** UX calcula preview no frontend. Inconsistente com o motor real — preview pode divergir do resultado real.

## Consequências

- **Positivas:** Reutilização total do motor (mesma lógica, mesmo código path); timeline de auditoria limpa (sem eventos falsos); schema de endpoints inalterado (25 endpoints); UX pode chamar dry-run livremente sem impacto no audit trail.
- **Negativas:** Motor precisa de branch condicional `if (!dry_run) { emit(event) }` no passo 6; dry-runs não são rastreáveis (se precisar auditar quem fez preview, não há registro).
- **Mitigações:** Log de acesso HTTP (access log) registra todas as chamadas ao endpoint, incluindo dry-runs, via `X-Correlation-ID`. Se auditoria de dry-runs for necessária no futuro, pode-se adicionar log dedicado sem domain event.
