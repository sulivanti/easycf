# BR-006 — Tratamento de Erros: Inline vs Toast por Código HTTP

- **Regra:** O MOD-002 aplica tratamento de erros diferenciado por código HTTP, seguindo o padrão Server-Driven UI (DOC-DEV-001 §4.5 UX-010.1 item 4). Erros de validação (422, 409) são exibidos inline por campo. Erros de servidor (5xx) são exibidos em toast com correlationId. Erros de acesso (401, 403) disparam ações de navegação. Em nenhum caso detalhes técnicos ou stacktraces são expostos ao usuário.
- **Exemplo:**
  - 422 com `extensions.invalid_fields`: cada campo inválido exibe mensagem inline; foco move para primeiro erro (a11y)
  - 409 (e-mail duplicado): "Este e-mail já está cadastrado no sistema." exibido inline sob o campo e-mail, NÃO em toast
  - 500: "Não foi possível criar o usuário." + correlationId em toast
  - 401: modal "Sessão expirada. Faça login novamente."
  - 403: redirect + toast "Sem permissão"
- **Exceções:** Nenhuma. As regras de mapping são declarativas nos Screen Manifests e não devem ser sobrescritas em código.
- **Impacto:** fluxo, segurança, compliance

### Matriz de Tratamento por Tela

| HTTP Status | UX-USR-001 (Lista) | UX-USR-002 (Form) | UX-USR-003 (Convite) |
|---|---|---|---|
| 401 | Modal — sessão expirada | Modal — sessão expirada | Modal — sessão expirada |
| 403 | Toast — sem permissão | Toast — sem permissão | Toast — sem permissão |
| 404 | — | — | Inline — usuário não encontrado |
| 409 | — | Inline — e-mail duplicado | — |
| 422 | Toast — filtros inválidos | Form validation — por campo | Toast — não é possível reenviar |
| 500 | Toast + correlationId | Toast + correlationId | Toast + correlationId |
