## FR-001.1 — CRUD de Vínculos Organizacionais (user_org_scopes)

- **Descrição:** API para vincular, listar e remover vínculos entre usuários e nós organizacionais (N1–N4), com invalidação de cache Redis e emissão de domain events.
- **Prioridade:** Must
- **Perfis impactados:** Administrador de acesso, Usuário (consulta própria)
- **Done funcional:**
  1. Admin cria vínculo PRIMARY (máximo 1) ou SECONDARY (sem limite) para um usuário em um nó N1–N4
  2. Admin lista vínculos org do usuário com breadcrumb do nó organizacional
  3. Admin revoga vínculo via soft delete (status=INACTIVE, deleted_at preenchido)
  4. Usuário consulta seus próprios vínculos via endpoint self-service (sem scope adicional)
  5. Nós N5 (tenant) são rejeitados com 422
  6. Background job expira vínculos com `valid_until < now()`
- **Efeito colateral?** [x] Sim — Invalidação de cache Redis `DEL auth:org_scope:user:{userId}` + domain events `identity.org_scope_granted|revoked|expired`
- **Idempotência:** required — POST com `Idempotency-Key` TTL 60s
- **Timeline/Notifications:** yes — domain events emitidos via Outbox Pattern (ver DATA-003, SEC-EventMatrix)
- **Dependências (IDs):** BR-001.1, BR-001.2, BR-001.3, DATA-001, INT-001.1, INT-001.2, SEC-001, NFR-001.4

### Endpoints

| Método | Path | operationId | Scope | Idempotência |
|---|---|---|---|---|
| GET | /api/v1/admin/users/:id/org-scopes | `admin_user_org_scopes_list` | `identity:org_scope:read` | N/A |
| POST | /api/v1/admin/users/:id/org-scopes | `admin_user_org_scopes_create` | `identity:org_scope:write` | `Idempotency-Key` TTL 60s |
| DELETE | /api/v1/admin/users/:id/org-scopes/:scopeId | `admin_user_org_scopes_delete` | `identity:org_scope:write` | Idempotente por natureza |
| GET | /api/v1/my/org-scopes | `my_org_scopes` | — (próprio usuário) | N/A |

### Critérios de aceite (Gherkin)

```gherkin
Funcionalidade: CRUD de Vínculos Organizacionais

  Cenário: Criar vínculo PRIMARY
    Dado que admin tem scope identity:org_scope:write
    E o usuário não tem nenhum vínculo PRIMARY
    Quando POST /api/v1/admin/users/:id/org-scopes com { org_unit_id, scope_type: "PRIMARY" }
    Então status 201
    E vínculo criado com status=ACTIVE
    E cache Redis DEL auth:org_scope:user:{userId} executado
    E evento identity.org_scope_granted emitido com correlation_id

  Cenário: Rejeitar segundo PRIMARY
    Dado que o usuário já tem um vínculo PRIMARY ativo
    Quando POST /api/v1/admin/users/:id/org-scopes com scope_type=PRIMARY
    Então 409: "Usuário já possui uma área principal (PRIMARY)."

  Cenário: Listar vínculos com breadcrumb
    Dado que o usuário tem 2 vínculos org
    Quando GET /api/v1/admin/users/:id/org-scopes
    Então retorna array com { id, scope_type, org_unit: { id, codigo, nome, nivel, breadcrumb }, valid_until, status }

  Cenário: Soft delete de vínculo
    Quando DELETE /api/v1/admin/users/:id/org-scopes/:scopeId
    Então deleted_at preenchido, status=INACTIVE
    E cache Redis invalidado
    E evento identity.org_scope_revoked emitido

  Cenário: Consulta self-service
    Dado que o usuário está autenticado
    Quando GET /api/v1/my/org-scopes
    Então retorna apenas os vínculos desse usuário
    E não requer scope adicional

  Cenário: RBAC — sem scope retorna 403
    Dado que o caller não tem identity:org_scope:read
    Quando GET /api/v1/admin/users/:id/org-scopes
    Então retorna 403 RFC 9457
```

## FR-001.2 — Compartilhamento Controlado (access_shares)

- **Descrição:** API para criar, listar, revogar compartilhamentos controlados com vigência obrigatória, motivo documentado e autorizador identificado (distinto ou auto-autorizado via scope `identity:share:authorize`).
- **Prioridade:** Must
- **Perfis impactados:** Administrador de acesso (CRUD admin), Usuário (consulta acessos recebidos)
- **Done funcional:**
  1. Admin cria compartilhamento com `reason`, `authorized_by`, `valid_until`, `allowed_actions`, `resource_type`, `resource_id`
  2. Validação de autorização condicional ao scope `identity:share:authorize` (auto-auth permitida COM scope, bloqueada SEM)
  3. Admin lista shares com filtros (status, grantee_id) e paginação cursor-based
  4. Admin/autorizador revoga share (status=REVOKED, revoked_at, revoked_by preenchidos)
  5. Grantee consulta acessos recebidos via endpoint self-service
  6. Background job expira shares com `valid_until < now()`
- **Efeito colateral?** [x] Sim — Domain events `identity.share_created|revoked|expired`
- **Idempotência:** required — POST com `Idempotency-Key` TTL 60s
- **Timeline/Notifications:** yes — domain events via Outbox Pattern (ver DATA-003, SEC-EventMatrix)
- **Dependências (IDs):** BR-001.7, BR-001.8, BR-001.9, DATA-001, INT-001.2, SEC-001, NFR-001.4

### Endpoints

| Método | Path | operationId | Scope | Idempotência |
|---|---|---|---|---|
| POST | /api/v1/admin/access-shares | `admin_access_shares_create` | `identity:share:write` | `Idempotency-Key` TTL 60s |
| GET | /api/v1/admin/access-shares | `admin_access_shares_list` | `identity:share:read` | N/A |
| DELETE | /api/v1/admin/access-shares/:id | `admin_access_shares_revoke` | `identity:share:revoke` | Idempotente por natureza |
| GET | /api/v1/my/shared-accesses | `my_shared_accesses` | — | N/A |

### Critérios de aceite (Gherkin)

```gherkin
Funcionalidade: Compartilhamento Controlado

  Cenário: Criar compartilhamento válido
    Dado que admin tem scope identity:share:write
    E grantor_id != authorized_by
    E valid_until no futuro e reason preenchido
    Quando POST /api/v1/admin/access-shares com todos os campos
    Então status 201 e status=ACTIVE
    E evento identity.share_created emitido

  Cenário: Auto-autorização com scope
    Dado que grantor possui identity:share:authorize
    E grantor_id = authorized_by
    Quando POST /api/v1/admin/access-shares
    Então status 201

  Cenário: Auto-autorização bloqueada sem scope
    Dado que grantor NÃO possui identity:share:authorize
    E grantor_id = authorized_by
    Quando POST /api/v1/admin/access-shares
    Então 422: "Sem scope 'identity:share:authorize', o autorizador deve ser diferente do solicitante."

  Cenário: Revogar compartilhamento
    Dado que existe share com status=ACTIVE
    Quando DELETE /api/v1/admin/access-shares/:id
    Então status=REVOKED, revoked_at=now(), revoked_by=caller.id
    E evento identity.share_revoked emitido

  Cenário: Listagem admin com paginação
    Dado que admin tem scope identity:share:read
    Quando GET /api/v1/admin/access-shares?status=ACTIVE&grantee_id=uuid
    Então retorna shares filtrados com paginação cursor-based

  Cenário: Grantee consulta recebidos
    Dado que o usuário tem 2 shares ativos para ele
    Quando GET /api/v1/my/shared-accesses
    Então retorna apenas shares com status=ACTIVE destinados ao caller

  Cenário: RBAC — sem scope retorna 403
    Dado que o caller não tem identity:share:write
    Quando POST /api/v1/admin/access-shares
    Então retorna 403 RFC 9457
```

## FR-001.3 — Delegação Temporária (access_delegations)

- **Descrição:** API para criar, listar e revogar delegações temporárias. Delegator transfere subconjunto de seus escopos (exceto `:approve`/`:execute`/`:sign`) para outro usuário por tempo limitado.
- **Prioridade:** Must
- **Perfis impactados:** Usuário (delegator e delegatee)
- **Done funcional:**
  1. Delegator cria delegação com subconjunto de seus escopos, TTL obrigatório e motivo
  2. Escopos `:approve`, `:execute`, `:sign` são rejeitados com 422
  3. Escopos não possuídos pelo delegator são rejeitados com 422
  4. Escopos obtidos por delegação são rejeitados (sem re-delegação)
  5. Listagem retorna delegações "dadas" (caller=delegator) e "recebidas" (caller=delegatee)
  6. Delegator revoga delegação própria
  7. Background job expira delegações com `valid_until < now()`
- **Efeito colateral?** [x] Sim — Domain events `identity.delegation_created|revoked|expired`
- **Idempotência:** required — POST com `Idempotency-Key` TTL 60s
- **Timeline/Notifications:** yes — domain events via Outbox Pattern (ver DATA-003, SEC-EventMatrix)
- **Dependências (IDs):** BR-001.4, BR-001.5, BR-001.6, BR-001.8, DATA-001, INT-001.2, SEC-001, NFR-001.4

### Endpoints

| Método | Path | operationId | Scope | Idempotência |
|---|---|---|---|---|
| POST | /api/v1/access-delegations | `access_delegations_create` | — (próprio usuário) | `Idempotency-Key` TTL 60s |
| GET | /api/v1/access-delegations | `access_delegations_list` | — | N/A |
| DELETE | /api/v1/access-delegations/:id | `access_delegations_revoke` | — | Idempotente por natureza |

### Critérios de aceite (Gherkin)

```gherkin
Funcionalidade: Delegação Temporária

  Cenário: Criar delegação válida
    Dado que delegator possui "finance:invoice:read" no token
    Quando POST /api/v1/access-delegations com { delegatee_id, delegated_scopes: ["finance:invoice:read"], valid_until, reason }
    Então status 201 e status=ACTIVE
    E evento identity.delegation_created emitido

  Cenário: Rejeitar escopo de aprovação
    Dado que delegated_scopes contém "finance:invoice:approve"

    Então 422: "Delegações não podem incluir escopos de aprovação, execução ou assinatura."

  Cenário: Rejeitar escopo não possuído
    Dado que delegator não tem "hr:salary:read" no token
    Quando POST /api/v1/access-delegations com delegated_scopes: ["hr:salary:read"]
    Então 422: "Não é possível delegar um escopo que você não possui."

  Cenário: Rejeitar sub-delegação

    Quando User-B tenta criar delegação com os mesmos escopos para User-C
    Então 422: "Escopos obtidos por delegação não podem ser re-delegados."

  Cenário: Listar delegações dadas e recebidas
    Dado que o usuário tem 2 delegações dadas e 1 recebida
    Quando GET /api/v1/access-delegations
    Então retorna seção "given" e "received", filtrando apenas status=ACTIVE

  Cenário: Revogar delegação própria
    Dado que delegator chama DELETE /api/v1/access-delegations/:id
    Então status=REVOKED e revoked_at=now()
    E evento identity.delegation_revoked emitido
```

## FR-001.4 — Background Job de Expiração

- **Descrição:** Job BullMQ (`expire_identity_grants`) que executa a cada 5 minutos via cron, expirando registros com `valid_until < now()` nas 3 tabelas e emitindo domain events via Outbox Pattern.
- **Prioridade:** Must
- **Perfis impactados:** Sistema (automação)
- **Done funcional:**
  1. Executa UPDATE em lote: `user_org_scopes` (status=INACTIVE, deleted_at=now()), `access_shares` (status=EXPIRED), `access_delegations` (status=EXPIRED)
  2. Para cada registro expirado, emite domain event correspondente via Outbox Pattern
  3. Job é idempotente — re-execução não duplica EXPIRED
  4. Falha → DLQ com retry 3x, NÃO bloqueia operações da aplicação
- **Efeito colateral?** [x] Sim — Domain events de expiração (`identity.org_scope_expired`, `identity.share_expired`, `identity.delegation_expired`)
- **Idempotência:** inerente — WHERE clause filtra apenas status=ACTIVE
- **Timeline/Notifications:** yes — domain events via Outbox Pattern
- **Dependências (IDs):** DATA-001, DATA-003, INT-001.2, NFR-001.1, NFR-001.2
- **Gatilho ADR:** N/A (padrão BullMQ já documentado em DOC-DEV-001 DATA-012)

### Critérios de aceite (Gherkin)

```gherkin
Funcionalidade: Background Job de Expiração

  Cenário: Expirar user_org_scopes
    Dado que um vínculo org tem valid_until < now() e status=ACTIVE
    Quando o job expire_identity_grants roda
    Então status=INACTIVE e deleted_at=now()
    E evento identity.org_scope_expired emitido
    E cache Redis DEL auth:org_scope:user:{userId} executado

  Cenário: Expirar access_shares
    Dado que um share tem valid_until < now() e status=ACTIVE
    Quando o job roda
    Então status=EXPIRED
    E evento identity.share_expired emitido

  Cenário: Expirar access_delegations
    Dado que uma delegação tem valid_until < now() e status=ACTIVE
    Quando o job roda
    Então status=EXPIRED
    E evento identity.delegation_expired emitido

  Cenário: Job idempotente
    Dado que o job já expirou registros no ciclo anterior
    Quando o job roda novamente
    Então nenhum registro é atualizado (WHERE status=ACTIVE não encontra resultados)

  Cenário: Falha com DLQ
    Dado que o job falha por erro transitório
    Então o BullMQ faz retry até 3x
    E se persistir, move para DLQ
    E operações da aplicação não são bloqueadas
```
