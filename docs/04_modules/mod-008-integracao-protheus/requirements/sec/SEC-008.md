# SEC-008 — Segurança e Compliance da Integração Dinâmica Protheus

- **estado_item:** READY
- **owner:** arquitetura
- **data_ultima_revisao:** 2026-03-23
- **rastreia_para:** US-MOD-008, US-MOD-008-F01, US-MOD-008-F03, FR-008, BR-008, DATA-008, DATA-003, INT-008, SEC-002, DOC-FND-000, DOC-ARC-003
- **referencias_exemplos:** EX-SEC-001, EX-PII-001, EX-AUTH-001
- **evidencias:** N/A

---

## 1. Autenticação

- **Modelo:** JWT Bearer via MOD-000 (Foundation). Não há fluxo de autenticação próprio (DOC-FND-000 §1).
- **Todos os 15 endpoints** do MOD-008 exigem JWT válido no header `Authorization: Bearer {token}`.
- **Worker BullMQ:** service account com scopes internos (`integration:execute`). O service account é provisionado pelo Foundation com token de longa duração renovado via refresh token rotativo. O token do worker carrega `tenant_id` do job (propagado do call_log enfileirado).
- **Service-to-service:** Worker BullMQ acessa o Protheus/TOTVS com credenciais do `auth_config` (descriptografadas em runtime, nunca expostas via API). O JWT do worker NÃO é propagado ao Protheus — a autenticação com o Protheus usa as credenciais configuradas no `integration_services.auth_config`.

## 2. Autorização (RBAC)

- **Modelo:** RBAC via `@RequireScope` (DOC-FND-000 §2). Scopes definidos no Amendment MOD-000-F12.
- **Enforcement:** `preHandler` em cada rota Fastify. Scope verificado contra JWT claims.

### 2.1 Catálogo de Scopes do MOD-008

| Scope | Descrição | Endpoints protegidos |
|---|---|---|
| `integration:service:read` | Ver catálogo de serviços de destino | GET /admin/integration-services |
| `integration:service:write` | Criar/editar serviços (credenciais mascaradas em resposta) | POST, PATCH /admin/integration-services |
| `integration:routine:write` | Configurar rotinas de integração (HTTP, mapeamentos, params) | POST integration-config, POST/PATCH/DELETE field-mappings, POST/PATCH params |
| `integration:execute` | Disparar execução manual de integração | POST /integration-engine/execute |
| `integration:log:read` | Ver logs de chamadas | GET /admin/integration-logs, GET /admin/integration-logs/:id |
| `integration:log:reprocess` | Reprocessar chamadas em DLQ | POST /admin/integration-logs/:id/reprocess |

> **Nota:** Endpoints de rotinas herdados do MOD-007 (GET/POST /admin/routines) usam scopes `param:routine:read` e `param:routine:write` respectivamente, conforme definido no MOD-007.

### 2.3 Separação de Privilégios

- **Leitura vs. Escrita vs. Execução:** Scopes granulares permitem que operadores monitorem logs (`log:read`) sem poder reprocessar (`log:reprocess`) ou configurar rotinas (`routine:write`).
- **Reprocessamento isolado:** `integration:log:reprocess` separado de `log:read` — reprocessar tem impacto operacional direto no Protheus e requer justificativa (BR-010).
- **Execução isolada:** `integration:execute` separado de `routine:write` — configurar a integração e executá-la são responsabilidades distintas (Arquiteto vs. Motor/trigger).

## 3. Classificação de Dados

| Classificação | Entidades/Campos | Justificativa |
|---|---|---|
| **Confidencial** | `integration_services.auth_config` | Credenciais criptografadas AES-256 — acesso restrito ao worker em runtime. NUNCA retornado em GET (BR-002). NUNCA incluído em payloads de domain events (DATA-003). |
| **Restrito** | `integration_params` com `is_sensitive=true` | Parâmetros técnicos sensíveis (ex: tokens derivados, chaves API) — mascarados em logs, UI e domain events (BR-004). |
| **Restrito** | `integration_call_logs.request_headers` (campos sensíveis) | Headers como Authorization, api_key — mascarados antes de persistir no banco (BR-004). |
| **Interno** | `integration_call_logs.request_payload`, `response_body` | Payloads de integração — visíveis para usuários com `integration:log:read`, não públicos. |
| **Interno** | `integration_services` (exceto auth_config), `integration_routines`, `integration_field_mappings`, `integration_params` (exceto is_sensitive) | Configuração de integração — dados organizacionais sem PII. |
| **Interno** | `integration_reprocess_requests` | Registros de auditoria de reprocessamento — sem PII. |

## 4. Mascaramento & LGPD

### 4.1 Política de Mascaramento

| Campo | Classificação | Máscara | Onde | Justificativa |
|---|---|---|---|---|
| `integration_services.auth_config` | Confidencial | `"***"` em GET responses | API (serializer) | BR-002 — credenciais NUNCA retornadas. Valor real acessível apenas pelo worker. |
| `integration_params.value` (is_sensitive=true) | Restrito | `"••••••"` em listagens | API + UI | BR-004 — parâmetros sensíveis mascarados em todos os contextos. |
| `integration_call_logs.request_headers` (Authorization, api_key, x-api-key) | Restrito | `"***"` antes de INSERT | Persistência (service layer) | BR-004 — headers sensíveis mascarados ANTES de gravar no banco. |
| `auth_config` em domain events | Confidencial | NUNCA incluído | Domain events payload | DATA-003 — auth_config explicitamente excluído de todos os payloads de eventos. |
| `request_payload` (quando is_sensitive) | Restrito | Campos sensíveis mascarados | Domain events payload | DATA-003 — campos derivados de params is_sensitive mascarados nos eventos. |

### 4.2 LGPD / Proteção de Dados

- **Impacto LGPD:** Moderado. O MOD-008 não armazena dados pessoais diretamente, mas payloads de integração PODEM conter PII transitória (dados de casos do MOD-006 propagados ao Protheus).
- **PII transitória em payloads:** O `request_payload` e `response_body` dos call_logs PODEM conter dados pessoais dependendo do mapeamento configurado. O mascaramento é aplicado via `is_sensitive` nos params e `transform_expr` nos field_mappings.
- **Direito de acesso/exclusão:** Como call_logs são registros de auditoria imutáveis (BR-009), a exclusão por LGPD requer anonimização dos campos PII nos payloads, não deleção do registro.
- **Retenção:** call_logs seguem política de retenção do Foundation para dados de auditoria. Dados Confidenciais (`auth_config`) são criptografados em repouso (AES-256 via secret do ambiente).

## 5. Auditoria

### 5.1 Domain Events como Audit Trail

- **8 domain events** definidos em DATA-003 cobrem todas as operações de escrita do módulo.
- **Tabela `domain_events`** é a única fonte para auditoria/timeline (DOC-ARC-003 §1 Dogma 6).
- **`integration_call_logs`** como audit trail complementar específico de execução — registros imutáveis (BR-009), nunca deletados.

### 5.2 Rastreabilidade (Correlation ID)

- **X-Correlation-ID** propagado em toda a cadeia: UI → API → Outbox → BullMQ Worker → Protheus → call_log → domain_event (OKR-4).
- **Structured logging:** `correlationId` + `operationId` em todos os logs aplicacionais.
- **Causation chain:** `causation_id` em domain events encadeia: `call_queued` → `call_completed`/`call_failed` → `call_dlq` → `call_reprocessed`.

### 5.3 Imutabilidade de Registros

- **call_logs:** NÃO possuem soft-delete. Status REPROCESSED marca o log original, mas o registro permanece intacto (BR-009).
- **reprocess_requests:** Registros de auditoria — não deletáveis.
- **Reprocessamento:** Cria novo call_log com `parent_log_id` apontando para o original. O log original permanece inalterado.

## 6. Multi-Tenancy e Row-Level Security

### 6.1 Tenant Isolation

- **MUST:** `tenant_id` obrigatório em TODAS as 6 tabelas do MOD-008 (DOC-FND-000 §2).
- **Derivação:** `tenant_id` é extraído do JWT do usuário autenticado e injetado em todas as queries.
- **Worker BullMQ:** O `tenant_id` é propagado do call_log enfileirado (persistido na transação de negócio) — o worker herda o contexto do tenant.

### 6.2 Row-Level Security (RLS) Considerations

| Tabela | Filtro tenant_id | Considerações adicionais |
|---|---|---|
| `integration_services` | WHERE tenant_id = :jwt_tenant_id | Serviços são isolados por tenant — tenant A não vê serviços de tenant B |
| `integration_routines` | WHERE tenant_id = :jwt_tenant_id | Herda isolamento via FK chain (behavior_routines também filtrada por tenant) |
| `integration_field_mappings` | WHERE tenant_id = :jwt_tenant_id | Acessível apenas via rotina do mesmo tenant |
| `integration_params` | WHERE tenant_id = :jwt_tenant_id | Acessível apenas via rotina do mesmo tenant |
| `integration_call_logs` | WHERE tenant_id = :jwt_tenant_id | Logs de execução isolados por tenant |
| `integration_reprocess_requests` | WHERE tenant_id = :jwt_tenant_id | Reprocessamentos isolados por tenant |
| `domain_events` (MOD-000) | WHERE tenant_id = :jwt_tenant_id | Eventos filtrados por tenant + ACL (SEC-002) |

### 6.3 Enforcement

- **Application layer:** Toda query gerada pelo repository MUST incluir `WHERE tenant_id = :tenant_id` (DOC-FND-000 §2).
- **Database layer (recomendado):** PostgreSQL RLS policies como defesa em profundidade para tabelas de alto volume (`integration_call_logs`).
- **Testes:** Testes de integração MUST validar que tenant A não acessa dados de tenant B em todos os 15 endpoints.

## 7. Proteções Específicas do Módulo

### 7.1 Credenciais em Trânsito

- `auth_config` é descriptografado APENAS no momento da chamada HTTP pelo worker, em memória (nunca em disco/log).
- Após a chamada, o valor descriptografado é descartado (não cacheado).
- Em caso de erro, `auth_config` NÃO é incluído no `error_message` do call_log.

### 7.2 Imutabilidade de Rotinas PUBLISHED

- Rotinas PUBLISHED são imutáveis em todas as camadas (BR-001): `integration_routines`, `field_mappings`, `params`.
- Esta é uma medida de segurança de integridade — chamadas em andamento referenciam a configuração imutável.
- Enforcement no service layer + validação de status antes de qualquer PATCH/POST/DELETE em entities dependentes.

### 7.3 Reprocessamento Governado

- Reprocessamento de DLQ requer scope `integration:log:reprocess` + justificativa obrigatória (min 10 chars — BR-010).
- O operador que reprocessou é registrado em `reprocessed_by` e no domain event `integration.call_reprocessed`.
- Apenas chamadas em status DLQ podem ser reprocessadas (BR-012).

### 7.4 Concorrência BullMQ

- Deduplicação via `jobId = call_log.id` previne execução duplicada (DATA-008 §2.5).
- `INTEGRATION_CONCURRENCY` (env var, default 10) limita conexões simultâneas ao Protheus (NFR-008).
- Worker idempotente: se job já processado (status != QUEUED/RUNNING), ignora silenciosamente.
