### Seção afetada: FR-007 — Gestão de Roles/RBAC

### Alteração 1: Campo `status` no PUT /roles/:id

**Antes:** `PUT /api/v1/roles/:id` aceita `{ name?, description?, scopes }`.

**Depois:** `PUT /api/v1/roles/:id` aceita `{ name?, description?, scopes, status? }`.

- `status` é **opcional** (string enum: `ACTIVE` | `INACTIVE`)
- Se omitido, status permanece inalterado (backwards-compatible)
- Validação: apenas `ACTIVE` ou `INACTIVE` aceitos (alinhado com CHECK constraint `roles_status_check`)

### Alteração 2: UpdateRoleUseCase

O `UpdateRoleInput` deve incluir `status?: 'ACTIVE' | 'INACTIVE'`.

O use case deve:
1. Se `status` presente → `role.changeStatus(input.status)`
2. Persistir via `roleRepo.update(updated.toProps(), tx)`
3. Se status mudou → emitir evento `role.status_changed` (além do `role.updated` existente)

### Alteração 3: UpdateRoleBody (DTO Zod)

```typescript
export const updateRoleBody = z.object({
  name: z.string().min(1).max(255).optional(),
  description: z.string().max(1000).nullable().optional(),
  scopes: z.array(scopeString).min(1),
  status: z.enum(['ACTIVE', 'INACTIVE']).optional(),  // NOVO
});
```

### Alteração 4: Frontend type

```typescript
export interface UpdateRoleRequest {
  name?: string;
  description?: string | null;
  scopes: string[];
  status?: 'ACTIVE' | 'INACTIVE';  // NOVO
}
```
