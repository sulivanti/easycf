# Emenda: INT-001-C01

- **Documento base:** [INT-001](../../requirements/int/INT-001.md)
- **estado_item:** MERGED
- **Natureza:** C (Correção)
- **Data:** 2026-03-30
- **owner:** Marcos Sulivan
- **Motivação:** Handler GET /tree envia `reply.send(result)` sem mapeamento camelCase→snake_case. O repositório retorna `tenantId` (camelCase) mas o schema Zod `orgUnitTenantSummary` espera `tenant_id` (snake_case). Fastify valida resposta contra schema e retorna 500.
- **rastreia_para:** FR-001-C03 (correção anterior do mesmo tipo em outros endpoints do MOD-003)

---

## Detalhamento

### Endpoint afetado

| Método | Path | operationId | Linha |
|--------|------|-------------|-------|
| GET | /api/v1/org-units/tree | `org_units_tree` | 144 |

### Causa raiz

O handler em `org-units.route.ts:144` faz:

```typescript
return reply.status(200).send(result);
```

O use case `getOrgUnitTreeUseCase.execute()` retorna nós de árvore com campos camelCase (`tenantId`), mas o schema `orgUnitTreeResponse` → `orgUnitTreeNode` → `orgUnitTenantSummary` espera `tenant_id` (snake_case).

### Correção

Adicionar mapeamento recursivo camelCase→snake_case no handler, seguindo o mesmo padrão já usado em GET /:id, POST /, etc. do mesmo módulo (FR-001-C03).

```typescript
function mapTreeNode(node: Record<string, any>): Record<string, any> {
  return {
    id: node.id,
    codigo: node.codigo,
    nome: node.nome,
    descricao: node.descricao,
    nivel: node.nivel,
    status: node.status,
    tenants: node.tenants.map((t: Record<string, any>) => ({
      tenant_id: t.tenantId,
      codigo: t.codigo,
      name: t.name,
    })),
    children: node.children.map(mapTreeNode),
  };
}

return reply.status(200).send({
  tree: result.tree.map(mapTreeNode),
});
```
