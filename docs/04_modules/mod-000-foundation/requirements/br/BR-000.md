# BR-000 — Regras de Negócio do Foundation

---

## BR-001 — Prevenção de Enumeração de Usuários (User Enumeration Prevention)

- **Regra:** Respostas de falha de login DEVEM ser sempre genéricas ("E-mail ou senha incorretos."), sem revelar se o e-mail existe ou não no sistema.
- **Exemplo:** Usuário tenta login com e-mail inexistente → resposta 401 idêntica à de senha errada. `actorId=undefined` e `entityId=UUID_NULO` na auditoria.
- **Exceções:** Nenhuma. Regra inegociável de segurança.
- **Impacto:** SEGURANÇA, COMPLIANCE
- **Gherkin:**

```gherkin
Funcionalidade: Prevenção de Enumeração de Usuários

Cenário: Login com e-mail inexistente
  Dado que o e-mail "naoexiste@test.com" não está cadastrado
  Quando POST /auth/login é chamado com esse e-mail
  Então o status deve ser 401
  E a mensagem deve ser "E-mail ou senha incorretos."
  E actorId na auditoria deve ser undefined
```

## BR-002 — Kill-Switch de Sessão em Banco

- **Regra:** O `sessionId` embutido no JWT DEVE ser validado no banco a cada requisição autenticada via `verifyActiveSession()`. JWT válido + sessão revogada = 401 imediato.
- **Exemplo:** Admin revoga sessão do usuário via DELETE /auth/sessions/:id → próxima requisição do usuário retorna 401 mesmo com JWT ainda dentro do TTL.
- **Exceções:** Nenhuma. Desvio intencional de stateless JWT puro, justificado pela necessidade de revogação imediata.
- **Impacto:** SEGURANÇA, ESTADO

```gherkin
Funcionalidade: Kill-Switch de Sessão

Cenário: Requisição com sessão revogada
  Dado que o JWT do usuário é válido (não expirou)
  E a sessão foi revogada via Kill-Switch
  Quando qualquer endpoint protegido é chamado
  Então deve retornar 401
```

## BR-003 — Rate Limiting de Autenticação

- **Regra:** O endpoint de login DEVE impor limite de 10 tentativas por 15 minutos por IP.
- **Exemplo:** IP faz 11ª tentativa em 15 min → retorna 429 (RFC 9457) com `retry_after` em segundos.
- **Exceções:** Nenhuma.
- **Impacto:** SEGURANÇA

```gherkin
Funcionalidade: Rate Limiting

Cenário: Excedeu limite de tentativas
  Dado que o IP fez 10 tentativas em 15 minutos
  Quando nova tentativa é realizada
  Então deve retornar 429 com retry_after em segundos
```

## BR-004 — Soft Delete Obrigatório (LGPD)

- **Regra:** Nenhuma entidade do Foundation DEVE ser removida fisicamente (hard delete). Exclusão DEVE preencher `deletedAt` e alterar status para `INACTIVE`, preservando dados para auditoria e compliance LGPD.
- **Exemplo:** DELETE /users/:id → `users.deletedAt = NOW()`, `users.status = 'INACTIVE'`, `content_users.deletedAt = NOW()`.
- **Exceções:** Nenhuma. Regra de compliance legal.
- **Impacto:** COMPLIANCE, DADOS

```gherkin
Funcionalidade: Soft Delete LGPD

Cenário: Exclusão de usuário
  Dado que o admin tem escopo users:user:delete
  Quando DELETE /users/:id é chamado
  Então deletedAt deve ser preenchido em users E content_users
  E status deve ser INACTIVE
  E dados NÃO devem ser removidos fisicamente
```

## BR-005 — Formato Canônico de Escopos RBAC

- **Regra:** Todas as permissões DEVEM seguir o formato canônico de 3 segmentos `dominio:entidade:acao` (regex: `^[a-z][a-z0-9_]*(:[a-z][a-z0-9_]*){1,2}$`), conforme DOC-FND-000 §2.1. O formato com 2 segmentos (`recurso:acao`) é aceito pelo regex mas o padrão canônico é 3 segmentos. Escopos com maiúsculas, segmentos vazios ou fora do padrão DEVEM ser rejeitados com 422.
- **Exemplo:** `users:user:read` ✅ (3 seg. canônico), `tenants:branch:write` ✅, `system:audit:read` ✅, `Roles:Write` ❌ (maiúsculas), `finance::approve` ❌ (segmento vazio), `a` ❌ (sem separador).
- **Exceções:** Nenhuma. Padrão imutável. Catálogo de scopes fundacionais em DOC-FND-000 §2.2.
- **Impacto:** SEGURANÇA, DADOS

```gherkin
Funcionalidade: Validação de Formato de Escopo

Cenário: Escopo canônico de 3 segmentos aceito
  Dado que o body contém escopo "users:user:read"
  Quando o endpoint é chamado
  Então deve aceitar o escopo

Cenário: Escopo de 2 segmentos aceito (retrocompatível)
  Dado que o body contém escopo "storage:upload"
  Quando o endpoint é chamado
  Então deve aceitar o escopo

Cenário: Escopo com formato inválido
  Dado que o body contém escopo "Roles:Write"
  Quando o endpoint é chamado
  Então deve retornar 422 informando o regex correto
```

## BR-006 — Substituição Total de Escopos em Roles

- **Regra:** A atualização de escopos de uma role (PUT) DEVE substituir completamente os escopos anteriores. NÃO é append. Implementado via DELETE + INSERT em transação Drizzle.
- **Exemplo:** Role "Admin" tem `[users:user:read, users:role:write]` → PUT com `[finance:invoice:approve]` → resultado: apenas `[finance:invoice:approve]`.
- **Exceções:** Nenhuma.
- **Impacto:** DADOS, FLUXO

```gherkin
Funcionalidade: Substituição de Escopos

Cenário: PUT substitui escopos completamente
  Dado que a role tem escopos [users:read, roles:write]
  Quando PUT /roles/:id com scopes [finance:approve]
  Então os escopos finais devem ser apenas [finance:approve]
```

## BR-007 — Isolamento Multi-Tenant Obrigatório

- **Regra:** Todo acesso a dados DEVE ser filtrado por `tenant_id`. Queries sem filtro de tenant são proibidas em endpoints protegidos. O `tenant_id` é extraído do JWT/sessão do usuário autenticado.
- **Exemplo:** GET /users lista apenas usuários do tenant do requisitante. Nunca retorna dados de outros tenants.
- **Exceções:** Endpoints de super-admin (se existirem) podem operar cross-tenant com escopo específico.
- **Impacto:** SEGURANÇA, COMPLIANCE, DADOS

```gherkin
Funcionalidade: Isolamento Multi-Tenant

Cenário: Acesso filtrado por tenant
  Dado que o usuário pertence ao tenant "tenant-A"
  Quando GET /users é chamado
  Então apenas usuários de "tenant-A" devem ser retornados
```

## BR-008 — Desvio MFA com Token Temporário

- **Regra:** Se o usuário tem `mfa_secret` configurado, o login NÃO DEVE emitir tokens de acesso. DEVE retornar apenas `temp_token` com escopo `mfa-only` e TTL de 5 minutos. O fluxo continua em US-MOD-000-F02.
- **Exemplo:** Login com credenciais válidas + MFA ativo → `{mfa_required: true, temp_token: "...", expires_in: 300}`. Sem `access_token`, sem sessão criada.
- **Exceções:** Nenhuma.
- **Impacto:** SEGURANÇA, FLUXO

```gherkin
Funcionalidade: Desvio MFA

Cenário: Login com MFA configurado
  Dado que o usuário tem mfa_secret preenchido
  Quando login com credenciais válidas
  Então deve retornar mfa_required=true e temp_token
  E NÃO deve conter access_token
  E NÃO deve criar sessão
```

## BR-009 — Campo Código Constitucional

- **Regra:** Tabelas de entidades principais (users, roles, tenants) DEVEM conter o campo `codigo` (`varchar(100)`, NOT NULL, UNIQUE) como identificador amigável de negócio, distinto do UUID e de outros campos.
- **Exemplo:** `users.codigo = "usr-00042"`, `roles.codigo = "admin"`, `tenants.codigo = "filial-sp"`.
- **Exceções:** Tabelas auxiliares/junction não requerem campo codigo.
- **Impacto:** DADOS

## BR-010 — Cookies httpOnly para Tokens

- **Regra:** Tokens de autenticação (`accessToken` e `refreshToken`) DEVEM ser setados como cookies `httpOnly=true`, `sameSite=lax`. O `refreshToken` DEVE ter `path=/api/v1/auth/refresh` restrito.
- **Exemplo:** Resposta de login seta cookies: `accessToken` (path=/, httpOnly) e `refreshToken` (path=/api/v1/auth/refresh, httpOnly).
- **Exceções:** Nenhuma. Regra de segurança.
- **Impacto:** SEGURANÇA

## BR-011 — Cache Redis para RBAC com Invalidação Automática

- **Regra:** Escopos de roles DEVEM ser cacheados no Redis (key `auth:scopes:role:{roleId}`). Toda mutação em roles (PUT, DELETE) DEVE invalidar o cache com DEL. Falha no Redis NÃO DEVE derrubar as rotas — capturar silenciosamente e prosseguir.
- **Exemplo:** PUT /roles/:id → Redis DEL `auth:scopes:role:{id}` → próximo request busca do banco.
- **Exceções:** Nenhuma.
- **Impacto:** DADOS, FLUXO

```gherkin
Funcionalidade: Cache RBAC com Resiliência

Cenário: Invalidação de cache após mutação
  Dado que existe cache para role "admin"
  Quando PUT /roles/:id é executado
  Então Redis DEL deve ser chamado para a key da role

Cenário: Falha no Redis não derruba a rota
  Dado que o Redis está indisponível
  Quando PUT /roles/:id é executado
  Então a mutação deve prosseguir normalmente
  E o erro Redis deve ser logado silenciosamente
```

## BR-012 — Idempotência em Endpoints com Efeito Colateral

- **Regra:** Endpoints que causam efeito colateral (login, registro, criação) DEVEM suportar `Idempotency-Key` no header. Reenvios com a mesma chave dentro do TTL retornam a resposta cacheada sem duplicar o efeito.
- **Exemplo:** POST /auth/login com `Idempotency-Key: abc123` → 2ª chamada em 30s retorna mesma resposta sem criar nova sessão.
- **Exceções:** Endpoints de leitura (GET) não requerem idempotência.
- **Impacto:** DADOS, FLUXO

## BR-013 — Token de Recuperação de Senha Single-Use com TTL

- **Regra:** O token de recuperação de senha (forgot/reset) DEVE ser um UUID v4 armazenado como hash (SHA-256), DEVE ser single-use (invalidado imediatamente após consumo) e DEVE expirar em 1 hora (TTL 3600s). Após expiração ou uso, qualquer tentativa de reset com o mesmo token DEVE retornar 422 (Unprocessable Entity, RFC 9457 §16.11) — o token não é processável porque expirou ou já foi consumido. O código 401 é reservado exclusivamente para falha de autenticação real (JWT/sessão).
- **Exemplo:** POST /auth/forgot-password → gera UUID token, armazena SHA-256 no banco com `expires_at = NOW() + 1h`. POST /auth/reset-password com token válido → senha alterada, token marcado como `used_at = NOW()`. Segunda tentativa com mesmo token → 422.
- **Exceções:** Nenhuma. Regra de segurança.
- **Impacto:** SEGURANÇA, FLUXO

```gherkin
Funcionalidade: Token de Recuperação de Senha

Cenário: Token válido reseta a senha
  Dado que o usuário solicitou recuperação e recebeu token UUID
  E o token foi gerado há menos de 1 hora
  Quando POST /auth/reset-password com token e nova senha
  Então a senha deve ser atualizada
  E o token deve ser invalidado (used_at preenchido)

Cenário: Token expirado é rejeitado
  Dado que o token foi gerado há mais de 1 hora
  Quando POST /auth/reset-password com token expirado
  Então deve retornar 422 com type "/problems/token-expired"
  E detail "Link expirado ou inválido. Solicite um novo link."

Cenário: Token já utilizado é rejeitado
  Dado que o token já foi utilizado anteriormente
  Quando POST /auth/reset-password com token reutilizado
  Então deve retornar 422 com type "/problems/token-used"
  E detail "Link expirado ou inválido. Solicite um novo link."
```

## BR-014 — Alteração de Senha Requer Verificação da Senha Atual

- **Regra:** O endpoint POST /auth/change-password DEVE exigir o campo `currentPassword` no body e validá-lo via bcrypt compare antes de aceitar a nova senha. Falha na verificação DEVE retornar 400 (RFC 9457) com mensagem genérica ("Senha atual incorreta.") — 401 é reservado para falha de autenticação real (JWT/sessão). O domain event `auth.password_changed` DEVE ser emitido apenas após sucesso.
- **Exemplo:** POST /auth/change-password com `{currentPassword: "old", newPassword: "new"}` → bcrypt compare OK → senha atualizada → `auth.password_changed` emitido.
- **Exceções:** Nenhuma. Conforme DOC-FND-000 §1.3.
- **Impacto:** SEGURANÇA

```gherkin
Funcionalidade: Alteração de Senha Autenticada

Cenário: Senha atual correta permite alteração
  Dado que o usuário está autenticado
  Quando POST /auth/change-password com senha atual válida
  Então a senha deve ser atualizada
  E domain event auth.password_changed deve ser emitido

Cenário: Senha atual incorreta bloqueia alteração
  Dado que o usuário está autenticado
  Quando POST /auth/change-password com senha atual inválida
  Então deve retornar 400 "Senha atual incorreta."
  E a senha NÃO deve ser alterada
```
