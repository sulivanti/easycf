## INT-001 — Consumo de Tenants (MOD-000-F07)

- **Sistema externo:** MOD-000 (Foundation API) — tabela `tenants`
- **Modo:** API (SQL direto via FK — mesmo banco)
- **Auth:** N/A (acesso interno via FK, não via HTTP)
- **Contrato:**
  - **Leitura:** `SELECT id, codigo, nome, status FROM tenants WHERE id = :tenant_id`
  - **Validação:** Tenant deve existir e estar acessível para vinculação
- **Comportamento de falha:**
  - **Timeout:** N/A (acesso interno, mesma transação DB)
  - **Retries:** N/A
  - **Backoff:** N/A
  - **DLQ:** Não
- **Idempotência:** UNIQUE (org_unit_id, tenant_id) garante idempotência natural
- **Fallback:** 404 se tenant não existir; 422 se tenant inativo
- **Observabilidade:** `["correlationId", "logs"]` — X-Correlation-ID propagado em domain_events

## INT-004 — Emissão de Domain Events (MOD-000 — domain_events)

- **Sistema externo:** MOD-000 (Foundation) — tabela `domain_events`
- **Propósito:** Registrar todas as operações de escrita do MOD-003 na tabela unificada de auditoria/timeline do Foundation
- **Modo:** API (SQL direto — insert na tabela `domain_events` dentro da mesma transação DB)
- **Auth:** N/A (acesso interno, mesma transação)
- **Contrato:**
  - **Escrita:** INSERT em `domain_events` com campos: `event_type`, `entity_type='org_unit'`, `entity_id`, `tenant_id`, `actor_id`, `correlation_id`, `payload`, `created_at`
  - **Eventos emitidos:** `org.unit_created`, `org.unit_updated`, `org.unit_deleted`, `org.unit_restored`, `org.tenant_linked`, `org.tenant_unlinked` (ver DATA-003)
- **Comportamento de falha:**
  - **Timeout:** N/A (mesma transação DB — se falhar, a operação principal também falha via rollback)
  - **Retries:** N/A
  - **DLQ:** Não (a consistência é garantida pela transação)
- **Idempotência:** Garantida pela transação atômica — evento só é persistido se a operação principal for commitada
- **Observabilidade:** `["correlationId", "logs"]` — cada evento inclui `correlation_id` para rastreio end-to-end
