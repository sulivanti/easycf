### Alterações neste pilar

#### 1. Novos itens de Done Funcional

Adicionar ao done criteria de FR-001:

- [ ] Ação "Editar" no dropdown para TODOS os status — navega para /usuarios/:id (modo edição)
- [ ] Ação "Resetar senha" no dropdown para status ACTIVE — modal de confirmação + POST reset-password
- [ ] Ação "Bloquear" no dropdown para status ACTIVE, INACTIVE e PENDING — modal de confirmação + PATCH status
- [ ] Ação "Reativar" no dropdown para status INACTIVE — PATCH status (sem modal)
- [ ] Ação "Desbloquear" no dropdown para status BLOCKED — PATCH status (sem modal)
- [ ] Ação "Cancelar convite" no dropdown para status PENDING — modal de confirmação + DELETE invite
- [ ] Modal de bloqueio: usa nome, nunca e-mail (BR-002). Botão isLoading durante chamada
- [ ] Modal de cancelamento de convite: usa nome, nunca e-mail (BR-002). Botão isLoading durante chamada
- [ ] Após bloqueio (200): linha atualizada inline para BLOCKED, modal fechado, toast de sucesso
- [ ] Após desbloqueio (200): linha atualizada inline para ACTIVE, toast de sucesso
- [ ] Após reativação (200): linha atualizada inline para ACTIVE, toast de sucesso
- [ ] Após reset de senha (200): toast de sucesso
- [ ] Após cancelamento de convite (200): linha atualizada inline, toast de sucesso
- [ ] Dropdown "•••" visível para TODOS os status (nunca oculto)

#### 2. Novos operationIds Consumidos

| operationId | Ação | Método |
|---|---|---|
| `users_update_status` | Bloquear/Desbloquear/Reativar | PATCH /api/v1/users/:id/status |
| `users_reset_password` | Resetar senha | POST /api/v1/users/:id/reset-password |
| `users_cancel_invite` | Cancelar convite | DELETE /api/v1/users/:id/invite |

#### 3. Novos cenários Gherkin

```gherkin
  Cenário: Editar usuário via dropdown
    Dado que o admin com scope "users:user:read" vê a listagem
    Quando clica "Editar" no dropdown de qualquer usuário
    Então é navegado para /usuarios/:id

  Cenário: Bloquear usuário ATIVO com modal
    Dado que o admin com scope "users:user:write" clica "Bloquear" para um usuário ATIVO
    Quando o modal de confirmação é exibido (com nome, sem e-mail)
    E o admin confirma
    Então PATCH /api/v1/users/:id/status é chamado com body { "status": "BLOCKED" }
    E após 200, a linha muda para BLOQUEADO sem reload

  Cenário: Reativar usuário INATIVO
    Dado que o admin com scope "users:user:write" clica "Reativar" para um usuário INATIVO
    Então PATCH /api/v1/users/:id/status é chamado com body { "status": "ACTIVE" }
    E após 200, a linha muda para ATIVO sem reload
    E toast "Usuário reativado com sucesso."

  Cenário: Desbloquear usuário BLOQUEADO
    Dado que o admin com scope "users:user:write" clica "Desbloquear" para um usuário BLOQUEADO
    Então PATCH /api/v1/users/:id/status é chamado com body { "status": "ACTIVE" }
    E após 200, a linha muda para ATIVO sem reload

  Cenário: Cancelar convite de usuário PENDENTE
    Dado que o admin com scope "users:user:delete" clica "Cancelar convite" para um PENDENTE
    Quando o modal de confirmação é exibido (com nome, sem e-mail)
    E o admin confirma
    Então DELETE /api/v1/users/:id/invite é chamado
    E após 200, toast "Convite cancelado com sucesso."
```
