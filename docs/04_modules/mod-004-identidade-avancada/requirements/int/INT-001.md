## INT-001.1 — Cache Redis (Invalidação de Escopo Organizacional)

- **Sistema externo:** Redis (interno, mesmo cluster)
- **Propósito:** Invalidar cache de escopos organizacionais ao criar/remover `user_org_scopes`, forçando refetch no próximo acesso a processos que filtrem por área.
- **Modo:** Chave-Valor (comando DEL)
- **Auth:** Conexão interna via `REDIS_URL` (sem auth externo, rede privada)

### Contrato

```
Key pattern: auth:org_scope:user:{userId}

Operação SET (populate):
  Trigger: Cache miss no middleware de resolução de scopes organizacionais
  Comando: SET auth:org_scope:user:{userId} <json_payload> EX 300
  TTL: 300s (5 minutos) — alinhado com intervalo do job de expiração (INT-001.2)
  Payload: JSON array de { org_unit_id, scope_type, status }

Operação DEL (invalidate):
  Trigger: POST /admin/users/:id/org-scopes (create) | DELETE /admin/users/:id/org-scopes/:scopeId (revoke) | Background job (expire)
  Comando: DEL auth:org_scope:user:{userId}
```

> **Safety net (PEN-004 PENDENTE-003):** O TTL de 300s garante que, mesmo em falha dupla (Worker parado + DEL falhando), o cache auto-expira em no máximo 5 minutos. O custo é 1 query extra por usuário a cada 5 min sem mutação — aceitável como trade-off de segurança. Alinhado com OKR-3 (expiração < 5min).

### Comportamento de Falha

| Parâmetro | Valor | Justificativa |
|---|---|---|
| Timeout | 500ms | Cache é best-effort; não bloquear operação principal |
| Retries | 1 | Falha rara em rede local; retry único suficiente |
| Backoff | N/A (retry imediato) | Latência mínima |
| DLQ | Não | Operação não-crítica |
| Idempotência | Inerente (DEL é idempotente) | |
| Fallback | Log warning (nível WARN) + continue | Próximo acesso fará refetch automaticamente |

### Observabilidade

- `correlation_id` propagado no log de invalidação
- Métrica: `identity.cache.invalidation.count` (sucesso/falha)
- Log: `{ level: "warn", msg: "redis_del_failed", key, correlation_id, error }` em caso de falha

> **Nota (DOC-DEV-001 DATA-013):** Redis é usado exclusivamente como cache efêmero. O estado reside no PostgreSQL. Proibido usar Redis como banco primário ou Pub/Sub.

## INT-001.2 — Background Job BullMQ (Expiração de Grants)

- **Sistema externo:** BullMQ via Redis (interno)
- **Propósito:** Expirar automaticamente registros com `valid_until < now()` em `user_org_scopes`, `access_shares` e `access_delegations`, emitindo domain events via Outbox Pattern.
- **Modo:** Fila (cron job)
- **Auth:** Conexão interna via `REDIS_URL`

### Contrato

```
Job name: expire_identity_grants
Queue: identity-expiration
Cron: */5 * * * * (a cada 5 minutos)
Concurrency: 1 (processamento sequencial — evita race conditions no batch UPDATE)

SQL executado:
  UPDATE user_org_scopes SET status='INACTIVE', deleted_at=now(), updated_at=now()
    WHERE valid_until < now() AND status='ACTIVE' AND valid_until IS NOT NULL;
  UPDATE access_shares SET status='EXPIRED', updated_at=now()
    WHERE valid_until < now() AND status='ACTIVE';
  UPDATE access_delegations SET status='EXPIRED', updated_at=now()
    WHERE valid_until < now() AND status='ACTIVE';

Para cada registro expirado → INSERT em domain_events (Outbox Pattern)
Para cada user_org_scopes expirado → DEL auth:org_scope:user:{userId} (INT-001.1)
```

### Comportamento de Falha

| Parâmetro | Valor | Justificativa |
|---|---|---|
| Timeout | 30000ms (30s) | Batch UPDATE pode envolver muitos registros |
| Retries | 3 | Erros transitórios de DB (lock timeout, conexão) |
| Backoff | Exponencial, 1000ms inicial, jitter ±200ms | Evitar thundering herd após falha de DB |
| DLQ | Sim | Após 3 falhas, move para DLQ com alerting |
| Idempotência | Inerente (WHERE status='ACTIVE' não re-processa) | |
| Fallback | NÃO bloqueia operações da aplicação | Job é independente do request pipeline |

### Observabilidade

- `correlation_id` gerado por execução do job (UUID v4)
- Métricas:
  - `identity.expiration.job.duration_ms`
  - `identity.expiration.records.expired` (por tabela)
  - `identity.expiration.job.failures`
- Log: `{ level: "info", msg: "expire_grants_completed", expired_scopes, expired_shares, expired_delegations, duration_ms, correlation_id }`

> **Nota (DOC-DEV-001 DATA-012):** O Worker NÃO roda dentro da API. Worker separado faz claim do outbox, processa e marca `processed_at`.

## INT-001.5 — Contrato de Exposição: `user_org_scopes` para Módulos Consumidores

- **Tipo:** Contrato de exposição (provider contract)
- **Propósito:** Documentar como módulos dependentes (MOD-005, MOD-006, MOD-007, MOD-008) DEVEM consumir a tabela `user_org_scopes` do MOD-004 para filtrar dados por área organizacional.
- **Modo:** Banco compartilhado (PostgreSQL) — JOIN direto
- **Auth:** Mesma conexão de banco (pool compartilhado, RLS por `tenant_id`)

### Tabela Exposta

| Campo | Tipo DB | Descrição | Disponível para JOIN |
|---|---|---|:---:|
| `id` | uuid (PK) | Identificador único do vínculo | Sim |
| `tenant_id` | uuid (FK) | Isolamento multi-tenant (RLS) | Sim |
| `user_id` | uuid (FK → users) | Usuário vinculado | Sim |
| `org_unit_id` | uuid (FK → org_units) | Nó organizacional N1–N4 | Sim |
| `scope_type` | varchar(16) | `PRIMARY` ou `SECONDARY` | Sim |
| `status` | varchar(16) | `ACTIVE` ou `INACTIVE` | Sim |
| `valid_from` | timestamptz | Início da vigência | Sim |
| `valid_until` | timestamptz | Fim da vigência (null = permanente) | Sim |

> **Campos internos NÃO expostos para JOIN:** `granted_by`, `created_at`, `updated_at`, `deleted_at`. Consumidores NÃO DEVEM referenciar esses campos.

### Regras de Consumo (Obrigatórias)

| # | Regra | Justificativa |
|---|---|---|
| R1 | Filtrar SEMPRE por `tenant_id` (RLS) | Isolamento multi-tenant obrigatório |
| R2 | Filtrar SEMPRE por `status = 'ACTIVE'` | Registros INACTIVE/soft-deleted não devem ser considerados |
| R3 | Usar índice `idx_user_org_scopes_tenant_user` para queries por usuário | Hot path: `WHERE tenant_id = ? AND user_id = ? AND status = 'ACTIVE'` |
| R4 | Usar índice `idx_user_org_scopes_org_unit` para queries por área | Query: `WHERE tenant_id = ? AND org_unit_id = ? AND status = 'ACTIVE'` |
| R5 | NÃO fazer UPDATE/DELETE direto na tabela | Mutações são exclusivas do MOD-004 (integridade de cache + domain events) |
| R6 | Considerar `scope_type` se distinção PRIMARY/SECONDARY for relevante | Ex: MOD-005 pode filtrar apenas PRIMARY para owner do processo |

### Padrão de JOIN Recomendado

```sql
-- Filtrar registros de um módulo consumidor pela área organizacional do usuário
SELECT p.*
FROM <tabela_consumidor> p
INNER JOIN user_org_scopes uos
  ON uos.user_id = p.owner_id
  AND uos.org_unit_id = p.org_unit_id
  AND uos.tenant_id = p.tenant_id
  AND uos.status = 'ACTIVE'
WHERE p.tenant_id = :tenant_id;
```

### Invalidação e Consistência

| Aspecto | Comportamento |
|---|---|
| Cache Redis | MOD-004 invalida `auth:org_scope:user:{userId}` em toda mutação (INT-001.1) |
| TTL | 300s — cache auto-expira mesmo sem invalidação explícita (safety net) |
| Domain Events | `identity.org_scope.created`, `identity.org_scope.deleted`, `identity.org_scope.expired` — consumidores PODEM assinar para invalidar caches próprios |
| Expiração automática | Background job a cada 5min marca `INACTIVE` registros com `valid_until < now()` (INT-001.2) |

> **Nota:** Se um módulo consumidor mantiver cache local de `user_org_scopes`, DEVE assinar os domain events acima para invalidação. O uso direto via JOIN (sem cache local) é o padrão recomendado — o cache Redis do MOD-004 já otimiza o hot path.

### Impacto de Mudanças

Alterações na estrutura de `user_org_scopes` (adição/remoção de campos, mudança de constraints) DEVEM:

1. Ser documentadas neste contrato (INT-001.5)
2. Avaliar impacto nos módulos consumidores listados
3. Comunicar via amendment se o módulo estiver READY
