# ADR-003 — tenant_id em org_units para Isolamento Multi-Tenant (RLS)

## Contexto

O sistema é multi-tenant. A tabela `org_units` define a hierarquia organizacional que contém tenants no nível N5 (via `org_unit_tenant_links`). A questão é: org_units precisa de um campo `tenant_id` próprio para Row-Level Security (RLS)?

## Decisão

**Sim. Adicionar `tenant_id` como FK obrigatória em `org_units`.** Cada árvore organizacional pertence a um tenant, e o campo `tenant_id` é usado como filtro obrigatório em todas as queries (RLS na camada de repositório).

## Alternativas Consideradas

1. **Sem tenant_id (derivar do contexto via org_unit_tenant_links):** Complicaria queries; CTE recursivo precisaria de JOIN extra para validar pertencimento; violaria o padrão Foundation onde toda entidade tem `tenant_id`.
2. **RLS via PostgreSQL Policies:** Elegante mas acoplaria a segurança ao engine DB; preferimos filtro explícito na camada de aplicação per DOC-FND-000.

## Consequências

- **Positivas:** Padrão uniforme com Foundation; queries simples com `WHERE tenant_id = :ctx`; índices compostos (`tenant_id, codigo`) para unicidade por tenant (BR-008); CTE recursivo filtrado na raiz.
- **Negativas:** Redundância conceitual — org_units já definem a hierarquia que contém tenants. O `tenant_id` em org_units identifica "quem é dono da árvore", não "qual tenant está neste nó".
- **Nota:** O `tenant_id` em `org_units` e o `tenant_id` em `org_unit_tenant_links` têm semânticas diferentes: o primeiro é RLS (isolamento), o segundo é vínculo funcional (N4→N5).
