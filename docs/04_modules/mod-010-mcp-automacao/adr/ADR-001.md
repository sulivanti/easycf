# ADR-001 — Gateway Síncrono de 8 Passos (Por Que Síncrono e Não Async)

## Contexto

O MOD-010 define um gateway MCP (`McpGateway`) que recebe requisições de agentes automatizados e executa 8 passos sequenciais de validação antes de despachar a ação. A decisão central é se esses 8 passos devem ser executados de forma síncrona (blocking request-response) ou se a validação deve ser enfileirada (assíncrona, com callback/polling).

A cadeia de validação inclui: autenticação bcrypt, verificação de status, resolução de ação, verificação de vínculo, verificação de escopos, dupla checagem de blocklist, registro em `mcp_executions` e avaliação de política. Algumas dessas operações envolvem I/O (banco de dados, bcrypt compare). A latência esperada do bcrypt compare é 60-80ms (rounds >= 12).

## Decisão

**O gateway executa os 8 passos de forma síncrona dentro de uma única requisição HTTP.** O agente MCP recebe resposta imediata: 200 (DIRECT/EVENT_ONLY), 202 (CONTROLLED), ou 4xx (falha de validação). A assincronicidade fica reservada para o pós-despacho (integração MOD-008 via BullMQ) e para o fluxo CONTROLLED (MOD-009 retorna `movement_id` imediatamente).

## Alternativas Consideradas

1. **Validação assíncrona com fila:** O agente envia requisição, recebe 202 imediato, e polled para resultado. Simplificaria o timeout mas adicionaria complexidade de polling, aumentaria latência efetiva, e dificultaria o tratamento de erros de validação (agente não saberia imediatamente se API key é inválida).
2. **Pipeline parcialmente async:** Passos 1-6 síncronos (validação), passo 7-8 assíncronos (registro + despacho). Reduziria latência do response mas criaria janela onde a execução foi aceita sem registro em `mcp_executions` — violando a invariante de rastreabilidade integral.
3. **Saga pattern com compensação:** Cada passo como etapa de saga. Overhead excessivo para uma cadeia de validação que normalmente completa em < 200ms.

## Consequências

- **Positivas:** Feedback imediato ao agente (sabe na mesma requisição se foi bloqueado, se precisa de aprovação ou se executou). Rastreabilidade integral — passo 7 garante registro ANTES do despacho. Simplicidade de implementação e debug (single request/response). bcrypt P95 < 100ms é aceitável para operação síncrona.
- **Negativas:** A latência do gateway fica acoplada à latência do bcrypt compare (~60-80ms). Se o MOD-009 estiver lento, execuções CONTROLLED sofrem (mitigado pelo timeout de 5000ms com fallback para 503). Threads/connections ocupadas durante bcrypt compare.
- **Riscos:** Se o volume de requisições crescer significativamente, o custo computacional do bcrypt pode exigir offloading para um pool dedicado ou migração para hardware com aceleração crypto.
- **Timebox:** Reavaliar se P99 do gateway exceder 1s consistentemente por 7 dias.
