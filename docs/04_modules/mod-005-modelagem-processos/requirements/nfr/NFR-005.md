### 1.1 SLO do Endpoint /flow (Crítico)

O endpoint `/flow` é consumido pelo editor visual (UX-PROC-001) em cada carregamento de tela. O SLA < 200ms é um OKR do módulo (OKR-4).

**Estratégia de otimização:**
- Query única com JOINs (DATA-005 §5.1) + agrupamento no service
- Indexes otimizados: `idx_macro_stages_cycle_ordem`, `idx_stages_macro_ordem`, `idx_gates_stage_ordem`
- Se ultrapassar 200ms com grafos grandes (>50 nós): considerar CTE ou queries separadas paralelas
- Monitorar com métricas de latência P95/P99

## 8. Concorrencia e Consistencia

| Cenario | Estrategia | Justificativa |
|---|---|---|
| Edicao concorrente de ciclo DRAFT | Optimistic locking via `updated_at` | Dois analistas editando o mesmo ciclo DRAFT simultaneamente — o segundo recebe 409 Conflict |
| Reordenacao de macroetapas/estagios/gates | Transacao serializable no UPDATE de `ordem` | BR-012 exige reordenacao atomica sem lacunas |
| Fork atomico | Transacao com isolation READ COMMITTED | Copia de 7 tabelas — rollback completo se qualquer etapa falhar |
| Publicacao (DRAFT -> PUBLISHED) | SELECT FOR UPDATE no ciclo | Evita race condition entre publicacao e edicao |

> **Nota:** Optimistic locking usa `WHERE updated_at = :expected_updated_at` no PATCH. Se o timestamp divergir, retorna 409 com `extensions.conflict_field: "updated_at"`.
