# ADR-001 — Módulo UX-First: Frontend Exclusivo sem Endpoints Próprios

## Contexto

O MOD-002 (Gestão de Usuários — Backoffice) precisa de 3 telas para gerenciar o ciclo de vida de usuários: listagem, cadastro e fluxo de convite. Todos os endpoints REST necessários já existem no MOD-000-F05 (Users API) e MOD-000-F06 (Roles API).

A questão é: **MOD-002 deve criar seus próprios endpoints (módulo full-stack) ou apenas consumir os existentes (módulo UX-First)?**

## Decisão

**MOD-002 é exclusivamente frontend (UX-First).** Não cria endpoints, schemas Drizzle, migrações ou lógica de backend. Consome 6 operationIds existentes do MOD-000:

- `users_list`, `users_create`, `users_get`, `users_delete` (F05)
- `users_invite_resend` (F05 — via amendment)
- `roles_list` (F06)

A estrutura de código segue DOC-ESC-001 §6.3 (Nível 1 — Web only):

```text
apps/web/src/modules/users/
  ui/screens/     ← 3 telas (list, form, invite)
  ui/components/  ← componentes compartilhados
  ui/forms/       ← formulário com dois modos
  domain/         ← view-model, regras de visibilidade
  data/           ← queries, mappers
```

**Não existe** `apps/api/src/modules/users/`.

## Alternativas Consideradas

### Alternativa A: Módulo full-stack com endpoints proxy

- Criar endpoints `/api/v1/users-backoffice/*` que delegam para MOD-000-F05
- **Rejeitada:** Duplicação de endpoints sem valor. Aumenta superfície de ataque, manutenção e complexidade sem benefício funcional.

### Alternativa B: Módulo full-stack com endpoints agregados

- Criar endpoints BFF (Backend for Frontend) que combinam chamadas (ex: `users_list` + `roles_list` em uma só)
- **Rejeitada:** Prematura. Com apenas 6 operationIds e chamadas paralelas no frontend, o overhead de um BFF não se justifica no MOD-002 v1. Reavaliável se performance se tornar problema (ver NFR-001 §1).

### Alternativa C: UX-First (escolhida)

- Frontend consome diretamente os operationIds do MOD-000
- **Vantagens:** Zero duplicação, menor superfície de código, deploy independente do backend, testes focados em UI
- **Tradeoffs:** Sem BFF para otimizar chamadas; dependência direta de 6 contratos do MOD-000

## Consequências

1. **Scaffolding:** `forge-module` não gera código em `apps/api/`. Apenas `apps/web/`.
2. **Testes:** Testes de integração focam em componentes UI com mock dos operationIds (Vitest + Testing Library). Sem testes de use case backend.
3. **Code review:** Checklist PR segue DOC-ESC-001 §6.4 (Nível 1 — Web only). Sem checklist de API.
4. **Amendment obrigatório:** `users_invite_resend` precisa ser criado no MOD-000-F05 antes do scaffolding da F03.
5. **Evolução futura:** Se MOD-002 precisar de lógica backend (ex: importação em massa, BFF), será necessário um ADR para promover a full-stack.

## Status

**Aceita** — decisão validada pelo épico US-MOD-002 §1 (separação MOD-002 UX vs MOD-000-F05 API).
