### D1 — Expandir resposta de GET /admin/access-shares (FR-001.2)

O endpoint `admin_access_shares_list` DEVE retornar objetos expandidos para `grantee_id` e `grantor_id`:

**Resposta atual:**
```json
{
  "id": "uuid",
  "grantee_id": "uuid",
  "grantor_id": "uuid",
  "resource_type": "...",
  "resource_id": "uuid",
  "status": "ACTIVE"
}
```

**Resposta requerida (expandida):**
```json
{
  "id": "uuid",
  "grantee_id": "uuid",
  "grantee": { "id": "uuid", "name": "Ana Oliveira", "email": "ana@a1.com.br" },
  "grantor_id": "uuid",
  "grantor": { "id": "uuid", "name": "Marcos Silva", "email": "marcos@a1.com.br" },
  "resource_type": "...",
  "resource_id": "uuid",
  "status": "ACTIVE"
}
```

- Campos `grantee` e `grantor` sao objetos com `{ id, name, email }`
- Os campos `_id` originais permanecem para retrocompatibilidade
- Implementacao via LEFT JOIN na query ou populate pos-query
- Impacto no endpoint `my_shared_accesses` tambem: expandir `grantor`

### D3 — Expandir resposta de GET /admin/users/:id/org-scopes (FR-001.1)

O endpoint `admin_user_org_scopes_list` DEVE incluir info resumida do usuario no contexto da listagem agrupada:

**Nota:** Este endpoint ja opera no contexto de um userId especifico. A nova tabela UX (D1 do UX-001-M01) lista TODOS os usuarios com escopos, nao apenas um. Para suportar essa view, e necessario um NOVO endpoint:

### D4 — Novo endpoint: Listagem agrupada de usuarios com escopos

Adicionar endpoint para listar usuarios com seus escopos organizacionais (view administrativa da tabela):

| Metodo | Path | operationId | Scope | Idempotencia |
|---|---|---|---|---|
| GET | /api/v1/admin/org-scopes | `admin_org_scopes_list` | `identity:org_scope:read` | N/A |

**Query parameters:**
- `q` (string, opcional): busca textual por nome ou email do usuario
- `scope_type` (enum, opcional): filtro por tipo PRIMARY/SECONDARY
- `status` (enum, opcional): filtro por ACTIVE/INACTIVE/EXPIRED
- `cursor` (string, opcional): cursor para paginacao
- `limit` (int, default 20): itens por pagina

**Resposta:**
```json
{
  "data": [{
    "user": { "id": "uuid", "name": "Marcos Silva", "email": "marcos.silva@a1.com.br" },
    "primary_scope": {
      "id": "uuid",
      "org_unit": { "id": "uuid", "codigo": "UN-0003", "nome": "Diretoria", "nivel": 3, "breadcrumb": "Grupo A1 > A1 Engenharia > Diretoria" },
      "status": "ACTIVE",
      "valid_until": null
    },
    "secondary_scopes": [{
      "id": "uuid",
      "org_unit": { "id": "uuid", "codigo": "UN-0005", "nome": "Engenharia Civil", "nivel": 3, "breadcrumb": "..." },
      "status": "ACTIVE",
      "valid_until": "2026-06-30"
    }],
    "total_scopes": 3
  }],
  "next_cursor": "abc123",
  "has_more": true
}
```

- Agrupa escopos por usuario com `primary_scope` separado de `secondary_scopes`
- `total_scopes` suporta o pill '+N' para overflow no frontend
- Busca textual `q` filtra por nome ou email (ILIKE no PostgreSQL)

### D5 — Suporte a busca textual de usuarios (autocomplete)

Os drawers de atribuicao (OrgScopeDrawer), compartilhamento (ShareDrawer) e delegacao (DelegationDrawer) precisam de autocomplete de usuarios.

**Endpoint existente em MOD-002:** `GET /api/v1/users` (`users_list`) ja suporta listagem. Verificar se aceita query parameter `q` para busca textual:

- Se ja suporta `q`: sem alteracao necessaria em MOD-002 (apenas documentar uso)
- Se nao suporta: criar amendment FR-001-M01 em MOD-002 para adicionar `?q=`

**Uso no frontend:**
- OrgScopeDrawer (D3 do UX-001-M01): campo USUARIO com autocomplete via GET /api/v1/users?q={term}
- ShareDrawer (D11 do UX-001-M01): campo BENEFICIARIO com autocomplete
- DelegationDrawer (D12 do UX-001-M01): campo DELEGADO com autocomplete
- Debounce de 300ms no input antes de disparar a requisicao
- Filtrar apenas usuarios ativos do tenant corrente

