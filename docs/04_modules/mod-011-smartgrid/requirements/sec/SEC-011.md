# SEC-011 — Segurança e Compliance do SmartGrid

- **estado_item:** DRAFT
- **owner:** arquitetura
- **data_ultima_revisao:** 2026-03-19
- **rastreia_para:** US-MOD-011, US-MOD-011-F01, US-MOD-011-F02, US-MOD-011-F03, US-MOD-011-F04, US-MOD-011-F05, BR-011, FR-011, DATA-011, DATA-003, INT-011, DOC-FND-000, DOC-ARC-003
- **referencias_exemplos:** EX-AUTH-001, EX-PII-001
- **evidencias:** N/A

---

> **Nota arquitetural:** MOD-011 é Nível 1 (UX Consumer). Não cria novos escopos RBAC, não possui tabelas de banco de dados e não expõe endpoints API próprios. Toda segurança server-side é delegada ao MOD-007 (motor de avaliação) e ao módulo dono do registro. Este documento foca em: (1) escopos RBAC herdados, (2) segurança client-side, (3) soft delete delegado, (4) auditoria via domain events delegados, (5) classificação de dados, (6) telemetria de segurança.

---

## 1. Autenticação

- **Modelo:** JWT Bearer via MOD-000 (Foundation). Não há fluxo de autenticação próprio no MOD-011.
- **Todas as chamadas** do SmartGrid ao motor MOD-007 e ao módulo destino exigem JWT válido no header `Authorization: Bearer {token}`.
- **Tenant isolation:** Todas as chamadas ao motor passam `tenant_id` implícito via JWT claims. O MOD-007 aplica filtro por `tenant_id` nas avaliações (DOC-FND-000 §2).
- **Session kill-switch:** Se a sessão do usuário for invalidada (kill-switch MOD-000), o interceptor HTTP global do Application Shell redireciona para `/login`. Dados não salvos na grade são perdidos (comportamento esperado — sem persistência server-side).

---

## 2. Autorização (RBAC) — Escopos Herdados

- **Modelo:** RBAC via `@RequireScope` (DOC-FND-000 §2). MOD-011 **não cria** novos scopes.
- **Enforcement:** Realizado pelo MOD-007 e pelo módulo destino nos seus respectivos endpoints. O SmartGrid consome endpoints protegidos que já aplicam `preHandler` com scope check.

### 2.1 Catálogo de Scopes Consumidos (Herdados)

| Scope consumido | Módulo origem | Uso no SmartGrid | Telas |
|---|---|---|---|
| `param:engine:evaluate` | MOD-007 | Cada chamada de validação de linha no motor (POST /routine-engine/evaluate) | UX-SGR-001, UX-SGR-002, UX-SGR-003 |
| `param:framer:read` | MOD-007 | Carregar configuração da Operação (context_framer) no mount da grade | UX-SGR-001 |

### 2.2 Scopes Dinâmicos do Módulo Destino

Para operações de persistência (save batch F02, save changes F03, confirm delete F04), o SmartGrid chama endpoints do módulo dono do registro. Os scopes necessários são definidos pelo módulo destino e verificados dinamicamente:

| Operação SmartGrid | Endpoint destino (dinâmico) | Scope destino (exemplo) |
|---|---|---|
| Salvar lote (F02) | POST /{modulo}/{entity} | `{dominio}:{entidade}:write` |
| Salvar alteração (F03) | PUT/PATCH /{modulo}/{entity}/{id} | `{dominio}:{entidade}:write` |
| Exclusão lógica (F04) | DELETE /{modulo}/{entity}/{id} | `{dominio}:{entidade}:delete` |

> **Nota:** O SmartGrid NÃO faz verificação de scopes no frontend como decisão de segurança. O frontend pode ocultar ações para UX (DOC-FND-000 §2.3), mas o enforcement real é no backend do módulo destino via `preHandler`.

### 2.3 Regra de Proteção Frontend (UX Guard)

- Menus e rotas das telas UX-SGR-001/002/003 DEVEM ser ocultados se o usuário não possuir `param:engine:evaluate` (verificação via `/auth/me` scopes).
- Esta é uma decisão de UX — não substitui proteção de rota server-side (DOC-FND-000 §2.3).
- Se o usuário navegar diretamente para a rota sem scope, a primeira chamada ao motor retorna HTTP 403 e a tela exibe mensagem de erro (error_handling: toast).

## 3. Classificação de Dados

| Classificação | Dados | Justificativa |
|---|---|---|
| **Interno** | Configuração da Operação (context_framer, routine_items) | Regras de negócio organizacionais — visível para usuários autorizados, não público |
| **Interno** | Estado da grade (linhas, valores, status visual) | Dados efêmeros em memória do navegador — sem persistência server-side |
| **Interno** | Export JSON (arquivo .json client-side) | Contém dados de registros em edição — sem PII do sistema, mas pode conter dados de negócio |
| **Público (tenant)** | Domain events de avaliação (`param.routine_applied`) | `sensitivity_level: 0` — sem PII |
| **Conforme módulo destino** | Domain events de CRUD (`{modulo}.{entity}.created/updated/deleted`) | Classificação e mascaramento definidos pelo módulo dono do registro |

> **Nota:** MOD-011 não armazena dados. Dados em memória (grid state) são descartados ao fechar o navegador. Export JSON é responsabilidade do usuário.

## 6. Soft Delete como Política de Exclusão

- **MUST:** Exclusão de registros via UX-SGR-003 é **sempre lógica** (`deleted_at` + `status=INACTIVE`) conforme padrão MOD-000 / DOC-FND-000.
- **Enforcement:** O soft delete é executado pelo endpoint do módulo destino (`DELETE /{modulo}/{entity}/{id}`), não pelo SmartGrid.
- **Rastreabilidade:** O evento `{modulo}.{entity}.deleted` emitido pelo módulo destino inclui `source: "smartgrid"` e `delete_type: "soft"` no payload (DATA-003).
- **Hard delete:** Exclusão física NUNCA acontece via SmartGrid (BR-009).
- **Pré-validação:** Antes de executar a exclusão, o SmartGrid valida cada registro via motor com `current_record_state` (F01/F04). Registros bloqueados não são excluídos.

## 7. Client-Side Security

### SEC-001 — Export JSON: Exclusão de Dados Sensíveis

- **MUST:** O arquivo JSON exportado via Export (UX-SGR-001) NÃO DEVE conter:
  - Tokens JWT ou session data
  - Credentials ou API keys
  - tenant_id ou user_id (identificadores internos do sistema)
  - Metadados de scopes ou permissões
- **Conteúdo permitido:** `operacao_id`, definição de colunas, dados das linhas (valores preenchidos pelo usuário), status de validação.
- **Referência:** FR-004, BR-006

### SEC-002 — Import JSON: Validação de Schema

- **MUST:** Arquivo JSON importado DEVE ser validado contra schema esperado antes de popular a grade.
- **MUST:** Rejeitar JSON com campos desconhecidos (whitelist approach — apenas campos esperados são aceitos).
- **MUST:** Rejeitar JSON com tamanho acima do limite configurável (previne DoS client-side por arquivo massivo).
- **MUST:** Após importação, todas as linhas retornam ao status neutro (sem ícone) e requerem revalidação.
- **Referência:** FR-004, BR-006, NFR-007

### SEC-003 — Limite de Linhas como Proteção contra DoS Client-Side

- **MUST:** Limite máximo de linhas por sessão configurável (FR-005, NFR-002).
- **MUST:** Botão "Adicionar" desabilitado ao atingir o limite.
- **MUST:** Importação JSON que excede o limite é rejeitada com mensagem explicativa.
- **MUST:** Duplicação que excederia o limite é rejeitada com mensagem explicativa (BR-007).

### SEC-004 — Proteção contra Injeção via Dados da Grade

- **MUST:** Valores preenchidos pelo usuário nas células da grade DEVEM ser sanitizados antes do envio ao motor (POST /routine-engine/evaluate) e ao módulo destino.
- **SHOULD:** Aplicar encoding de saída (output encoding) ao renderizar valores na grade para prevenir XSS.
- **SHOULD:** Campos do tipo texto livre DEVEM ter limite de caracteres configurável.

## 8. Auditoria

### 8.1 Auditoria via Domain Events (Delegada)

MOD-011 não persiste domain events diretamente. A auditoria é rastreável via:

| Ação do usuário no SmartGrid | Evento emitido por | event_type | Rastreabilidade |
|---|---|---|---|
| Validar linha (motor) | MOD-007 | `param.routine_applied` | `correlation_id` propagado pela UI |
| Salvar lote de novos registros (F02) | Módulo destino | `{modulo}.{entity}.created` | `correlation_id` + `source: "smartgrid"` + `batch_id` |
| Salvar alteração de registro (F03) | Módulo destino | `{modulo}.{entity}.updated` | `correlation_id` + `source: "smartgrid"` + `changes[]` |
| Excluir registros em massa (F04) | Módulo destino | `{modulo}.{entity}.deleted` | `correlation_id` + `source: "smartgrid"` + `delete_type: "soft"` |

> O campo `source: "smartgrid"` permite distinguir ações originadas pelo SmartGrid de ações feitas via interface padrão do módulo destino.

### 8.2 Telemetria de Ações UI (UIActionEnvelope)

Todas as ações definidas nos Screen Manifests (UX-SGR-001/002/003) emitem telemetria via `@easycf/ui-telemetry` conforme DOC-ARC-003 e DOC-FND-000 §4:

| Campo obrigatório | Valor |
|---|---|
| `screen_id` | `UX-SGR-001`, `UX-SGR-002` ou `UX-SGR-003` |
| `action` | action_id do manifest (ex: `validate_line`, `save_batch`, `export_json`) |
| `tenant_id` | Extraído do JWT claims |
| `correlation_id` | Propagado via `X-Correlation-ID` |
| `status` | `requested`, `succeeded`, `failed` |
| `duration_ms` | Capturado pelo pacote (`capture_duration_ms: true`) |

### 8.3 Ações Client-Only (Sem Auditoria Server-Side)

As seguintes ações são puramente client-side e NÃO geram domain events ou chamadas de API:

| Ação | Tipo | Telemetria UI |
|---|---|---|
| `export_json` | client_only | Sim (UIActionEnvelope) |
| `import_json` | client_only | Sim (UIActionEnvelope) |
| `add_line` | client_only | Sim (UIActionEnvelope) |
| `remove_selected_lines` | client_only | Sim (UIActionEnvelope) |
| `apply_value_mass` | client_only | Sim (UIActionEnvelope) |
| `clear_column_mass` | client_only | Sim (UIActionEnvelope) |
| `duplicate_line` | client_only | Sim (UIActionEnvelope) |
| `navigate_back` | client_only | Sim (UIActionEnvelope) |

> Embora client-only, estas ações DEVEM emitir UIActionEnvelope para observabilidade (rastreio de uso, métricas de adoção, debugging).

## 9. LGPD / Proteção de Dados

- MOD-011 **não armazena dados pessoais** (0 tabelas, 0 persistência server-side).
- Dados em memória da grade podem conter dados de negócio do módulo destino, mas não são persistidos pelo SmartGrid.
- Export JSON é arquivo local — responsabilidade do usuário quanto ao armazenamento e compartilhamento.
- A classificação LGPD dos dados operados (registros do módulo destino) é responsabilidade do módulo destino.
