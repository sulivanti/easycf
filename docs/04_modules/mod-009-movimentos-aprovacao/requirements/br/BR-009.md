# BR-009 — Regras de Negócio de Movimentos sob Aprovação

---

## BR-001 — Origem Não É Autorização

- **Regra:** Uma chamada de API, integração sistêmica ou agente MCP pode iniciar uma solicitação de operação, mas a origem por si só NÃO contorna a exigência de alçada. Se a regra de controle define que a origem é controlada, a operação DEVE ser interceptada e convertida em movimento controlado.
- **Exemplo:** API tenta executar `integration_call` com `origin_type=API`. Regra `RCG-API-INTEGRACAO` define `origin_types=["API","MCP","AGENT"]`. Motor intercepta e cria `controlled_movement` com `status=PENDING_APPROVAL`.
- **Exceções:** Nenhuma. Princípio inegociável.
- **Impacto:** SEGURANÇA, COMPLIANCE, FLUXO

```gherkin
Funcionalidade: Origem Não É Autorização

Cenário: API como origem é interceptada por regra de controle
  Dado que existe regra de controle ACTIVE com origin_types=["API","MCP","AGENT"]
  E uma chamada de API com origin_type=API tenta executar operação controlada
  Quando POST /movement-engine/evaluate é chamado
  Então motor cria controlled_movement com status=PENDING_APPROVAL
  E retorna HTTP 202

Cenário: MCP como origem é interceptada da mesma forma
  Dado que existe regra de controle ACTIVE com origin_types=["API","MCP","AGENT"]
  E uma chamada MCP com origin_type=MCP tenta executar operação controlada
  Quando POST /movement-engine/evaluate é chamado
  Então motor cria controlled_movement com status=PENDING_APPROVAL

Cenário: Operação sem regra aplicável é livre
  Dado que não existe regra de controle para object_type=relatorio e origin_type=HUMAN
  Quando POST /movement-engine/evaluate é chamado
  Então retorna { controlled: false } com HTTP 200
```

## BR-002 — Segregação de Funções (Solicitante ≠ Aprovador)

- **Regra:** O solicitante de um movimento (requested_by) NÃO PODE aprovar o próprio movimento, EXCETO quando possui o `required_scope` exigido pela alçada E `allow_self_approve=true` na `approval_rules` (auto-aprovação por suficiência de escopo — épico §3.1).
- **Exemplo:** João solicita pedido de venda R$ 75.000. João tenta aprovar. Sistema retorna 422: "O solicitante não pode aprovar o próprio movimento (segregação de funções)."
- **Exceções:** Auto-aprovação por suficiência de escopo (ver BR-003).
- **Impacto:** SEGURANÇA, COMPLIANCE

```gherkin
Funcionalidade: Segregação de Funções

Cenário: Solicitante tenta aprovar próprio movimento
  Dado que João solicitou movimento MOV-001 (requested_by=João)
  E João NÃO possui required_scope com allow_self_approve=true
  Quando POST /my/approvals/:approvalId/approve é chamado por João
  Então 422: "O solicitante não pode aprovar o próprio movimento (segregação de funções)."

Cenário: Solicitante tenta reprovar próprio movimento
  Dado que João solicitou movimento MOV-001 (requested_by=João)
  Quando POST /my/approvals/:approvalId/reject é chamado por João
  Então 422: "O solicitante não pode reprovar o próprio movimento (segregação de funções)."

Cenário: Outro aprovador aprova movimento de João com sucesso
  Dado que João solicitou movimento MOV-001
  E Maria é aprovadora da alçada com scope approval:decide
  Quando POST /my/approvals/:approvalId/approve é chamado por Maria com parecer
  Então 200 com approval_instance atualizada para decision=APPROVED
```

## BR-003 — Auto-Aprovação por Suficiência de Escopo

- **Regra:** Se o solicitante possui o `required_scope` exigido pela `approval_rules` aplicável E `allow_self_approve=true`, o sistema cria e aprova automaticamente o movimento sem enviá-lo ao inbox. Status passa direto para `AUTO_APPROVED` → `EXECUTED`. Registrado em `movement_history` com `event_type=AUTO_APPROVED_BY_SCOPE`.
- **Exemplo:** Maria possui scope `approval:decide` e a alçada requer exatamente esse scope com `allow_self_approve=true`. Motor cria movimento e auto-aprova imediatamente. Movimento NÃO aparece no inbox de ninguém.
- **Exceções:** Nenhuma. Se `allow_self_approve=false`, a segregação padrão prevalece.
- **Impacto:** EFICIÊNCIA, COMPLIANCE

```gherkin
Funcionalidade: Auto-Aprovação por Suficiência de Escopo

Cenário: Solicitante com scope suficiente e allow_self_approve=true
  Dado que Maria possui scope approval:decide
  E a regra de alçada aplicável tem required_scope=approval:decide e allow_self_approve=true
  Quando POST /movement-engine/evaluate é chamado por Maria
  Então motor cria movimento com status=AUTO_APPROVED
  E movement_history registra event_type=AUTO_APPROVED_BY_SCOPE
  E movimento NÃO aparece no inbox de ninguém
  E operação é executada imediatamente

Cenário: Solicitante com scope suficiente mas allow_self_approve=false
  Dado que Maria possui scope approval:decide
  E a regra de alçada aplicável tem required_scope=approval:decide e allow_self_approve=false
  Quando POST /movement-engine/evaluate é chamado por Maria
  Então motor cria movimento com status=PENDING_APPROVAL
  E movimento aparece no inbox de aprovadores da alçada
```

## BR-004 — 4 Critérios Combináveis de Alçada

- **Regra:** Uma `approval_rules` pode usar um ou mais critérios em combinação:
  1. **VALUE:** `operation_value > value_threshold` (ex: pedido > R$ 50.000 → alçada diretoria)
  2. **HIERARCHY:** `user.org_level < required_level` (ex: N3 exige aprovador N2 ou superior)
  3. **ORIGIN:** `origin_type IN origin_types` (ex: API/MCP/AGENT → sempre exige aprovação humana)
  4. **OBJECT:** `object_type + operation_type` (ex: DELETE de cadastro_produto → sempre controlado)
- **Exemplo:** Pedido de venda R$ 75.000 com `origin_type=API`. Regra combina VALUE (> 50.000) + ORIGIN (API). Ambos incidem → movimento controlado.
- **Exceções:** Se nenhum critério incide, a operação é livre (controlled=false).
- **Impacto:** FLUXO, SEGURANÇA

```gherkin
Funcionalidade: 4 Critérios Combináveis de Alçada

Cenário: Critério VALUE incide isoladamente
  Dado que existe regra com criteria_type=VALUE e value_threshold=50000
  E operação tem operation_value=75000
  Quando POST /movement-engine/evaluate é chamado
  Então motor intercepta e cria controlled_movement

Cenário: Critérios VALUE + ORIGIN combinados
  Dado que existe regra com criteria_types=[VALUE, ORIGIN] e value_threshold=50000 e origin_types=["API"]
  E operação tem operation_value=75000 e origin_type=API
  Quando POST /movement-engine/evaluate é chamado
  Então motor intercepta — ambos os critérios incidem

Cenário: Critério OBJECT+OPERATION incide
  Dado que existe regra com criteria_type=OBJECT e object_type=cadastro_produto e operation_type=DELETE
  E operação é DELETE de cadastro_produto
  Quando POST /movement-engine/evaluate é chamado
  Então motor intercepta e cria controlled_movement

Cenário: Nenhum critério incide — operação livre
  Dado que existe regra com criteria_type=VALUE e value_threshold=50000
  E operação tem operation_value=10000
  Quando POST /movement-engine/evaluate é chamado
  Então retorna { controlled: false }
```

## BR-005 — Override com Justificativa Obrigatória

- **Regra:** Override de movimento pendente requer scope `approval:override` e justificativa obrigatória com mínimo de 20 caracteres. O override é registrado em `movement_override_log` (imutável) e `movement_history` com `event_type=OVERRIDDEN`. A operação é executada imediatamente após o override.
- **Exemplo:** Gestor com scope `approval:override` faz POST /movements/:id/override com justificativa "Urgência operacional aprovada pelo diretor financeiro em reunião extraordinária". Override registrado, operação executada, `domain_event: movement.overridden` emitido.
- **Exceções:** Justificativa com menos de 20 chars → 422.
- **Impacto:** COMPLIANCE, AUDITORIA

```gherkin
Funcionalidade: Override com Justificativa Obrigatória

Cenário: Override com justificativa válida
  Dado que movimento MOV-001 está com status=PENDING_APPROVAL
  E gestor possui scope approval:override
  Quando POST /movements/:id/override com justificativa "Urgência operacional aprovada pelo diretor financeiro em reunião extraordinária"
  Então 200 com status=OVERRIDDEN
  E movement_override_log registra override imutável
  E movement_history registra event_type=OVERRIDDEN
  E domain_event movement.overridden é emitido
  E operação é executada imediatamente

Cenário: Override com justificativa curta demais
  Dado que movimento MOV-001 está com status=PENDING_APPROVAL
  E gestor possui scope approval:override
  Quando POST /movements/:id/override com justificativa "Urgente"
  Então 422: "Justificativa deve ter no mínimo 20 caracteres."

Cenário: Override sem scope approval:override
  Dado que movimento MOV-001 está com status=PENDING_APPROVAL
  E usuário NÃO possui scope approval:override
  Quando POST /movements/:id/override
  Então 403: "Escopo insuficiente."
```

## BR-006 — Cadeia de Aprovação Multinível

- **Regra:** Uma regra de alçada pode ter múltiplos níveis (1, 2, ...). Aprovação do nível N avança para nível N+1. Aprovação do último nível executa a operação. Rejeição em qualquer nível encerra a cadeia (status=REJECTED). Timeout pode escalar para outro nível via `escalation_rule_id`.
- **Exemplo:** Nível 1 = gerente_comercial (timeout 24h). Nível 2 = diretoria (timeout 48h). Nível 1 aprova → nível 2 criado. Nível 2 aprova → operação executada.
- **Exceções:** Timeout sem escalada → status TIMEOUT no nível.
- **Impacto:** FLUXO, COMPLIANCE

```gherkin
Funcionalidade: Cadeia de Aprovação Multinível

Cenário: Aprovação nível 1 avança para nível 2
  Dado que movimento MOV-001 tem cadeia com 2 níveis
  E approval_instance nível 1 está PENDING
  Quando aprovador nível 1 aprova com parecer
  Então approval_instance nível 1 recebe decision=APPROVED
  E approval_instance nível 2 é criada com status=PENDING
  E movement_history registra evento LEVEL_APPROVED

Cenário: Aprovação do último nível executa operação
  Dado que movimento MOV-001 tem cadeia com 2 níveis
  E approval_instance nível 1 já foi APPROVED
  E approval_instance nível 2 está PENDING
  Quando aprovador nível 2 aprova com parecer
  Então controlled_movement recebe status=APPROVED
  E operação é executada
  E movement_history registra evento EXECUTED

Cenário: Rejeição em qualquer nível encerra cadeia
  Dado que movimento MOV-001 tem cadeia com 2 níveis
  E approval_instance nível 1 está PENDING
  Quando aprovador nível 1 rejeita com parecer
  Então controlled_movement recebe status=REJECTED
  E cadeia é encerrada — nível 2 não é criado

Cenário: Timeout com escalada
  Dado que approval_instance nível 1 tem timeout_hours=24 e escalation_rule_id definido
  E 24 horas se passaram sem decisão
  Quando job de timeout executa
  Então approval_instance nível 1 recebe decision=TIMEOUT
  E nova approval_instance é criada conforme escalation_rule_id
```

## BR-007 — Prioridade de Regras de Controle

- **Regra:** Regras de controle são avaliadas em ordem de `priority` (menor = maior precedência). Apenas a primeira regra que incide gera o movimento controlado. Regras fora de vigência (`valid_until` expirado) são ignoradas.
- **Exemplo:** Regra A (priority=10) e Regra B (priority=50) ambas se aplicam ao `object_type=pedido_venda`. Apenas Regra A (priority=10) gera o movimento.
- **Exceções:** Nenhuma.
- **Impacto:** FLUXO, CONSISTÊNCIA

```gherkin
Funcionalidade: Prioridade de Regras de Controle

Cenário: Apenas a regra de maior precedência gera movimento
  Dado que regra A tem priority=10 e regra B tem priority=50
  E ambas se aplicam ao object_type=pedido_venda
  Quando POST /movement-engine/evaluate é chamado para pedido_venda
  Então apenas regra A (priority=10) gera o controlled_movement
  E regra B é ignorada

Cenário: Regra expirada é ignorada
  Dado que regra A tem priority=10 e valid_until=2026-01-01 (expirada)
  E regra B tem priority=50 e valid_until=null (sem expiração)
  Quando POST /movement-engine/evaluate é chamado
  Então regra A é ignorada
  E regra B gera o controlled_movement
```

## BR-008 — Cancelamento Apenas em PENDING_APPROVAL

- **Regra:** Somente o solicitante pode cancelar um movimento, e apenas quando o status é `PENDING_APPROVAL`. Movimentos em qualquer outro status (APPROVED, EXECUTED, etc.) NÃO podem ser cancelados.
- **Exemplo:** Movimento status=APPROVED → POST /movements/:id/cancel → 422: "Movimentos aprovados não podem ser cancelados."
- **Exceções:** Nenhuma.
- **Impacto:** FLUXO, STATE

```gherkin
Funcionalidade: Cancelamento Apenas em PENDING_APPROVAL

Cenário: Solicitante cancela movimento pendente
  Dado que movimento MOV-001 tem status=PENDING_APPROVAL
  E João é o solicitante (requested_by=João)
  Quando POST /movements/:id/cancel é chamado por João com motivo
  Então 200 com status=CANCELLED
  E movement_history registra event_type=CANCELLED

Cenário: Cancelamento recusado para movimento aprovado
  Dado que movimento MOV-001 tem status=APPROVED
  Quando POST /movements/:id/cancel é chamado pelo solicitante
  Então 422: "Movimentos aprovados não podem ser cancelados."

Cenário: Cancelamento recusado para movimento executado
  Dado que movimento MOV-001 tem status=EXECUTED
  Quando POST /movements/:id/cancel é chamado pelo solicitante
  Então 422: "Movimentos executados não podem ser cancelados."

Cenário: Outro usuário não pode cancelar movimento alheio
  Dado que movimento MOV-001 foi solicitado por João
  E Maria NÃO é a solicitante
  Quando POST /movements/:id/cancel é chamado por Maria
  Então 403: "Apenas o solicitante pode cancelar este movimento."
```

## BR-009 — Código Imutável

- **Regra:** O campo `codigo` de `movement_control_rules` e `controlled_movements` é imutável após criação. Não pode ser alterado via PATCH.
- **Exemplo:** POST cria regra com `codigo=RCG-PEDIDO-VALOR`. PATCH tenta alterar `codigo` → 422.
- **Exceções:** Nenhuma.
- **Impacto:** RASTREABILIDADE, DATA

```gherkin
Funcionalidade: Código Imutável

Cenário: PATCH tenta alterar código de regra de controle
  Dado que regra foi criada com codigo=RCG-PEDIDO-VALOR
  Quando PATCH /admin/control-rules/:id com { codigo: "RCG-NOVO" }
  Então 422: "O campo codigo é imutável após criação."

Cenário: PATCH tenta alterar código de movimento controlado
  Dado que movimento foi criado com codigo=MOV-20260319-001
  Quando qualquer operação tenta alterar o campo codigo
  Então operação é rejeitada — campo imutável

Cenário: Outros campos da regra podem ser alterados
  Dado que regra foi criada com codigo=RCG-PEDIDO-VALOR e nome="Pedido Valor"
  Quando PATCH /admin/control-rules/:id com { nome: "Pedido Alto Valor" }
  Então 200 com nome atualizado e codigo inalterado
```
