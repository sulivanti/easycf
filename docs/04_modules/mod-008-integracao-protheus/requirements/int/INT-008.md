## 1. INT-001 — MOD-007 (Parametrização Contextual) — Herança

### Propósito

O MOD-008 herda a estrutura de rotinas versionadas do MOD-007 (`behavior_routines` com `routine_type='INTEGRATION'`). A tabela `integration_routines` é uma extensão 1:1 de `behavior_routines`, adicionando configuração HTTP. Todo o ciclo de vida (DRAFT→PUBLISHED→DEPRECATED), fork e imutabilidade é controlado pelo MOD-007.

### Contrato

| Propriedade | Valor |
|---|---|
| **external_system** | MOD-007 (Parametrização Contextual) |
| **purpose** | Herança de behavior_routines para rotinas de integração HTTP |
| **mode** | FK direto (síncrono — shared database) |
| **auth** | N/A — acesso direto via FK no mesmo banco de dados |
| **contract** | FK `integration_routines.routine_id` → `behavior_routines.id` WHERE `routine_type = 'INTEGRATION'` |
| **timeout** | N/A (query local) |
| **retry** | N/A |
| **backoff** | N/A |
| **idempotência** | N/A — transação atômica |
| **fallback** | N/A — FK constraint garante integridade referencial |

### Dados Consumidos do MOD-007

| Dado | Tabela | Uso no MOD-008 |
|---|---|---|
| `behavior_routines.id` | behavior_routines | FK em integration_routines, field_mappings, params, call_logs |
| `behavior_routines.status` | behavior_routines | Determina imutabilidade (PUBLISHED = congelado — BR-001) |
| `behavior_routines.routine_type` | behavior_routines | Filtro: apenas `routine_type='INTEGRATION'` |

## 2. INT-002 — MOD-006 (Execução de Casos) — Inbound Events

### Propósito

O MOD-006 emite eventos de transição de estágio (`case.stage_transitioned`, `case.opened`, etc.) que disparam rotinas de integração configuradas via `trigger_events` em `integration_routines`. O MOD-008 consome esses eventos para iniciar execuções automáticas.

### Contrato

| Propriedade | Valor |
|---|---|
| **external_system** | MOD-006 (Execução de Casos) |
| **purpose** | Eventos de caso disparam rotinas de integração automaticamente |
| **mode** | Event-driven (assíncrono via domain_events) |
| **auth** | N/A — eventos consumidos internamente (listener no mesmo processo ou via event bus) |
| **contract** | Listener em `domain_events` WHERE `event_type IN (trigger_events)` |
| **timeout** | N/A — processamento assíncrono |
| **retry** | Gerenciado pelo motor de execução do MOD-008 (retry_max da rotina) |
| **backoff** | Exponencial: `retry_backoff_ms × 2^(attempt-1)` |
| **idempotência** | `call_log.id` usado como `jobId` no BullMQ — previne duplicação de enfileiramento |
| **fallback** | Se listener falhar ao processar evento: log de erro + alerta. Evento permanece na tabela `domain_events` para reprocessamento manual |

### Eventos Consumidos do MOD-006

| event_type | Ação no MOD-008 |
|---|---|
| `case.stage_transitioned` | Busca rotinas com `trigger_events` contendo este event_type → enfileira execução |
| `case.opened` | Idem — rotinas configuradas para disparar na abertura de caso |
| `case.event_registered` | Idem — rotinas configuradas para disparar em eventos customizados |

### Fluxo de Disparo Automático

```
MOD-006 emite domain_event (case.stage_transitioned)
  → Listener MOD-008 identifica rotinas PUBLISHED com trigger_events match
  → Para cada rotina: monta payload (field_mappings + params)
  → INSERT integration_call_logs (status=QUEUED) — Outbox Pattern
  → Enfileira job BullMQ: integration-execution
```

### Observabilidade

- `correlation_id` do evento MOD-006 propagado para o `integration_call_logs.correlation_id`
- Log structured: `{ event_type, routine_id, case_id, correlation_id }`

## 4. INT-004 — Protheus/TOTVS — Outbound HTTP REST

### Propósito

Chamadas HTTP REST ao ERP Protheus/TOTVS para execução de integrações dinâmicas. O motor BullMQ executa as chamadas de forma assíncrona com retry, backoff exponencial e DLQ.

### Contrato

| Propriedade | Valor |
|---|---|
| **external_system** | Protheus/TOTVS (ERP) |
| **purpose** | Executar integrações HTTP dinâmicas configuradas por rotina |
| **mode** | HTTP REST assíncrono via BullMQ worker |
| **auth** | Configurável por serviço: NONE, BASIC (user:pass base64), BEARER (token fixo), OAUTH2 (client_credentials → token endpoint → access_token) |
| **contract** | Dinâmico — definido por rotina: `{http_method} {base_url}{endpoint_tpl}` com payload montado via `integration_field_mappings` e `integration_params` |
| **timeout** | `integration_routines.timeout_ms` (override) ou `integration_services.timeout_ms` (default 30000ms) |
| **retry** | `integration_routines.retry_max` (default 3, max 10). Retry apenas para: timeout, 5xx, ECONNREFUSED, ECONNRESET |
| **backoff** | Exponencial: `retry_backoff_ms × 2^(attempt-1)`. Default: 1000ms → 1s, 2s, 4s |
| **idempotência** | `integration_call_logs.id` como `jobId` no BullMQ. Protheus SHOULD implementar idempotência via header `X-Idempotency-Key` (quando suportado) |
| **fallback** | Após `retry_max` esgotado: status=DLQ, domain_event `integration.call_dlq`, alerta no monitor UX-INTEG-002. Reprocessamento manual governado |

### Fluxo de Execução (Worker BullMQ)

```
Worker recebe job da fila "integration-execution"
  1. Busca integration_routine + service (com auth_config descriptografado)
  2. Monta URL: base_url + endpoint_tpl (com variáveis substituídas)
  3. Monta payload: field_mappings + params + defaults
  4. Monta headers: Content-Type, Authorization, X-Correlation-ID, params HEADER
  5. UPDATE call_log SET status='RUNNING', started_at=now()
  6. Executa HTTP call com timeout
  7. Avalia resposta:
     - 2xx → status=SUCCESS, response_body, duration_ms
     - 4xx → status=FAILED (sem retry — erro de contrato)
     - 5xx/timeout/network → status=FAILED (com retry se attempt < retry_max)
  8. UPDATE call_log SET status, response_*, completed_at, duration_ms
  9. Emite domain_event correspondente
  10. Se FAILED e retries restantes → re-enfileira com delay (backoff)
  11. Se retries esgotados → status=DLQ, emite integration.call_dlq
```

### Autenticação por Tipo

| auth_type | Fluxo |
|---|---|
| `NONE` | Sem header Authorization |
| `BASIC` | Header `Authorization: Basic {base64(user:pass)}`. Credenciais descriptografadas de `auth_config` |
| `BEARER` | Header `Authorization: Bearer {token}`. Token fixo de `auth_config` |
| `OAUTH2` | Fluxo detalhado abaixo (PEN-008/PENDENTE-003) |

#### Fluxo OAUTH2 Detalhado (PEN-008/PENDENTE-003)

**Grant type:** `client_credentials` (sem refresh_token — padrão Protheus/TOTVS)

**Cache de token em Redis (compartilhado entre workers BullMQ):**

1. Worker verifica Redis: `GET oauth2:token:{service_id}:{tenant_id}`
2. Se presente e válido (TTL > 0): usa o `access_token` cacheado
3. Se expirado ou ausente:
   a. Adquire lock distribuído: `SET oauth2:lock:{service_id}:{tenant_id} {worker_id} NX EX 30`
   b. **Com lock:** chama `POST {token_endpoint}` com `client_id` + `client_secret` (`grant_type=client_credentials`), armazena em Redis com `SET oauth2:token:{service_id}:{tenant_id} {access_token} EX {expires_in - 60}`
   c. **Sem lock** (outro worker renovando): retry com backoff (100ms, 200ms, 400ms) até 5s, depois falha com `OAUTH2_TOKEN_UNAVAILABLE`
4. **Mid-flight expiry:** interceptor HTTP detecta resposta 401 do Protheus → força renovação do token (repete passo 3), retenta a chamada original 1x
5. Header: `Authorization: Bearer {access_token}`

**Chave Redis:** `oauth2:token:{service_id}:{tenant_id}` — isolamento por serviço e tenant
**Lock Redis:** `oauth2:lock:{service_id}:{tenant_id}` — TTL 30s, previne thundering herd
**TTL do token:** `expires_in - 60s` (margem de segurança de 60s)

**Erros:**

| Cenário | Comportamento |
|---|---|
| Token endpoint indisponível | Retry conforme Retry Policy padrão. Se esgotar, job falha com `OAUTH2_TOKEN_ENDPOINT_UNAVAILABLE` |
| Token endpoint retorna erro 4xx | Falha imediata (credenciais inválidas). Job falha com `OAUTH2_INVALID_CREDENTIALS` |
| Mid-flight 401 + renovação falha | Job falha com `OAUTH2_TOKEN_EXPIRED_UNRECOVERABLE` |
| Lock não adquirido após 5s | Job falha com `OAUTH2_TOKEN_UNAVAILABLE` (outro worker travou) |

### Retry Policy

| Tentativa | Delay | Cumulativo |
|---|---|---|
| 1 (inicial) | 0 | 0 |
| 2 (retry 1) | 1000ms | 1s |
| 3 (retry 2) | 2000ms | 3s |
| 4 (retry 3) | 4000ms | 7s |

> Fórmula: `delay = retry_backoff_ms × 2^(attempt - 2)` para attempt >= 2

### Condições de Retry

| Condição | Retry? | Justificativa |
|---|---|---|
| HTTP 2xx | NÃO | Sucesso |
| HTTP 4xx | NÃO | Erro de contrato/payload — retry não resolve |
| HTTP 5xx | SIM | Erro temporário do servidor |
| Timeout | SIM | Protheus pode estar sobrecarregado |
| ECONNREFUSED | SIM | Serviço indisponível temporariamente |
| ECONNRESET | SIM | Conexão resetada |
| Outros erros de rede | SIM | Falhas transitórias |

### Headers Enviados

| Header | Valor | Obrigatório |
|---|---|---|
| `Content-Type` | `integration_routines.content_type` (default: application/json) | SIM |
| `Authorization` | Conforme `auth_type` do serviço | Depende do auth_type |
| `X-Correlation-ID` | `integration_call_logs.correlation_id` | SIM |
| `X-Idempotency-Key` | `integration_call_logs.id` | SHOULD |
| Headers de params | `integration_params` com `param_type='HEADER'` | Dinâmico |

### Limites Operacionais

| Recurso | Limite | Configurável |
|---|---|---|
| Concurrency BullMQ | 10 (default) | env `INTEGRATION_CONCURRENCY` |
| Timeout por chamada | 30000ms (default) | Por serviço e por rotina |
| Payload máximo | 1MB | Fixo — limite prático JSON HTTP |
| Rate limit (execute manual) | 30 req/min | Por tenant |

### Observabilidade

- `X-Correlation-ID` propagado em todos os requests HTTP ao Protheus
- Structured logging: `{ routine_id, service_id, http_method, endpoint, status, duration_ms, attempt_number, correlation_id }`
- Métricas Prometheus: `integration_calls_total{status,routine_id}`, `integration_calls_duration_ms{routine_id}`, `integration_dlq_count{routine_id}`

```json
{

    "has_more": true,
    "limit": 50
```

> O cursor é opaco e baseado em `(created_at, id)` para estabilidade em inserções concorrentes.
