## FR-001 — Login de Usuário (UX-AUTH-001 — Painel login)

- **Descrição:** O sistema DEVE prover tela de login com campos e-mail e senha, integrada ao endpoint `POST /api/v1/auth/login` (auth_login) do MOD-000. Login bem-sucedido redireciona para /dashboard e exibe o Application Shell.
- **Prioridade:** Must
- **Perfis impactados:** Administradores
- **Done funcional:**
  - Campos e-mail e senha renderizados no painel login
  - Botão "Entrar" dispara POST /auth/login com isLoading imediato
  - Resposta 200 → redirect para /dashboard + exibição do Shell (UX-SHELL-001)
  - Resposta 200 com `mfa_required=true` → **fallback defensivo:** exibir Toast "Autenticação multifator requerida. Contate o administrador para configuração." + NÃO redirecionar para /login/mfa (rota não especificada — MFA é roadmap futuro, mod.md §2). Quando UX-MFA-001 for implementado, substituir fallback pelo redirect para /login/mfa?session=temp_token (PENDENTE-004 §Opção B)
  - Resposta 401 → Toast "E-mail ou senha inválidos." + correlationId (BR-001)
  - Resposta 403 → Toast "Sua conta está bloqueada. Entre em contato com o suporte." + correlationId
  - Resposta 429 → Toast "Muitas tentativas. Tente novamente em X segundos." + botão desabilitado por retry_after
- **Efeito colateral?** [ ] Sim [x] Não *(login é endpoint do MOD-000)*
- **Idempotência:** n/a (delegada ao MOD-000)
- **Timeline/Notifications:** yes — domain events MOD-000: `auth.login_success`, `auth.login_failed` (via X-Correlation-ID)
- **Dependências (IDs):** BR-001, BR-003, BR-004, SEC-001, SEC-002, UX-001, INT-001, DATA-003

```gherkin
Funcionalidade: Login de Usuário

Cenário: Login bem-sucedido redireciona para Dashboard
  Dado que o usuário está na tela /login
  Quando ele preenche email e senha válidos e clica em "Entrar"
  Então o botão deve entrar em isLoading
  E POST /api/v1/auth/login deve ser chamado
  E ao receber 200, redirecionar para /dashboard
  E o Application Shell deve ser exibido

Cenário: Login com MFA obrigatório (fallback — UX-MFA-001 não implementado)
  Dado que o usuário tem MFA configurado
  Quando o login retorna 200 com mfa_required=true
  Então exibir Toast "Autenticação multifator requerida. Contate o administrador para configuração."
  E NÃO redirecionar para /login/mfa (rota inexistente)
  E NÃO exibir o Application Shell
  E manter o usuário no painel login

Cenário: Erro 401 exibe mensagem genérica
  Dado que o login retorna 401
  Então exibir Toast "E-mail ou senha inválidos."
  E incluir correlationId no rodapé
  E retornar botão ao estado normal

Cenário: Conta bloqueada (403)
  Dado que o login retorna 403
  Então exibir Toast "Sua conta está bloqueada. Entre em contato com o suporte."

Cenário: Rate limit (429)
  Dado que o login retorna 429 com retry_after
  Então exibir Toast "Muitas tentativas. Tente novamente em X segundos."
  E desabilitar botão pelo período retry_after
```

## FR-003 — Reset de Senha via Token (UX-AUTH-001 — Painel reset-password)

- **Descrição:** O sistema DEVE prover painel de redefinição de senha ativado automaticamente quando a URL contém `?token=<uuid>`. Submit dispara `POST /api/v1/auth/reset-password` (auth_reset_password).
- **Prioridade:** Must
- **Perfis impactados:** Administradores
- **Done funcional:**
  - Acesso a `/login?token=<uuid>` ativa painel reset-password automaticamente
  - Campos nova_senha e confirmar_senha visíveis
  - Token NÃO validado antecipadamente (apenas no submit)
  - Resposta 200 → Toast de sucesso + retorno ao painel login
  - Resposta 400/422 → Toast "Link inválido ou expirado. Solicite um novo link." + link para forgot-password
- **Efeito colateral?** [ ] Sim [x] Não *(endpoint do MOD-000)*
- **Idempotência:** n/a (delegada ao MOD-000)
- **Timeline/Notifications:** yes — domain event MOD-000: `auth.password_reset_completed` (via X-Correlation-ID)
- **Dependências (IDs):** BR-003, BR-004, SEC-001, UX-001, INT-004, DATA-003

```gherkin
Funcionalidade: Reset de Senha via Token

Cenário: Token na URL ativa painel reset-password

  Então o painel reset-password DEVE ser ativado
  E campos nova_senha e confirmar_senha devem ser visíveis

Cenário: Token expirado exibe erro
  Dado que o usuário submete reset com token expirado
  Quando POST /auth/reset-password retorna 400/422
```

## FR-004 — Application Shell com Sidebar e Header (UX-SHELL-001)

- **Descrição:** O sistema DEVE prover Application Shell persistente após autenticação, composto por Sidebar (menu filtrado por scopes), Header (Breadcrumb + ProfileWidget) e ContentArea. O Shell monta chamando `GET /api/v1/auth/me` (auth_me) via React Query com cache TTL de 30s (PENDENTE-001 §Opção C) e exibe skeleton durante loading.
- **Prioridade:** Must
- **Perfis impactados:** Administradores autenticados
- **Done funcional:**
  - Shell monta após login bem-sucedido e chama auth_me via React Query (query key: `['auth', 'me']`, staleTime: 30s) — resultado é compartilhado automaticamente com Dashboard (FR-005) sem acoplamento via Context
  - Skeleton exibido no Header (avatar, nome, tenant) e Sidebar (itens de menu) durante loading
  - Sidebar filtrada por `auth_me.scopes[]` (BR-005)
  - **Empty state Sidebar:** Quando `auth_me.scopes=[]`, Sidebar exibe item placeholder com ícone `Info` e texto "Nenhum módulo configurado para seu perfil." em vez de ficar completamente vazia (PENDENTE-002 §Opção B)
  - ProfileWidget exibe nome completo, e-mail, nome do tenant ativo
  - Dropdown ProfileWidget contém "Alterar Senha" e "Sair"
  - Logout dispara `POST /api/v1/auth/logout` (auth_logout) com isLoading; após 200 ou erro de rede, redirect para /login
- **Efeito colateral?** [ ] Sim [x] Não
- **Idempotência:** n/a (leitura auth_me + logout delegado ao MOD-000)
- **Timeline/Notifications:** yes — domain event MOD-000: `auth.logout` (via X-Correlation-ID)
- **Dependências (IDs):** BR-003, BR-004, BR-005, SEC-001, UX-001, INT-002, INT-005, INT-006, FR-007, NFR-001, DATA-003

```gherkin
Funcionalidade: Application Shell

Cenário: Shell carrega com Sidebar filtrada por scopes
  Dado que o usuário está autenticado com scopes ["users:user:read", "tenants:branch:read"]
  Quando o Shell carrega após auth_me retornar
  Então a Sidebar DEVE exibir apenas "Usuários" e "Filiais"

Cenário: Skeleton loading durante auth_me
  Dado que o Shell está montando e aguardando auth_me
  Então Header deve exibir skeleton nos campos avatar, nome e tenant
  E Sidebar deve exibir skeleton nos itens de menu

Cenário: Sidebar exibe empty state quando scopes estão vazios
  Dado que o usuário está autenticado com scopes=[]
  Quando o Shell carrega após auth_me retornar
  Então a Sidebar DEVE exibir item placeholder com ícone Info
  E texto "Nenhum módulo configurado para seu perfil."
  E a Sidebar NÃO deve estar completamente vazia

Cenário: Sessão expirada redireciona para login
  Dado que qualquer requisição HTTP retorna 401
  Então o interceptor HTTP global deve redirecionar para /login
  E exibir Toast "Sua sessão expirou. Faça login novamente."

Cenário: Logout encerra sessão
  Dado que o usuário clica em "Sair" no ProfileWidget
  Então POST /auth/logout deve ser chamado
  E após resposta (sucesso ou erro), redirecionar para /login
```

## FR-005 — Dashboard Executivo com Saudação e Atalhos (UX-DASH-001)

- **Descrição:** O sistema DEVE prover Dashboard pós-login com WelcomeWidget (saudação personalizada por período do dia + nome + filial ativa) e ModuleShortcuts (cards filtrados por scopes). Dados vindos exclusivamente de `GET /api/v1/auth/me` via React Query com cache TTL de 30s (PENDENTE-001 §Opção C), compartilhando resultado com Shell (FR-004).
- **Prioridade:** Must
- **Perfis impactados:** Administradores autenticados
- **Done funcional:**
  - Dashboard monta e consome auth_me via React Query (query key: `['auth', 'me']`, staleTime: 30s) — se Shell (FR-004) já buscou, o resultado vem do cache sem requisição adicional (BR-008)
  - WelcomeWidget exibe saudação: "Bom dia/Boa tarde/Boa noite, {nome}!" + subtexto "{tenant}"
  - Horários: 5h–12h=dia, 12h–18h=tarde, 18h–5h=noite (horário local do browser)
  - ModuleShortcuts exibe cards filtrados por `auth_me.scopes[]`
  - Catálogo de atalhos: Usuários (users:user:read), Perfis e Permissões (users:role:read), Filiais (tenants:branch:read), Auditoria (system:audit:read)
  - Skeleton durante loading; timeout 3s → estado de erro com correlationId + botão retry (BR-009)
  - Estado vazio (sem scopes) exibe "Nenhum módulo disponível para seu perfil."
  - Erro 5xx → Toast com correlationId + retry, sem desconectar o usuário (BR-010)
  - Erro 401 → redirect /login via interceptor
- **Efeito colateral?** [ ] Sim [x] Não
- **Idempotência:** n/a (leitura auth_me)
- **Timeline/Notifications:** no
- **Dependências (IDs):** BR-004, BR-005, BR-008, BR-009, BR-010, SEC-001, UX-001, INT-005, NFR-001

## FR-006 — Telemetria UI via UIActionEnvelope (Todas as Telas)

- **Descrição:** O sistema DEVE integrar `@easycf/ui-telemetry` nas 3 telas do MOD-001, emitindo `UIActionEnvelope` correto para cada ação rastreável. Ações pré-auth (UX-AUTH-001) sem tenant_id; ações pós-auth com tenant_id. X-Correlation-ID propagado em toda requisição HTTP não-client_only.
- **Prioridade:** Must
- **Perfis impactados:** Time de engenharia (observabilidade)
- **Done funcional:**
  - UIActionEnvelope emitido para todas as actions não-client_only dos 3 manifests
  - Ciclo de vida: requested → succeeded (2xx) ou failed (4xx/5xx)
  - duration_ms preenchido em succeeded/failed
  - problem_type derivado de RFC 9457 em failed
  - tenant_id ausente em UX-AUTH-001 (BR-006)
  - tenant_id presente em UX-SHELL-001 e UX-DASH-001
  - X-Correlation-ID em todo header HTTP não-client_only
  - Cada ação gera correlation_id UUID v4 distinto
  - Ações client_only (navigate_to_forgot, navigate_to_login, navigate_sidebar, navigate_breadcrumb) NÃO geram requisição HTTP
- **Efeito colateral?** [ ] Sim [x] Não
- **Idempotência:** n/a (telemetria fire-and-forget)
- **Timeline/Notifications:** yes — UIActionEnvelopes correlacionados com domain events MOD-000 via X-Correlation-ID
- **Dependências (IDs):** BR-006, SEC-001, SEC-002, INT-001, INT-002, INT-003, INT-004, INT-005, DATA-003, NFR-001

```gherkin
Funcionalidade: Telemetria UI

Cenário: Ação pré-auth emite envelope sem tenant_id
  Dado que o usuário está na tela /login (UX-AUTH-001)
  Quando dispara submit_login
  Então UIActionEnvelope DEVE ter screen_id=UX-AUTH-001, action_id=submit_login
  E tenant_id NÃO deve estar presente

Cenário: Ação pós-auth emite envelope com tenant_id
  Dado que o usuário está no Shell (UX-SHELL-001)
  Quando dispara load_current_user
  Então UIActionEnvelope DEVE incluir tenant_id do JWT

Cenário: X-Correlation-ID em toda requisição HTTP
  Dado que uma action não-client_only é disparada
  Quando a requisição HTTP é enviada
  Então X-Correlation-ID DEVE estar no header
  E o mesmo correlation_id DEVE estar no UIActionEnvelope
```
