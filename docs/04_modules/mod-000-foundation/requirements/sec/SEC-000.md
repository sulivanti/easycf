# SEC-000 — Segurança e Compliance do Foundation

---

## 1. Autenticação

- **Modelo:** JWT com sessionId ancorado em banco (desvio intencional de stateless puro — BR-002)
- **Fluxos:**
  - Login nativo: e-mail + senha (bcrypt compare)
  - SSO OAuth2: Google, Microsoft/Azure AD, Sign in with Apple (OIDC/JWKS)
  - MFA/TOTP (RFC 6238): segundo fator opcional por usuário
- **Sessões:** Armazenadas em `user_sessions` com Kill-Switch individual e global
- **Tokens:**
  - `accessToken` e `refreshToken` em cookies httpOnly, sameSite=lax (BR-010)
  - TTL access: 12h (normal), 30d (remember_me)
  - **Refresh Token Rotation (PENDENTE-002, OAuth2 BCP):**
    - A cada POST /auth/refresh, novo refresh_token é emitido e o anterior é invalidado
    - Token families: cada refresh_token pertence a uma família (raiz = login original)
    - **Reuse detection:** Se um refresh_token já utilizado for apresentado novamente, toda a família é invalidada (kill-switch automático — sessão comprometida)
    - Grace period: 10s para o token antigo, mitigando race conditions em múltiplas abas
    - Evento de segurança: `auth.token_reuse_detected` emitido na detecção de reuso (sensitivity_level=2)
- **Desvio MFA:** Quando ativo, login emite apenas `temp_token` (escopo `mfa-only`, TTL 5min — BR-008)
- **SSO Identity Linking (ADR-004):** Quando SSO detecta e-mail existente com conta nativa, exige confirmação via senha nativa antes de vincular provider. `temp_sso_token` (escopo `sso-link-only`, TTL 10min, máx. 5 tentativas)

## 2. Autorização

- **Modelo:** RBAC por escopos (`dominio:entidade:acao`)
- **Formato:** Regex `^[a-z][a-z0-9_]*(:[a-z][a-z0-9_]*){1,2}$` (BR-005, DOC-FND-000 §2.1)
- **Enforcement:**
  - Backend: `requireScope('dominio:entidade:acao')` no `preHandler` de todo endpoint protegido
  - Frontend: Ocultação de menus/rotas (decisão UX, não substitui enforcement)
- **Cache:** Redis `auth:scopes:role:{roleId}` com invalidação automática (BR-011)
- **Multi-tenant:** `requireTenantScope` verifica status do vínculo no pivot `tenant_users` — BLOCKED/INACTIVE = 403

### §2.3 — Anti-escalação na atribuição de roles

#### Regra SEC-ANTI-ESC-001 — Validação de anti-escalação

**Definição:** O backend DEVE rejeitar `POST /api/v1/users` com `role_id` quando a role referenciada contém scopes que o usuário autenticado NÃO possui.

**Algoritmo:**

```text
1. Extrair scopes do JWT do requisitante (auth.scopes[])
2. Consultar scopes da role referenciada (role_permissions WHERE role_id = request.role_id)
3. Computar diferença: target_scopes - requester_scopes
4. Se diferença não-vazia → 403 Forbidden
```

**Response de erro (RFC 9457):**

```json
{
  "type": "/problems/privilege-escalation-denied",
  "title": "Privilege escalation denied",
  "status": 403,
  "detail": "Cannot assign role with scopes you do not possess: [system:audit:sensitive, users:user:delete].",
  "extensions": {
    "missing_scopes": ["system:audit:sensitive", "users:user:delete"],
    "correlationId": "{correlation_id}"
  }
}
```

**Exceção:** Usuários com scope `users:role:write` (gestores de roles) podem atribuir qualquer role. Justificativa: quem pode criar/editar roles já tem capacidade implícita de escalar privilégios.

#### Cenários Gherkin

```gherkin
Cenário: Admin com scopes limitados tenta atribuir role superior
  Dado que o admin tem scopes ["users:user:read", "users:user:write", "users:role:read"]
  E que a role "Super Admin" tem scopes ["users:user:read", "users:user:write", "users:user:delete", "system:audit:sensitive"]
  Quando POST /api/v1/users com role_id = id da role "Super Admin"
  Então status 403 com type "/problems/privilege-escalation-denied"
  E detail contém "system:audit:sensitive, users:user:delete"

Cenário: Admin com todos os scopes atribui qualquer role
  Dado que o admin tem todos os scopes do catálogo canônico
  Quando POST /api/v1/users com role_id = id da role "Super Admin"
  Então status 201 com usuário criado

Cenário: Gestor de roles pode atribuir qualquer role (exceção)
  Dado que o admin tem scope "users:role:write"
  E NÃO possui scope "system:audit:sensitive"
  Quando POST /api/v1/users com role_id = id da role que inclui "system:audit:sensitive"
  Então status 201 com usuário criado (exceção de gestores de roles)

Cenário: Criação sem role_id não requer validação de escalação
  Dado que o admin tem scope "users:user:write"
  Quando POST /api/v1/users sem role_id
  Então status 201 com usuário criado sem vinculação
```

#### Problem Type adicionado ao catálogo de erros

| Type | Status | Quando |
|------|--------|--------|
| `/problems/privilege-escalation-denied` | 403 | `role_id` contém scopes que o requisitante não possui |

## 3. Classificação dos Dados

| Classificação | Entidades/Campos | Tratamento |
|---|---|---|
| **Público** | tenant.name, tenant.codigo, role.name | Sem restrições |
| **Interno** | user.email, user.full_name, session metadata | Acesso autenticado + scope |
| **Confidencial** | user.cpf_cnpj, user.password_hash, mfa_secret | Mascaramento em logs, nunca em response |
| **Restrito** | system:audit:sensitive events, PII em domain_events | Acesso via `system:audit:sensitive` scope |

## 4. Política de Retenção

| Dado | Retenção | Justificativa |
|---|---|---|
| `user_sessions` (ativas) | TTL da sessão (12h ou 30d) | Limpeza automática de sessões expiradas |
| `user_sessions` (revogadas) | 90 dias | Auditoria de segurança |
| `domain_events` | Indefinida (soft) | Compliance LGPD — dados de auditoria |
| Usuários soft-deleted | 5 anos | LGPD — prazo legal de retenção |
| Arquivos em storage | Purge de efêmeros (pendentes > 24h) | Limpeza de uploads não confirmados |

## 5. Mascaramento e LGPD

- **Campos mascaráveis:** `cpf_cnpj`, `email` (em contextos de log), `password_hash` (NUNCA expor), `mfa_secret` (NUNCA expor)
- **Princípio de minimização:** Payloads de domain_events DEVEM conter snapshot mínimo sem PII desnecessária
- **Regra de response:** Nunca retornar `passwordHash` ou `mfaSecret` em qualquer endpoint
- **User Enumeration Prevention:** Respostas de falha de login sempre genéricas (BR-001)
- **Soft Delete obrigatório:** Dados preservados para auditoria, nunca hard delete (BR-004)

## 6. Auditoria

- **Fonte única:** Tabela `domain_events` (DATA-003) — proibido criar tabelas satélites de logs
- **Eventos auditáveis:** Todos catalogados em DATA-003 (34 event_types)
- **Trilha de mudanças:** correlation_id + causation_id para rastreabilidade E2E
- **Quem pode ver:**
  - `system:audit:read` → eventos com sensitivity_level 0-1
  - `system:audit:sensitive` → todos os níveis (0-3)
- **Endpoint canônico:** `GET /entities/{entityType}/{entityId}/history` (DOC-ARC-003 §4)

## 7. Autorização de Eventos (Emit/View/Notify)

- **Emit:** Controlado pela permissão do **comando** que gera o evento (não pelo evento em si)
- **View:** Controlado pela ACL da **entity originária** + filtro `tenant_id` obrigatório
- **Notify:** Resolvido por regras baseadas em papéis (watchers, owner, approver, admin)
- **Detalhes completos:** Ver SEC-002

## 8. Proteções

| Proteção | Implementação |
|---|---|
| **Rate Limiting** | 10 req/15min/IP em /auth/login (BR-003) |
| **Validação de entrada** | Zod na borda (shape validation) |
| **Logs seguros** | Logs estruturados sem PII/segredos |
| **Secrets** | Fora do repositório, scanner de secrets no CI |
| **CORS** | Configurável por ambiente |
| **X-Correlation-ID** | Propagado em toda resposta + eventos |
| **Idempotency-Key** | Endpoints com efeito colateral (BR-012) |
| **Problem Details (RFC 9457)** | Erros estruturados sem expor detalhes internos |
