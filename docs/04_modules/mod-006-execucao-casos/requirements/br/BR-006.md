# BR-006 — Regras de Negócio da Execução de Casos

---

## BR-001 — Caso Referencia Versão Frozen do Ciclo

- **Regra:** Ao abrir um caso, o sistema captura o `cycle_version_id` vigente. Qualquer fork ou atualização do ciclo no MOD-005 NÃO afeta instâncias em andamento.
- **Exemplo:** Ciclo v1 está PUBLISHED e caso é aberto. Ciclo v1 é DEPRECATED e v2 é publicado. O caso existente ainda referencia `cycle_version_id` de v1.
- **Exceções:** Nenhuma. Regra inegociável de integridade.
- **Impacto:** STATE, COMPLIANCE

```gherkin
Funcionalidade: Freeze de cycle_version_id

Cenário: Fork do ciclo não afeta casos em andamento
  Dado que ciclo v1 está PUBLISHED e caso foi aberto com cycle_version_id=v1
  Quando ciclo v1 é forked para v2 e estágio-A é renomeado em v2
  Então GET /cases/:id ainda retorna blueprint com estrutura da v1
  E case.cycle_version_id permanece apontando para v1
```

## BR-002 — Motor de Transição Bloqueia Gate Pendente

- **Regra:** Se algum `gate_instance` com `required=true` do estágio atual tem status diferente de RESOLVED ou WAIVED, o motor DEVE bloquear a transição com 422.
- **Exemplo:** Gate "Aprovação Gerencial" APPROVAL required=true com status=PENDING → 422: "Gate 'Aprovação Gerencial' ainda não foi resolvido."
- **Exceções:** Gates com `gate_type=INFORMATIVE` NUNCA bloqueiam transição, mesmo com required=true.
- **Impacto:** STATE, COMPLIANCE

```gherkin
Funcionalidade: Motor bloqueia gate pendente

Cenário: Gate APPROVAL pending bloqueia transição
  Dado que estágio atual tem gate "Aprovação Gerencial" APPROVAL required=true com status=PENDING
  Quando POST /cases/:id/transition
  Então 422: "Gate 'Aprovação Gerencial' não foi resolvido."
  E extensions.pending_gates inclui o gate pendente
  E current_stage_id permanece inalterado

Cenário: Gate WAIVED não bloqueia transição
  Dado que estágio atual tem gate APPROVAL required=true com status=WAIVED
  Quando POST /cases/:id/transition
  Então transição ocorre normalmente
```

## BR-003 — Históricos de Estágio e Responsável são Independentes

- **Regra:** Uma reatribuição de responsável NÃO gera entrada em `stage_history`. Uma transição de estágio NÃO altera automaticamente `case_assignments`.

## BR-007 — Reatribuição Desativa Atribuição Anterior

- **Regra:** Ao atribuir um novo usuário a um papel que já tem atribuição ativa, o sistema DEVE desativar a atribuição anterior (`is_active=false`) antes de criar a nova. Não pode haver duas atribuições ativas para o mesmo papel no mesmo estágio/caso.
- **Exemplo:** user-A é RESPONSAVEL ativo. POST /assignments com user-B para mesmo papel → atribuição de user-A desativada, user-B criado ativo.

## BR-012 — Transições de Status do Caso

- **Regra:** O caso segue transições de status válidas: OPEN → ON_HOLD (hold), ON_HOLD → OPEN (resume), OPEN|ON_HOLD → CANCELLED (cancel), OPEN → COMPLETED (transição para estágio terminal). Qualquer transição fora dessas é rejeitada com 422.
- **Exemplo:** Caso ON_HOLD → POST /transition → 422: "Caso não está aberto." (deve primeiro fazer resume).
- **Exceções:** REOPENED pode reverter COMPLETED → OPEN (requer scope `process:case:reopen`).
- **Impacto:** STATE

```gherkin
Funcionalidade: Máquina de estados do caso

Cenário: ON_HOLD bloqueia transições de estágio
  Dado que caso está ON_HOLD
  Quando POST /cases/:id/transition
  Então 422: "Caso não está aberto."

Cenário: Resume retorna caso para OPEN
  Dado que caso está ON_HOLD
  Quando POST /cases/:id/resume
  Então case.status=OPEN
  E case_events registra event_type=RESUMED
```

- **estado_item:** READY
- **owner:** Marcos Sulivan
- **data_ultima_revisao:** 2026-03-23
- **rastreia_para:** US-MOD-006-F01, FR-006

- **Regra:** Para resolver um gate do tipo CHECKLIST, TODOS os itens do checklist devem ser marcados como `checked=true`. Resolução parcial DEVE retornar 422.
- **Exemplo:** Gate CHECKLIST com 3 itens, 1 não marcado → 422: "Todos os itens do checklist são obrigatórios."

- **Regra:** Dispensar um gate obrigatório requer scope `process:case:gate_waive` e campo `motivo` obrigatório (min 20 caracteres). A ação DEVE ser registrada em case_events como EXCEPTION para auditoria.
- **Exemplo:** POST /cases/:id/gates/:gid/waive sem scope gate_waive → 403. Com scope mas motivo < 20 chars → 422.
- **Exceções:** Nenhuma.
- **Impacto:** COMPLIANCE, SECURITY

```gherkin
Funcionalidade: Dispensa de gate auditada

Cenário: Waive com escopo e motivo válido
  Dado que usuário tem scope process:case:gate_waive

  Então gate_instance.status=WAIVED
  E case_events registra EXCEPTION com gate_id e motivo

Cenário: Waive sem escopo especial
  Dado que usuário NÃO tem scope process:case:gate_waive
  Quando POST /cases/:id/gates/:gateId/waive
```

## BR-016 — Reabertura de Caso COMPLETED

- **Regra:** Registrar evento REOPENED em caso COMPLETED reverte `status` para OPEN e limpa `completed_at`. Requer scope `process:case:reopen` (PEN-006/PENDENTE-001 Opção B). Body DEVE incluir `target_stage_id` obrigatório — o motor não adivinha, o operador decide para qual estágio retornar (PEN-006/PENDENTE-005 Opção B). Validação: `target_stage_id` deve pertencer ao mesmo `cycle_id` do caso. Gates do estágio destino são recriados como PENDING. Ação excepcional de alto impacto e auditada.
- **Exemplo:** POST /cases/:id/events com event_type=REOPENED e target_stage_id="Em Análise" → case.status=OPEN, case.completed_at=null, current_stage_id="Em Análise", gates recriados PENDING.
- **Exceções:** Nenhuma.
- **Impacto:** STATE, COMPLIANCE

```gherkin
Funcionalidade: Reabertura de caso

Cenário: Reabrir caso COMPLETED com scope e target_stage_id
  Dado que caso está COMPLETED no estágio terminal "Encerrado"
  E usuário tem scope process:case:reopen
  Quando POST /cases/:id/events com { event_type: "REOPENED", descricao: "Correção necessária", target_stage_id: "Em Análise" }
  Então case.status=OPEN
  E case.completed_at=null
  E current_stage_id atualizado para "Em Análise"
  E gate_instances do estágio "Em Análise" recriados como PENDING
  E case_events registra REOPENED
  E domain event case.reopened emitido

Cenário: Rejeitar REOPENED sem target_stage_id
  Dado que caso está COMPLETED
  E usuário tem scope process:case:reopen
  Quando POST /cases/:id/events com { event_type: "REOPENED" } sem target_stage_id
  Então 422: "target_stage_id é obrigatório para reabertura."

Cenário: Rejeitar target_stage_id de outro ciclo
  Dado que caso pertence ao cycle_id=ciclo-A
  Quando POST REOPENED com target_stage_id de ciclo-B
  Então 422: "Estágio não pertence ao ciclo deste caso."
```
