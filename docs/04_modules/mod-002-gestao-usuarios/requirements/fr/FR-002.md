# FR-002 — Formulário de Cadastro de Usuário (Dois Modos)

- **Descrição:** Formulário de criação de usuário com dois modos mutuamente exclusivos via abas: "Enviar Convite" (padrão) e "Senha Temporária". Validação inline por campo, select de perfil carregado via API, indicador de força de senha client-side no modo senha, e idempotência via `Idempotency-Key` gerado no mount. Redirect automático 1.5s após sucesso. Modo edição: formulário preenchido via GET /users/:id, campos editáveis (nome, perfil), e-mail readonly, toggle de status, PATCH /users/:id.
- **Prioridade:** Must
- **Perfis impactados:** Administrador (gestão), Administrador (completo)
- **Done funcional:**
  - [ ] Acesso a /usuarios/novo requer scope `users:user:write` — redirect sem ele (BR-001)
  - [ ] Select "Perfil de acesso" carregado via `roles_list` com skeleton durante loading
  - [ ] Falha ao carregar roles: Toast + correlationId, campo com retry inline (BR-006)
  - [ ] Modo padrão: "Enviar Convite" — campos de senha ocultos (BR-003)
  - [ ] Troca de modo reseta todo o formulário (campos + erros) (BR-003)
  - [ ] Modo convite: POST com `{ fullName, email, roleId, mode: "invite" }` (BR-003)
  - [ ] Modo senha: POST com `{ fullName, email, roleId, password, forcePasswordReset: true }` (BR-003)
  - [ ] Indicador de força de senha client-side (Fraca/Média/Forte) — sem chamada à API
  - [ ] Validação de confirmação de senha: "As senhas não coincidem."
  - [ ] Campos obrigatórios em branco: botão submit disabled + mensagem inline
  - [ ] E-mail inválido: validação onBlur com mensagem inline
  - [ ] Erro 409 (e-mail duplicado): inline no campo e-mail, sem toast (BR-006)
  - [ ] Erro 422 (invalid_fields): inline por campo + foco no primeiro erro (BR-006)
  - [ ] Erro 500: toast + correlationId, formulário preservado para retry (BR-006)
  - [ ] `Idempotency-Key`: UUID v4 no header, double-click = 1 requisição (BR-005)
  - [ ] Toast sucesso convite: "Usuário criado com sucesso. Convite enviado." — sem e-mail (BR-002)
  - [ ] Toast sucesso senha: "Usuário criado com sucesso." — sem e-mail (BR-002)
  - [ ] Redirect para /usuarios após 1.5s de sucesso
  - [ ] Cancelar: descarta formulário, nenhuma chamada API, redirect para /usuarios
  - [ ] Acesso a /usuarios/:id carrega dados via GET /api/v1/users/:id com skeleton
  - [ ] Campos editáveis: nome completo, perfil (roleId). E-mail readonly.
  - [ ] Toggle de status visível se scope `users:user:write` (ACTIVE/BLOCKED/INACTIVE)
  - [ ] Botão "Resetar Senha" (secondary) se scope `users:user:write` + status ACTIVE

  - [ ] Erro 404 → redirect /usuarios + toast "Usuário não encontrado."
  - [ ] Erro 422 → inline por campo
  - [ ] Erro 500 → toast + correlationId, formulário preservado

### operationIds Consumidos

| operationId | Ação | Método |
|---|---|---|
| `users_create` | Criar usuário (convite ou senha) | POST /api/v1/users |
| `roles_list` | Popula select de perfil | GET /api/v1/roles |
| `users_get` | Carregar dados do usuário para edição | GET /api/v1/users/:id |
| `users_update` | Atualizar usuário | PATCH /api/v1/users/:id |

### Critérios de aceite (Gherkin)

```gherkin
Funcionalidade: Formulário de Cadastro — FR-002

  Cenário: Criação com convite — happy path
    Dado que o admin com scope "users:user:write" abre /usuarios/novo
    E preenche nome, e-mail e seleciona perfil
    Quando clica "Criar usuário"
    Então POST /api/v1/users é chamado com mode="invite" e Idempotency-Key
    E ao receber 201, Toast exibe "Usuário criado com sucesso. Convite enviado."
    E após 1.5s, redireciona para /usuarios

  Cenário: Criação com senha temporária
    Dado que o admin seleciona aba "Senha Temporária"
    E preenche todos os campos + senha válida + checkbox "Exigir troca"
    Quando clica "Criar usuário"
    Então POST /api/v1/users é chamado com password e forcePasswordReset=true

  Cenário: Troca de modo reseta formulário
    Dado que o admin preencheu campos no modo convite
    Quando clica na aba "Senha Temporária"
    Então todos os campos são limpos e erros removidos

  Cenário: Erros 422 exibidos por campo
    Dado que POST /api/v1/users retorna 422 com extensions.invalid_fields
    Então cada campo inválido exibe mensagem inline
    E foco é movido para o primeiro campo com erro

  # --- Cenários de Edição (FR-002-M01) ---

  Cenário: Edição de usuário — happy path
    Dado que o admin com scope "users:user:read" acessa /usuarios/:id
    Então GET /api/v1/users/:id é chamado
    E o formulário é preenchido com os dados retornados
    E o campo e-mail está readonly

  Cenário: Salvar alterações com sucesso
    Dado que o admin editou nome e/ou perfil
    Quando clica "Salvar Alterações"
    Então PATCH /api/v1/users/:id é chamado com os campos alterados
    E ao receber 200, toast exibe "Usuário atualizado com sucesso."
    E redireciona para /usuarios

  Cenário: Toggle de status visível por scope
    Dado que o admin com scope "users:user:write" acessa /usuarios/:id
    Então o toggle de status é visível (ACTIVE/BLOCKED/INACTIVE)

  Cenário: Toggle de status oculto sem scope
    Dado que o admin sem scope "users:user:write" acessa /usuarios/:id
    Então o toggle de status não é exibido

  Cenário: Botão Resetar Senha visível para usuário ativo
    Dado que o admin com scope "users:user:write" acessa /usuarios/:id de um usuário ACTIVE
    Então o botão "Resetar Senha" (secondary) é visível

  Cenário: Usuário não encontrado (404)
    Dado que o admin acessa /usuarios/:id com ID inexistente
    Quando GET /api/v1/users/:id retorna 404
    Então redireciona para /usuarios
    E toast exibe "Usuário não encontrado."

  Cenário: Erro de validação (422)
    Dado que PATCH /api/v1/users/:id retorna 422 com extensions.invalid_fields
    Então cada campo inválido exibe mensagem inline
    E foco é movido para o primeiro campo com erro

  Cenário: Erro de servidor (500)
    Dado que PATCH /api/v1/users/:id retorna 500
    Então toast exibe mensagem de erro + correlationId
    E formulário é preservado para retry
```
