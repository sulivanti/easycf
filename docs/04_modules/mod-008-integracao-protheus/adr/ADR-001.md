# ADR-001 — Outbox Pattern para Garantia de Entrega de Chamadas

## Contexto

O MOD-008 precisa garantir que toda chamada de integração disparada por um evento de negócio (ex: transição de estágio no MOD-006) seja efetivamente executada, mesmo que o worker BullMQ esteja indisponível no momento do disparo. Três abordagens foram avaliadas para garantir a atomicidade entre a transação de negócio e o enfileiramento da chamada HTTP.

## Decisão

**Adotar Outbox Pattern:** O INSERT em `integration_call_logs` com `status=QUEUED` ocorre **dentro da mesma transação** da operação de negócio que originou a chamada. Um worker BullMQ escaneia periodicamente logs com status QUEUED e enfileira jobs. O `call_log.id` é usado como `jobId` no BullMQ para deduplicação automática.

## Alternativas Consideradas

1. **Fire-and-forget (enfileirar diretamente no BullMQ):**
   - Prós: Simplicidade de implementação; menor latência (sem poll).
   - Contras: Se o enqueue no Redis falhar ou o worker não consumir antes de um crash, a chamada se perde irrecuperavelmente. Viola OKR-1 (zero perda de chamadas). Não há registro persistente do intent de chamada.

2. **Two-Phase Commit (2PC) entre PostgreSQL e Redis:**
   - Prós: Atomicidade real entre banco e fila.
   - Contras: Complexidade operacional elevada; 2PC entre PostgreSQL e Redis não é nativamente suportado; latência adicional significativa; ponto único de falha no coordinator. Overengineering para o volume esperado.

3. **Outbox Pattern (escolhido):**
   - Prós: Atomicidade garantida via transação SQL única; zero perda de chamadas por crash de worker; registro de auditoria integrado (call_log já serve como log e outbox); deduplicação via `jobId = call_log.id`; compatível com BullMQ sem modificações.
   - Contras: Polling introduz latência adicional (mitigado com scan interval curto); INSERT no caminho crítico da transação de negócio (mitigado com SLO < 50ms p95).

## Consequências

- **Positivas:** Zero perda de chamadas mesmo com crash de worker (OKR-1); a tabela `integration_call_logs` serve simultaneamente como outbox e audit trail; deduplicação nativa via BullMQ `jobId`; RPO = 0 para jobs enfileirados.
- **Negativas:** Latência de polling entre INSERT e processamento (mitigável com scan interval de 1-5s); INSERT adicional no caminho crítico da transação de negócio (SLO < 50ms p95 — validar com testes de performance).
- **Riscos:** Se o scan interval for muito alto, chamadas ficam em QUEUED por tempo excessivo. Mitigação: monitorar `integration_queue_latency_ms` e alertar se > 10s.
