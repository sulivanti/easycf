# BR-001 — Regras de Negócio do Backoffice Admin

---

## BR-001 — Prevenção de Enumeração de Usuários no Shell (User Enumeration Prevention)

- **Regra:** Mensagens de erro de login DEVEM ser sempre genéricas: `"E-mail ou senha inválidos."` — sem variações por tipo de erro (e-mail inexistente, senha incorreta, conta inativa). Aplica-se a todas as respostas 401 na tela UX-AUTH-001.
- **Exemplo:** Usuário tenta login com e-mail inexistente → UI exibe Toast "E-mail ou senha inválidos." com correlationId. Usuário tenta com senha errada → mesma mensagem exata.
- **Exceções:** Conta bloqueada (403) exibe mensagem específica: "Sua conta está bloqueada. Entre em contato com o suporte." — esta é uma exceção aceita pois não revela existência de e-mail (requer credenciais corretas para chegar ao 403).
- **Impacto:** COMPLIANCE, PERMISSIONS

```gherkin
Funcionalidade: Prevenção de Enumeração no Shell

Cenário: Login com credenciais inválidas exibe mensagem genérica
  Dado que o usuário está na tela /login (UX-AUTH-001)
  Quando POST /auth/login retorna 401 (e-mail inexistente OU senha incorreta)
  Então a mensagem exibida DEVE ser exatamente "E-mail ou senha inválidos."
  E um Toast de erro deve aparecer com correlationId no rodapé
```

## BR-002 — Resposta Genérica de Recuperação de Senha (Anti-Enumeration)

- **Regra:** A solicitação de recuperação de senha DEVE sempre exibir mensagem positiva: `"Se o e-mail estiver cadastrado, você receberá um link em breve."` — independentemente de o e-mail existir ou não no sistema.
- **Exemplo:** Usuário preenche "naoexiste@test.com" no painel forgot-password → Toast de sucesso com mensagem genérica. Usuário preenche e-mail válido → mesma mensagem.
- **Exceções:** Nenhuma. Regra inegociável de segurança.
- **Impacto:** COMPLIANCE, PERMISSIONS

```gherkin
Funcionalidade: Recuperação de Senha sem Enumeração

Cenário: Solicitação de recuperação com resposta genérica
  Dado que o usuário está no painel forgot-password da UX-AUTH-001
  Quando ele preenche qualquer e-mail e clica em "Enviar link"
  Então o Toast de sucesso DEVE exibir "Se o e-mail estiver cadastrado, você receberá um link em breve."
  E esta mensagem DEVE ser idêntica para e-mail existente e inexistente
```

- **estado_item:** DRAFT

## BR-003 — Estado isLoading Obrigatório em Botões de Submit

- **Regra:** Todo botão de submit (login, forgot-password, reset-password, logout) DEVE entrar em estado `isLoading` imediatamente após o clique — desabilitado com spinner — permanecendo assim até a resposta da API ou timeout.
- **Exemplo:** Usuário clica "Entrar" → botão fica desabilitado com spinner → ao receber 200 redireciona para /dashboard; ao receber 401 o botão volta ao estado normal.
- **Exceções:** Nenhuma.
- **Impacto:** FLOW, STATE

```gherkin
Funcionalidade: Estado isLoading em Submits (BR-003)

Cenário: Botão de login entra em isLoading durante requisição
  Dado que o usuário clica em "Entrar" na tela /login
  Então o botão DEVE ser desabilitado com spinner imediatamente
  E DEVE permanecer desabilitado até resposta da API
  E ao receber resposta (sucesso ou erro) DEVE retornar ao estado normal
```

- **estado_item:** DRAFT
- **owner:** arquitetura

## BR-004 — Toast de Erro com CorrelationId Obrigatório

- **Regra:** Todo Toast de erro HTTP DEVE incluir o `correlationId` no rodapé (texto pequeno, cor cinza). Nunca exibir detalhes técnicos (stack trace, IDs internos, mensagem raw do servidor).
- **Exemplo:** Login falha com 500 → Toast "Erro ao processar. Tente novamente." + rodapé "ID: abc123-def456"
- **Exceções:** Nenhuma.
- **Impacto:** FLOW, COMPLIANCE

```gherkin
Funcionalidade: Toast de Erro com CorrelationId (BR-004)

Cenário: Erro HTTP exibe Toast com correlationId
  Dado que qualquer requisição HTTP do Shell retorna erro (4xx/5xx)
  Então um Toast de erro DEVE ser exibido
  E o rodapé do Toast DEVE conter o correlationId da requisição
  E NENHUM detalhe técnico deve ser visível ao usuário
```

## BR-005 — Sidebar Populada por Scopes do auth_me (Nunca Hardcoded)

- **Regra:** A Sidebar do Application Shell DEVE ser populada exclusivamente pelos scopes retornados por `GET /auth/me`. Itens de menu sem scope correspondente NÃO devem aparecer na Sidebar — nem desabilitados, nem com ícone de cadeado, nem com tooltip "sem acesso".
- **Exemplo:** auth_me retorna `scopes=["users:user:read", "tenants:branch:read"]` → Sidebar exibe apenas "Usuários" e "Filiais". "Perfis e Permissões" e "Auditoria" não aparecem.
- **Exceções:** Nenhuma. Regra de segurança e UX.
- **Impacto:** PERMISSIONS, FLOW

```gherkin
Funcionalidade: Sidebar Filtrada por Scopes

Cenário: Sidebar exibe apenas itens com scope autorizado
  Dado que auth_me retorna scopes=["users:user:read", "tenants:branch:read"]
  Quando o Shell renderiza a Sidebar
  Então APENAS "Usuários" e "Filiais" devem aparecer
  E "Perfis e Permissões" (requer users:role:read) NÃO deve aparecer
  E "Auditoria" (requer system:audit:read) NÃO deve aparecer
  E nenhum item deve aparecer desabilitado ou com indicação de "sem acesso"
```

## BR-006 — tenant_id Ausente em Ações Pré-Autenticação

- **Regra:** Nenhuma ação da tela UX-AUTH-001 (pré-autenticação) DEVE incluir `tenant_id` no `UIActionEnvelope`. O campo deve ser `null` ou omitido. Ações pós-autenticação (UX-SHELL-001, UX-DASH-001) DEVEM incluir `tenant_id` do JWT.
- **Exemplo:** submit_login emite UIActionEnvelope sem tenant_id. load_current_user (após auth) emite com tenant_id do JWT.
- **Exceções:** Nenhuma. Regra de privacidade e consistência.
- **Impacto:** DATA, COMPLIANCE

```gherkin
Funcionalidade: tenant_id por Fase de Autenticação

Cenário: Ação pré-auth não inclui tenant_id
  Dado que o usuário está na tela /login (UX-AUTH-001, pre_auth=true)
  Quando ele dispara submit_login
  Então o UIActionEnvelope NÃO deve conter tenant_id

Cenário: Ação pós-auth inclui tenant_id
  Dado que o usuário está autenticado no Shell (UX-SHELL-001)
  Quando o Shell dispara load_current_user
  Então o UIActionEnvelope DEVE conter tenant_id do JWT ativo
```

- **estado_item:** DRAFT

- **Regra:** A transição entre o painel `login` e o painel `forgot-password` na UX-AUTH-001 DEVE ser client-only: sem reload de página, sem alteração da URL `/login`, sem requisição HTTP.
- **Exemplo:** Usuário clica "Esqueci minha senha" → painel muda para forgot-password → URL permanece `/login` → nenhuma requisição HTTP é disparada.
- **Exceções:** Nenhuma.

```gherkin
Funcionalidade: Transição Client-Only Login ↔ Forgot

Cenário: Abertura do painel de recuperação sem reload
  Dado que o usuário está no painel login da UX-AUTH-001
  Quando ele clica em "Esqueci minha senha"
  Então o painel DEVE mudar para forgot-password sem reload
  E a URL DEVE permanecer /login
  E nenhuma requisição HTTP deve ser disparada
```

## BR-008 — Dashboard Usa auth_me como Fonte Única (Nunca Cache)

- **Regra:** O Dashboard (UX-DASH-001) DEVE chamar `GET /auth/me` ao montar para obter nome, tenant e scopes. NUNCA usar localStorage, JWT decodificado ou cache desatualizado como fonte de dados de exibição.
- **Exemplo:** localStorage contém nome "João" (desatualizado) mas auth_me retorna "João Carlos" → WelcomeWidget exibe "João Carlos".
- **Exceções:** Nenhuma.
- **Impacto:** DATA, STATE

```gherkin
Funcionalidade: auth_me como Fonte Única do Dashboard

Cenário: Dashboard usa dados de auth_me, não localStorage
  Dado que o localStorage contém um nome desatualizado
  Quando o Dashboard monta e chama GET /auth/me
  Então o WelcomeWidget DEVE exibir o nome retornado por auth_me
  E NUNCA o valor do localStorage ou do JWT decodificado
```
