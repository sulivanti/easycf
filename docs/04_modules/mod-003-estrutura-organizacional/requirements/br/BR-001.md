# BR-001 — Regras de Negócio da Estrutura Organizacional

---

## BR-001 — Hierarquia Organizacional de 5 Níveis (N1–N5)

- **Regra:** O sistema DEVE manter uma hierarquia organizacional formal de exatamente 5 níveis: N1 (Grupo Corporativo), N2 (Unidade), N3 (Macroárea), N4 (Subunidade Organizacional), N5 (Entidade Jurídica/Estabelecimento). N1–N4 são nós na tabela `org_units`. N5 é o tenant existente em MOD-000-F07, vinculado via `org_unit_tenant_links`.
- **Exemplo:** Grupo Alpha (N1) → Unidade SP (N2) → Macroárea Tech (N3) → Subunidade Dev (N4) → Filial SP-01 (N5/tenant)
- **Exceções:** Nenhuma. A árvore nunca deve exceder 5 níveis.
- **Impacto:** DATA, FLOW, PERMISSIONS

## BR-002 — Nível Derivado Automaticamente do Pai

- **Regra:** O campo `nivel` DEVE ser calculado pelo backend como `parent.nivel + 1`. O valor NUNCA deve ser aceito do cliente. Nós N1 (raiz) têm `parent_id=null` e `nivel=1`.
- **Exemplo:** Criar nó com `parent_id` apontando para N2 → backend calcula `nivel=3`.
- **Exceções:** Nenhuma.
- **Impacto:** DATA, STATE

## BR-003 — Código Imutável Após Criação

- **Regra:** O campo `codigo` é identificador de negócio estável e DEVE ser imutável após a criação. Tentativas de alteração via PATCH devem retornar 422.
- **Exemplo:** PATCH /org-units/:id com { codigo: "NOVO" } → 422 "O campo 'codigo' é imutável após criação."
- **Exceções:** Nenhuma. Regra inegociável.
- **Impacto:** DATA, COMPLIANCE

## BR-004 — Prevenção de Loop na Árvore

- **Regra:** Antes de criar ou mover um nó, o sistema DEVE verificar via CTE recursivo que o `parent_id` proposto não é descendente do nó atual, prevenindo loops circulares na hierarquia.
- **Exemplo:** Nó A é pai de B, que é pai de C → tentar mover A para ser filho de C → 422 "Operação criaria loop na hierarquia."
- **Exceções:** Nenhuma.
- **Impacto:** DATA, STATE

## BR-005 — Soft Delete Condicional (Sem Filhos Ativos)

- **Regra:** O soft delete (DELETE /org-units/:id) só é permitido se o nó NÃO possuir filhos com `status=ACTIVE` e `deleted_at IS NULL`. Caso contrário, retorna 422 com a lista de filhos ativos.
- **Exemplo:** Nó N2 com 3 filhos N3 ativos → DELETE retorna 422 com `extensions.active_children[]`.
- **Exceções:** Nenhuma.
- **Impacto:** DATA, STATE, FLOW

## BR-006 — Vinculação N5 Exclusivamente em Nós N4

- **Regra:** A vinculação de tenant (N5) via `org_unit_tenant_links` só é aceita em nós com `nivel=4`. Tentativas em outros níveis retornam 422.
- **Exemplo:** POST /org-units/:id/tenants em nó N3 → 422 "Vinculação de tenant só é permitida em nós de nível N4."
- **Exceções:** Nenhuma.
- **Impacto:** DATA, STATE

## BR-007 — N5 Não Duplica Tenant (MOD-000-F07)

- **Regra:** O MOD-003 NUNCA cria tabela paralela para N5. O N5 é o tenant existente gerenciado por MOD-000-F07. A vinculação é feita exclusivamente via `org_unit_tenant_links`.
- **Exemplo:** Um tenant com `id=uuid` é vinculado ao N4 → não é criado como `org_unit`.
- **Exceções:** Nenhuma. Decisão arquitetural central do módulo.
- **Impacto:** DATA, COMPLIANCE

## BR-008 — Unicidade Global de Código

- **Regra:** O campo `codigo` em `org_units` DEVE ser único globalmente (constraint `UNIQUE(codigo)`). A tabela `org_units` é cross-tenant por design — a hierarquia organizacional abrange múltiplos tenants. Tentativa de criar nó com código duplicado retorna 409.
- **Exemplo:** Nó "GC-001" já existe → qualquer tentativa de criar outro "GC-001" → 409 Conflict.
- **Exceções:** Nenhuma.
- **Impacto:** DATA, COMPLIANCE

## BR-009 — Restore Condicional (Pai Deve Estar Ativo)

- **Regra:** O restore (PATCH /org-units/:id/restore) de um nó com soft delete só é permitido se o `parent_id` referencia um nó com `deleted_at IS NULL`. Caso contrário, retorna 422 "Não é possível restaurar: o nó pai está inativo."
- **Exemplo:** Nó N3 foi soft-deleted. Seu pai N2 também foi soft-deleted → restore do N3 retorna 422.
- **Exceções:** Nós N1 (raiz, sem parent_id) podem ser restaurados sem restrição de pai.
- **Impacto:** DATA, STATE, FLOW

## BR-010 — Parent_id Imutável Após Criação (Sem Movimentação de Nós)

- **Regra:** O campo `parent_id` é imutável após a criação do nó. Tentativas de alteração via PATCH devem retornar 422. A movimentação de nós na hierarquia está fora do escopo (roadmap futuro — MOD-003 v2).
- **Exemplo:** PATCH /org-units/:id com { parent_id: "novo-pai-uuid" } → 422 "O campo 'parent_id' é imutável após criação. Movimentação de nós não é suportada nesta versão."
- **Exceções:** Nenhuma. Movimentação de nós requer ADR específica e está no roadmap futuro.
- **Impacto:** DATA, STATE

## BR-011 — Nível Máximo N4 para Org Units (N5 Exclusivo via Vinculação)

- **Regra:** O sistema DEVE rejeitar a criação de um nó filho direto de N4 na tabela `org_units`. O nível máximo de org_units é N4 (nivel=4). Nível 5 é representado exclusivamente pelo tenant vinculado via `org_unit_tenant_links`. Tentativa de criar filho de N4 retorna 422.
- **Exemplo:** POST /org-units com parent_id apontando para nó N4 → 422 "Nível máximo (N4) atingido. Use vinculação de tenant para N5."
- **Exceções:** Nenhuma.
- **Impacto:** DATA, STATE, FLOW

## BR-012 — Idempotência Obrigatória em Operações de Escrita com Efeito Colateral

- **Regra:** Os endpoints `POST /org-units` (create) e `POST /org-units/:id/tenants` (link_tenant) DEVEM aceitar o header `Idempotency-Key`. Requisições com a mesma chave dentro do TTL (60s) retornam o resultado original sem criar duplicatas. Conforme DOC-ESC-001 §3.6 (Nível 1 MUST).
- **Exemplo:** POST /org-units com Idempotency-Key "key-abc" cria o nó. Mesmo POST com "key-abc" → retorna 201 com o mesmo recurso, sem duplicação.
- **Exceções:** Endpoints de leitura (GET) e PATCH (update/restore com idempotência natural) não requerem Idempotency-Key.
- **Impacto:** DATA, STATE
