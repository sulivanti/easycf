## Princípios (MUST)

- **Não use "permissão no evento" como fonte de verdade.**
  - **Emit** é controlado pela permissão do **comando** que gera o evento.
  - **View** é controlado pela permissão de leitura da **entity originária** (ACL) + `tenant_id`.
- `visibility_level`/`sensitivity_level` **não substituem** ACL/RBAC/ABAC: servem apenas como **guard-rail/otimizador**.
- **Autorização de Linha (MUST):** toda leitura em `domain_events` e `notifications` MUST filtrar por `tenant_id` e respeitar a ACL do registro originário.

## Glossário

- **Emit:** quem pode disparar o evento (derivado do comando/ação que o origina).
- **View:** quem pode ler/visualizar eventos (timeline/auditoria).
- **Notify:** quem recebe notificações (inbox/real-time), resolvido por regra.
- **Sensitivity:** 0 = administrativo, 1 = operacional, 2 = alto (escalada de privilégio).
- **Retenção:** tempo mínimo de retenção do evento antes de elegibilidade para purge.

## Regras de Filtragem (MUST)

1. **`tenant_id` obrigatório** em todas as consultas a `domain_events` — sem exceção.
2. **Validação ACL** antes de retornar eventos na timeline/auditoria:
   - Eventos EVT-001 a EVT-007: requer `mcp:agent:read`
   - Eventos EVT-008 a EVT-009: requer `mcp:log:read`
   - Evento EVT-010: requer `mcp:log:read` + `system:audit:sensitive`
3. **Eventos com sensitivity_level=2** (EVT-010) requerem scope adicional `system:audit:sensitive` para visualização completa. Sem este scope, campos `maskable_fields` são substituídos por `"***"`.
4. **Eventos de execução** (EVT-008/009/010) NÃO incluem payloads completos no corpo do evento — consultar via `GET /admin/mcp-executions/:id` para detalhes.

## Regras de Mascaramento por Evento

| event_type | maskable_fields | Regra de mascaramento |
|---|---|---|
| `mcp.agent.created` | `owner_user_id` | Mascarar em exports LGPD; visível para admin com `mcp:agent:read` |
| `mcp.agent.updated` | — | Nenhum campo sensível |
| `mcp.agent.scopes_updated` | — | Escopos adicionados/removidos são dados operacionais |
| `mcp.agent.revoked` | — | `revoked_by` é UUID operacional |
| `mcp.agent.key_rotated` | — | NUNCA inclui key nova ou antiga (payload_policy DATA-003) |
| `mcp.agent.action_granted` | — | Nenhum campo sensível |
| `mcp.agent.action_revoked` | — | Nenhum campo sensível |
| `mcp.execution.completed` | `request_payload`, `result_payload` | NÃO incluídos no evento (volume + potencial PII). Acessíveis via GET detalhado com mascaramento de campos `password\|api_key\|secret\|token\|cpf\|email\|phone` |
| `mcp.execution.blocked` | `request_payload` | Mesma regra de execution.completed |
| `mcp.privilege_escalation_attempt` | `request_payload`, `origin_ip` | Visíveis apenas com `system:audit:sensitive`. Sem este scope, mascarados como `"***"` |

## Retenção por Categoria

| Categoria | Eventos | Retenção | Justificativa |
|---|---|---|---|
| **Gestão de agentes** | EVT-001 a EVT-005 | Padrão Foundation (exceto EVT-004: 2 anos) | Operações administrativas. Revogação retida mais tempo por ser irreversível. |
| **Vínculos** | EVT-006, EVT-007 | Padrão Foundation | Vínculo/desvínculo de ações — dados operacionais. |
| **Execuções normais** | EVT-008 | 1 ano | Volume alto de execuções. Após 1 ano, manter apenas aggregates (contadores). |
| **Execuções bloqueadas** | EVT-009 | 1 ano | Falhas de validação. Retenção alinhada com execuções normais. |
| **Escalação de privilégio** | EVT-010 | **2 anos** | Incidentes de segurança. Retenção estendida para compliance e investigação. |

## Gherkin BDD — Autorização de Eventos

```gherkin
Funcionalidade: Autorização de Visualização de Eventos MCP

Cenário: Admin com mcp:agent:read visualiza eventos de gestão de agentes
  Dado que o usuário tem scope "mcp:agent:read" no tenant "TENANT-A"
  Quando GET /timeline?entity_type=mcp.agent
  Então retorna eventos EVT-001 a EVT-007 filtrados por tenant_id = "TENANT-A"
  E campo owner_user_id mascarado se sensitivity_level >= 1 e sem scope audit

Cenário: Admin com mcp:log:read visualiza execuções
  Dado que o usuário tem scope "mcp:log:read" no tenant "TENANT-A"
  Quando GET /timeline?entity_type=mcp.execution
  Então retorna eventos EVT-008 e EVT-009 filtrados por tenant_id
  E payloads request_payload e result_payload ausentes (acessíveis via GET detalhado)

Cenário: Evento de privilege escalation requer scope adicional
  Dado que o usuário tem scope "mcp:log:read" mas NÃO tem "system:audit:sensitive"
  Quando GET /timeline?entity_type=mcp.execution com eventos EVT-010
  Então evento EVT-010 é retornado mas campos maskable_fields são "***"
  E detalhes completos indisponíveis

Cenário: Evento de privilege escalation com scope de auditoria sensível
  Dado que o usuário tem scopes "mcp:log:read" + "system:audit:sensitive"
  Quando GET /timeline?entity_type=mcp.execution com eventos EVT-010
  Então evento EVT-010 retornado com detalhes completos
  E campos request_payload e origin_ip visíveis para investigação

Cenário: Cross-tenant bloqueado em timeline
  Dado que o usuário pertence ao tenant "TENANT-A"
  Quando GET /timeline com filtro tentando tenant_id = "TENANT-B"
  Então 403 ou resultado vazio (sem vazamento cross-tenant)
```
