# INT-000 — Integrações e Contratos do Foundation

## INT-001 — Google OAuth2 (SSO)

- **Sistema externo:** Google Identity Platform (`accounts.google.com`)
- **Modo:** API (Authorization Code Flow — OAuth2)
- **Auth:** OAuth2 Client ID + Client Secret (server-to-server)
- **Contrato:**
  - **Redirect:** `GET /auth/google` → redirect para `accounts.google.com/o/oauth2/v2/auth`
  - **Callback:** `GET /auth/google/callback?code=...` → troca `code` por `access_token`
  - **Userinfo:** `GET googleapis.com/oauth2/v2/userinfo` → `{email, name, picture}`
  - **Response:** Redirect para `{FRONTEND_URL}/sso-success?token=JWT&provider=google&is_new_user=bool`
- **Timeout:** 5000ms
- **Retry:** 3 tentativas (apenas token exchange, não redirect)
- **Backoff:** Exponencial, 500ms inicial
- **DLQ:** [ ] Não (fluxo síncrono)
- **Idempotência:** Se usuário já existe (por email), reutiliza registro
- **Fallback:** Redirect para `/login?error=sso_failed&provider=google`
- **Observabilidade:** correlation_id propagado, log de provider + is_new_user

- **DLQ:** [ ] Não
- **Idempotência:** Se usuário já existe (por email), reutiliza registro

- **Observabilidade:** correlation_id propagado

## INT-003 — Sign in with Apple (OIDC/JWKS)

- **Sistema externo:** Apple Identity Service (`appleid.apple.com`)
- **Modo:** API (Authorization Code Flow — OIDC)
- **Auth:** Service ID + Private Key (JWT signed)
- **Contrato:**
  - **Redirect:** `GET /auth/apple` → redirect para `appleid.apple.com/auth/authorize`
  - **Callback:** `POST /auth/apple/callback` (form_post) → `{code, id_token}`
  - **JWKS:** Validação via `appleid.apple.com/auth/keys` (cache JWKS com TTL 24h)
  - **Userinfo:** Extraído do `id_token` (sub, email, name — nome só no primeiro login)
- **Timeout:** 5000ms
- **Retry:** 3 tentativas (token exchange)
- **Backoff:** Exponencial, 500ms inicial
- **DLQ:** [ ] Não
- **Idempotência:** Se usuário já existe (por email ou Apple sub), reutiliza registro
- **Fallback:** Redirect para `/login?error=sso_failed&provider=apple`
- **Observabilidade:** correlation_id propagado

## INT-004 — Redis (Cache RBAC)

- **Sistema externo:** Redis (interno, infraestrutura)
- **Modo:** API (TCP — biblioteca ioredis/redis)
- **Auth:** Connection string com senha (env var `REDIS_URL`)
- **Contrato:**
  - **SET:** `auth:scopes:role:{roleId}` → JSON array de escopos
  - **GET:** `auth:scopes:role:{roleId}` → leitura com fallback para DB
  - **DEL:** `auth:scopes:role:{roleId}` → invalidação após mutação
- **Timeout:** 1000ms
- **Retry:** 0 (fail-fast com fallback)
- **Backoff:** N/A
- **DLQ:** [ ] Não
- **Idempotência:** SET/DEL são naturalmente idempotentes
- **Fallback:** Se Redis indisponível, busca direto do banco (BR-011)
- **Observabilidade:** correlation_id em logs, métricas de hit/miss

## INT-005 — E-mail Service (Boas-Vindas e Notificações)

- **Sistema externo:** SMTP / Provider de e-mail (ex: SendGrid, SES)
- **Modo:** API (HTTP ou SMTP)
- **Auth:** API Key (env var `MAIL_API_KEY`)
- **Contrato:**
  - **Welcome:** `sendWelcomeEmail({email, name})` — disparado após user.created
  - **Password Reset:** `sendPasswordResetEmail({email, token, ttl})` — disparado após forgot_password
- **Timeout:** 5000ms
- **Retry:** 3 tentativas
- **Backoff:** Exponencial, 1000ms inicial
- **DLQ:** [x] Sim — registrar falha para reprocessamento
- **Idempotência:** Dedupe por `correlation_id + event_type`
- **Fallback:** Fire-and-forget para welcome (não bloqueia 201). Reset email é crítico — retry obrigatório.
- **Observabilidade:** correlation_id, log de envio/falha, métricas de delivery rate

## INT-006 — Storage Provider (MinIO / S3)

- **Sistema externo:** MinIO (local) ou AWS S3 (cloud) — provider-agnóstico
- **Modo:** API (HTTP — AWS SDK v3)
- **Auth:** Access Key + Secret Key (env vars `STORAGE_ACCESS_KEY`, `STORAGE_SECRET_KEY`)
- **Contrato:**
  - **Presign Upload:** `generatePresignedUrl({bucket, key, contentType, maxBytes})` → `{url, expires_in}`
  - **Confirm:** `headObject({bucket, key})` → verifica existência e tamanho
  - **Signed Read:** `generateSignedGetUrl({bucket, key, ttl})` → URL temporária de leitura
  - **Delete:** `deleteObject({bucket, key})` → remoção do objeto
- **Timeout:** 5000ms (presign), 10000ms (head/delete)
- **Retry:** 3 tentativas
- **Backoff:** Exponencial, 500ms inicial
- **DLQ:** [ ] Não (operações síncronas)
- **Idempotência:** Presign é naturalmente idempotente; confirm valida estado
- **Fallback:** Se provider indisponível, retornar 503 com retry_after
- **Observabilidade:** correlation_id, métricas de upload size/latency

### Validação condicional

| Condição | Resultado |
|----------|-----------|
| `mode='invite'` + `password` presente | `422` — "Password must not be sent when mode is 'invite'" |
| `mode='password'` + `password` ausente | `422` — "Password is required when mode is 'password'" |
| `role_id` + role inexistente | `422` — "Role not found" |
| `role_id` + sem auth | `403` — "Role assignment requires authentication" |

## INT-005 — Auth Refresh API — Schema `RefreshResponse` (FR-000-C04)

- **Sistema:** MOD-000 Foundation (interno)
- **Modo:** REST API (OpenAPI v3.1)
- **Base path:** `/api/v1/auth`

O endpoint `POST /auth/refresh` usa schema **`RefreshResponse`** separado do `LoginResponse`. O refresh retorna apenas tokens (sem campo `user`), pois `RefreshTokenUseCase` não busca dados do usuário.

```yaml
RefreshResponse:
  type: object
  required: [access_token, refresh_token, token_type, expires_in]
  properties:
    access_token: { type: string }
    refresh_token: { type: string }
    token_type: { type: string, enum: [Bearer] }
    expires_in: { type: integer }
```

> **Nota:** O `LoginResponse` (usado em `POST /auth/login`) inclui campo `user`. O `RefreshResponse` **não inclui** `user` — o frontend já possui esses dados via login ou `GET /auth/me`.

## INT-007 — Cache RBAC — Convenções Redis (DOC-PADRAO-002 §3.4)

- **Sistema:** MOD-000 Foundation (interno — infraestrutura Redis)
- **Modo:** Cache (ioredis singleton)
- **Referência normativa:** DOC-PADRAO-002 §3.4.1–3.4.6

### Requisitos

1. **Conexão ioredis (§3.4.1):** RBAC cache MUST usar instância singleton com configuração padrão (retryStrategy, lazyConnect, connectTimeout/commandTimeout 5s).
2. **Key naming (§3.4.3):** Chaves RBAC devem seguir padrão `mod-000:rbac:user:{uuid}`. MUST usar `:` como separador, MUST prefixar com `mod-000:`.
3. **TTL (§3.4.4):** Cache RBAC MUST usar TTL de 5 min (300s). MUST NOT usar `SET` sem `EX`/`PX`.
4. **Database (§3.4.5):** Cache RBAC MUST usar db0 (`allkeys-lru`). Conexão ioredis MUST especificar `db: 0`.
5. **Health check (§3.4.6):** MUST expor Redis health check no endpoint `/health` com latência do PING.
