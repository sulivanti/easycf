# ADR-002 — Estratégia Fail-Safe na Integração MOD-005 → MOD-006 (Deleção Protegida)

- **status:** accepted
- **timebox_days:** 90

---

## Contexto

A regra BR-005 exige que estágios com instâncias ativas no MOD-006 não possam ser deletados. O MOD-005 precisa consultar o MOD-006 para verificar a contagem de instâncias antes de permitir soft delete. Surge a questão: o que fazer quando o MOD-006 está indisponível?

## Decisão

**Fail-safe: bloquear deleção** quando MOD-006 está indisponível. Retornar HTTP 503: "Não foi possível verificar instâncias ativas. Tente novamente."

## Alternativas Consideradas

| Alternativa | Prós | Contras |
|---|---|---|
| **A) Fail-safe — bloquear deleção (escolhida)** | Máxima segurança; nunca deleta estágio que pode ter instâncias ativas | Operação de administração fica indisponível se MOD-006 cair |
| **B) Fail-open — permitir deleção** | Admin nunca fica bloqueado | Risco de corromper instâncias ativas no MOD-006; viola BR-005 |
| **C) Cache local de contagem** | Reduz dependência; admin pode deletar com dados recentes | Cache pode estar desatualizado; janela de inconsistência |
| **D) Event-driven (MOD-006 publica contagem)** | Desacoplamento total; sem chamada síncrona | Complexidade; eventual consistency; contagem pode não refletir estado real |

## Consequências

- MOD-005 depende da disponibilidade do MOD-006 para operações de DELETE em estágios
- Endpoint interno: `GET /internal/instances/count-active?stage_id={uuid}` (INT-005 §4.1)
- Timeout: 3000ms, 1 retry, sem backoff
- Monitorar taxa de 503 para identificar problemas de disponibilidade do MOD-006
- Se a taxa de indisponibilidade for alta, reavaliar para Opção C (cache) como evolução futura
