## 1. Autenticação

### 1.1 Endpoints Admin (humanos)

- **Modelo:** JWT Bearer via MOD-000 (Foundation). Não há fluxo de autenticação próprio.
- **Todos os endpoints** `/api/v1/admin/mcp-*` exigem JWT válido no header `Authorization: Bearer {token}`.
- **Enforcement:** `preHandler` em cada rota Fastify. Scope verificado contra JWT claims.

### 1.2 Endpoint Gateway MCP (agentes)

- **Modelo:** API key no header `X-MCP-Agent-Key`. Endpoint `POST /api/v1/mcp/execute` **NÃO aceita JWT humano**.
- **API key:** 256 bits (32 bytes) codificados em base64url (44 caracteres). Gerada via `crypto.randomBytes(32)`.
- **Armazenamento:** Hash bcrypt com `rounds >= 12`. Apenas o hash é persistido na coluna `api_key_hash` de `mcp_agents`.
- **Exibição única:** Key retornada UMA VEZ no response body de `POST /admin/mcp-agents` (criação) e `POST /admin/mcp-agents/:id/rotate-key` (rotação). Nunca exposta em GET, logs, domain events ou monitor UX.
- **Validação:** `bcrypt.compare(received_key, stored_hash)` no passo 1 do gateway (McpGateway). Timeout de compare <= 100ms P99.
- **Falha de authn:** 401 `{ type: "mcp-authn-failed", title: "MCP Authentication Failed", detail: "Invalid or expired API key", status: 401 }`. Sem revelar se o agente existe.

```gherkin
Funcionalidade: Autenticação de Agentes MCP via API key

Cenário: Agente autentica com sucesso via API key
  Dado que o agente "AGENT-01" está ACTIVE e possui API key válida
  Quando POST /mcp/execute com header X-MCP-Agent-Key contendo a key válida
  Então o gateway aceita e prossegue para o passo 2 (verificação de status)

Cenário: Agente envia API key inválida
  Quando POST /mcp/execute com header X-MCP-Agent-Key contendo key incorreta
  Então 401 "Invalid or expired API key"
  E nenhum detalhe sobre existência do agente é revelado

Cenário: Endpoint /mcp/execute rejeita JWT
  Quando POST /mcp/execute com header Authorization: Bearer <jwt>
  Então 401 "Este endpoint requer autenticação via X-MCP-Agent-Key"
```

## 2. Autorização (RBAC)

- **Modelo:** RBAC via `@RequireScope` (DOC-FND-000 §2). Scopes definidos no Amendment MOD-000-F12.
- **Enforcement:** `preHandler` em cada rota Fastify. Scope verificado contra JWT claims.

### 2.1 Catálogo de Scopes do MOD-010

| Scope | Descrição | Endpoints protegidos |
|---|---|---|
| `mcp:agent:read` | Visualizar catálogo de agentes | `GET /admin/mcp-agents`, `GET /admin/mcp-agents/:id` |
| `mcp:agent:write` | Criar/editar agentes, gerenciar vínculos de ação | `POST /admin/mcp-agents`, `PATCH /admin/mcp-agents/:id`, `POST /admin/mcp-agents/:id/rotate-key`, `POST /admin/mcp-agents/:id/actions`, `DELETE /admin/mcp-agents/:id/actions/:actionId` |
| `mcp:agent:revoke` | Revogar agente (ação crítica separada) | `POST /admin/mcp-agents/:id/revoke` |
| `mcp:action:read` | Visualizar catálogo de ações | `GET /admin/mcp-actions` |
| `mcp:action:write` | Criar/editar ações | `POST /admin/mcp-actions`, `PATCH /admin/mcp-actions/:id` |
| `mcp:log:read` | Visualizar execuções e logs MCP | `GET /admin/mcp-executions`, `GET /admin/mcp-executions/:id` |

### 2.2 Separação de Privilégios

- **Leitura vs. Escrita:** `mcp:agent:read` e `mcp:agent:write` são scopes separados — operadores de suporte podem visualizar agentes sem poder editá-los.
- **Revogação isolada:** `mcp:agent:revoke` é separado de `mcp:agent:write` — quem cria não necessariamente pode revogar (BR-006, SEC-004 segregação).
- **Ações separadas de agentes:** `mcp:action:read` e `mcp:action:write` — o catálogo de ações pode ser gerido por equipe diferente da que gere agentes.
- **Monitor read-only:** `mcp:log:read` — acesso somente-leitura ao monitor de execuções. Sem escrita em nenhum log.

### 2.3 Agentes MCP — Scopes Próprios (Não RBAC)

- **Agentes MCP NÃO usam RBAC.** Autenticam via API key e possuem `allowed_scopes` registrados no cadastro.
- **Agentes NÃO herdam scopes do owner** (BR-001 — regra-mãe).
- **Verificação em runtime:** Passo 5 do gateway verifica `required_scopes ⊆ allowed_scopes`. Se falhar → 403 + evento `mcp.execution.blocked`.

## 3. Blocklist de Escopos — Phase 1 (Permanente)

### 3.1 Escopos Bloqueados

Escopos permanentemente bloqueados para agentes MCP (ScopeBlocklistValidator):

| Padrão | Descrição | Match |
|---|---|---|
| `*:delete` | Exclusões | qualquer scope terminado em `:delete` |
| `*:approve` | Aprovações | qualquer scope terminado em `:approve` |
| `approval:decide` | Decisão de aprovação | match exato |
| `approval:override` | Override de aprovação | match exato |
| `*:sign` | Assinaturas | qualquer scope terminado em `:sign` |
| `*:execute` | Execuções de gates | qualquer scope terminado em `:execute` |

### 3.2 Pontos de Enforcement

| Ponto | Momento | Ação em violação |
|---|---|---|
| **WRITE-time** (POST/PATCH agente) | Criação ou edição de `allowed_scopes` | 422 com lista de escopos bloqueados |
| **RUNTIME** (gateway passo 6) | Dupla verificação antes do despacho | 403 + `mcp.execution.blocked` + `mcp.privilege_escalation_attempt` (EVT-010) |

### 3.3 Phase 2 — Liberação per-agent

- Escopos `*:create` podem ser liberados por agente individual via flag `phase2_create_enabled`.
- **Condições:** Aprovação explícita do owner, configuração per-agent (nunca global), registro em auditoria (EVT-003 scopes_updated).

```gherkin
Funcionalidade: Blocklist de Escopos Phase 1

Cenário: Rejeitar criação de agente com escopo bloqueado
  Dado que o admin tem scope "mcp:agent:write"
  Quando POST /admin/mcp-agents com allowed_scopes contendo "org:unit:delete"
  Então 422: "Escopo bloqueado para agentes MCP: org:unit:delete (corresponde ao padrão *:delete)"

Cenário: Dupla verificação no gateway detecta escopo bloqueado em runtime
  Dado que um agente possui "process:case:read" em allowed_scopes
  E uma ação requer "process:case:gate_resolve" (não corresponde a blocklist)
  E o agente NÃO possui "process:case:gate_resolve" em allowed_scopes
  Quando o agente tenta executar a ação
  Então 403: "Escopo insuficiente para esta ação"
  E evento "mcp.execution.blocked" é emitido

Cenário: Dupla verificação detecta tentativa de privilege escalation
  Dado que um agente de alguma forma possui escopo bloqueado (data corrompida)
  Quando o gateway executa passo 6 (dupla verificação blocklist)
  Então 403: "Privilege escalation detectada — escopo bloqueado"
  E evento "mcp.privilege_escalation_attempt" é emitido com sensitivity_level=2
  E execução registrada como BLOCKED em mcp_executions
```

## 4. Detecção de Privilege Escalation

- **sensitivity_level:** 2 (classificação mais alta do módulo).
- **Gatilhos de detecção:**
  1. Agente tenta executar ação cujo `required_scopes` inclui escopo da blocklist Phase 1.
  2. Payload de execução contém tentativa de injeção de escopos (campo ignorado pelo gateway).
  3. Dados corrompidos — `allowed_scopes` contém escopo bloqueado (dupla verificação no passo 6).
- **Ação imediata:**
  - Execução bloqueada (status=BLOCKED).
  - Evento `mcp.privilege_escalation_attempt` (EVT-010) emitido com detalhes completos.
  - Notificação para admin + owner + security_admin via in-app e e-mail.
  - Badge vermelho no monitor UX-MCP-002.
- **Registro:** `mcp_executions` com `status=BLOCKED`, `blocked_reason`, `blocked_at_step=6`.

```gherkin
Funcionalidade: Detecção de Privilege Escalation

Cenário: Tentativa de escalada via escopo bloqueado
  Dado que o agente "AGENT-01" tenta executar ação que requer "approval:decide"
  Quando POST /mcp/execute com action_code referenciando ação de aprovação
  Então 403 "Privilege escalation detectada"
  E evento "mcp.privilege_escalation_attempt" emitido com sensitivity_level=2
  E notificação enviada para admin, owner do agente e security_admin
  E badge vermelho exibido no UX-MCP-002

Cenário: Tentativa de injeção de escopos no payload
  Dado que o agente envia payload com campo "override_scopes": ["approval:decide"]
  Quando o gateway processa o payload
  Então campo "override_scopes" é ignorado
  E evento "mcp.privilege_escalation_attempt" emitido
  E execução registrada com status=BLOCKED
```

## 5. Segregação de Funções

| Função | Scope necessário | Justificativa |
|---|---|---|
| Criar/editar agente | `mcp:agent:write` | Operação de gestão padrão |
| Revogar agente | `mcp:agent:revoke` | Ação crítica irreversível — isolada para controle de alçada |
| Gerenciar catálogo de ações | `mcp:action:write` | Equipe de definição de catálogo pode ser distinta |
| Monitorar execuções | `mcp:log:read` | Acesso somente-leitura para operadores e auditores |

- **Owner != Admin:** `owner_user_id` registrado em `mcp_agents` identifica o responsável pelo agente. O owner pode não ter `mcp:agent:revoke`.
- **Separação catálogo/agentes:** Ações (`mcp:action:write`) e agentes (`mcp:agent:write`) são geridos por scopes diferentes.
- **Ação de revogação auditada:** `POST /admin/mcp-agents/:id/revoke` requer motivo obrigatório (`revocation_reason` — BR-015) e emite EVT-004.

## 7. Payload Sanitization

### 7.1 Campos Proibidos em Respostas

| Campo | Onde é omitido | Regra |
|---|---|---|
| `api_key_hash` | Todos os endpoints GET/PATCH | NUNCA retornado em nenhum response body (BR-004) |
| `api_key` (plaintext) | Retornado APENAS em POST create e POST rotate-key | Exibição única — campo ausente em todas as demais respostas |

### 7.2 Mascaramento em Logs e Monitor

| Padrão de campo | Máscara | Aplicação |
|---|---|---|
| `password`, `api_key`, `secret`, `token`, `credential` | `"***"` | Payloads de `mcp_executions.request_payload` e `result_payload` exibidos no monitor UX-MCP-002 |
| `api_key_hash` | campo removido | Nunca incluído em domain events, logs ou respostas |

### 7.3 Sanitização de Domain Events

- Payloads de domain events seguem `payload_policy` definida em DATA-003.
- EVT-005 (`mcp.agent.key_rotated`): NUNCA inclui key nova ou antiga.
- EVT-008 (`mcp.execution.completed`): NÃO inclui `request_payload` ou `result_payload` (volume). Consultar via GET detalhado.
- EVT-010 (`mcp.privilege_escalation_attempt`): Inclui detalhes completos para investigação mas SEM credenciais.

## 8. Isolamento Multi-tenant

- **MUST:** `tenant_id` obrigatório em 4 tabelas (`mcp_agents`, `mcp_actions`, `mcp_executions`, `mcp_agent_action_links`) com FK para `tenants.id`. A tabela `mcp_action_types` é catálogo global (seed data) sem tenant_id.
- **Row Level Security:** Todas as queries MUST filtrar por `tenant_id` do contexto JWT (admin) ou derivado do agente autenticado (gateway).
- **Cross-tenant bloqueado:** Agente MCP de tenant A não pode executar ações de tenant B — `tenant_id` do agente é verificado contra `tenant_id` da ação no passo 4 do gateway.
- **Domain events:** `domain_events` e `notifications` MUST filtrar por `tenant_id` + ACL antes de retornar resultados (DOC-FND-000 §3).

```gherkin
Funcionalidade: Isolamento Multi-tenant

Cenário: Agente só acessa dados do próprio tenant
  Dado que o agente "AGENT-01" pertence ao tenant "TENANT-A"
  Quando POST /mcp/execute com ação pertencente a "TENANT-B"
  Então 404 "Ação não encontrada" (não revela existência em outro tenant)

Cenário: Admin só visualiza agentes do próprio tenant
  Dado que o admin pertence ao tenant "TENANT-A"
  Quando GET /admin/mcp-agents
  Então retorna apenas agentes com tenant_id = "TENANT-A"
```

## 10. LGPD / Proteção de Dados

### 10.1 Impacto LGPD

- **Impacto:** Moderado. MOD-010 armazena referências a dados pessoais (`owner_user_id`, `created_by`) mas não dados pessoais diretamente.
- **PII indireta:** `owner_user_id` e `created_by` são FKs para `users.id` (MOD-000). O dado pessoal (nome, email) reside no Foundation.
- **PII em payloads:** `request_payload` e `result_payload` de `mcp_executions` PODEM conter PII dependendo da ação executada.

### 10.2 Direitos do Titular

| Direito LGPD | Tratamento MOD-010 |
|---|---|
| **Acesso (Art. 18, II)** | Dados do titular acessíveis via endpoint Foundation (MOD-000). MOD-010 retorna execuções vinculadas ao owner_user_id. |
| **Correção (Art. 18, III)** | Dados pessoais corrigidos no MOD-000. FKs em MOD-010 apontam para o registro atualizado automaticamente. |
| **Eliminação (Art. 18, VI)** | `owner_user_id` permanece como UUID órfão em `mcp_agents` (necessário para auditoria — LGPD Art. 16, II). Agentes do titular DEVEM ser revogados se o titular solicitar eliminação. |
| **Portabilidade (Art. 18, V)** | MOD-010 contribui com dados de agentes e execuções vinculados ao titular no pacote de portabilidade. |
| **Oposição (Art. 18, §2)** | Titular pode solicitar revogação de agentes MCP vinculados à sua identidade. |

### 10.3 Mascaramento de PII

| Campo | Entidade | Tratamento |
|---|---|---|
| `owner_user_id` | `mcp_agents`, domain events | Maskable — substituído por hash anônimo em exports LGPD |
| `created_by` | `mcp_agents`, `mcp_actions`, `mcp_action_types` | Maskable — UUID referência, sem PII direta |
| `request_payload` | `mcp_executions` | Sanitizado — campos matching padrão PII mascarados como `"***"` |
| `result_payload` | `mcp_executions` | Sanitizado — mesma regra de request_payload |
| `origin_ip` | domain events (EVT-010) | Maskable — incluído apenas em eventos sensitivity_level=2 para investigação |

## 12. Rate Limits

### 12.1 Limites por Operação

| Endpoint | Rate Limit | Janela | Justificativa |
|---|---|---|---|
| `POST /mcp/execute` | 60 req/min por agente | Sliding window (hard limit) | Proteção contra automação descontrolada; alinhado com NFR-005 |
| `POST /mcp/execute` | 1000 req/min por tenant | Sliding window | Fair-use entre tenants |
| `POST /admin/mcp-agents` | 10 req/min por usuário | Sliding window | Proteção contra criação em massa (cada criação gera bcrypt hash) |
| `POST .../rotate-key` | 5 req/min por agente | Sliding window | Rotação excessiva indica comprometimento ou abuso |
| `POST .../revoke` | 10 req/min por usuário | Sliding window | Ação irreversível — rate limit conservador |
| `GET /admin/mcp-executions` | 60 req/min por usuário | Sliding window | Proteção de queries em tabela append-only potencialmente grande |

### 12.2 Resposta em Rate Limit

- HTTP 429 `{ type: "rate-limit-exceeded", title: "Too Many Requests", detail: "Rate limit exceeded for this operation", status: 429, retry_after: <seconds> }`.
- Header `Retry-After` com segundos até próxima janela.

### 12.3 Brute Force Protection no Gateway

- **Após 5 falhas consecutivas de authn** (API key inválida) para o mesmo agent_codigo: bloqueio temporário de 15 minutos.
- **Evento:** `mcp.execution.blocked` emitido a cada tentativa falhada. Após bloqueio, notificação a admin + owner.
- **Implementação:** Contador em Redis com TTL de 15 minutos. Reset em authn bem-sucedida.

```gherkin
Funcionalidade: Rate Limiting e Brute Force Protection

Cenário: Rate limit no gateway MCP
  Dado que o agente "AGENT-01" já executou 100 requests neste minuto
  Quando POST /mcp/execute
  Então 429 "Rate limit exceeded" com header Retry-After

Cenário: Bloqueio por brute force no gateway
  Dado que houve 5 tentativas consecutivas com API key inválida para "AGENT-01"
  Quando POST /mcp/execute com API key para "AGENT-01"
  Então 429 "Agent temporarily blocked due to repeated authentication failures"
  E notificação enviada para admin e owner do agente
```

## 13. Proteção de API Key

### 13.1 Ciclo de Vida da Key

| Momento | Ação | Key visível? |
|---|---|---|
| Criação (`POST /admin/mcp-agents`) | Gera key + hash bcrypt | Sim — UMA VEZ no response body |
| Rotação (`POST .../rotate-key`) | Gera nova key + hash, invalida anterior | Sim — UMA VEZ no response body |
| Qualquer GET | Retorna metadados do agente | Não — campo `api_key` ausente |
| Monitor UX-MCP-002 | Exibe execuções | Não — nenhuma key visível |
| Domain events | Payloads de eventos | Não — NUNCA incluída |
| Logs de aplicação | Structured logs | Não — sanitizada antes de log |

### 13.2 Armazenamento Seguro

- Hash bcrypt com `rounds >= 12` (~60-80ms de compare em hardware moderno).
- **NEVER** armazenar key em plaintext em banco, cache (Redis), logs ou filas.
- **NEVER** incluir `api_key_hash` em responses — é dado interno do banco.
