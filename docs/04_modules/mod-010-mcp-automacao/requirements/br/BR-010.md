## BR-001 — Agente Não Herda Escopos de Aprovação

- **Regra:** O fato de o usuário poder aprovar não significa que seu agente associado também possa aprovar. Agente MCP tem escopos próprios — não herda escopos do usuário vinculado. Os `allowed_scopes` do agente são definidos explicitamente no cadastro, sem qualquer derivação dos scopes do owner.
- **Exemplo:** João tem scope `approval:decide`. Seu agente MCP "AGENT-COMERCIAL-01" tenta executar ação que requer `approval:decide`. Resultado: 403 "Agentes MCP não podem exercer escopos de aprovação."
- **Exceções:** Nenhuma. Princípio inegociável — regra-mãe do módulo.
- **Impacto:** PERMISSIONS, COMPLIANCE

```gherkin
Funcionalidade: Agente Não Herda Escopos de Aprovação

Cenário: Agente MCP bloqueado ao tentar ação com escopo de aprovação
  Dado que o usuário "João" tem scope "approval:decide"
  E o agente MCP "AGENT-COMERCIAL-01" pertence a "João"
  E o agente NÃO possui "approval:decide" em allowed_scopes
  Quando o agente tenta executar ação que requer "approval:decide"
  Então 403: "Agentes MCP não podem exercer escopos de aprovação."
  E evento "mcp.privilege_escalation_attempt" é emitido com sensitivity_level=2

Cenário: Agente MCP executa ação com escopo próprio válido
  Dado que o agente "AGENT-COMERCIAL-01" possui "mcp:action:read" em allowed_scopes
  E a ação requer "mcp:action:read"
  Quando o agente executa a ação
  Então a execução prossegue conforme a política da ação
```

## BR-002 — Blocklist de Escopos Phase 1 (Permanente)

- **Regra:** Os seguintes escopos NUNCA podem ser atribuídos a agentes MCP: `*:delete`, `*:approve`, `approval:decide`, `approval:override`, `*:sign`, `*:execute`. A validação ocorre tanto na criação (POST) quanto na edição (PATCH) de agentes. O `ScopeBlocklistValidator` aplica matching por wildcard (`*:delete` bloqueia qualquer scope terminado em `:delete`).
- **Exemplo:** POST /admin/mcp-agents com `allowed_scopes: ["org:unit:delete"]` → 422 "Escopo bloqueado para agentes MCP: org:unit:delete (corresponde ao padrão *:delete)"
- **Exceções:** Nenhuma. Phase 1 é permanente e irrevogável.
- **Impacto:** PERMISSIONS, COMPLIANCE

```gherkin
Funcionalidade: Blocklist de Escopos Phase 1

Cenário: Rejeitar criação de agente com escopo bloqueado explícito
  Dado que o admin tem scope "mcp:agent:write"
  Quando POST /admin/mcp-agents com allowed_scopes contendo "approval:decide"
  Então 422: "Escopo bloqueado para agentes MCP: approval:decide"
  E evento "mcp.privilege_escalation_attempt" é emitido

Cenário: Rejeitar edição que adiciona escopo bloqueado por wildcard
  Dado que o agente "AGENT-01" existe com allowed_scopes=["process:case:read"]
  Quando PATCH /admin/mcp-agents/:id com allowed_scopes contendo "process:case:delete"
  Então 422: "Escopo bloqueado para agentes MCP: process:case:delete (corresponde ao padrão *:delete)"

Cenário: Validação bloqueia todos os padrões Phase 1
  Dado que os padrões bloqueados são: *:delete, *:approve, approval:decide, approval:override, *:sign, *:execute
  Quando qualquer allowed_scope corresponde a algum padrão
  Então a operação é rejeitada com 422
```

## BR-003 — Blocklist Phase 2 (Liberação per-agent)

- **Regra:** O escopo `*:create` pode ser liberado sob condições cumulativas: (1) MCP testado e validado em ambiente de produção, (2) aprovação explícita do owner do agente, (3) configuração per-agent (não global — cada agente recebe liberação individual), (4) registro em auditoria com data, motivo e responsável pela liberação.
- **Exemplo:** Admin deseja que agente "AGENT-VENDAS-01" possa criar pedidos. Configura `phase2_create_enabled=true` para esse agente, com registro em auditoria: "Liberação de *:create para AGENT-VENDAS-01 — aprovado por João em 2026-04-15 — motivo: automação de pedidos recorrentes."
- **Exceções:** Liberação individual por agente, nunca global. Reversível (pode ser desabilitada a qualquer momento).
- **Impacto:** PERMISSIONS, COMPLIANCE

## BR-004 — API Key Retornada Uma Única Vez

- **Regra:** A API key do agente é gerada como 256 bits base64url (44 caracteres), retornada UMA VEZ no body da resposta na criação (POST) ou rotação (POST rotate-key). Armazenada internamente como hash bcrypt com rounds >= 12. GET NUNCA retorna a key em texto plano nem o hash bcrypt. O campo `api_key` é omitido (não presente como `null` — ausente do payload).
- **Exemplo:** POST /admin/mcp-agents → 201 com `{ "id": "uuid", "api_key": "k3j...base64url...", ... }`. GET /admin/mcp-agents/:id → campo `api_key` **ausente** do JSON de resposta.
- **Exceções:** Nenhuma. Se o admin perder a key, deve usar rotate-key para gerar uma nova.
- **Impacto:** DATA, COMPLIANCE

```gherkin
Funcionalidade: API Key Retornada Uma Única Vez

Cenário: Criação retorna API key no body
  Quando POST /admin/mcp-agents com dados válidos
  Então 201 com campo "api_key" presente no body (44 caracteres base64url)

Cenário: GET nunca retorna API key
  Dado que o agente "AGENT-01" foi criado
  Quando GET /admin/mcp-agents/:id
  Então o campo "api_key" NÃO está presente no body de resposta

Cenário: Rotação retorna nova API key e invalida anterior
  Dado que o agente "AGENT-01" tem api_key="key_antiga"
  Quando POST /admin/mcp-agents/:id/rotate-key
  Então 200 com nova "api_key" no body
  E autenticação com "key_antiga" retorna 401
```

## BR-006 — Revogação Imediata e Irreversível

- **Regra:** A revogação de um agente é imediata e irreversível. Status muda para REVOKED, `revoked_at` preenchido com timestamp UTC, `revocation_reason` obrigatório no body. Qualquer tentativa de execução posterior retorna 403. Qualquer tentativa de reativar (PATCH com status=ACTIVE) retorna 422. Vínculos agente-ação permanecem no banco para auditoria, mas são inefetivos.
- **Exemplo:** POST /admin/mcp-agents/:id/revoke com `{ "reason": "Agente comprometido" }` → 200. Posterior POST /mcp/execute com API key do agente → 403 "Agente revogado."
- **Exceções:** Nenhuma. Para substituir, criar novo agente.
- **Impacto:** STATE, COMPLIANCE

```gherkin
Funcionalidade: Revogação Imediata e Irreversível

Cenário: Revogar agente com sucesso
  Dado que o agente "AGENT-01" está com status=ACTIVE
```

## BR-007 — Política DIRECT Apenas para Baixo Risco

- **Regra:** Política DIRECT só é disponível para ações cujo `action_type` tenha `can_be_direct=true`. Na criação/edição de uma ação MCP, se `execution_policy=DIRECT` e o `action_type` associado tem `can_be_direct=false`, a operação DEVE retornar 422. Tipos com `can_be_direct=true` por padrão: CONSULTAR, MONITORAR. Tipos com `can_be_direct=false`: SUBMETER, EXECUTAR. PREPARAR tem default `false` (conservador, princípio de menor privilégio — PEN-010/PENDENTE-002), ajustável por tenant.
- **Exemplo:** POST /admin/mcp-actions com `{ action_type_id: <EXECUTAR>, execution_policy: "DIRECT" }` → 422 "Tipo de ação EXECUTAR não permite política DIRECT."
- **Exceções:** Nenhuma.

## BR-008 — Política CONTROLLED Exige Decisão Humana

- **Regra:** Política CONTROLLED sempre exige que a decisão final seja de um humano com `approval:decide`. A chamada ao MOD-009 é feita com `origin_type=MCP` e `origin_agent_id` preenchido. O motor do MOD-009 cria um `controlled_movement` que aparece na fila de aprovação humana. O agente MCP recebe 202 com `movement_id` e `status=PENDING_APPROVAL` — nunca recebe resultado final síncrono.
- **Exemplo:** Agente MCP solicita ação "SUBMETER-PEDIDO" com policy=CONTROLLED. Gateway MCP chama POST /movement-engine/evaluate com `{ origin_type: "MCP", origin_agent_id: "uuid-agent" }`. Motor retorna `{ controlled: true, movement_id: "uuid-mov" }`. Agente recebe 202 `{ movement_id: "uuid-mov", status: "PENDING_APPROVAL" }`.
- **Exceções:** Nenhuma. CONTROLLED sem aprovação humana é violação de governança.
- **Impacto:** FLOW, COMPLIANCE

```gherkin
Funcionalidade: Política CONTROLLED Exige Decisão Humana

Cenário: Ação CONTROLLED gera movimento controlado
  Dado que a ação tem execution_policy=CONTROLLED
  Quando o agente solicita execução via POST /mcp/execute
  Então POST /movement-engine/evaluate é chamado com origin_type=MCP
  E controlled_movement é criado no MOD-009
  E agente recebe 202 com movement_id e status="PENDING_APPROVAL"

Cenário: Movimento CONTROLLED registrado em mcp_executions
  Dado que a ação CONTROLLED foi despachada
  Então mcp_executions registra status=CONTROLLED_PENDING
  E linked_movement_id aponta para o controlled_movement criado
```

## BR-009 — Política EVENT_ONLY Sem Escrita

- **Regra:** Política EVENT_ONLY apenas emite domain_event com o contexto do agente e da ação. NENHUMA escrita transacional é permitida (zero side-effects em banco além do registro em `mcp_executions`). O domain_event emitido contém `origin_type=MCP`, `agent_id`, `action_code` e o payload fornecido pelo agente.
- **Exemplo:** Agente MCP solicita ação "NOTIFICAR-STATUS" com policy=EVENT_ONLY. Gateway MCP emite `mcp.event_only.dispatched` com payload do agente. Agente recebe 200 `{ status: "EVENT_EMITTED", event_type: "mcp.event_only.dispatched" }`.
- **Exceções:** O registro em `mcp_executions` (status=EVENT_EMITTED) é a única escrita permitida — é rastreabilidade, não side-effect de negócio.

## BR-010 — Gateway MCP Valida 8 Passos Sequenciais

- **Regra:** O gateway MCP (`POST /mcp/execute`) DEVE executar 8 passos de validação em ordem estrita. Falha em qualquer passo aborta a execução e registra o erro em `mcp_executions`. Os passos são: (1) autenticar agente via API key + bcrypt compare, (2) verificar status=ACTIVE, (3) buscar ação pelo `action_code`, (4) verificar vínculo agente-ação (`mcp_agent_action_links`) + `valid_until` não expirado, (5) verificar `required_scopes` da ação contidos em `allowed_scopes` do agente, (6) dupla verificação blocklist de escopos Phase 1, (7) INSERT em `mcp_executions` com status=RECEIVED, (8) avaliar `execution_policy` e despachar.
- **Exemplo:** Agente envia POST /mcp/execute com `{ action_code: "CONSULTAR-ESTOQUE", payload: {...} }`. Passo 1: API key válida. Passo 2: status=ACTIVE. Passo 3: ação encontrada. Passo 4: vínculo ativo e não expirado. Passo 5: escopos satisfeitos. Passo 6: nenhum escopo bloqueado. Passo 7: execução registrada. Passo 8: policy=DIRECT → execução imediata → 200.
- **Exceções:** Nenhuma. Ordem dos passos é invariável.
- **Impacto:** FLOW, STATE

```gherkin
Funcionalidade: Gateway MCP 8 Passos Sequenciais

Cenário: Passo 1 falha — API key inválida
  Quando POST /mcp/execute com API key inválida
  Então 401: "API key inválida ou agente não encontrado."

Cenário: Passo 2 falha — agente inativo
  Dado que API key é válida mas agente tem status=INACTIVE
  Quando POST /mcp/execute
  Então 403: "Agente inativo."

Cenário: Passo 4 falha — vínculo expirado
  Dado que o vínculo agente-ação tem valid_until no passado
  Quando POST /mcp/execute com action_code da ação
  Então 403: "Vínculo agente-ação expirado."

Cenário: Todos os 8 passos passam
  Dado que o agente está ACTIVE, ação existe, vínculo ativo, escopos válidos
  Quando POST /mcp/execute
  Então mcp_executions registra status=RECEIVED
  E despacho ocorre conforme execution_policy
```

## BR-011 — Vínculo Agente-Ação Único e com Vigência

- **Regra:** Cada par (agent_id, action_id) é único na tabela `mcp_agent_action_links` (constraint UNIQUE). O campo `valid_until` é nullable — quando preenchido, o vínculo expira automaticamente após a data. Vínculos expirados são ignorados pelo gateway MCP (passo 4). O campo `granted_by` registra o admin responsável pela concessão.
- **Exemplo:** POST /admin/mcp-agents/:id/actions com `{ action_id: "uuid", valid_until: "2026-06-30T23:59:59Z" }` → 201. Após 2026-06-30, gateway ignora o vínculo.
- **Exceções:** Nenhuma. Tentativa de duplicar vínculo retorna 409.
- **Impacto:** DATA, PERMISSIONS

## BR-012 — Detecção e Registro de Privilege Escalation

- **Regra:** Toda tentativa de privilege escalation (agente tentando usar escopos bloqueados, injetar escopos no payload, ou contornar a blocklist) DEVE ser: (1) bloqueada com status=BLOCKED em `mcp_executions`, (2) classificada com `sensitivity_level=2`, (3) emitir evento `mcp.privilege_escalation_attempt`, (4) exibir badge vermelho "Escalada de Privilégio" no UX-MCP-002. O campo de escopos injetados no payload é ignorado (nunca sobrescreve `allowed_scopes`).
- **Exemplo:** Agente envia `{ action_code: "X", scopes: ["approval:decide"] }` no payload. Campo `scopes` é ignorado. Se a ação requer `approval:decide` e o agente não tem, evento de escalada é emitido.
- **Exceções:** Nenhuma.
- **Impacto:** COMPLIANCE, STATE
