# FR-006 — Requisitos Funcionais da Execução de Casos

## FR-001 — Abertura de Caso

- **Descrição:** Abrir instância de caso vinculada a ciclo PUBLISHED, com freeze de `cycle_version_id`, geração automática de código e criação de gate_instances PENDING para o estágio inicial.
- **Endpoint:** POST /api/v1/cases
- **Scope:** `process:case:write`
- **Dependências:** BR-001 (freeze), BR-009 (PUBLISHED only), BR-010 (codigo), DATA-006, MOD-005 (process_cycles, process_stages, process_gates)
- **Done Funcional:**
  1. Valida ciclo PUBLISHED (422 se DRAFT/DEPRECATED)
  2. Captura `cycle_version_id` vigente (freeze imutável)
  3. Gera `codigo` no padrão CASO-{YYYY}-{SEQ5}
  4. Cria `case_instances` com status=OPEN e current_stage_id=estágio is_initial
  5. Cria `gate_instances` PENDING para todos os gates do estágio inicial
  6. Insere `stage_history` (from_stage_id=null, to_stage_id=inicial)
  7. Emite domain event `case.opened`
- **Idempotência:** required (Idempotency-Key com TTL 60s)
- **Timeline/Notifications:** yes (case.opened → DATA-003, SEC-002)

```gherkin
Funcionalidade: Abertura de Caso

Cenário: Abrir caso em ciclo PUBLISHED
  Dado que existe ciclo com status=PUBLISHED e estágio inicial definido
  Quando POST /api/v1/cases com { cycle_id, tenant_id }
  Então 201 com case_instance criada
  E current_stage_id = estágio is_initial do ciclo
  E cycle_version_id = versão vigente (frozen)
  E gate_instances PENDING criadas para gates do estágio inicial
  E stage_history registra abertura (from=null, to=inicial)
  E evento case.opened emitido

Cenário: Rejeitar abertura em ciclo DRAFT
  Dado que ciclo tem status=DRAFT
  Quando POST /api/v1/cases com esse cycle_id
  Então 422: "Apenas ciclos publicados podem ter novos casos abertos."
```

## FR-002 — Motor de Transição de Estágios

- **Descrição:** Executar transição entre estágios com validação sequencial: (1) caso OPEN, (2) transição existe no blueprint, (3) papel autorizado, (4) gates required resolvidos, (5) evidência se required.
- **Endpoint:** POST /api/v1/cases/:id/transition
- **Scope:** `process:case:write`
- **Request Body:** `{ transition_id, evidence?, motivo? }`
- **Dependências:** BR-002 (gate bloqueante), BR-004 (terminal completa), BR-005 (INFORMATIVE), BR-006 (papel required), BR-012 (status), DATA-006, MOD-005 (stage_transitions, process_gates, stage_role_links)
- **Done Funcional:**
  1. Valida caso.status=OPEN (422 se diferente)
  2. Valida transição existe no blueprint e parte do current_stage (422)
  3. Valida papel do usuário em allowed_roles da transição (403)
  4. Valida gates required=true do estágio atual RESOLVED ou WAIVED (422 com pending_gates)
  5. Valida papel required=true do estágio tem atribuição ativa (422)
  6. Valida evidence fornecida se transition.evidence_required=true (422)
  7. INSERT stage_history (from=current, to=next)
  8. UPDATE case_instances SET current_stage_id=to_stage_id
  9. Se to_stage.is_terminal: UPDATE status=COMPLETED, completed_at=now()
  10. INSERT case_events (event_type=STAGE_TRANSITIONED)
  11. Cria gate_instances PENDING para gates do novo estágio
  12. Emite domain event `case.stage_transitioned` (ou `case.completed` se terminal)
- **Idempotência:** n/a (transição é uma ação com side-effect sequencial; repetir altera estado)
- **Timeline/Notifications:** yes (case.stage_transitioned → DATA-003, SEC-002)

```gherkin
Funcionalidade: Motor de Transição

Cenário: Transição bem-sucedida sem gates pendentes
  Dado que todos os gates required do estágio atual estão RESOLVED
  E o usuário tem papel com allowed_role na transição
  Quando POST /cases/:id/transition com { transition_id }
  Então stage_history registra a transição
  E current_stage_id é atualizado para o novo estágio
  E gate_instances PENDING criadas para gates do novo estágio
  E evento case.stage_transitioned emitido

Cenário: Motor bloqueia sem papel autorizado
  Dado que transição tem allowed_roles=["APROVADOR"] e o usuário é "RESPONSAVEL"
  Quando POST /cases/:id/transition
  Então 403: "Seu papel (Responsável) não autoriza esta transição."

Cenário: Transição para estágio terminal completa caso
  Dado que to_stage.is_terminal=true
  Quando transição é executada
  Então case.status=COMPLETED e completed_at=now()
  E evento case.completed emitido

Cenário: Gates recriados como PENDING após reabertura (REOPENED)
  Dado que caso foi reaberto para estágio "Em Análise" via REOPENED com target_stage_id
  E estágio "Em Análise" tem 3 gates definidos no blueprint
  Quando REOPENED é processado (FR-007)
  Então 3 novos gate_instances criados com status=PENDING para estágio "Em Análise"
  E gate_instances do estágio terminal mantidos com status original (histórico preservado)
```

> **Nota sobre REOPENED:** Quando um caso é reaberto (FR-007, BR-016), os gate_instances do estágio destino são recriados com status=PENDING. Gates do estágio terminal (anterior) são mantidos como registro histórico — NÃO são deletados. Isto preserva o audit trail completo e permite que a constraint UNIQUE(case_id, gate_id) seja estendida para UNIQUE(case_id, gate_id, stage_history_entry_id) se necessário. Ref: PEN-006/PENDENTE-005 Opção B.

## FR-003 — Controles do Caso (ON_HOLD, RESUME, CANCEL)

- **Descrição:** Colocar caso em espera, retomar e cancelar com motivo obrigatório.
- **Endpoints:**
  - POST /api/v1/cases/:id/hold — Scope: `process:case:write`
  - POST /api/v1/cases/:id/resume — Scope: `process:case:write`
  - POST /api/v1/cases/:id/cancel — Scope: `process:case:cancel`
- **Dependências:** BR-011 (cancel motivo), BR-012 (transições de status), DATA-006
- **Done Funcional:**
  - **HOLD:** caso OPEN → ON_HOLD; case_events registra ON_HOLD com motivo; emite case.on_hold

- **Idempotência:** n/a
- **Timeline/Notifications:** yes (case.on_hold, case.resumed, case.cancelled → DATA-003, SEC-002)

## FR-004 — Resolução de Gates

- **Descrição:** Resolver gates por tipo: APPROVAL (com decision APPROVED/REJECTED e parecer), DOCUMENT (com upload/URL), CHECKLIST (com todos os itens marcados).
- **Endpoint:** POST /api/v1/cases/:id/gates/:gateId/resolve
- **Scope:** `process:case:gate_resolve`
- **Dependências:** BR-008 (can_approve), BR-013 (checklist completo), DATA-006 (gate_instances), SEC-006
- **Done Funcional:**
  - **APPROVAL:** decision=APPROVED|REJECTED + parecer obrigatório; se REJECTED, registra EXCEPTION em case_events; motor NÃO avança automaticamente em rejeição
  - **DOCUMENT:** evidence { type: "file", url, filename } salva em gate_instance.evidence
  - **CHECKLIST:** todos os itens checked=true obrigatório (422 se parcial)
  - Para todos: gate_instance.status=RESOLVED, resolved_by=user.id, resolved_at=now()
  - Emite domain event `case.gate_resolved`
- **Idempotência:** n/a (resolução é ação idempotente por natureza — UNIQUE(case_id, gate_id) impede duplicação)
- **Timeline/Notifications:** yes (case.gate_resolved → DATA-003, SEC-002)

```gherkin

Cenário: Resolver gate APPROVAL com APPROVED

  Quando POST resolve com { decision: "APPROVED", parecer: "OK" }
  Então gate_instance.status=RESOLVED, decision=APPROVED
  E evento case.gate_resolved emitido
```

## FR-007 — Registro de Eventos Avulsos

- **Descrição:** Registrar eventos no caso sem mudança de estágio: COMMENT, EXCEPTION, REOPENED, EVIDENCE, REASSIGNED, ON_HOLD, RESUMED. Para REOPENED, scope especial `process:case:reopen` e `target_stage_id` obrigatório.
- **Endpoint:** POST /api/v1/cases/:id/events
- **Scope:** `process:case:write` (geral) | `process:case:reopen` (para event_type=REOPENED)
- **Request Body:** `{ event_type, descricao, metadata?, target_stage_id? }`
- **Dependências:** BR-016 (REOPENED), DATA-006 (case_events), SEC-006 (scope reopen)
- **Done Funcional:**
  1. Valida event_type em enum permitido
  2. INSERT case_events com stage_id=current_stage_id, created_by, created_at
  3. Se event_type=REOPENED e caso COMPLETED:
     - Valida scope `process:case:reopen` (403 se ausente)
     - Valida `target_stage_id` presente e obrigatório (422 se ausente)
     - Valida `target_stage_id` pertence ao mesmo `cycle_id` do caso (422 se inválido)
     - Reverte status=OPEN, limpa completed_at
     - UPDATE current_stage_id = target_stage_id
     - Cria gate_instances PENDING para gates do estágio destino (gates do estágio terminal mantidos como histórico)
     - INSERT stage_history (from=terminal, to=target_stage_id, motivo=descricao)
     - Emite domain event `case.reopened` com { case_id, from_stage_id, target_stage_id, reopened_by, reason }
  4. Para outros event_types: current_stage_id NÃO muda
  5. Emite domain event `case.event_recorded` (para tipos != REOPENED)
- **Idempotência:** n/a
- **Timeline/Notifications:** yes (case.event_recorded / case.reopened → DATA-003, SEC-002)

```gherkin
Funcionalidade: Eventos Avulsos

Cenário: Registrar comentário sem mudar estágio
  Dado que caso está no Estágio B
  Quando POST /cases/:id/events com { event_type: "COMMENT", descricao: "Aguardando..." }
  Então case_events criado
  E current_stage_id permanece Estágio B

Cenário: Reabertura com target_stage_id move caso para estágio destino
  Dado que caso está COMPLETED no estágio terminal "Encerrado"
  E usuário tem scope process:case:reopen
  Quando POST /cases/:id/events com { event_type: "REOPENED", descricao: "Correção necessária", target_stage_id: "Em Análise" }
  Então case.status=OPEN, case.completed_at=null
  E current_stage_id atualizado para "Em Análise"
  E gate_instances do estágio "Em Análise" recriados como PENDING
  E stage_history registra (from="Encerrado", to="Em Análise")
  E domain event case.reopened emitido com target_stage_id

Cenário: Rejeitar REOPENED sem target_stage_id
  Dado que caso está COMPLETED
  E usuário tem scope process:case:reopen
  Quando POST /cases/:id/events com { event_type: "REOPENED", descricao: "Correção" }
  Então 422: "target_stage_id é obrigatório para reabertura."

Cenário: Rejeitar REOPENED sem scope reopen
  Dado que caso está COMPLETED
  E usuário NÃO tem scope process:case:reopen
  Quando POST /cases/:id/events com { event_type: "REOPENED", target_stage_id: "uuid" }
  Então 403

Cenário: Rejeitar REOPENED com estágio de outro ciclo
  Dado que caso está COMPLETED com cycle_id=ciclo-A

  Quando POST /cases/:id/events com { event_type: "REOPENED", target_stage_id: "estágio-ciclo-B" }
```

## FR-009 — Listagem de Casos com Filtros

- **Descrição:** Listagem paginada cursor-based com filtros por ciclo, status, estágio, date range e "Minha responsabilidade".
- **Endpoint:** GET /api/v1/cases
- **Scope:** `process:case:read`
- **Query Params:** `cycle_id?, status?, stage_id?, assigned_to_me?, opened_after?, opened_before?, search?, object_id?, cursor?, limit?`
- **Dependências:** DATA-006 (case_instances, case_assignments, gate_instances), NFR-006 (SLO < 300ms)
- **Done Funcional:**
  1. Filtro obrigatório por tenant_id
  2. Paginação cursor-based (sem offset)
  3. Filtro `assigned_to_me=true`: retorna apenas casos com case_assignment ativa para o usuário
  4. Filtro `search`: busca textual por codigo (debounce 400ms no client)
  5. Filtro `object_id` (UUID, opcional): exact match em `case_instances.object_id` via índice `idx_case_instances_object` (B-tree parcial). Mutuamente exclusivo com `search` — se ambos enviados, `object_id` prevalece. Usado por integrações programáticas (MOD-007, MOD-008). Ref: PEN-006/PENDENTE-003 Opção A
  6. Cada item inclui: codigo, cycle_name, current_stage_name, status, pending_gates_count
  7. nextCursor retornado quando há mais resultados
- **Idempotência:** n/a (leitura)
- **Timeline/Notifications:** no

```gherkin
Funcionalidade: Listagem de Casos

Cenário: Filtro "Minha responsabilidade"
  Dado que usuário ativa filtro assigned_to_me=true
  Quando GET /cases?assigned_to_me=true
  Então retorna apenas casos onde o usuário tem atribuição ativa

Cenário: Paginação cursor-based
  Dado que há 50 casos
  Quando GET /cases?limit=20
  Então retorna 20 casos e nextCursor

Cenário: Filtro por object_id (exact match)
  Dado que existe caso vinculado a object_id="abc-123-uuid"
  Quando GET /cases?object_id=abc-123-uuid
  Então retorna apenas casos vinculados a esse object_id
  E response time < 300ms (SLO NFR-006)

Cenário: object_id prevalece sobre search
  Quando GET /cases?object_id=abc-123-uuid&search=CASO-2026
  Então filtra por object_id (exact match)
  E ignora search
```

## FR-014 — Expiração Automática de Atribuições (Background Job)

- **Descrição:** Background job periódico (cron 5 min) que detecta e desativa atribuições expiradas: (1) delegações MOD-004 expiradas via `DelegationCheckerPort.getExpiredDelegations()`, (2) atribuições temporárias com `valid_until < now()`.
- **Trigger:** Cron a cada 5 minutos
- **Scope:** N/A (job interno, sem endpoint HTTP)
- **Dependências:** BR-015 (delegação expira), BR-017 (valid_until expira), ADR-005, INT-006 §3.2 (contrato getExpiredDelegations), DATA-006 (case_assignments), NFR-006 §3 (healthcheck)
- **Done Funcional:**
  1. Query `case_assignments WHERE is_active=true AND valid_until < now()` — desativar atribuições temporárias expiradas
  2. Chama `DelegationCheckerPort.getExpiredDelegations()` — obtém atribuições vinculadas a delegações MOD-004 expiradas
  3. Para cada atribuição expirada: UPDATE `is_active=false`
  4. INSERT `case_events` (event_type=REASSIGNED, metadata={ reason: "auto-expirado", source: "expiration-job", expired_at })
  5. Emite domain event `case.assignment_replaced` por assignment desativada
  6. Registra métricas: total processado, total desativado, duração
  7. Healthcheck via `/health/jobs` com `last_run_at` e `status` (NFR-006 §3)
- **Fallback:** Se `getExpiredDelegations()` falhar (timeout 3s, 1 retry): log error + skip delegações nesta iteração. Atribuições por `valid_until` continuam processando normalmente. Fail-safe: nenhuma atribuição é desativada por engano.
- **Tolerância:** Até 5 minutos entre expiração real e desativação (intervalo do cron)
- **Evolução:** Quando MOD-004 emitir domain event `delegation.expired`, complementar com handler reativo (ADR-005)
- **Timeline/Notifications:** yes (case.assignment_replaced → DATA-003, SEC-002)

```gherkin
Funcionalidade: Expiração Automática de Atribuições

Cenário: Delegação MOD-004 expirada desativa atribuição
  Dado que case_assignment tem delegation_id vinculado
  E a delegação expirou há mais de 5 minutos
  Quando o job de expiração executa
  Então case_assignment.is_active=false
  E case_events registra REASSIGNED com reason="auto-expirado"
  E domain event case.assignment_replaced emitido

  Dado que case_assignment tem valid_until=2026-03-18T23:59:59Z
  E agora é 2026-03-19T00:05:00Z
  Quando o job de expiração executa
  Então case_assignment.is_active=false
  E case_events registra REASSIGNED com reason="auto-expirado"

Cenário: Falha no DelegationCheckerPort não afeta valid_until
  Dado que DelegationCheckerPort.getExpiredDelegations() retorna timeout
  E existem atribuições com valid_until expirado
  Quando o job executa
  Então atribuições por valid_until são desativadas normalmente
  E log de erro registrado para delegações
  E job retenta delegações no próximo ciclo
```
