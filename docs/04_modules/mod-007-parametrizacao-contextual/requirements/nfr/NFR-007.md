## NFR-001 — Performance do Motor de Avaliacao

- **Requisito:** O endpoint POST /routine-engine/evaluate deve responder em < 200ms (p95) para contextos com ate 10 regras de incidencia ativas e 50 routine_items. p99 < 500ms.
- **Justificativa:** O motor e chamado no fluxo critico de transicao do MOD-006 — latencia adicional bloqueia a operacao do usuario. SLA do MOD-006 depende do SLA deste motor.
- **Mecanismos de garantia:**
  - Sem cache Redis (decisao tecnica 2026-03-15 — BR-009). Performance garantida por queries otimizadas e indexes.
  - Index `idx_incidence_status_vigencia` para passo 1 (encontrar regras ativas)
  - Index `idx_links_incidence` para passo 2 (join rotinas vinculadas)
  - Index `idx_items_routine_ordem` para passo 3 (avaliacao sequencial de itens)
  - Queries com `WHERE status = 'ACTIVE'` e partial indexes para reduzir scan
- **Monitoramento:** Alerta se p95 > 150ms por 5 minutos consecutivos (pre-breach warning)

- **estado_item:** DRAFT
- **data_ultima_revisao:** 2026-03-20
- **rastreia_para:** US-MOD-007-F03, BR-009, FR-009, INT-007, DATA-007

## NFR-003 — Disponibilidade

- **Requisito:** O motor de avaliacao (POST /routine-engine/evaluate) deve ter disponibilidade >= 99.9%. Endpoints CRUD admin devem ter disponibilidade >= 99.5%.
- **Justificativa:** Motor de avaliacao e dependencia sincrona do MOD-006. Se o motor estiver indisponivel, transicoes de caso ficam bloqueadas (INT-001 fail-closed). Endpoints admin tem tolerancia maior por nao serem path critico de operacao.
- **Mecanismos:**
  - Health check dedicado para o motor de avaliacao
  - Circuit breaker no lado do MOD-006 (INT-001)
  - Zero downtime deploys (rolling update)
- **Calculo:** 99.9% = maximo 8.76h de indisponibilidade/ano ou 43.8min/mes

## NFR-007 — Topologia

- **Requisito:** O modulo opera em topologia **sincrona** para todas as operacoes.

| Componente | Topologia | Justificativa |
|---|---|---|
| **Motor de avaliacao** | sync | Chamada sincrona do MOD-006 no fluxo de transicao |
| **CRUD admin** | sync | Operacoes de configuracao com resposta imediata |
| **Domain events** | sync (persist) + async (notify) | Persistencia sincrona na transacao; notificacoes assincronas via Foundation |
| **Background job expiracao** | async (cron) | Expiracao de enquadradores via job periodico |

### Background Job de Expiração — Decisão de Isolamento (PEN-007/PEN-002)

> **Decisão:** Job independente no MOD-007 (`framer-expiration.job.ts`). Isolamento total — falha no job de expiração do MOD-004 não afeta expiração de enquadradores. Duplicação de lógica de scheduling aceita nesta fase. Revisitar quando houver 3+ módulos com jobs de expiração.

| Propriedade | Valor |
|---|---|
| **Job** | `framer-expiration.job.ts` |
| **Scheduler** | Cron próprio do MOD-007 (não compartilhado com MOD-004) |
| **Frequência** | A cada 5 minutos (configurável via env `FRAMER_EXPIRATION_INTERVAL_MS`, default: 300000) |
| **Query** | `SELECT FROM framers WHERE valid_until < now() AND status = 'ACTIVE'` |
| **Ação** | Atualiza `status → EXPIRED`, emite domain event `framer.expired` (DATA-003 EVT-004) |
| **Concorrência** | Lock distribuído (advisory lock) para evitar dupla execução em múltiplas instâncias |
| **Healthcheck** | Dedicado: `framer-expiration-job` (ver NFR-008) |
| **Monitoramento** | Alerta se job não executar por > 15 minutos |
| **Idempotência** | Seguro para re-execução (só processa enquadradores ainda ACTIVE com valid_until expirado) |

- **Nota:** A decisao de remover cache Redis (BR-009) simplifica a topologia — nao ha componente async de invalidacao de cache.
