## NFR-001 — Performance do Gateway MCP

- **Topologia:** sync (gateway executa 8 passos sequenciais + despacho)

### SLOs

| Endpoint | Métrica | Baseline | Alvo P95 | Alvo P99 |
|---|---|---|---|---|
| `POST /mcp/execute` (DIRECT) | Latência | N/A (módulo novo) | 200ms | 500ms |
| `POST /mcp/execute` (CONTROLLED) | Latência | N/A | 500ms | 1000ms |
| `POST /mcp/execute` (EVENT_ONLY) | Latência | N/A | 100ms | 200ms |
| `POST /mcp/execute` (authn bcrypt) | Latência bcrypt compare | N/A | 80ms | 100ms |

### Detalhamento

- **DIRECT:** 8 passos de validação + execução da ação. bcrypt compare domina (~60-80ms). Restante deve completar em < 120ms.
- **CONTROLLED:** Inclui chamada síncrona ao MOD-009 (`POST /movement-engine/evaluate`) com timeout de 5000ms. P95 inclui a latência do MOD-009.
- **EVENT_ONLY:** Apenas validação (passos 1-7) + emit de domain event. Sem side-effect de negócio.
- **bcrypt:** rounds >= 12 (BR-004). Tempo de compare é proporcional aos rounds — 12 rounds ~ 60-80ms em hardware moderno.

- **rastreia_para:** FR-007, BR-010

## NFR-004 — Segurança da API Key

### SLOs

| Métrica | Alvo |
|---|---|
| bcrypt compare latency | < 100ms P99 |
| bcrypt hash rounds | >= 12 (configurable, never < 12) |
| API key entropy | 256 bits (base64url, 44 chars) |
| API key exposure in responses | ZERO (apenas POST create e POST rotate-key) |
| API key exposure in logs | ZERO (nunca logada, nem mascarada — ausente) |

### Detalhamento

- Key gerada via `crypto.randomBytes(32).toString('base64url')` (256 bits).
- Hash via `bcrypt.hash(key, 12)`.
- Resposta de criação/rotação inclui `api_key` em texto plano UMA VEZ. Client deve armazenar de forma segura.
- GET, LIST, LOG endpoints NUNCA retornam `api_key` nem `api_key_hash`.
- Logs estruturados excluem o campo `api_key` (não aparece nem mascarado).

- **rastreia_para:** FR-001, FR-003, BR-004

## NFR-005 — Escalabilidade

### Limites

| Recurso | Limite | Tipo | Observação |
|---|---|---|---|
| Agentes ativos por tenant | 1.000 | Soft limit | Configurável via tenant settings |
| Ações no catálogo por tenant | 5.000 | Soft limit | |
| Execuções por dia por tenant | 10.000 | Soft limit | Rate limiting via token bucket |
| Execuções por minuto por agente | 60 | Hard limit | Proteção contra agente out-of-control |
| Vínculos agente-ação por agente | 100 | Soft limit | |
| Payload size (request) | 64 KB | Hard limit | 413 Payload Too Large |
| Payload size (response) | 256 KB | Hard limit | Truncamento com warning |

### Rate Limiting

- **Gateway (`POST /mcp/execute`):** Token bucket por `agent_id` — 60 req/min burst, 30 req/min sustained.
- **Admin endpoints:** Rate limiting padrão do Foundation (100 req/min por JWT).
- **Response header:** `X-RateLimit-Remaining`, `X-RateLimit-Reset`.
- **Quando excedido:** 429 Too Many Requests com `Retry-After` header.

```json
{
  "type": "https://api.easya2.com/errors/rate-limit-exceeded",
  "title": "Limite de Taxa Excedido",
  "detail": "Agente excedeu o limite de 60 execuções por minuto.",
  "status": 429,
  "extensions": {
    "correlationId": "{correlation_id}",
    "retry_after_seconds": 30
  }
}
```

- **rastreia_para:** FR-007

## NFR-006 — Auditoria e Retenção

### SLOs

| Recurso | Política | Observação |
|---|---|---|
| `mcp_executions` | Retenção 5 anos | Compliance — registros de auditoria de automação |
| `domain_events` (DATA-003) | Retenção conforme DOC-ARC-003 | Eventos seguem política global |
| `mcp_agents` (revogados) | Indefinida | Agentes revogados mantidos para auditoria |
| `mcp_agent_action_links` (removidos) | Hard delete | Links são descartáveis (evento de auditoria captura) |

### Imutabilidade

- Registros em `mcp_executions` são **append-only**: uma vez criados, campos `status`, `result_payload`, `completed_at`, `error_message`, `duration_ms`, `linked_movement_id`, `linked_integration_log_id` podem ser atualizados UMA VEZ (de RECEIVED para o estado final). Nenhuma segunda atualização é permitida.
- Tentativa de atualizar execução já completada deve ser rejeitada silenciosamente (log warning).

### Particionamento

- Recomendação: particionamento por `received_at` (range mensal) quando volume exceder 10M registros — viabiliza queries eficientes com retenção de 5 anos.

- **rastreia_para:** FR-009, BR-010, DATA-010

## NFR-008 — Disaster Recovery (DR)

### SLOs

| Métrica | Alvo |
|---|---|
| **RPO** (Recovery Point Objective) | 1 hora (baseado em backup incremental do PostgreSQL) |
| **RTO** (Recovery Time Objective) | 4 horas (restore de banco + redeploy + validação) |

### Procedimentos

- **Backup:** PostgreSQL WAL archiving contínuo + snapshot diário.
- **Restore:** Restauração point-in-time (PITR) para `mcp_executions` (tabela mais crítica para compliance).
- **BullMQ:** Jobs em DLQ são reprocesados após restore. Jobs perdidos entre backup e falha são irrecuperáveis (aceito pelo RTO).
- **API keys:** Após DR, todas as keys permanecem válidas (hashes bcrypt no banco restaurado). Nenhuma rotação forçada necessária a menos que haja suspeita de comprometimento.

- **rastreia_para:** DATA-010, NFR-006

## NFR-009 — Healthcheck

### Endpoints

| Endpoint | Tipo | Checks | Response |
|---|---|---|---|
| `GET /health/live` | Liveness | Processo respondendo | `200 { "status": "ok" }` |
| `GET /health/ready` | Readiness | DB connection + Redis (BullMQ) + bcrypt self-test | `200 { "status": "ready", "checks": {...} }` |

### Detalhamento do Readiness Check

```json
{
  "status": "ready",
  "checks": {
    "database": {
      "status": "up",
      "latency_ms": 5
    },
    "redis": {
      "status": "up",
      "latency_ms": 2
    },
    "bcrypt_selftest": {
      "status": "up",
      "latency_ms": 65
    }
  },
  "version": "1.0.0",
  "uptime_seconds": 86400
}
```

### Degradação

- Se `database` down: readiness retorna `503 { "status": "not_ready" }`. Gateway rejeitará requisições.
- Se `redis` down: readiness retorna `200` com warning. Gateway funciona (DIRECT/CONTROLLED/EVENT_ONLY sem BullMQ). Integrações MOD-008 ficarão indisponíveis.
- Se `bcrypt_selftest` lento (> 150ms): readiness retorna `200` com warning. Log de alerta emitido.

- **rastreia_para:** NFR-003
