# ADR-002 — Freeze de cycle_version_id na Abertura do Caso

- **status:** proposed
- **timebox_days:** 90

---

## Contexto

O MOD-006 instancia casos sobre blueprints do MOD-005. Blueprints podem evoluir (fork de ciclo, renomeação de estágios, adição de gates). Sem proteção, um caso em andamento poderia "ver" mudanças no blueprint que não existiam quando o caso foi aberto — gerando inconsistências (ex: gate novo aparece retroativamente, transição que existia desaparece).

BR-001 define que "qualquer fork ou atualização do ciclo no MOD-005 NÃO afeta instâncias em andamento". A questão é: qual mecanismo técnico garante esse isolamento?

## Decisão

**Capturar e congelar o `cycle_version_id` no momento da abertura do caso.** O campo `case_instances.cycle_version_id` é preenchido com o ID da versão vigente do ciclo no POST /cases e torna-se imutável após criação. Todas as leituras de blueprint (estágios, gates, transições, papéis) durante a vida do caso referenciam exclusivamente essa versão frozen — nunca a versão "latest" do ciclo.

## Alternativas Consideradas

| Alternativa | Prós | Contras |
|---|---|---|
| **A) Freeze por cycle_version_id FK (escolhida)** | Isolamento total; simples de implementar (FK imutável); leituras sempre consistentes; sem custo de armazenamento extra | Caso nunca "vê" melhorias no blueprint (ex: novo gate adicionado); se MOD-005 deletar versão, FK ficaria órfã (mitigado por soft-delete) |
| **B) Snapshot completo do blueprint em JSON** | Independência total do MOD-005; caso portátil; zero FKs cross-module | Duplicação massiva de dados; inconsistência se snapshot divergir do real; dificuldade de migrar snapshots; sem integridade referencial |
| **C) Referência "latest" com versionamento de mudanças** | Casos sempre veem a versão mais recente; migração automática | Viola BR-001 fundamentalmente; complexidade de migração de instâncias; risco de inconsistência em transições |
| **D) Freeze com migração opcional (admin)** | Permite que admin promova caso para nova versão do blueprint | Complexidade de migração (gates novos, transições removidas); risco de estado inválido pós-migração |

## Consequências

- `case_instances.cycle_version_id` é NOT NULL e imutável (nenhum UPDATE deve alterá-lo — enforced via application)
- FKs para `process_stages`, `process_gates`, `stage_transitions`, `process_roles` e `stage_role_links` sempre referem entidades da versão frozen
- MOD-005 DEVE manter soft-delete (não DELETE físico) para preservar integridade das FKs
- GET /cases/:id retorna blueprint da versão frozen (FR-010)
- Se no futuro for necessária migração de versão (Alternativa D), criar ADR dedicada com fluxo de migração seguro
- INT-006 §3.1 documenta que a integração com MOD-005 é DB read no mesmo schema — FKs garantem integridade
