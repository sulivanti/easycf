## Detalhamento

### BUG-001: Response schema literal incompatível com serializerCompiler

**Localização:** `apps/api/src/modules/org-units/presentation/routes/org-units.route.ts` — endpoints DELETE /:id (linha 279) e PATCH /:id/restore (linha 305)

**Causa raiz:** O `serializerCompiler` do `fastify-type-provider-zod` (registrado globalmente em `apps/api/src/index.ts:103`) chama `schema.safeParse()` para validar responses. Os dois endpoints usam JSON Schema literal:

```typescript
// ANTES (incorreto):
response: {
  200: { type: 'object' as const, properties: { message: { type: 'string' as const } } },
}
```

O objeto `{ type: 'object', properties: ... }` não possui método `.safeParse()`, gerando TypeError. O handler já executou com sucesso antes da serialização, por isso a operação persiste mas o client recebe erro 500.

**Correção:**

1. Criar schema Zod reutilizável em `org-units.dto.ts`:

```typescript
export const genericMessageResponse = z.object({
  message: z.string(),
});
```

2. Substituir nos dois endpoints:

```typescript
// DEPOIS (correto):
import { genericMessageResponse } from '../dtos/org-units.dto.js';

// DELETE /:id
response: { 200: genericMessageResponse }

// PATCH /:id/restore
response: { 200: genericMessageResponse }
```

### BUG-002: Tree endpoint não retorna unidades inativas

**Localização:**
- `apps/api/src/modules/org-units/presentation/routes/org-units.route.ts` — GET /tree (linha 134)
- `apps/api/src/modules/org-units/infrastructure/drizzle-repositories.ts` — `getTree()` (linha 100)
- `apps/api/src/modules/org-units/application/use-cases/get-org-unit-tree.use-case.ts`
- `apps/api/src/modules/org-units/application/ports/repositories.ts`

**Causa raiz:** O repository `getTree()` filtra com `eq(orgUnits.status, 'ACTIVE')`, e não aceita parâmetro para incluir inativos. O endpoint não expõe query parameter.

**Correção:**

1. **DTO — novo schema de querystring** em `org-units.dto.ts`:

```typescript
export const orgUnitTreeQuery = z.object({
  include_inactive: z
    .enum(['true', 'false'])
    .optional()
    .default('false')
    .transform((v) => v === 'true'),
});
```

2. **Route — adicionar querystring ao GET /tree:**

```typescript
app.get<{ Querystring: z.infer<typeof orgUnitTreeQuery> }>('/tree', {
  // ...
  schema: {
    querystring: orgUnitTreeQuery,
    // ... demais props mantidas
  },
  handler: async (request, reply) => {
    const result = await request.dipiContainer.getOrgUnitTreeUseCase.execute({
      includeInactive: request.query.include_inactive,
    });
    // ... mapper mantido
  },
});
```

3. **Port (interface do repository):**

```typescript
// repositories.ts
getTree(includeInactive?: boolean, tx?: TransactionContext): Promise<readonly OrgUnitTreeNode[]>;
```

4. **Use case — propagar parâmetro:**

```typescript
// get-org-unit-tree.use-case.ts
async execute(input?: { includeInactive?: boolean }) {
  const tree = await this.orgUnitRepo.getTree(input?.includeInactive);
  return { tree };
}
```

5. **Repository — condicionar filtro:**

```typescript
// drizzle-repositories.ts — getTree()
async getTree(includeInactive = false, tx?: TransactionContext): Promise<readonly OrgUnitTreeNode[]> {
  const c = conn(this.db, tx);

  const conditions = [isNull(orgUnits.deletedAt)];
  if (!includeInactive) {
    conditions.push(eq(orgUnits.status, 'ACTIVE'));
  }

  const allUnits = await c
    .select()
    .from(orgUnits)
    .where(and(...conditions))
    .orderBy(orgUnits.nivel, orgUnits.nome);

  // ... resto do método inalterado
}
```

**Nota:** Unidades com `deletedAt IS NOT NULL` continuam sempre ocultas — o filtro `isNull(orgUnits.deletedAt)` é mantido incondicionalmente. Apenas o filtro de `status` é removido quando `includeInactive=true`.

### Acceptance Criteria

```gherkin
Cenário: DELETE retorna 200 sem erro de serialização
  Dado que existe uma unidade organizacional ativa sem filhos ativos
  Quando DELETE /api/v1/org-units/:id é chamado
  Então a response tem status 200
  E o body é { "message": "Unidade organizacional desativada." }
  E nenhum erro de serialização ocorre

Cenário: PATCH restore retorna 200 sem erro de serialização
  Dado que existe uma unidade organizacional inativa cujo pai é ativo
  Quando PATCH /api/v1/org-units/:id/restore é chamado
  Então a response tem status 200
  E o body é { "message": "Unidade organizacional restaurada." }
  E nenhum erro de serialização ocorre

Cenário: GET /tree sem parâmetro retorna apenas ativas
  Quando GET /api/v1/org-units/tree é chamado sem query params
  Então todas as unidades retornadas têm status "ACTIVE"

Cenário: GET /tree com include_inactive=true retorna ativas e inativas
  Dado que existem unidades com status "ACTIVE" e "INACTIVE" (com deletedAt IS NULL)
  Quando GET /api/v1/org-units/tree?include_inactive=true é chamado
  Então unidades com status "ACTIVE" e "INACTIVE" são retornadas
  E unidades com deletedAt != NULL não são retornadas
```
