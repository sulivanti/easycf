# ADR-002 — Campo priority Removido de incidence_rules — Resolução por Restritividade no Runtime

## Contexto

O Documento Mestre original previa um campo `priority` (integer) na tabela `incidence_rules` para resolver conflitos quando múltiplos enquadradores incidissem sobre o mesmo objeto-alvo. A proposta era que a regra com maior prioridade prevalecesse. Contudo, prioridade numérica apresenta problemas operacionais graves: (a) administradores não conseguem definir prioridade relativa de forma intuitiva, especialmente com muitos enquadradores; (b) prioridade global gera acoplamento — adicionar uma nova regra pode exigir reordenar todas as existentes; (c) não fica claro o que "prioridade maior vence" significa quando os efeitos não são mutuamente exclusivos.

## Decisão

**Campo `priority` removido da tabela `incidence_rules`.** Resolução de conflitos é feita em duas camadas:

1. **Camada 1 — Config-time (422):** UNIQUE constraint `(framer_id, target_object_id)` impede que o mesmo enquadrador incida duas vezes sobre o mesmo objeto. Conflito detectado no cadastro retorna 422 (BR-003).
2. **Camada 2 — Runtime (safety net):** Se por exceção (dados legados, race condition) dois contextos conflitantes coexistirem, o motor aplica a regra mais RESTRITIVA: HIDE > SHOW, SET_REQUIRED > SET_OPTIONAL, domínio menor (interseção) prevalece. Campos não conflitantes são mesclados via union (BR-004).

## Alternativas Consideradas

1. **Manter campo `priority` (integer):** Simples de implementar, mas gera carga cognitiva elevada para administradores e acoplamento entre regras.
2. **Prioridade por tipo de enquadrador:** Menos flexível; engessa a hierarquia de tipos.
3. **Resolução manual via UI (dialog de conflito):** UX complexa; exige decisão humana em runtime, o que é inviável para chamadas do MOD-006.

## Consequências

- **Positivas:** UX mais simples (sem campo priority); resolução determinística e previsível; sem acoplamento entre regras; safety net automático sem intervenção humana.
- **Negativas:** Administrador não pode forçar que um enquadrador "vença" explicitamente sobre outro; a regra de restritividade pode não ser intuitiva em todos os cenários.
- **Mitigações:** UX deve exibir preview do motor (dry-run) para que o configurador veja o resultado antes de publicar; documentação clara das regras de restritividade em UX-007.
