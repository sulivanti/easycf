# FR-001 — Requisitos Funcionais da Estrutura Organizacional

---

## FR-001 — CRUD de Unidades Organizacionais (N1–N4)

- **Descrição:** O sistema DEVE prover API CRUD completa para unidades organizacionais, incluindo: listagem flat com filtros e cursor-based pagination (`GET /org-units`), criação com nivel derivado (`POST /org-units`), detalhe com breadcrumb/ancestors (`GET /org-units/:id`), atualização de nome/descrição/status (`PATCH /org-units/:id`), e soft delete condicional (`DELETE /org-units/:id`).
- **Prioridade:** Must
- **Perfis impactados:** Administrador organizacional
- **Done funcional:**
  - POST /org-units cria nó com nivel calculado automaticamente
  - GET /org-units lista com cursor pagination, filtros por nivel/status — response DEVE mapear `parentId`→`parent_id`, `createdAt`→`created_at` (ISO string); não enviar raw do repository (FR-001-C03)
  - GET /org-units/:id retorna nó + ancestors[] + breadcrumb — response DEVE mapear campo a campo (sem spread `...result`), datas como ISO string (FR-001-C03)
  - PATCH /org-units/:id atualiza nome/descrição/status e **campos cadastrais** (cnpj, razao_social, filial, responsavel, telefone, email_contato) — codigo e parent_id são imutáveis — response DEVE retornar apenas os campos do schema, não o result raw (FR-001-C03)
  - DELETE /org-units/:id executa soft delete (deleted_at + status=INACTIVE) se sem filhos ativos
- **Efeito colateral?** [x] Sim *(create e link_tenant suportam Idempotency-Key)*
- **Idempotência:** required (POST /org-units aceita `Idempotency-Key`)
- **Timeline/Notifications:** yes — domain events: `org.unit_created`, `org.unit_updated`, `org.unit_deleted` (`tenant_id = SYSTEM_TENANT_ID` — cross-tenant, ADR-003; FR-001-C02)
- **Soft Limit Warning:** `POST /org-units` DEVE emitir header `X-Limit-Warning` quando total de nós ativos > 400 (ver NFR-001 §7.1)
- **Dependências (IDs):** BR-001, BR-002, BR-003, BR-004, BR-005, BR-008, BR-010, BR-011, BR-012, DATA-001, DATA-003, SEC-001, SEC-002, NFR-001

## FR-002 — Tree Query via CTE Recursivo

- **Descrição:** O sistema DEVE prover endpoint `GET /org-units/tree` que retorna a hierarquia completa N1–N5 em estrutura aninhada, usando CTE recursivo. Nós N4 incluem array de tenants vinculados (N5). Nós com `deleted_at` preenchido são excluídos. Latência alvo < 200ms para 100 nós.
- **Prioridade:** Must
- **Perfis impactados:** Administrador organizacional
- **Done funcional:**
  - Retorna árvore completa N1→N2→N3→N4→(tenants N5)
  - Cada nó inclui breadcrumb e path
  - Nós soft-deleted são excluídos da árvore
  - Sem cache — todas as chamadas executam query ao vivo (índice em `parent_id`)
- **Efeito colateral?** [ ] Não
- **Idempotência:** n/a (leitura)
- **Timeline/Notifications:** no
- **Dependências (IDs):** BR-001, DATA-001, NFR-001

## FR-003 — Vinculação N4 → Tenant (N5)

- **Descrição:** O sistema DEVE prover endpoints para vincular (`POST /org-units/:id/tenants`) e desvincular (`DELETE /org-units/:id/tenants/:tenantId`) tenants existentes (MOD-000-F07) a nós N4. Vinculação rejeita nós que não sejam N4. Tenant inexistente retorna 404. Vínculo duplicado retorna 409.
- **Prioridade:** Must
- **Perfis impactados:** Administrador organizacional
- **Done funcional:**
  - POST /org-units/:id/tenants com { tenant_id } cria vínculo (201)
  - Validação: org_unit.nivel == 4, tenant existe, par não duplicado
  - DELETE /org-units/:id/tenants/:tenantId executa soft unlink (deleted_at)
  - Domain events emitidos: org.tenant_linked, org.tenant_unlinked (`tenant_id` = UUID do tenant vinculado/desvinculado — passado via `tenantId` explícito no `createOrgUnitEvent()`; FR-001-C02)
- **Efeito colateral?** [x] Sim *(link_tenant suporta Idempotency-Key)*
- **Idempotência:** required (POST /org-units/:id/tenants aceita `Idempotency-Key`)
- **Timeline/Notifications:** yes — domain events: `org.tenant_linked`, `org.tenant_unlinked`
- **Dependências (IDs):** BR-006, BR-007, DATA-001, DATA-003, SEC-001, SEC-002

## FR-004 — Restore de Unidade Organizacional (Soft-Deleted)

- **Descrição:** O sistema DEVE prover endpoint `PATCH /org-units/:id/restore` para restaurar um nó previamente soft-deleted. A restauração só é permitida se o nó pai (`parent_id`) estiver ativo (`deleted_at IS NULL`). Nós N1 (raiz) podem ser restaurados sem restrição de pai. Após restore, `deleted_at` é setado para `NULL` e `status` volta para `ACTIVE`.
- **Prioridade:** Must
- **Perfis impactados:** Administrador organizacional
- **Done funcional:**
  - PATCH /org-units/:id/restore restaura nó soft-deleted (200)
  - Validação: nó deve ter `deleted_at IS NOT NULL`
  - Validação: pai deve estar ativo (se não for raiz)
  - `deleted_at` → NULL, `status` → ACTIVE
  - Domain event emitido: `org.unit_restored` (`tenant_id = SYSTEM_TENANT_ID` — cross-tenant, ADR-003; FR-001-C02)
- **Efeito colateral?** [x] Sim
- **Idempotência:** required (restore de nó já ativo retorna 200 sem efeito)
- **Timeline/Notifications:** yes — domain event: `org.unit_restored`
- **Dependências (IDs):** BR-009, DATA-001, DATA-003, SEC-001, SEC-002

## FR-005 — Listagem Flat com Cursor-Based Pagination e Filtros

- **Descrição:** O sistema DEVE prover endpoint `GET /api/v1/org-units` para listagem flat (não hierárquica) das unidades organizacionais com cursor-based pagination. Suporta filtros por `nivel` (1..4), `status` (ACTIVE/INACTIVE), `parent_id` (filhos diretos de um nó), e busca textual por `nome` e `codigo`. A paginação segue o padrão DOC-ARC-001 com `cursor` + `limit` (default 20, max 100).
- **Prioridade:** Must
- **Perfis impactados:** Administrador organizacional
- **Done funcional:**
  - GET /api/v1/org-units retorna lista flat com cursor-based pagination
  - Filtros: `?nivel=2`, `?status=ACTIVE`, `?parent_id=uuid`, `?search=termo`
  - Resposta inclui `data[]`, `pagination.next_cursor`, `pagination.has_more`
  - Cada item inclui `id`, `codigo`, `nome`, `nivel`, `status`, `parent_id`, `created_at`
  - Nós com `deleted_at` preenchido excluídos por default (filtro `?status=INACTIVE` para incluir)
- **Efeito colateral?** [ ] Não
- **Idempotência:** n/a (leitura)
- **Timeline/Notifications:** no
- **Dependências (IDs):** BR-001, DATA-001, NFR-001

## FR-006 — Campos Cadastrais nas Unidades Organizacionais (FR-001-M01)

- **Descrição:** Os endpoints CRUD de org_units DEVEM aceitar e retornar 6 campos cadastrais opcionais: `cnpj`, `razao_social`, `filial`, `responsavel`, `telefone`, `email_contato`. Estes campos suportam o DetailPanel e FormPanel definidos em UX-001-M01.
- **Prioridade:** Must
- **Perfis impactados:** Administrador organizacional
- **Done funcional:**
  - POST /org-units aceita os 6 campos opcionais no body
  - PATCH /org-units/:id aceita os 6 campos como partial update (omitido = não altera)
  - GET /org-units/:id retorna os 6 campos (null se não preenchidos)
  - GET /org-units/tree NÃO inclui campos cadastrais (performance)
  - GET /org-units (lista flat) NÃO inclui campos cadastrais
- **Validação dos campos:**
  - `cnpj`: varchar max 18, formato livre
  - `razao_social`: varchar max 300
  - `filial`: varchar max 100
  - `responsavel`: varchar max 200
  - `telefone`: varchar max 20
  - `email_contato`: varchar max 254, validação básica (contém @)
- **Nenhum campo é obrigatório. Nenhum participa de regras de unicidade.**
- **Dependências (IDs):** DATA-001-M01, UX-001-M01

### Critérios de aceite (Gherkin)

```gherkin
Funcionalidade: Requisitos Funcionais da Estrutura Organizacional

Cenário: Criar nó raiz N1
  Dado que POST /api/v1/org-units é chamado com { nome: "Grupo Alpha", codigo: "GC-001" }
  E não há parent_id no body
  Então o status deve ser 201 e nivel deve ser 1

Cenário: Tree query retorna hierarquia completa N1–N5
  Dado que existe a árvore: GC-001 → UN-SP → MA-TECH → SU-DEV → (tenant: sp01)
  Quando GET /api/v1/org-units/tree é chamado
  Então a resposta deve ser estrutura aninhada com N1 na raiz
  E nós N4 devem ter array "tenants" com os tenants vinculados

Cenário: Vincular tenant existente a nó N4
  Dado que existe um nó N4 e um tenant existente
  Quando POST /api/v1/org-units/:id/tenants com { tenant_id }
  Então deve retornar 201 e o evento org.tenant_linked é emitido

Cenário: Soft delete emite domain event
  Dado que existe um nó N3 sem filhos ativos
  Quando DELETE /api/v1/org-units/:id é chamado
  Então o status deve ser 200 e o evento org.unit_deleted é emitido
  E o campo deleted_at deve estar preenchido

Cenário: Idempotency-Key previne duplicação na criação
  Dado que POST /api/v1/org-units já foi chamado com Idempotency-Key "key-abc"
  Quando o mesmo POST é chamado novamente com Idempotency-Key "key-abc"
  Então deve retornar 201 com o mesmo recurso sem criar duplicado

Cenário: Restore de nó soft-deleted com pai ativo
  Dado que existe um nó N3 soft-deleted cujo pai N2 está ativo
  Quando PATCH /api/v1/org-units/:id/restore é chamado
  Então deve retornar 200 e deleted_at deve ser NULL
  E o evento org.unit_restored deve ser emitido

Cenário: Restore bloqueado quando pai está inativo
  Dado que existe um nó N3 soft-deleted cujo pai N2 também está soft-deleted
  Quando PATCH /api/v1/org-units/:id/restore é chamado
  Então deve retornar 422: "Não é possível restaurar: o nó pai está inativo."

Cenário: Desvincular tenant de nó N4
  Dado que existe um nó N4 com tenant vinculado
  Quando DELETE /api/v1/org-units/:id/tenants/:tenantId
  Então deve retornar 200 e o evento org.tenant_unlinked é emitido

Cenário: Listagem flat com cursor-based pagination
  Dado que existem 25 nós na tabela org_units
  Quando GET /api/v1/org-units?limit=10 é chamado
  Então a resposta deve conter data[] com 10 itens
  E pagination.has_more deve ser true
  E pagination.next_cursor deve estar preenchido

Cenário: Filtro por nivel na listagem flat
  Dado que existem nós de nivel 1 a 4
  Quando GET /api/v1/org-units?nivel=2 é chamado
  Então a resposta deve conter apenas nós com nivel=2

Cenário: Busca textual por nome na listagem flat
  Dado que existem nós com nomes variados
  Quando GET /api/v1/org-units?search=Tech é chamado
  Então a resposta deve conter nós cujo nome ou codigo contém "Tech"

Cenário: Warning header emitido ao criar nó acima de 80% do soft limit
  Dado que existem 401 nós ativos na tabela org_units
  Quando POST /api/v1/org-units é chamado com dados válidos
  Então deve retornar 201
  E o response header X-Limit-Warning deve conter "org_units_count=402, soft_limit=500, threshold=80%"

Cenário: Criação permitida mesmo acima do soft limit de 500
  Dado que existem 500 nós ativos na tabela org_units
  Quando POST /api/v1/org-units é chamado com dados válidos
  Então deve retornar 201
  E o response header X-Limit-Warning deve conter "exceeded=true"

Cenário: Parent_id imutável via PATCH
  Dado que existe um nó com parent_id="uuid-pai"
  Quando PATCH /api/v1/org-units/:id com { parent_id: "novo-uuid" }
  Então deve retornar 422: "O campo 'parent_id' é imutável após criação."
```
