# Emenda: DATA-000-C02

- **Documento base:** [DATA-000](../../requirements/data/DATA-000.md)
- **estado_item:** DRAFT
- **Natureza:** C (Correcao)
- **Data:** 2026-03-25
- **owner:** ECF Core Team
- **Motivacao:** Login e todos os 13 use-cases MOD-000 falham ao inserir na tabela `domain_events` porque `tenant_id` e passado como string vazia (`''`), violando a constraint `uuid NOT NULL` do PostgreSQL. Campos opcionais (`causation_id`, `dedupe_key`) tambem sao serializados incorretamente como `undefined` em vez de `NULL`.
- **rastreia_para:** spec-fix-domain-events-tenant-id, FR-000, DATA-003

---

## Detalhamento

### 1. Problema observado

Ao executar `POST /api/v1/auth/login` com credenciais validas, a API retorna erro 500 com:

```
Failed query: insert into "domain_events" (...) values (default, $1, ...)
params: ,session,2143faaf-...,auth.login_success,...
```

O parametro `$1` (`tenant_id`) e string vazia ŌĆö PostgreSQL rejeita porque a coluna e `uuid NOT NULL`.

### 2. Causa-raiz

O codegen (commit `fed0682`) gerou todos os use-cases com placeholder `tenantId: ''` e comentario `// resolved by caller/middleware`. O middleware/caller de resolucao nunca foi implementado.

**13 use-cases afetados:**

|----------|-----------|-------------------|
| `auth/login.use-case.ts` | `auth.login_success`, `session.created` | Pre-auth (sem tenant) |
| `auth/logout.use-case.ts` | `auth.logout`, `session.revoked` | Pos-auth (session.tenantId) |
| `auth/refresh-token.use-case.ts` | `auth.token_reuse_detected` | Pos-auth (session.tenantId) |

| `auth/update-profile.use-case.ts` | `user.profile_updated` | Pos-auth (session.tenantId) |
| `users/create-user.use-case.ts` | `user.created` | Pos-auth (input.tenantId) |

### 3. Correcao ŌĆö 3 acoes

#### Acao 1: Constante `SYSTEM_TENANT_ID`

Adicionar em `foundation-events.ts`:

```typescript
/** UUID fixo para eventos que ocorrem fora de contexto de tenant (login, forgot-password, reset-password) */
export const SYSTEM_TENANT_ID = '00000000-0000-0000-0000-000000000000';
```

**Justificativa:** Eventos pre-autenticacao genuinamente nao tem tenant. O `SYSTEM_TENANT_ID` e padrao em sistemas multi-tenant para operacoes de sistema. A coluna `domain_events.tenant_id` NAO tem FK para `tenants`, entao basta ser UUID valido.

#### Acao 2: Fallback na factory `createFoundationEvent()`

```typescript
export function createFoundationEvent(params: {
  tenantId?: string;  // opcional ŌĆö usa SYSTEM_TENANT_ID como fallback
  // ... demais campos inalterados
}): DomainEventBase {
  return {
    tenantId: params.tenantId || SYSTEM_TENANT_ID,
    // ... demais campos inalterados
    causationId: params.causationId ?? null,  // explicito null (nao undefined)
    dedupeKey: params.dedupeKey ?? null,      // explicito null (nao undefined)
  };
}
```

#### Acao 3: Campos nullable explicitamente `null` no repository

Em `DrizzleDomainEventRepository.create()` e `createMany()`:

```typescript
await c.insert(domainEvents).values({
  // ...
  createdBy: event.createdBy ?? null,
  causationId: event.causationId ?? null,   // era undefined ŌåÆ agora null
  dedupeKey: event.dedupeKey ?? null,       // era undefined ŌåÆ agora null
});
```

### 4. Regra para codegen futuro

**DATA-000 ┬¦8 Nota adicional (proposta):**

> Eventos pre-autenticacao (login, forgot-password, reset-password) DEVEM usar a constante `SYSTEM_TENANT_ID` (`00000000-0000-0000-0000-000000000000`) como `tenant_id`. Eventos pos-autenticacao DEVEM propagar o `tenantId` da sessao ativa (`request.session.tenantId`). O codegen NAO DEVE gerar `tenantId: ''` como placeholder.

### 5. Criterios de aceitacao

- **AC-001:** `POST /api/v1/auth/login` com credenciais validas retorna 200 com tokens (sem erro 500).
- **AC-002:** Tabela `domain_events` contem registro com `event_type = 'auth.login_success'` e `tenant_id = '00000000-0000-0000-0000-000000000000'`.
- **AC-003:** Nenhum registro em `domain_events` tem `tenant_id = ''`.
- **AC-004:** Campos `causation_id` e `dedupe_key` sao `NULL` (nao string vazia) quando nao fornecidos.
