# UX-001 — UX e Jornadas (Gestão de Usuários)

---

## Jornada 1 — Listagem de Usuários (UX-USR-001)

### Happy path

1. Admin acessa /usuarios → skeleton na tabela
2. Dados carregam → tabela com nome, perfil, status, data
3. Admin filtra por status/perfil ou busca por texto → tabela atualiza com overlay leve
4. Admin clica "Carregar mais" → registros adicionados ao final (append)

### Alternativas/erros

- Sem scope `users:user:read` → redirect /dashboard + toast "Sem permissão para acessar esta seção."
- GET 500 → toast + correlationId + estado de erro com botão "Tentar novamente"
- Lista vazia → ilustração + "Nenhum usuário encontrado" + CTA "Criar primeiro usuário" (se scope write)

### Estados

| Estado | Componente | Comportamento |
|---|---|---|
| **Loading** | Skeleton | N linhas placeholder animadas na tabela |
| **Empty** | Empty state | Ilustração + texto + CTA condicional |
| **Error** | Error state | Toast + botão "Tentar novamente" |
| **Refetch** | Overlay | Overlay leve sobre tabela durante filtro/busca |

| `filter` | Filtrar | query | collection | `users_list` (GET /api/v1/users?status=...) | — |

## Jornada 2 — Formulário de Cadastro (UX-USR-002)

### Happy path (modo convite)

1. Admin acessa /usuarios/novo → select de perfil carrega com skeleton
2. Preenche nome, e-mail, seleciona perfil
3. Clica "Criar usuário" → botão isLoading
4. 201 → toast "Usuário criado com sucesso. Convite enviado." → redirect /usuarios (1.5s)

### Happy path (modo senha)

1. Admin clica aba "Senha Temporária" → formulário reseta
2. Preenche todos os campos + senha + confirmação
3. Indicador de força atualiza em tempo real
4. Clica "Criar usuário" → 201 → toast "Usuário criado com sucesso." → redirect

### Alternativas/erros

- Sem scope `users:user:write` → redirect /usuarios + toast "Sem permissão para criar usuários."
- Roles 500 → toast + correlationId + campo com retry inline
- E-mail duplicado (409) → inline no campo e-mail: "Este e-mail já está cadastrado no sistema."
- Campos inválidos (422) → inline por campo + foco no primeiro erro
- Servidor (500) → toast + correlationId, formulário preservado

### Estados

| Estado | Componente | Comportamento |
|---|---|---|
| **Loading (roles)** | Skeleton | Select de perfil com skeleton |
| **Loading (submit)** | isLoading | Botão submit com spinner, disabled |
| **Error (roles)** | Inline error | Mensagem + botão retry no campo |
| **Error (submit)** | Toast ou inline | Inline para 409/422, toast para 500 |

### Ações (catálogo UX-010)

| `cancel` | Cancelar | view | single | — (navegação para /usuarios) | — |

| Toast sucesso (senha) | "Usuário criado com sucesso." |
| Toast sem permissão | "Sem permissão para criar usuários." |

## Jornada 3 — Convite e Ativação (UX-USR-003)

### Happy path

1. Admin acessa /usuarios/:id/convite → skeleton no card de status
2. Dados carregam → card com badge de status + nome no título
3. Status PENDING → botão "Reenviar convite" habilitado
4. Admin clica → botão isLoading → 200 → toast "Convite reenviado com sucesso." → countdown 60s

### Alternativas/erros

- Sem scope `users:user:read` → redirect /usuarios + toast
- 404 → mensagem inline "Usuário não encontrado." + sem botão de reenvio
- Status ACTIVE → badge verde, botão não renderizado
- Status BLOCKED → badge vermelho, botão desabilitado + tooltip
- Status INACTIVE → badge cinza, botão não renderizado
- Token expirado (PENDING) → alerta âmbar + botão destacado
- Reenvio 500 → toast + correlationId, botão habilitado sem cooldown
- Reenvio 422 (status mudou) → toast genérico + reload da tela

### Estados

| Estado | Componente | Comportamento |
|---|---|---|
| **Loading** | Skeleton | Card de status + breadcrumb sem nome |
| **PENDING** | Badge âmbar | "Aguardando ativação" + botão habilitado |
| **ACTIVE** | Badge verde | "Ativo" + sem botão |
| **BLOCKED** | Badge vermelho | "Bloqueado" + botão desabilitado + tooltip |
| **INACTIVE** | Badge cinza | "Inativo" + sem botão |
| **Cooldown** | Countdown | Botão desabilitado, "Aguarde X segundos..." |
| **Error** | Inline/Toast | 404 inline, 500 toast |

### Ações (catálogo UX-010)

| action_id | label_pt | kind | scope | endpoint | event_type |
|---|---|---|---|---|---|
| `view` | Ver status do convite | query | single | `users_get` (GET /api/v1/users/:id) | — |
| `submit` | Reenviar convite | command | single | `users_invite_resend` (POST /invite/resend) | `user.invite_resent` |

### Copy (mensagens)

| Contexto | Mensagem |
|---|---|

| PENDING texto | "Convite enviado. O usuário ainda não ativou a conta." |
| Token expirado alerta | "O link de ativação expirou. Reenvie o convite." |
| ACTIVE texto | "O usuário ativou a conta com sucesso." |
| Tooltip BLOCKED | "Desbloqueie o usuário antes de reenviar o convite." |

## Acessibilidade (todas as telas)

- Formulários com labels associados via `htmlFor`
- Mensagens de erro associadas por `aria-describedby`
- Foco movido para primeiro campo com erro após submit (422)
- Modais com trap de foco e fechamento via ESC
- Botões com `aria-disabled` e `aria-busy` durante isLoading
- Contraste mínimo WCAG AA em badges de status e alertas

## Telemetria (UX Action Events)

Para cada ação, emitir (DOC-DEV-001 §4.5 UX-010):

- `ux.action.requested` — ao iniciar a ação
- `ux.action.succeeded` — ao receber resposta de sucesso
- `ux.action.failed` — ao receber erro

**Payload mínimo:** `correlation_id`, `screen_id`, `tenant_id`, `action_id`, `timestamp`

## Error Recovery Flows (INT-001, NFR-001 §10)

### Mapeamento de Erros por Tela (RFC 9457 — INT-001)

| Tela | HTTP Status | Componente de UI | Ação de recovery | Ref |
|---|---|---|---|---|
| UX-USR-001 | 401 | Modal sessão expirada | Redirect para login | SEC-001 §1 |
| UX-USR-001 | 500 | Toast + correlationId + botão retry | Retry manual (re-fetch lista) | BR-006, NFR-001 §10 |
| UX-USR-002 | 409 | Inline no campo e-mail | Admin corrige e-mail e re-submete | BR-002, BR-006 |
| UX-USR-002 | 422 | Inline por campo + foco no primeiro erro | Admin corrige campos | BR-006, NFR-001 §4 |
| UX-USR-002 | 500 | Toast + correlationId. Formulário preservado. | Admin re-submete sem perder dados | BR-006, NFR-001 §10 |
| UX-USR-003 | 404 | Inline "Usuário não encontrado" | Sem recovery — botão de reenvio oculto | FR-003 |
| UX-USR-003 | 422 | Toast genérico + reload | Status mudou entre load e ação | FR-003, BR-006 |
| UX-USR-003 | 500 | Toast + correlationId. Botão reabilitado (sem cooldown). | Admin re-tenta | BR-006, NFR-001 §10 |

### Resiliência de Navegação (NFR-001 §10)

- **Request pendente + navegação:** AbortController cancela request. Nenhum toast pós-navegação.
- **Cooldown sobrevive a erro:** Se cooldown está ativo e ocorre erro em outra ação, cooldown NÃO é resetado (NFR-001 §10).
- **Erro de rede:** Toast genérico "Sem conexão com o servidor." + correlationId (se disponível).
- **Timeout (15s):** Cancelar request, toast de timeout, botão habilitado para retry.

## View-Model Mapping (DATA-001)

O mapeamento entre DTOs da API e view-model de renderização está documentado em DATA-001 §View-Model. As jornadas acima consomem:

| Jornada | DTO consumido | View-Model | Regras de transformação |

## Segurança na UX (SEC-001 §9-10)

| Aspecto | Regra na jornada | Ref |
|---|---|---|
| **PII em toasts** | Nenhum toast exibe e-mail. Copy fixa sem interpolação de PII. | BR-002, SEC-001 §5 |
| **Scope enforcement** | Guards de rota são UX. Enforcement real no backend. | SEC-001 §2, DOC-FND-000 §2.3 |
| **Enumeração de e-mail** | Erro 409 usa mensagem genérica inline. Não revela e-mail existente. | SEC-001 §10 |
| **XSS** | React auto-escape. Nunca `dangerouslySetInnerHTML`. | NFR-001 §11 |
