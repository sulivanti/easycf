## Detalhamento

### UX-001 — Modo Editar (InlineEditCard) — Campo `status`

**Antes (UX-001-M02):**

```markdown
| `status` | Toggle | Toggle ativo/inativo — PENDENTE-010 | — |
```

**Depois:**

```markdown
| `status` | ReadOnlyBadge + BtnDesativar | Badge "ATIVO"/"INATIVO" (read-only) + botão "Desativar" (danger secondary) que abre DeactivateModal. NÃO é toggle editável — usa fluxo DELETE existente (BR-005). | — |
```

### Comportamento detalhado

1. **Unidade ATIVA:** Badge verde "ATIVO" (read-only) + botão "Desativar" ao lado (secondary danger: border #E74C3C, text #E74C3C, hover fill #E74C3C text #FFF)
2. **Ao clicar "Desativar":** Abre o `DeactivateModal` existente (UX-001-M01 §DeactivateModal) com contagem de subdivisões/colaboradores
3. **Fluxo DELETE:** `DELETE /api/v1/org-units/:id` → soft delete (status=INACTIVE, deleted_at=now()) → modal fecha → DetailPanel atualiza → badge muda para "INATIVO"
4. **Unidade INATIVA:** Badge cinza "INATIVO" (read-only) + botão "Restaurar" ao lado (secondary: border #27AE60, text #27AE60)
5. **Ao clicar "Restaurar":** `PATCH /api/v1/org-units/:id/restore` diretamente (sem modal) → badge muda para "ATIVO"

### Layout do campo status no InlineEditCard

### Impacto

- **UX-001 §Modo Editar:** Substituir linha `status Toggle` por `status ReadOnlyBadge + BtnDesativar`
- **Sem mudança no backend:** Nenhum endpoint novo. Usa DELETE e PATCH /restore existentes.
- **Sem mudança no modelo de dados:** BR-005 (soft delete condicional) e BR-009 (restore condicional) permanecem inalterados.
- **DeactivateModal:** Reutiliza o modal existente definido em UX-001-M01.
