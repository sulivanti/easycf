### Alteracoes neste pilar

#### 1. Novos itens de Done Funcional (Edicao)

Adicionar ao done criteria de FR-002:

- [ ] Acesso a /usuarios/:id carrega dados via GET /api/v1/users/:id com skeleton
- [ ] Campos editaveis: nome completo, perfil (roleId). E-mail readonly.
- [ ] Toggle de status visivel se scope `users:user:write` (ACTIVE/BLOCKED/INACTIVE)
- [ ] Botao "Resetar Senha" (secondary) se scope `users:user:write` + status ACTIVE
- [ ] Submit via PATCH /api/v1/users/:id -> toast "Usuario atualizado com sucesso." -> redirect /usuarios
- [ ] Erro 404 -> redirect /usuarios + toast "Usuario nao encontrado."
- [ ] Erro 422 -> inline por campo
- [ ] Erro 500 -> toast + correlationId, formulario preservado

#### 2. Novos operationIds Consumidos

| operationId | Acao | Metodo |
|---|---|---|
| `users_get` | Carregar dados do usuario para edicao | GET /api/v1/users/:id |
| `users_update` | Atualizar usuario | PATCH /api/v1/users/:id |

#### 3. Novos cenarios Gherkin

```gherkin
Funcionalidade: Formulario de Edicao — FR-002-M01

  Cenario: Edicao de usuario — happy path
    Dado que o admin com scope "users:user:read" acessa /usuarios/:id
    Entao GET /api/v1/users/:id e chamado
    E o formulario e preenchido com os dados retornados
    E o campo e-mail esta readonly

  Cenario: Salvar alteracoes com sucesso
    Dado que o admin editou nome e/ou perfil
    Quando clica "Salvar Alteracoes"
    Entao PATCH /api/v1/users/:id e chamado com os campos alterados
    E ao receber 200, toast exibe "Usuario atualizado com sucesso."
    E redireciona para /usuarios

  Cenario: Toggle de status visivel por scope
    Dado que o admin com scope "users:user:write" acessa /usuarios/:id
    Entao o toggle de status e visivel (ACTIVE/BLOCKED/INACTIVE)

  Cenario: Toggle de status oculto sem scope
    Dado que o admin sem scope "users:user:write" acessa /usuarios/:id
    Entao o toggle de status nao e exibido

  Cenario: Botao Resetar Senha visivel para usuario ativo
    Dado que o admin com scope "users:user:write" acessa /usuarios/:id de um usuario ACTIVE
    Entao o botao "Resetar Senha" (secondary) e visivel

  Cenario: Usuario nao encontrado (404)
    Dado que o admin acessa /usuarios/:id com ID inexistente
    Quando GET /api/v1/users/:id retorna 404
    Entao redireciona para /usuarios
    E toast exibe "Usuario nao encontrado."

  Cenario: Erro de validacao (422)
    Dado que PATCH /api/v1/users/:id retorna 422 com extensions.invalid_fields
    Entao cada campo invalido exibe mensagem inline
    E foco e movido para o primeiro campo com erro

  Cenario: Erro de servidor (500)
    Dado que PATCH /api/v1/users/:id retorna 500
    Entao toast exibe mensagem de erro + correlationId
    E formulario e preservado para retry
```
