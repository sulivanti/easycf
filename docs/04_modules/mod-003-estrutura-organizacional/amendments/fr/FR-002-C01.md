## Detalhamento

### Correção 1 — Registro de rotas departments em `src/index.ts`

**Problema:** O arquivo `apps/api/src/index.ts` não importa nem registra `departmentsRoutes`. As 6 rotas de departments existem no código mas são inacessíveis em runtime.

**Correção:** Adicionar import e registro no entry point, seguindo o padrão existente dos outros módulos:

```typescript
// No bloco de imports (junto com orgUnitsRoutes):
import { departmentsRoutes } from './modules/departments/presentation/index.js';

// No bloco de registro de rotas (após orgUnitsRoutes, dentro do comentário MOD-003):
// MOD-003 Departments — relative paths
await app.register(departmentsRoutes, { prefix: '/api/v1/departments' });
```

**Prefixo:** `/api/v1/departments` — conforme FR-007 §Endpoints (POST `/api/v1/departments`, GET `/api/v1/departments`, etc.).

### Correção 2 — Response shape da lista departments

**Problema:** O handler `GET /departments` (linha 104-110 de `departments.route.ts`) envia:

```json
{
  "data": [...],
  "pagination": {
    "next_cursor": "...",
    "has_more": true
  }
}
```

Mas o schema Zod `paginatedResponse(departmentListItem)` espera formato flat:

```json
{
  "data": [...],
  "next_cursor": "...",
  "has_more": true
}
```

Isso causa falha de serialização em runtime (`Response doesn't match schema`).

**Correção:** Alterar o handler para enviar formato flat conforme `paginatedResponse`:

```typescript
// ANTES (INCORRETO):
return reply.send({
  data: result.data,
  pagination: {
    next_cursor: result.nextCursor,
    has_more: result.hasMore,
  },
});

// DEPOIS (CORRETO):
return reply.send({
  data: result.data,
  next_cursor: result.nextCursor ?? null,
  has_more: result.hasMore,
});
```

### Correção 3 — Repository Drizzle + DI container

**Problema:** Não existia implementação concreta do `DepartmentRepository` (Drizzle) e os 6 use cases de departments não estavam registrados no DI container (`src/plugins/di.ts`). Em runtime, `request.dipiContainer.createDepartmentUseCase` seria `undefined`.

**Correção:**
1. Criar `apps/api/src/modules/departments/infrastructure/drizzle-repositories.ts` — implementação `DrizzleDepartmentRepository` com 6 métodos (findById, findByCodigo, list, countActiveByTenant, create, update)
2. Adicionar imports + instanciação no `src/plugins/di.ts` — seção 4b após org-units
3. Registrar os 6 use cases no container: createDepartmentUseCase, updateDepartmentUseCase, deleteDepartmentUseCase, restoreDepartmentUseCase, getDepartmentUseCase, listDepartmentsUseCase

### Correção 4 (menor) — PATCH usa `createdBy` em vez de `updatedBy`

**Problema:** O handler `PATCH /departments/:id` (linha 173) passa `createdBy: user.userId` ao use case. Semanticamente deveria indicar quem está atualizando.

**Correção:** Adiada — o use case espera `createdBy` no input type. Mudança requer alteração no contrato do use case.
