### 2.4 Happy Path

1. Operador acessa /casos/:id -- skeleton carrega (header, barra de progresso, painel principal)
2. GET /cases/:id retorna dados completos -- header preenche com código, ciclo, status OPEN, estágio atual
3. Barra de progresso exibe macroetapas com a atual destacada
4. Operador verifica aba "Gates" -- gates PENDING exibidos com botão "Resolver"
5. Aprovador resolve gate APPROVAL com decisão APPROVED + parecer -- card muda para badge verde "Aprovado"
6. Todos os gates required RESOLVED -- botões de transição habilitados automaticamente
7. Operador clica "Avançar" -- modal de confirmação -- confirma -- POST /transition -- estágio atualiza
8. Timeline exibe novo item STAGE_CHANGE no topo

### 2.5 Alternativas e Erros

| Cenário | Comportamento |
|---|---|
| Gate APPROVAL resolvido com REJECTED | Card muda para badge vermelho "Reprovado" + parecer visível; transições permanecem bloqueadas |
| Transição bloqueada por gate pending | Botão desabilitado + tooltip: "Gate '{nome}' ainda pendente" |
| Transição bloqueada por papel | Botão desabilitado + tooltip: "Seu papel não autoriza esta transição" |
| Papel required sem atribuição | Card do papel com borda vermelha + label "Obrigatório -- não atribuído"; transições desabilitadas |
| Caso ON_HOLD | Banner amarelo "Caso em espera"; transições e gates bloqueados; botão "Retomar" no header |
| Caso COMPLETED | Badge azul "Concluído" + data; painel readonly exceto comentários |
| Caso CANCELLED | Badge cinza "Cancelado"; painel readonly total (sem comentários) |
| Transição com evidence_required | Mini-form inline: textarea (nota) ou upload de arquivo; botão "Confirmar" só habilita após preenchimento |
| Gate waive | Aviso: "Esta ação será auditada. Informe o motivo." + campo min 20 chars |
| Reatribuição | Autocomplete de usuários → modal: "Substituir {atual} por {novo} no papel {papel}?" |
| Erro 422 do motor | Toast com detail da RFC 9457 + correlationId copiável |
| Erro 403 | Toast: "Você não tem permissão para esta ação." |
| Erro 500 | Toast: "Erro inesperado. Tente novamente." + correlationId |

### 2.6 Estados de UI

| Estado | Comportamento |
|---|---|
| **Loading** | Skeleton no header, barra de progresso, e painel principal (skeleton por aba ao trocar) |
| **Empty (gates)** | "Este estágio não possui gates." (quando estágio sem gates) |
| **Empty (assignments)** | "Nenhum responsável atribuído." + botão "Atribuir" se scope `assign` |
| **Empty (timeline)** | "Nenhum evento registrado ainda." |
| **Error (load)** | Toast: "Não foi possível carregar o caso." + correlationId; botão "Tentar novamente" |
| **Error (action)** | Toast inline com detail RFC 9457; estado anterior mantido |
| **Optimistic update** | Comentários e reatribuições: atualização imediata na UI + rollback se erro |

### 3.4 Happy Path

1. Operador acessa /casos -- skeleton na tabela
2. GET /cases retorna lista paginada -- tabela preenche com casos do tenant
3. Operador aplica filtro "Minha responsabilidade" -- URL atualiza (?assigned_to_me=true)

## 7. Telemetria (UI Action Envelope)

Ambas as telas DEVEM emitir telemetria conforme DOC-ARC-003, utilizando o pacote `@easycf/ui-telemetry`:

| Campo | Valor |
|---|---|
| `screen_id` | UX-CASE-001 ou UX-CASE-002 |
| `action` | action_id selecionada (ex: `transition`, `approve`, `filter`) |
| `entity_type` | `case_instance`, `gate_instance`, `case_assignment`, `case_event` |
| `entity_id` | ID do caso (ou gate, assignment) |
| `tenant_id` | Do JWT |
| `operation_id` | operationId do OpenAPI (ex: `cases_transition`) |
| `correlation_id` | UUID v4 propagado via X-Correlation-ID |
| `status` | `requested` / `succeeded` / `failed` |
| `duration_ms` | Tempo da ação |
| `http_status` | Código HTTP da resposta |
| `problem_type` | Tipo de erro RFC 9457 (se erro) |
