# Emenda: FR-000-C04

- **Documento base:** [FR-000](../../requirements/fr/FR-000.md)
- **estado_item:** MERGED
- **Natureza:** C (Correção)
- **Data:** 2026-03-25
- **owner:** arquitetura
- **Motivação:** As rotas de auth (FR-001 login, FR-003 refresh, FR-004 /me) faziam `reply.send(result)` enviando objetos camelCase do use-case diretamente, mas os schemas Zod dos DTOs exigem snake_case. O `serializerCompiler` do Fastify rejeita com "Response doesn't match the schema", causando HTTP 500 em todas essas rotas. Adicionalmente, a rota `/refresh` reutiliza o schema `loginResponse` que exige campo `user`, mas `RefreshTokenUseCase` não retorna `user` — o que causa TypeError em runtime.
- **rastreia_para:** FR-001, FR-003, FR-004, spec-fix-auth-route-response-mapping

---

## Detalhamento

### DEF-001: Mismatch camelCase vs snake_case em 5 handlers

**Rotas afetadas:**

| Rota | Schema Zod | Campo(s) problemático(s) |
|------|-----------|--------------------------|
| `POST /auth/login` (success) | `loginResponse` | `accessToken`→`access_token`, `refreshToken`→`refresh_token`, `expiresIn`→`expires_in`, `user.fullName`→`user.full_name` + falta `token_type: 'Bearer'` |
| `POST /auth/login` (MFA) | `loginMfaResponse` | `mfaRequired`→`mfa_required`, `tempToken`→`temp_token`, `expiresIn`→`expires_in` |
| `POST /auth/refresh` | `loginResponse` | Mesmos do login success |
| `GET /auth/me` | `profileResponse` | `fullName`→`full_name`, `avatarUrl`→`avatar_url`, `activeTenantId`→`active_tenant_id` |
| `PATCH /auth/me` | `profileResponse` | Mesmos do GET /me |

**Causa raiz:** Os handlers faziam `return reply.status(200).send(result)` passando o objeto cru do use-case (camelCase) ao Fastify. O `serializerCompiler` Zod valida contra o schema declarado em `response: { 200: ... }` e rejeita keys desconhecidas.

**Correção:** Mapeamento explícito camelCase→snake_case em cada handler (Anti-Corruption Layer na camada de apresentação).

### DEF-002: `POST /auth/refresh` referencia `result.user` inexistente

**Retorno do `RefreshTokenUseCase`:**
```typescript
interface RefreshTokenOutput {
  readonly tokenPair: TokenPair;  // { accessToken, refreshToken, expiresIn }
  readonly sessionId: string;
  // NÃO tem "user"
}
```

**Retorno do `LoginUseCase` (para comparação):**
```typescript
interface LoginOutput {
  readonly tokenPair: TokenPair;
  readonly user: { id, email, fullName, status };
  readonly sessionId: string;
  // TEM "user"
}
```

A rota `/refresh` declara `response: { 200: loginResponse }` que exige o campo `user`. Porém o use-case não retorna `user`. O mapeamento copiado do login referencia `result.user.id`, `result.user.email` etc., causando `TypeError: Cannot read properties of undefined`.

**Correção recomendada:** Criar schema `refreshResponse` separado sem campo `user`:

```typescript
// auth.dto.ts — NOVO schema
export const refreshResponse = z.object({
  access_token: z.string(),
  refresh_token: z.string(),
  token_type: z.literal('Bearer'),
  expires_in: z.number().int(),
  // Sem "user" — refresh troca tokens, não busca perfil
});
```

**Justificativa da abordagem (schema separado):**
1. **Princípio da menor surpresa** — refresh troca tokens, não busca perfil. O contrato deve refletir a operação real.
2. **Performance** — evita query extra ao banco em cada refresh (operação frequente, a cada expiração do access_token).
3. **Menor blast radius** — só altera `auth.dto.ts` (novo schema) + schema na rota. Não toca no use-case.
4. **Frontend já tem o user** — recebeu no login ou via GET /me.

**Alternativa futura:** Se o frontend necessitar dados frescos do user no refresh, evoluir `RefreshTokenUseCase` para retornar `user` — como decisão deliberada via amendment separado.
