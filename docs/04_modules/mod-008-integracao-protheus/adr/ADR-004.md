# ADR-004 — Credenciais Criptografadas em Repouso com AES-256 via Secret do Ambiente

## Contexto

O campo `auth_config` da tabela `integration_services` armazena credenciais de autenticação para chamadas HTTP ao Protheus/TOTVS (username/password para BASIC, tokens para BEARER, client_id/client_secret para OAUTH2). Essas credenciais precisam ser protegidas em repouso no banco de dados. Três estratégias foram avaliadas.

## Decisão

**Criptografar `auth_config` com AES-256-GCM usando uma chave derivada de um secret do ambiente (env var).** O campo é armazenado como JSONB criptografado no PostgreSQL. A descriptografia ocorre exclusivamente no worker BullMQ em runtime, imediatamente antes da chamada HTTP. O valor descriptografado não é cacheado e é descartado após a chamada. Endpoints GET NUNCA retornam o valor real — sempre `"***"` (BR-002).

## Alternativas Consideradas

1. **Vault externo (HashiCorp Vault, AWS Secrets Manager):**
   - Prós: Gerenciamento centralizado de secrets; rotação automática de chaves; audit trail nativo; separação de responsabilidades (app não conhece a chave).
   - Contras: Dependência de infraestrutura externa; latência adicional por chamada (lookup no Vault antes de cada execução HTTP); custo operacional (deploy, configuração, HA do Vault); complexidade para MVP — overengineering para Wave 4.

2. **Environment variables diretamente (sem criptografia no banco):**
   - Prós: Simplicidade máxima; sem overhead de criptografia.
   - Contras: Credenciais em texto plano no banco; qualquer dump de banco ou acesso DBA expõe todas as credenciais; violaria compliance e BR-002; configuração por serviço de destino inviável via env vars (cada serviço tem credenciais diferentes).

3. **AES-256-GCM via secret do ambiente (escolhido):**
   - Prós: Credenciais nunca em texto plano no banco; secret rotation via redeploy; custo zero de infraestrutura adicional; GCM fornece autenticação do ciphertext (detecta tamper); compatível com evolução para Vault futuro (trocar o key provider sem mudar o schema).
   - Contras: Chave de criptografia em env var — se comprometida, todas as credenciais são expostas; rotação de chave exige re-criptografia de todos os registros; sem audit trail de acesso à chave.

## Consequências

- **Positivas:** Credenciais protegidas em repouso (OKR-3); compatível com compliance; mascaramento em GET via serializer (BR-002); logs de chamada nunca contêm credenciais descriptografadas; evolução para Vault possível sem breaking change.
- **Negativas:** Env var com a chave AES precisa ser protegida (acesso restrito ao processo do worker); re-criptografia necessária em key rotation; overhead de descriptografia por chamada (~1ms — negligível).
- **Riscos:** Comprometimento da env var expõe todas as credenciais. Mitigação: acesso restrito ao ambiente de produção; monitoramento de acesso ao secret; evolução para Vault no roadmap pós Wave 4.
