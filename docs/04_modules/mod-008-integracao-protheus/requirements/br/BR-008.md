# BR-008 — Regras de Negócio da Integração Dinâmica Protheus

---

## BR-001 — Rotina de Integração PUBLISHED é Imutável

- **Regra:** Uma rotina com `routine_type=INTEGRATION` e status PUBLISHED **NÃO PODE** ser editada via PATCH em nenhum campo (behavior_routines, integration_routines, field_mappings, params). Qualquer tentativa DEVE retornar 422. O único caminho para modificar é criar um fork (nova versão DRAFT). Regra herdada do MOD-007.
- **Exemplo:** Admin tenta `PATCH /admin/routines/:id/integration-config` em rotina PUBLISHED → 422: "Rotinas publicadas são imutáveis. Use o fork para criar nova versão."
- **Exceções:** Nenhuma. Regra inegociável de integridade arquitetural.
- **Impacto:** STATE, COMPLIANCE

```gherkin
Funcionalidade: Imutabilidade de Rotina de Integração Publicada

Cenário: Rotina PUBLISHED rejeita edição de config HTTP
  Dado que rotina routine_type=INTEGRATION está PUBLISHED
  Quando PATCH /admin/routines/:id/integration-config
  Então 422: "Rotinas publicadas são imutáveis. Use o fork para criar nova versão."

Cenário: Imutabilidade se estende a mapeamentos e parâmetros
  Dado que rotina routine_type=INTEGRATION está PUBLISHED
  Quando POST /admin/routines/:id/field-mappings
  Então 422: "Rotinas publicadas são imutáveis."
```

## BR-002 — Credenciais (auth_config) Nunca Retornadas em GET

- **Regra:** O campo `auth_config` da tabela `integration_services` é criptografado em repouso (AES-256 via secret do ambiente) e **NUNCA** retornado em endpoints GET. O endpoint DEVE retornar `auth_config: "***"` (mascarado). O valor real é acessível apenas pelo worker de execução em runtime.
- **Exemplo:** GET /admin/integration-services/:id → `{ ..., auth_type: "BASIC", auth_config: "***" }`. Senha NUNCA presente na resposta.
- **Exceções:** Nenhuma. Regra inegociável de segurança.
- **Impacto:** PERMISSIONS, COMPLIANCE

```gherkin

  Dado que serviço tem auth_type=BASIC com credenciais reais

  Então response.auth_config = "***" (mascarado)
```

## BR-003 — Serviço INACTIVE Bloqueia Execução de Rotinas

- **Regra:** Um serviço de destino com `status=INACTIVE` **NÃO PODE** ter rotinas vinculadas disparadas pelo motor de execução. Tentativa de execução DEVE resultar em `status=FAILED` com `error_message="Serviço inativo."` no `integration_call_logs`, sem chamada HTTP ser realizada.
- **Exemplo:** Motor tenta executar rotina vinculada a "PROTHEUS-PROD" que está INACTIVE → log criado com status=FAILED, error_message="Serviço inativo.", sem chamada HTTP.
- **Exceções:** Nenhuma.

## BR-004 — Campo Obrigatório Ausente Aborta Execução

- **Regra:** Se um mapeamento tem `required=true` e o campo de origem (`source_field`) está ausente ou vazio no contexto de execução, o motor DEVE abortar a execução **antes** de realizar a chamada HTTP. O log DEVE ser criado com `status=FAILED` e `error_message="Campo obrigatório ausente: {source_field}"`.
- **Exemplo:** Mapeamento required=true para `case.numero_nf`, caso não tem `numero_nf` → log com status=FAILED, error_message="Campo obrigatório ausente: case.numero_nf". Nenhuma chamada HTTP realizada.

## BR-005 — Parâmetros Sensíveis Mascarados em Todos os Logs

- **Regra:** Qualquer parâmetro com `is_sensitive=true` **DEVE** ter seu valor mascarado como `"***"` em TODOS os registros de `integration_call_logs` (campos `request_headers` e `request_payload`). O valor real NUNCA é gravado no log. Credenciais de autenticação são mascaradas independente de `is_sensitive`.
- **Exemplo:** Param `api_key` com `is_sensitive=true` → `request_headers["api_key"] = "***"` no log. Header `Authorization` → sempre `"***"` no log.
- **Exceções:** Nenhuma. Regra inegociável de segurança.
- **Impacto:** COMPLIANCE, DATA

```gherkin

Cenário: Parâmetro is_sensitive mascarado no log
```

## BR-006 — Outbox Pattern — Atomicidade da Transação de Negócio

- **Regra:** O INSERT em `integration_call_logs` com `status=QUEUED` DEVE ocorrer **dentro da mesma transação** do evento de negócio que originou a chamada (ex: transição de estágio no MOD-006). Se a transação falhar, o log NÃO é criado e a chamada NÃO é enfileirada. Tudo ou nada.
- **Exemplo:** Transição de estágio no MOD-006 cria: INSERT stage_history + UPDATE case_instances + INSERT integration_call_logs (QUEUED) — tudo na mesma transação. Se qualquer INSERT falhar, ROLLBACK total.
- **Exceções:** Nenhuma. Fundamento do Outbox Pattern.
- **Impacto:** DATA, FLOW, COMPLIANCE

```gherkin
Funcionalidade: Outbox Pattern — Atomicidade

Cenário: Outbox garante que chamada não se perde se worker cair
  Dado que job foi enfileirado mas worker reinicia antes de executar
  Quando worker volta
  Então job é reprocessado da fila sem duplicidade (BullMQ idempotência)
  E integration_call_logs não tem registro duplicado
```

## BR-007 — DLQ Após Esgotar Retries

- **Regra:** Quando uma chamada HTTP falha e `attempt_number` atinge `retry_max` (configurável por rotina), o motor DEVE atualizar o status para `DLQ` (Dead Letter Queue). O domain_event `integration.call_dlq` DEVE ser emitido e a notificação DEVE aparecer no monitor UX-INTEG-002 com badge vermelho.
- **Exemplo:** Rotina com retry_max=3. Tentativa 1 falha (503), tentativa 2 falha (503), tentativa 3 falha (503) → status=DLQ, domain_event emitido, badge DLQ no monitor.
- **Exceções:** Se retry_max=0, não há retry — falha vai direto para DLQ.
- **Impacto:** STATE, FLOW

```gherkin
Funcionalidade: Dead Letter Queue

Cenário: DLQ notifica operador após esgotar retries
  Dado que chamada falhou 3 vezes (retry_max=3)
  Então status=DLQ em integration_call_logs
  E alerta aparece no monitor UX-INTEG-002 com badge vermelho
  E domain_event: integration.call_dlq emitido
```

## BR-008 — Reprocessamento de DLQ Exige Justificativa e Preserva Log Original

- **Regra:** O reprocessamento de uma chamada em DLQ DEVE exigir uma justificativa obrigatória com mínimo de 10 caracteres. O log original DEVE permanecer inalterado (status=DLQ). Um NOVO `integration_call_logs` DEVE ser criado com `parent_log_id` apontando para o log original e `attempt_number=1` (novo ciclo de retries). O `integration_reprocess_requests` DEVE registrar quem, quando e por quê.
- **Exemplo:** Admin clica "Reprocessar" em chamada DLQ, preenche motivo "Protheus reiniciado após manutenção" (23 chars) e confirma → novo log criado, log original preservado como DLQ.
- **Exceções:** Nenhuma. Justificativa min 10 chars é obrigatória sem exceção.
- **Impacto:** STATE, COMPLIANCE, FLOW

```gherkin
Funcionalidade: Reprocessamento Governado

Cenário: Reprocessamento de DLQ exige justificativa
  Dado que chamada está em DLQ
  Quando admin clica "Reprocessar" sem preencher motivo
  Então o botão permanece desabilitado
  Quando preenche motivo (min 10 chars) e confirma
  Então nova tentativa é enfileirada e log original preservado
```

## BR-009 — Retry com Backoff Exponencial

- **Regra:** O retry de chamadas HTTP falhas DEVE usar backoff exponencial com a fórmula: `retry_backoff_ms × 2^(attempt-1)`. Os valores `retry_max` e `retry_backoff_ms` são configuráveis **por rotina** na tabela `integration_routines`. O retry é gerenciado pelo Outbox (re-enfileiramento), não pelo BullMQ.
- **Exemplo:** `retry_backoff_ms=1000`, `retry_max=3`. Attempt 1 falha → aguarda 1000ms. Attempt 2 falha → aguarda 2000ms. Attempt 3 falha → status=DLQ.
- **Exceções:** Se retry_max=0, nenhum retry é realizado.
- **Impacto:** FLOW, STATE

```gherkin
Funcionalidade: Retry Backoff Exponencial

Cenário: Falha transitória com retry
  Dado que Protheus retorna 503 (primeiro attempt)
  Então status=QUEUED (requeue) e attempt_number=2
  E worker aguarda retry_backoff_ms=1000ms antes de tentar novamente
  Quando Protheus retorna 503 novamente (segundo attempt)
  Então aguarda 2000ms (1000 × 2^1)
  Quando Protheus retorna 503 (terceiro e último attempt)
  Então status=DLQ
```
