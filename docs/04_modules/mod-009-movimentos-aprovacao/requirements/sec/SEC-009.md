# SEC-009 — Segurança e Compliance de Movimentos sob Aprovação

---

## 1. Autenticação

- **Método:** JWT Bearer (via MOD-000 Foundation)
- **Sessão:** Validada no banco (kill-switch disponível)
- **Header obrigatório:** `X-Correlation-ID` em todas as rotas
- **Service-to-service:** Chamadas internas ao motor de controle (`/movement-engine/evaluate`) por outros módulos utilizam internal JWT com scope `approval:engine:evaluate`. Não há shared secret.
- **Token refresh:** O módulo não gerencia refresh — consumidor responsável por renovar JWT antes de expiração.

## 2. Autorização — RBAC (7 scopes `approval:*`)

- **Modelo:** RBAC via `@RequireScope` (DOC-FND-000 §2). Enforcement em `preHandler` Fastify.
- **Scope verificado contra JWT claims** — sem cache local de permissões.

| Scope | Endpoints Protegidos | Descrição |
|---|---|---|
| `approval:rule:read` | GET /admin/control-rules | Ver regras de controle e alçadas |
| `approval:rule:write` | POST/PATCH /admin/control-rules, POST/PATCH /admin/approval-rules | Criar/editar regras |
| `approval:engine:evaluate` | POST /movement-engine/evaluate | Chamar motor de controle (usado por outros módulos) |
| `approval:movement:read` | GET /movements, GET /movements/:id | Ver movimentos controlados |
| `approval:movement:write` | POST /movements/:id/cancel | Cancelar movimentos (pelo solicitante) |
| `approval:decide` | POST /my/approvals/:id/approve\|reject | Aprovar ou reprovar movimentos |
| `approval:override` | POST /movements/:id/override | Override com justificativa obrigatória |

### 2.1 Separação de Privilégios

- **Leitura vs. Escrita:** `approval:rule:read` e `approval:rule:write` são scopes separados — analistas podem consultar regras sem editá-las.
- **Decisão isolada:** `approval:decide` é separado de `approval:movement:read` — ver movimentos não implica poder decidir sobre eles.
- **Override auditado:** `approval:override` é um scope exclusivo, separado de `approval:decide` — representa autoridade excepcional com compliance.

## 3. Regras de Autorização Complementares

| Regra | Tipo | Descrição |
|---|---|---|
| Segregação de funções | Business rule | `decided_by != movement.requested_by` — validação no service. Exceção: auto-aprovação por scope (épico §3.1) |
| Override auditado | Business rule | Requer scope `approval:override` + justificativa min 20 chars. Registro imutável em `movement_override_log` |
| Auto-aprovação | Business rule | Apenas quando `allow_self_approve=true` na `approval_rules` E solicitante possui `required_scope` |
| Inbox pessoal | Access control | `GET /my/approvals` retorna apenas movimentos atribuídos ao caller (filtro por alçada) |
| Cancelamento | Access control | Apenas solicitante pode cancelar (`movement.requested_by == caller_id`) e apenas em PENDING_APPROVAL |

### 3.1 Lógica de Validação — Segregação de Funções

A segregação é avaliada no service layer (`ApprovalService.decide()`) **antes** de persistir a decisão:

```
1. Extrair caller_id do JWT
2. Carregar controlled_movement pelo movement_id
3. SE caller_id == movement.requested_by:
   a. Verificar se existe approval_rule com allow_self_approve=true para a control_rule_id
   b. SE allow_self_approve=true E caller possui required_scope → permitir (emitir movement.auto_approved)
   c. SENÃO → rejeitar com 422 { code: "SEGREGATION_VIOLATION", detail: "Solicitante não pode aprovar o próprio movimento." }
4. SENÃO → prosseguir com fluxo normal de aprovação
```

- **Enforcement:** Service layer — não no middleware (requer dados do movimento).
- **Bypass impossível:** Nenhum scope sobrepõe a segregação exceto `allow_self_approve` configurado explicitamente na regra.
- **Log:** Toda tentativa de violação de segregação gera log de auditoria com `correlation_id`, `caller_id`, `movement_id`.

### 3.2 Lógica de Validação — Auto-Aprovação

```
1. Motor de controle (evaluate) identifica que operação é controlada
2. Verificar se approval_rule para o nível 1 possui allow_self_approve=true
3. SE sim E caller possui required_scope da regra:
   a. Criar controlled_movement com status=APPROVED (skip approval chain)
   b. Emitir movement.auto_approved (DATA-003 evento 13)
   c. Disparar execução imediata da operação
4. SENÃO → criar controlled_movement com status=PENDING_APPROVAL (fluxo normal)
```

- **Default:** `allow_self_approve=false` — configuração opt-in por regra.
- **Auditoria:** `movement.auto_approved` registra `scope_used` como justificativa automática.

### 3.3 Lógica de Validação — Cancelamento por Proprietário

```
1. Extrair caller_id do JWT
2. Carregar controlled_movement pelo movement_id
3. SE caller_id != movement.requested_by → 403 { code: "NOT_OWNER", detail: "Apenas o solicitante pode cancelar." }
4. SE movement.status != PENDING_APPROVAL → 422 { code: "INVALID_STATE", detail: "Cancelamento permitido apenas em PENDING_APPROVAL." }
5. Atualizar status para CANCELLED, emitir movement.cancelled
6. Notificar aprovadores com instâncias PENDING (libera da inbox)
```

### 3.4 Lógica de Validação — Override

```
1. Verificar scope approval:override no JWT
2. Validar justificativa: min 20 chars, max 2000 chars, não pode ser whitespace-only
3. Registrar em movement_override_log (append-only, imutável)
4. Atualizar status do movimento para APPROVED via override
5. Emitir movement.overridden (DATA-003 evento 10)
6. Notificar solicitante + admin + compliance
```

- **Dupla confirmação:** Frontend exige checkbox "Confirmo que estou aplicando override" antes de habilitar botão.
- **Imutabilidade:** Registros em `movement_override_log` não possuem UPDATE/DELETE — apenas INSERT.

## 4. Classificação dos Dados

| Tabela | Classificação | Campos Sensíveis | Justificativa |
|---|---|---|---|
| `movement_control_rules` | Interno | — | Configuração administrativa |
| `approval_rules` | Interno | — | Configuração administrativa |
| `controlled_movements` | Confidencial | `operation_payload` (pode conter dados de negócio) | Contém snapshot da operação interceptada |
| `approval_instances` | Interno | `parecer` | Opinião formal do aprovador |

| `movement_history` | Interno | `payload` | Append-only audit trail |
| `movement_override_log` | Restrito | `justificativa` (decisão formal auditada) | Decisão excepcional com compliance |

## 5. Política de Retenção

| Tabela / Categoria | Retenção | Justificativa |
|---|---|---|
| `movement_history` | Indefinida | Audit trail — compliance regulatório |
| `movement_override_log` | Indefinida | Compliance, imutável — evidência de decisão excepcional |
| `controlled_movements` | 5 anos após EXECUTED/CANCELLED/REJECTED | Retenção alinhada com requisitos fiscais/regulatórios |
| `approval_instances` | Acompanha `controlled_movements` | Ciclo de vida atrelado ao movimento |
| `movement_executions` | Acompanha `controlled_movements` | Ciclo de vida atrelado ao movimento |
| `movement_control_rules` | Indefinida (soft-delete) | Regras referenciadas por movimentos históricos |
| `approval_rules` | Indefinida (soft-delete) | Idem |

## 6. Mascaramento

| Campo | Regra | Quem pode ver sem máscara |
|---|---|---|
| `operation_payload` | Campos sensíveis mascarados pelo servidor antes de retornar ao frontend (sanitização via allowlist) | Sistema (execução pós-aprovação), aprovador com `approval:decide` vê apenas campos da allowlist |
| `execution_payload` | Mascarado em respostas de listagem; visível em detalhe para `approval:movement:read` | Admin com `approval:movement:read` |
| `justificativa` (override_log) | Visível apenas para roles com `approval:override` ou `system:audit:read` | Override actors, compliance, auditores |
| `parecer` (approval_instances) | Visível para participantes da cadeia de aprovação e solicitante | Aprovadores da cadeia + solicitante + admin |
| `error_message` (executions) | Mascarado em listagens — detalhes internos removidos | Admin com `approval:movement:read` |

## 7. LGPD

### 7.1 Princípios Aplicados

- **Minimização:** `operation_payload` contém snapshot mínimo necessário para decisão — sem duplicação de PII.
- **Anonimização:** `movement_history.payload` não duplica PII — referencia via `movement_id`.
- **Finalidade:** Dados coletados exclusivamente para controle de aprovação e auditoria regulatória.
- **Proporcionalidade:** `parecer` é texto livre limitado a 2000 chars; aprovadores instruídos a não incluir dados pessoais.

### 7.2 Direitos do Titular (LGPD Art. 18)

| Direito | Aplicabilidade | Implementação |
|---|---|---|
| **Acesso** | Sim | Titular pode consultar movimentos onde é `requested_by` via `GET /movements?requested_by=me` (scope `approval:movement:read`) |
| **Retificação** | Parcial | `operation_payload` é snapshot imutável; retificação aplica-se ao objeto original (módulo de origem). `parecer` não é retificável (decisão formal). |
| **Eliminação** | Não | Dados de auditoria (`movement_history`, `movement_override_log`) são retidos por obrigação legal (LGPD Art. 16, II). Movimentos finalizados podem ter `operation_payload` anonimizado após período de retenção. |
| **Portabilidade** | Sim | Export via API administrativa (formato JSON) dos movimentos do titular, incluindo decisões e histórico. |
| **Oposição** | Não aplicável | Tratamento baseado em obrigação legal/regulatória (LGPD Art. 7, II). |
| **Revogação de consentimento** | Não aplicável | Base legal não é consentimento. |

### 7.3 Anonimização Pós-Retenção

- Após período de retenção (5 anos), campos `operation_payload` e `execution_payload` podem ser substituídos por hash + metadados mínimos (job agendado).
- `requested_by`, `decided_by`: mantidos como UUID (pseudônimo) — não contêm PII diretamente.
- `parecer` e `justificativa`: mantidos por obrigação legal.

## 8. Auditoria

### 8.1 Eventos Auditáveis (13 domain events — DATA-003)

| Evento | O que é logado | Tabela de persistência |
|---|---|---|
| `approval.control_rule_created` | Snapshot mínimo: ids, status, campos configurados | `domain_events` |
| `approval.control_rule_updated` | Diff: campos alterados (before/after) | `domain_events` |
| `approval.approval_rule_created` | Snapshot: level, approver_type, timeout_hours | `domain_events` |
| `movement.created` | Snapshot: movement_id, object_type, origin_type, requested_by, operation_value | `domain_events` + `movement_history` |
| `movement.approved` | level, actor_id, parecer, is_final_level | `domain_events` + `movement_history` + `approval_instances` |
| `movement.rejected` | level, actor_id, parecer | `domain_events` + `movement_history` + `approval_instances` |
| `movement.executed` | result, execution_id, executed_by | `domain_events` + `movement_history` + `movement_executions` |
| `movement.failed` | error_message, retry_of | `domain_events` + `movement_history` + `movement_executions` |
| `movement.cancelled` | actor_id, cancellation_reason | `domain_events` + `movement_history` |
| `movement.overridden` | overridden_by, justificativa, scope_used | `domain_events` + `movement_history` + `movement_override_log` |
| `movement.escalated` | previous_level, new_level, timeout_at | `domain_events` + `movement_history` |
| `movement.timeout` | level, timeout_at, assigned_to | `domain_events` + `movement_history` |
| `movement.auto_approved` | requested_by, scope_used | `domain_events` + `movement_history` |

### 8.2 Trilha de Mudanças

- **`movement_history`:** Append-only, nunca alterado. Cada transição de estado gera um registro.
- **`movement_override_log`:** INSERT-only, imutável. Cada override gera um registro separado.
- **`domain_events`:** Fonte única de verdade para timeline e auditoria (DOC-ARC-003 §1 Dogma 6).

### 8.3 Controle de Acesso à Auditoria

| Dado | Quem pode ver | Filtro obrigatório |
|---|---|---|
| `movement_history` | `approval:movement:read` + participantes da cadeia | `tenant_id` (ACL) |
| `movement_override_log` | `approval:override` ou `system:audit:read` | `tenant_id` (ACL) |
| `domain_events` (movimentos) | `approval:movement:read` | `tenant_id` + `entity_type` (ACL) |
| `domain_events` (regras admin) | `approval:rule:read` | `tenant_id` + `entity_type` (ACL) |

### 8.4 Tentativas de Violação

- Toda tentativa de violação de segregação (§3.1) é logada em `application_log` com severity WARN.
- Toda tentativa de cancelamento por não-proprietário (§3.3) é logada com severity WARN.
- Toda tentativa de override sem scope é logada com severity WARN + correlation_id.
- Formato: `{ event: "security_violation", type: "SEGREGATION|NOT_OWNER|OVERRIDE_UNAUTHORIZED", caller_id, movement_id, correlation_id, timestamp }`.

## 9. Proteções

- **Rate limit:** Motor de controle (`/movement-engine/evaluate`) — 100 req/s por tenant
- **Rate limit:** Override (`/movements/:id/override`) — 10 req/min por tenant (operação excepcional)
- **Rate limit:** Decisões (`/my/approvals/:id/approve|reject`) — 60 req/min por usuário
- **Validação:** Zod schemas em todas as rotas
- **Logs seguros:** Sem PII em logs de aplicação; `correlation_id` para rastreamento
- **Secrets:** Nenhum secret específico do módulo
- **CSRF:** Não aplicável (API stateless, JWT Bearer)
- **Injection:** Zod + parametrized queries — sem concatenação SQL

## 10. Cenários de Segurança (Gherkin BDD)

```gherkin
Funcionalidade: Segregação de Funções na Aprovação

Cenário: Impedir aprovação do próprio movimento
  Dado que usuário "U1" criou movimento "M1" via motor de controle
  E a regra de alçada NÃO possui allow_self_approve=true
  Quando "U1" tenta POST /my/approvals/:approvalId/approve com parecer "Aprovado"
  Então 422 com code "SEGREGATION_VIOLATION"
  E detail "Solicitante não pode aprovar o próprio movimento."
  E log de tentativa de violação é registrado com severity WARN

Cenário: Permitir auto-aprovação quando configurado
  Dado que usuário "U1" criou movimento "M1" via motor de controle
  E a regra de alçada possui allow_self_approve=true
  E "U1" possui o required_scope da regra
  Quando o motor avalia a operação
  Então movimento é criado com status APPROVED
  E domain_event movement.auto_approved é emitido com scope_used
  E operação é executada imediatamente

Cenário: Auto-aprovação negada — scope insuficiente
  Dado que usuário "U1" criou movimento "M1" via motor de controle
  E a regra de alçada possui allow_self_approve=true
  Mas "U1" NÃO possui o required_scope da regra
  Quando o motor avalia a operação
  Então movimento é criado com status PENDING_APPROVAL
  E fluxo normal de aprovação é iniciado
```

```gherkin
Funcionalidade: Cancelamento por Proprietário

Cenário: Solicitante cancela próprio movimento pendente
  Dado que usuário "U1" é requested_by do movimento "M1"
  E "M1" está em PENDING_APPROVAL
  Quando "U1" POST /movements/M1/cancel com { reason: "Não necessário" }
  Então 200 com status CANCELLED
  E domain_event movement.cancelled é emitido
  E aprovadores com instâncias PENDING são notificados

Cenário: Outro usuário tenta cancelar movimento alheio
  Dado que usuário "U2" NÃO é requested_by do movimento "M1"
  Quando "U2" POST /movements/M1/cancel
  Então 403 com code "NOT_OWNER"
  E log de tentativa é registrado

Cenário: Solicitante tenta cancelar movimento já aprovado
  Dado que usuário "U1" é requested_by do movimento "M1"
  E "M1" está em APPROVED (já executado)
  Quando "U1" POST /movements/M1/cancel
  Então 422 com code "INVALID_STATE"
  E detail "Cancelamento permitido apenas em PENDING_APPROVAL."
```
