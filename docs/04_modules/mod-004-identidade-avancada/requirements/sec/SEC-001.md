# SEC-001 — Segurança e Compliance da Identidade Avançada

---

## 1. Autenticação

Herdada integralmente do MOD-000 (Foundation):

- JWT Bearer token validado no middleware Fastify (`preHandler`)
- Sessão persistida em banco (kill-switch disponível — DOC-FND-000 §1.2)
- MOD-004 **NÃO** implementa lógica de autenticação própria

## 2. Autorização

- **Modelo:** RBAC por scopes via `@RequireScope()` (DOC-FND-000 §2)
- **Enforcement:** Backend (preHandler) — frontend oculta elementos por conveniência UX, não por segurança
- **`tenant_id`:** Filtro obrigatório em todas as queries (RLS — DOC-DEV-001 DATA-001)

### 2.1 Catálogo de Escopos (Amendment em MOD-000-F12)

| Escopo | Descrição | Endpoints protegidos |
|---|---|---|
| `identity:org_scope:read` | Ver vínculos org de usuários | GET /admin/users/:id/org-scopes |
| `identity:org_scope:write` | Criar/remover vínculos org | POST /admin/users/:id/org-scopes, DELETE /admin/users/:id/org-scopes/:scopeId |
| `identity:share:read` | Ver compartilhamentos controlados | GET /admin/access-shares |
| `identity:share:write` | Criar compartilhamentos | POST /admin/access-shares |
| `identity:share:revoke` | Revogar compartilhamentos | DELETE /admin/access-shares/:id |
| `identity:share:authorize` | Permite auto-autorizar compartilhamentos (grantor = authorized_by) | Validação no service (POST /admin/access-shares) |
| `identity:delegation:read` | Ver delegações temporárias (admin) | GET /admin/access-delegations (futuro — admin view) |
| `identity:delegation:write` | Criar delegações temporárias (admin) | POST /admin/access-delegations (futuro — admin view) |

> **Nota:** Os scopes `identity:delegation:read` e `identity:delegation:write` estão registrados no catálogo do módulo (mod.md §4.1) para futuras telas administrativas de delegações. Os endpoints self-service atuais (`/access-delegations`) operam no contexto do próprio usuário e não requerem esses scopes.
>
> **Formato:** Todos seguem o padrão canônico `dominio:entidade:acao` com 3 segmentos (regex: `^[a-z][a-z0-9_]*(:[a-z][a-z0-9_]*){1,2}$` — DOC-FND-000 §2.1 v1.2.0)

### 2.2 Mapeamento Endpoint → Scope → operationId

| Endpoint | Scope requerido | operationId | Nota |
|---|---|---|---|
| GET /api/v1/admin/users/:id/org-scopes | `identity:org_scope:read` | `admin_user_org_scopes_list` | |
| POST /api/v1/admin/users/:id/org-scopes | `identity:org_scope:write` | `admin_user_org_scopes_create` | Idempotency-Key |
| DELETE /api/v1/admin/users/:id/org-scopes/:scopeId | `identity:org_scope:write` | `admin_user_org_scopes_delete` | |

| POST /api/v1/admin/access-shares | `identity:share:write` | `admin_access_shares_create` | Idempotency-Key |
| GET /api/v1/admin/access-shares | `identity:share:read` | `admin_access_shares_list` | |

### 2.3 Validações de Integridade com Impacto de Segurança

As seguintes regras de negócio possuem impacto direto em segurança e compliance e DEVEM ser enforçadas no service layer (não apenas na UI):

| Regra | Validação | HTTP | Justificativa SEC |
|---|---|---|---|
| BR-001.10 | `valid_until` deve ser estritamente no futuro | 422 | Impede criação de registros já expirados (bypass de job) |
| BR-001.11 | `org_unit_id` deve existir e estar ACTIVE em MOD-003 | 422 | Previne vinculação a nós organizacionais desativados |
| BR-001.12 | `grantee_id`/`delegatee_id` deve existir em `users` do mesmo tenant | 422 | Isolamento tenant + integridade referencial |

### 2.4 Autorização de Linha (Row-Level)

- **MUST:** Toda query MUST filtrar por `tenant_id = current_tenant` (RLS policy no PostgreSQL)
- **Endpoints self-service** (`/my/*`): filtram adicionalmente por `user_id = caller.id`
- **Delegações:** caller só vê delegações onde é `delegator_id` ou `delegatee_id`
- **Shares admin:** visíveis apenas com scope `identity:share:read`; grantee vê apenas via `/my/shared-accesses`

## 3. Classificação dos Dados

| Tabela | Classificação | Justificativa | PII? |
|---|---|---|---|
| `user_org_scopes` | Interno | Vínculo organizacional — referencia user_id (FK) mas não contém dados pessoais | Não (apenas FKs UUID) |
| `access_shares` | Confidencial | Expõe quem pode acessar o quê, com motivo e autorizador | Não (apenas FKs UUID + reason texto livre) |
| `access_delegations` | Confidencial | Escopos delegados entre usuários, com motivo | Não (apenas FKs UUID + scopes) |
| `domain_events` (MOD-004) | Confidencial | Payloads contêm IDs e metadados de operações de acesso | Não (payload policy: sem PII) |

## 4. Política de Mascaramento

| sensitivity_level | Regra | Campos afetados |
|---|---|---|
| 0 | Sem restrição | — |
| 1 (padrão MOD-004) | Guard-rail: payload filtrado em notificações; `allowed_actions` e `delegated_scopes` NÃO incluídos em notificações push | `access_shares.allowed_actions`, `access_delegations.delegated_scopes` |

> **Regra (DOC-FND-000 §3):** `sensitivity_level` é guard-rail/otimizador, não substitui ACL/RBAC. O enforcement forte é via `tenant_id` + ACL da entity originária.

## 6. LGPD

- **Minimização:** Payloads de domain events contêm apenas IDs e status — sem PII (nome, e-mail, CPF)
- **Anonimização:** N/A — tabelas MOD-004 não armazenam dados pessoais diretamente (apenas FKs UUID para `users`)
- **Direito ao esquecimento:** Soft delete em `user_org_scopes` e registros de shares/delegations. A exclusão real do usuário em MOD-000 (users) é bloqueada por `ON DELETE RESTRICT` nas FKs — requer tratamento coordenado
- **Portabilidade:** N/A — dados de autorização não são portáveis

---

## 7. Auditoria

- **Domain events:** 9 eventos catalogados em DATA-003, emitidos via Outbox Pattern
- **Autorização de eventos:** SEC-002 define Emit/View/Notify por evento
- **`X-Correlation-ID`:** Obrigatório em todas as respostas HTTP e em todos os domain events
- **Idempotência:** `Idempotency-Key` TTL 60s em todos os POST endpoints
- **Trilha completa:** Cada mutação registra `created_by` (ou `revoked_by`) + `correlation_id` + timestamp

## 8. Proteções

| Proteção | Implementação | Referência |
|---|---|---|
| Rate limiting | Herdado do MOD-000 (middleware global) | DOC-FND-000 |
| Validação de input | Zod schemas no Fastify (borda externa) | DOC-GNP-00 §3, EX-API-001 |
| Logs seguros | Logs estruturados sem PII; `correlation_id` em todos | DOC-ESC-001 §3.3 |
| Secrets | Fora do repositório; scanner no CI | DOC-ESC-001 §3.2 |
| SQL injection | Queries parametrizadas via Drizzle ORM | DOC-GNP-00 |
| RBAC bypass | `@RequireScope()` no preHandler (enforcement real, não apenas UX) | DOC-FND-000 §2.3 |
