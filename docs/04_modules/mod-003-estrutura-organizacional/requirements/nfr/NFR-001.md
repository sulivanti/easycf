- **Paginação:** Cursor-based para listagem flat (`GET /org-units`, FR-005). Tree query retorna tudo em uma chamada.

  - Filtros combinados: `nivel`, `status`, `parent_id`, `search` (ILIKE em nome/codigo)
- **Cache:** Sem cache — todas as chamadas tree query executam ao vivo. O índice em `parent_id WHERE deleted_at IS NULL` é o único mecanismo de otimização.
- **Topology:** `sync` — todas as operações são síncronas (sem filas/workers/outbox).

### 7.1 Comportamento do Soft Limit (PENDENTE-004)

Ao criar um nó via `POST /org-units`, o sistema DEVE contar o total de nós ativos (`deleted_at IS NULL`) na tabela `org_units`:

| Condição | Comportamento |
|---|---|
| `count ≤ 400` | Operação normal, sem warning |
| `400 < count < 500` | Operação prossegue + response header `X-Limit-Warning: org_units_count={N}, soft_limit=500, threshold=80%` |
| `count ≥ 500` | Operação prossegue + response header `X-Limit-Warning: org_units_count={N}, soft_limit=500, exceeded=true` |

- **Sem bloqueio:** O soft limit **nunca** impede a criação. Apenas sinaliza.
- **Métrica:** `org_units.soft_limit.warning_emitted` (counter) — incrementado a cada response com `X-Limit-Warning`.
- **Alerta:** Configurar alerta quando `count ≥ 450` (90%) para ação proativa da equipe de ops.
