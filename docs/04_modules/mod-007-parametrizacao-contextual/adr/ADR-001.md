# ADR-001 — Cache Redis Removido do Motor de Avaliação (Consistência > Performance)

## Contexto

O motor de avaliação do MOD-007 (`POST /routine-engine/evaluate`) é chamado sincronamente pelo MOD-006 durante transições de estágio. Em cenários de alta carga, uma camada de cache Redis foi considerada para reduzir latência. Contudo, o motor opera sobre dados altamente mutáveis (enquadradores com vigência, rotinas com ciclo DRAFT/PUBLISHED/DEPRECATED, regras de incidência ativas/inativas) e qualquer dado desatualizado pode resultar em comportamento incorreto: campos obrigatórios não exibidos, validações bloqueantes ignoradas ou defaults incorretos aplicados a objetos em transição.

Em 2026-03-15, a equipe de arquitetura decidiu que o custo de inconsistência supera o ganho de latência. O risco de cache stale em operações críticas (transição de estágio do MOD-006 depende de dados frescos) é inaceitável.

## Decisão

**Cache Redis removido do motor de avaliação inteiro.** Todas as chamadas a `POST /routine-engine/evaluate` executam queries ao vivo contra o banco de dados. Nenhuma camada de cache intermediária (Redis, in-memory, HTTP cache) é utilizada nos 6 passos do motor.

Performance é garantida por:

1. Indexes otimizados: `idx_incidence_status_vigencia` (passo 1), `idx_links_incidence` (passo 2), `idx_items_routine_ordem` (passo 3)
2. Partial indexes com `WHERE status = 'ACTIVE'` para reduzir scan
3. SLO: p95 < 200ms, p99 < 500ms para até 10 regras ativas e 50 items (NFR-001)

## Alternativas Consideradas

1. **Cache Redis com TTL curto (30s):** Reduziria latência mas introduziria janela de inconsistência. Uma rotina publicada ou enquadrador expirado poderia levar até 30s para refletir no motor — inaceitável para operações de transição do MOD-006.
2. **Cache com invalidação por evento:** Complexidade elevada. Cada mutação (create/update/delete/publish/fork/expire) precisaria invalidar chaves específicas. Race conditions entre invalidação e nova escrita são difíceis de eliminar.
3. **Cache read-through com lock distribuído:** Complexidade operacional desproporcional ao ganho. Adiciona dependência crítica do Redis (se Redis cair, motor para).
4. **Cache em nível de aplicação (in-memory):** Não funciona em cenário multi-instância sem sincronização.

## Consequências

- **Positivas:** Zero risco de dado desatualizado; simplicidade operacional (sem Redis como dependência); sem lógica de invalidação; debug facilitado (toda chamada é reproduzível).
- **Negativas:** Latência ligeiramente maior que solução com cache; dependência direta do banco para cada avaliação; se o banco estiver lento, o motor fica lento.
- **Mitigações:** Indexes otimizados; monitoramento com alerta se p95 > 150ms por 5 minutos; connection pooling adequado; possibilidade futura de read-replica se carga justificar.
