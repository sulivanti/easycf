## FR-001 — CRUD de Serviços de Destino

- **Goal:** Permitir criar, listar, visualizar e editar serviços de destino (Protheus/TOTVS) com URL, autenticação e configurações de ambiente.
- **Done funcional:**
  - POST /admin/integration-services cria serviço com codigo UNIQUE, base_url, auth_type, auth_config (criptografado), environment
  - GET /admin/integration-services lista com paginação cursor-based, filtro por status e environment
  - GET /admin/integration-services/:id retorna detalhe com auth_config mascarado ("***")
  - PATCH /admin/integration-services/:id edita (auth_config re-criptografado se presente)
  - auth_config armazenado criptografado (AES-256), NUNCA retornado em GET (BR-002)
  - Serviço INACTIVE bloqueia execução de rotinas vinculadas (BR-003)
- **Dependências:** BR-002, BR-003, DATA-008 (integration_services), SEC-008
- **Idempotency:** n/a (CRUD padrão)
- **Timeline/Notifications:** yes — emite `integration.service_created`, `integration.service_updated` (DATA-003)

```gherkin
Funcionalidade: CRUD de Serviços de Destino

Cenário: Criar serviço Protheus de produção
  Quando POST /admin/integration-services com {
    codigo: "PROTHEUS-PROD",
    base_url: "https://protheus.empresa.com/rest",
    auth_type: "BASIC",

  Então status 201

  E senha NUNCA presente na resposta
```

## FR-002 — Criação de Rotinas de Integração com Herança MOD-007

- **Goal:** Permitir criar rotinas de integração que herdam o versionamento do MOD-007 (`behavior_routines` com `routine_type='INTEGRATION'`) e configurar extensão HTTP.
- **Done funcional:**
  - POST /admin/routines com `routine_type=INTEGRATION` cria behavior_routines (herda MOD-007)
  - POST /admin/routines/:id/integration-config configura extensão HTTP: service_id, http_method, endpoint_tpl, timeout_ms, retry_max, retry_backoff_ms, trigger_events
  - endpoint_tpl suporta variáveis `{campo.path}` resolvidas em runtime (ex: `/api/pedidos/{case.object_id}`)
  - trigger_events define array de event_types do MOD-006 que disparam a rotina automaticamente
  - Rotina PUBLISHED imutável em todas as camadas (BR-001)
  - GET /admin/routines?type=INTEGRATION filtra apenas rotinas de integração
- **Dependências:** BR-001, BR-010, DATA-008 (integration_routines), INT-008
- **Idempotency:** n/a
- **Timeline/Notifications:** yes — emite `integration.routine_configured` (DATA-003)

```gherkin
Funcionalidade: Rotinas de Integração

Cenário: Criar rotina de integração com config HTTP
  Quando POST /admin/routines com { routine_type: "INTEGRATION", codigo, nome }
  E POST /admin/routines/:id/integration-config com {
    service_id, http_method: "POST",
    endpoint_tpl: "/WSRESTPV001/PedidoVenda",
    retry_max: 3, retry_backoff_ms: 1000,
    trigger_events: ["case.stage_transitioned"]
  }
  Então behavior_routines criada com routine_type=INTEGRATION
  E integration_routines criada com a configuração HTTP

Cenário: trigger_events vincula rotina a eventos do MOD-006
  Dado que trigger_events=["case.stage_transitioned"] e estágio = "EM_APROVAÇÃO"
  Quando caso transita para "EM_APROVAÇÃO"
  Então motor de integração é disparado automaticamente
  E correlation_id do case.stage_transitioned é propagado ao job

Cenário: endpoint_tpl suporta variáveis do contexto
  Dado que endpoint_tpl = "/api/pedidos/{case.object_id}"
  Quando motor de execução resolve o template com case_id
  Então a URL real usa o valor de case.object_id
```

## FR-003 — CRUD de Mapeamentos de Campos

- **Goal:** Permitir definir como cada campo do sistema se transforma no campo correspondente do Protheus, com 5 tipos de mapeamento, expressões de transformação e condições.
- **Done funcional:**
  - POST /admin/routines/:id/field-mappings cria mapeamento com source_field, target_field, mapping_type, required, transform_expr, condition_expr, default_value, ordem
  - PATCH /admin/field-mappings/:id edita mapeamento existente
  - DELETE /admin/field-mappings/:id remove mapeamento (apenas DRAFT — BR-001)
  - 5 tipos: FIELD (campo do caso), PARAM (parâmetro de contexto), HEADER, FIXED_VALUE (literal), DERIVED (expressão calculada)
  - transform_expr v1: UPPER, LOWER, TRIM, CONCAT, DATE_FORMAT
  - condition_expr v1: ==, !=, IN, NOT NULL (inclusão condicional no payload)
  - Campo `ordem` define a ordem de montagem do payload JSON
  - Campos required=true: ausência aborta execução antes do HTTP (BR-004)
- **Dependências:** BR-001, BR-004, BR-005, DATA-008 (integration_field_mappings)
- **Idempotency:** n/a
- **Timeline/Notifications:** no

```gherkin
Funcionalidade: Mapeamentos de Campos

Cenário: Criar mapeamento FIELD obrigatório
  Quando POST /admin/routines/:id/field-mappings com {
    source_field: "case.object.numero_pedido",
    target_field: "C5_NUM",
    mapping_type: "FIELD",
    required: true, ordem: 1
  }
  Então mapeamento criado e disponível para o motor

Cenário: Mapeamento FIXED_VALUE insere literal no payload
  Dado que mapeamento tem mapping_type=FIXED_VALUE e default_value="PV"
  Quando motor monta payload
  Então payload["C5_TIPO"] = "PV" independente do objeto

Cenário: transform_expr aplicada ao valor do campo
  Dado que mapeamento tem source_field="case.nome" e transform_expr="UPPER(value)"
  Quando motor resolve o mapeamento com case.nome="João Silva"
  Então payload inclui "JOÃO SILVA" (maiúsculas)

Cenário: condition_expr determina inclusão condicional
  Dado que mapeamento tem condition_expr="case.tipo == 'SERVICO'"
  E caso tem tipo="PRODUTO"
  Quando motor avalia
  Então campo NOT incluído no payload (condição falsa)

Cenário: Reordenação de mapeamentos afeta ordem no payload JSON
  Dado que mapeamentos têm ordem: C5_NUM(1), C5_FILIAL(2), C5_TIPO(3)
  Quando payload é montado
  Então JSON preserva a ordem declarada dos campos
```

## FR-005 — Motor de Execução Assíncrono (Outbox + BullMQ)

- **Goal:** Executar chamadas HTTP ao Protheus de forma assíncrona com garantia de entrega (Outbox Pattern), processamento via worker BullMQ e rastreabilidade completa.
- **Done funcional:**
  - INSERT integration_call_logs (status=QUEUED) dentro da transação de negócio (Outbox Pattern — BR-006)
  - Worker BullMQ escaneia logs QUEUED e enfileira jobs com jobId = call_log.id (dedupe automática)
  - Worker atualiza status: QUEUED → RUNNING → SUCCESS/FAILED
  - Em SUCCESS: grava response_body, http_status, completed_at, duration_ms; emite `integration.call_completed`
  - Em FAILED: reavalia retry conforme BR-009 (backoff exponencial)
  - Concurrency controlada por env var `INTEGRATION_CONCURRENCY` (default: 10)
  - Timeout da chamada HTTP respeitando integration_services.timeout_ms (default 30000)
  - Serviço INACTIVE bloqueia execução (BR-003)
- **Dependências:** BR-003, BR-006, BR-007, BR-009, BR-011, DATA-008 (integration_call_logs), DATA-003, INT-008
- **Idempotency:** required — BullMQ deduplica via jobId = call_log.id
- **Timeline/Notifications:** yes — emite `integration.call_queued`, `integration.call_completed`, `integration.call_failed`, `integration.call_dlq` (DATA-003, SEC-002)

```gherkin
Funcionalidade: Motor de Execução BullMQ

Cenário: Execução bem-sucedida
  Dado que job foi enfileirado com call_log.status=QUEUED
  Quando worker executa a chamada e Protheus retorna 200
  Então integration_call_logs.status=SUCCESS
  E response_body e http_status salvos
  E duration_ms calculado
  E domain_event: integration.call_completed emitido com correlation_id

Cenário: Outbox — chamada não se perde se worker reiniciar
  Dado que log está QUEUED e worker reinicia antes de executar
  Quando worker volta
  Então re-escaneia logs QUEUED e reenfileira o job
  E chamada é executada exatamente uma vez (sem duplicidade)

Cenário: Timeout da chamada HTTP
  Dado que service.timeout_ms=30000 e Protheus não responde em 30s
  Então chamada abortada: status=FAILED, error_message="Timeout após 30000ms"
  E retry é iniciado normalmente
```

## FR-007 — DLQ e Reprocessamento Governado

- **Goal:** Gerenciar chamadas em Dead Letter Queue (DLQ) e permitir reprocessamento com justificativa, preservando histórico completo.
- **Done funcional:**
  - Após retry_max esgotado: status=DLQ, domain_event `integration.call_dlq` (BR-007)
  - POST /admin/integration-logs/:id/reprocess com { reason } cria reprocessamento
  - Justificativa obrigatória min 10 chars (BR-008)
  - Novo log criado com parent_log_id, attempt_number=1
  - Log original permanece DLQ, imutável
  - integration_reprocess_requests registra who/when/why
  - Domain_event `integration.call_reprocessed` emitido
- **Dependências:** BR-007, BR-008, DATA-008 (integration_call_logs, integration_reprocess_requests), SEC-008 (scope integration:log:reprocess)
- **Idempotency:** n/a (cada reprocessamento cria novo log)
- **Timeline/Notifications:** yes — emite `integration.call_reprocessed` (DATA-003, SEC-002)

```gherkin
Funcionalidade: DLQ e Reprocessamento

Cenário: Reprocessamento de DLQ cria novo log vinculado ao original
  Dado que call_log está em DLQ
  E admin preencheu motivo e confirmou reprocessamento
  Quando POST /admin/integration-logs/:id/reprocess com { reason }
  Então integration_reprocess_requests criado
  E novo integration_call_logs criado com parent_log_id = id original
  E attempt_number = 1 (começa novo ciclo de retries)
  E log original preservado (status=DLQ, sem alteração)
```
