# ADR-001 — Garantia de Estágio Inicial Único por Ciclo (Trigger vs. Campo Denormalizado)

- **status:** accepted
- **timebox_days:** 90

---

## Contexto

A regra BR-002 exige que cada ciclo tenha **no máximo 1 estágio** com `is_initial=true`. Porém, a tabela `process_stages` não possui `cycle_id` diretamente — ela se relaciona via `macro_stage_id → process_macro_stages.cycle_id`. PostgreSQL não suporta partial unique index com FK indireta (não é possível criar `UNIQUE(cycle_id) WHERE is_initial=true` sem o campo `cycle_id` na tabela de estágios).

## Decisão

**Opção B (aceita): Campo denormalizado `cycle_id` em `process_stages`** com partial unique index nativo do PostgreSQL.

### Justificativa

1. **Estabilidade do campo:** `cycle_id` é derivável e estável — um estágio nunca muda de macroetapa/ciclo (só via fork, que cria novos IDs com novo `cycle_id`)
2. **Constraint nativo:** Partial unique index `UNIQUE(cycle_id) WHERE is_initial=true AND deleted_at IS NULL` — sem trigger, sem PL/pgSQL, visibilidade imediata no schema
3. **Benefício colateral:** Simplifica a validação de transição cross-ciclo (BR-008) — `WHERE s.cycle_id = :cycle_id` sem JOIN em `process_macro_stages`
4. **Fork atômico:** Campo copiado naturalmente durante fork (BR-004) sem complexidade adicional
5. **Risco mitigado:** Denormalização é segura porque macroetapas não mudam de ciclo — a relação `stage → macro_stage → cycle` é imutável após criação

### Implementação

```sql
-- 1. Adicionar coluna denormalizada
ALTER TABLE process_stages
  ADD COLUMN cycle_id UUID NOT NULL REFERENCES process_cycles(id);

-- 2. Partial unique index para is_initial
CREATE UNIQUE INDEX idx_stages_initial_unique
  ON process_stages (cycle_id)
  WHERE is_initial = true AND deleted_at IS NULL;

-- 3. Consistência: trigger para popular cycle_id automaticamente
CREATE OR REPLACE FUNCTION fn_stages_set_cycle_id()
RETURNS TRIGGER AS $$
BEGIN
  NEW.cycle_id := (SELECT cycle_id FROM process_macro_stages WHERE id = NEW.macro_stage_id);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_stages_set_cycle_id
  BEFORE INSERT ON process_stages
  FOR EACH ROW EXECUTE FUNCTION fn_stages_set_cycle_id();
```

> **Nota:** O trigger aqui é mínimo (apenas popula `cycle_id` a partir de `macro_stage_id`) — diferente da Opção A que usava trigger para validar unicidade. A unicidade é garantida pelo partial unique index nativo.

## Alternativas Consideradas

| Alternativa | Prós | Contras |
|---|---|---|
| **A) Trigger BEFORE INSERT/UPDATE** | Sem denormalização; modelo limpo | Trigger oculta lógica de validação; complexidade no fork |
| **B) Campo denormalizado `cycle_id` (aceita)** | Partial unique index nativo; simplifica BR-008; fork natural | Denormalização (risco mitigado — relação imutável) |
| **C) Validação apenas no service** | Simples | Risco de inconsistência em concorrência; sem garantia no DB |

## Consequências

- `process_stages` ganha campo `cycle_id UUID NOT NULL FK→process_cycles.id`
- Partial unique index garante BR-002 no nível de banco
- Validação de BR-008 (cross-ciclo) simplificada: `WHERE s.cycle_id = :cycle_id` direto
- Fork (BR-004) copia `cycle_id` do novo ciclo automaticamente
- Query do `/flow` (FR-011) pode filtrar por `cycle_id` direto em `process_stages`, eliminando 1 JOIN
