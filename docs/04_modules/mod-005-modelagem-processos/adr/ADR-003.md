# ADR-003 — Fork Atomico via Transacao Unica com Remapeamento de UUIDs

- **status:** accepted
- **timebox_days:** 90

---

## Contexto

A regra BR-004 exige que o fork de um ciclo PUBLISHED copie atomicamente **toda** a estrutura (7 tabelas: cycles, macro_stages, stages, gates, role_links, transitions, mais o campo denormalizado cycle_id em stages). Cada elemento copiado recebe novo UUID. Referencias internas (ex: `from_stage_id` e `to_stage_id` em `stage_transitions`) devem ser remapeadas para os novos UUIDs correspondentes. A operacao deve ser tudo-ou-nada.

O desafio tecnico principal e o remapeamento de FKs internas: transicoes referenciam stages pelo ID antigo, e os novos IDs so sao conhecidos apos a insercao. Alem disso, o timeout maximo e de 5s (NFR-005 §5) e o volume estimado e de ate 50 estagios + gates + transicoes por ciclo.

## Decisao

**Transacao unica com mapa de remapeamento em memoria (aceita).**

### Justificativa

1. **Mapa old→new em memoria:** O service cria um `Map<oldUUID, newUUID>` para cada entidade antes da insercao. Com volume maximo de ~500 registros por fork (DATA-005 §2.9), o mapa cabe confortavelmente em memoria
2. **Ordem de insercao deterministica:** Segue a mesma ordem da migracao (DATA-005 §4): cycles → macro_stages → stages → gates → role_links → transitions. Cada camada usa os novos IDs da camada anterior
3. **Atomicidade nativa:** `BEGIN...COMMIT` do PostgreSQL garante rollback completo em caso de falha em qualquer etapa — sem necessidade de sagas ou compensacao
4. **cycle_id denormalizado (ADR-001):** O campo `cycle_id` em `process_stages` e copiado com o novo `cycle_id` diretamente, sem depender do trigger (que so dispara em INSERT via macro_stage_id)
5. **Idempotencia via Idempotency-Key:** Header `Idempotency-Key` (INT-005 §1.8) previne forks duplicados em caso de retry do cliente

### Implementacao

```typescript
async forkCycle(cycleId: UUID, userId: UUID, idempotencyKey: string): Promise<Cycle> {
  return this.db.transaction(async (tx) => {
    // 1. Lock e validacao
    const original = await tx.selectForUpdate('process_cycles', cycleId);
    if (original.status !== 'PUBLISHED') throw new CycleNotPublishedError();

    // 2. Criar novo ciclo
    const newCycleId = uuid();
    const newCycle = { ...original, id: newCycleId, status: 'DRAFT', version: original.version + 1, parent_cycle_id: cycleId };

    // 3. Mapa de remapeamento
    const idMap = new Map<UUID, UUID>();

    // 4. Copiar macro_stages
    const macros = await tx.select('process_macro_stages', { cycle_id: cycleId });
    for (const m of macros) {
      const newId = uuid();
      idMap.set(m.id, newId);
      await tx.insert('process_macro_stages', { ...m, id: newId, cycle_id: newCycleId });
    }

    // 5. Copiar stages (com cycle_id denormalizado)
    const stages = await tx.select('process_stages', { cycle_id: cycleId });
    for (const s of stages) {
      const newId = uuid();
      idMap.set(s.id, newId);
      await tx.insert('process_stages', { ...s, id: newId, macro_stage_id: idMap.get(s.macro_stage_id), cycle_id: newCycleId });
    }

    // 6. Copiar gates, role_links (usando idMap para stage_id)
    // 7. Copiar transitions (remapeando from_stage_id e to_stage_id)
    const transitions = await tx.select('stage_transitions', { /* stages do ciclo */ });
    for (const t of transitions) {
      await tx.insert('stage_transitions', {
        ...t, id: uuid(),
        from_stage_id: idMap.get(t.from_stage_id),
        to_stage_id: idMap.get(t.to_stage_id)
      });
    }

    return newCycle;
  });
}
```

## Alternativas Consideradas

| Alternativa | Pros | Contras |
|---|---|---|
| **A) Transacao unica com mapa em memoria (aceita)** | Atomicidade nativa; simples; volume cabe em memoria; timeout 5s viavel | Lock na tabela source durante leitura (aceitavel — ciclo PUBLISHED e imutavel) |
| **B) Stored procedure PL/pgSQL** | Performance maxima (sem round-trips app↔DB); atomicidade nativa | Logica de negocio no banco; dificil de testar unitariamente; acoplamento com PostgreSQL |
| **C) CTAS + INSERT...SELECT com uuid_generate_v4()** | Performance (set-based); sem mapa em memoria | Remapeamento de FKs internas e extremamente complexo em SQL puro; ilegivel |
| **D) Event-driven (fork como saga)** | Desacoplamento; sem transacao longa | Complexidade de compensacao; eventual consistency inaceitavel para fork (usuario espera resultado sincrono) |

## Consequencias

- Fork e operacao sincrona com SLA < 2s (NFR-005 §1) para grafos de ate 50 estagios
- Rollback automatico em caso de falha (ex: constraint violation)
- `Idempotency-Key` previne duplicacao em caso de retry
- Domain event `process.cycle_forked` emitido apos COMMIT com `causation_id` (DATA-003)
- Teste de integracao obrigatorio: validar novos UUIDs, referencias remapeadas e rollback em falha (NFR-005 §10)
- Se volumes crescerem alem de 500 registros por fork, considerar Opcao B (stored procedure) como otimizacao
