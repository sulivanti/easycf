# Emenda: FR-000-C07

- **Documento base:** [FR-000](../../requirements/fr/FR-000.md)
- **estado_item:** MERGED
- **Natureza:** C (Correção)
- **Data:** 2026-03-25
- **owner:** ECF Core Team
- **Motivação:** 5 use cases Foundation pós-autenticação (change-password, logout, update-profile, create-role, update-role) passam `tenantId: ''` ao `createFoundationEvent()`. Graças ao fallback `|| SYSTEM_TENANT_ID` (DATA-000-C02), o INSERT não falha — mas perde rastreabilidade do tenant real que originou a ação. Estes são fluxos autenticados onde `request.session.tenantId` está disponível e deve ser propagado.
- **rastreia_para:** spec-fix-domain-events-tenant-id v2.0, DATA-000-C02, FR-001 (auth), FR-007 (roles)

---

## Detalhamento

### 1. Problema observado

Use cases autenticados emitem domain events com `tenant_id = SYSTEM_TENANT_ID` em vez do tenant real da sessão. O INSERT funciona (fallback de DATA-000-C02), mas todos os eventos de ações autenticadas ficam atribuídos ao "sistema" — perdendo a rastreabilidade de qual tenant originou a operação.

### 2. Causa-raiz

O codegen gerou `tenantId: ''` como placeholder com comentário `// resolved by caller/middleware`. O middleware de resolução nunca foi implementado. O fallback da factory mascara o problema.

### 3. Correção — Padrão para cada use case

Para cada um dos 5 use cases:

**3.1 — Adicionar `tenantId: string` na interface Input:**
```typescript
interface Input {
  // ... campos existentes mantidos
  tenantId: string;  // ← ADICIONAR
}
```

**3.2 — Passar no `createFoundationEvent()`:**
```typescript
createFoundationEvent({
  tenantId: input.tenantId,  // ← ADICIONAR (era '' → fallback SYSTEM_TENANT_ID)
  // ... demais params mantidos
})
```

**3.3 — Injetar na rota correspondente:**
```typescript
// Na rota Fastify, ao chamar o use case:
tenantId: request.session.tenantId,  // ← ADICIONAR no input
```

### 6. Use cases que NÃO precisam de alteração

| Use Case | Razão |
|----------|-------|
| Login | Pré-auth → `SYSTEM_TENANT_ID` correto (sem sessão) |
| ForgotPassword | Pré-auth → `SYSTEM_TENANT_ID` correto |
| ResetPassword | Pré-auth → `SYSTEM_TENANT_ID` correto |
| RefreshToken | Token flow → `SYSTEM_TENANT_ID` aceitável |
| CreateUser | `tenantId: ''` → fallback funciona; user pode ser cross-tenant |
| DeleteUser | `tenantId: ''` → fallback funciona; operação administrativa |

### 7. Critérios de aceitação

- **AC-001:** Given um usuário autenticado com `session.tenantId = 'xyz...'`, When executa change-password, Then o domain event `auth.password_changed` é persistido com `tenant_id = 'xyz...'` (não `SYSTEM_TENANT_ID`).
- **AC-002:** Given um usuário autenticado, When executa logout, Then o domain event `auth.logout` é persistido com `tenant_id` da sessão.
- **AC-003:** Given um usuário autenticado, When executa update-profile, Then o domain event `user.profile_updated` é persistido com `tenant_id` da sessão.
- **AC-004:** Given um administrador autenticado, When cria ou edita um role, Then os domain events `role.created`/`role.updated` são persistidos com `tenant_id` da sessão.
- **AC-005:** `pnpm -F @ecf/api build` compila sem erros de tipo.
- **AC-006:** Use cases pré-auth (login, forgot-password, reset-password) continuam emitindo eventos com `SYSTEM_TENANT_ID` — sem regressão.
