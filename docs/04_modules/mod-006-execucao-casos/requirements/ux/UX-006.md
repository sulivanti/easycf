## 2. UX-CASE-001 — Painel do Caso em Andamento

### 2.1 Jornada

Como **operador com `process:case:read`**, acesso /casos/:id para visualizar o estado completo do caso (estágio, gates, responsáveis, histórico) e, se tiver scopes adicionais, executar transições, resolver gates, atribuir responsáveis e registrar eventos.

### 2.2 Estrutura (4 painéis)

- **Header (overview):** código (monospace), ciclo + versão, status badge, estágio atual (breadcrumb macroetapa > estágio), ações contextuais (Suspender, Retomar, Cancelar)
- **Barra de Progresso (overview):** macroetapas do ciclo (concluída / atual / futura)
- **Transições Disponíveis (overview):** botões habilitados/desabilitados conforme gates e papéis
- **Aba Gates:** cards por gate com resolução por tipo + dispensa (waive)
- **Aba Responsáveis:** atribuição, reatribuição com confirmação modal
- **Aba Histórico:** timeline intercalando 3 históricos + campo de comentário

### 2.3 Ações Selecionadas do Catálogo UX-010

| action_id (UX-010) | kind | scope | label_pt | endpoint_hint | operation_id | domain_event |
|---|---|---|---|---|---|---|

| `view_history` | query | single | Ver histórico / Timeline | GET /api/v1/cases/:id/timeline | `cases_timeline` | — |
| `approve` | workflow | single | Aprovar gate | POST /api/v1/cases/:id/gates/:gateId/resolve | `cases_gates_resolve` | `case.gate_resolved` |

### 2.4 Happy Path

1. Operador acessa /casos/:id -- skeleton carrega (header, barra de progresso, painel principal)
2. GET /cases/:id retorna dados completos -- header preenche com código, ciclo, status OPEN, estágio atual
3. Barra de progresso exibe macroetapas com a atual destacada
4. Operador verifica aba "Gates" -- gates PENDING exibidos com botão "Resolver"
5. Aprovador resolve gate APPROVAL com decisão APPROVED + parecer -- card muda para badge verde "Aprovado"
6. Todos os gates required RESOLVED -- botões de transição habilitados automaticamente
7. Operador clica "Avançar" -- modal de confirmação -- confirma -- POST /transition -- estágio atualiza
8. Timeline exibe novo item STAGE_CHANGE no topo

### 2.5 Alternativas e Erros

| Cenário | Comportamento |
|---|---|
| Gate APPROVAL resolvido com REJECTED | Card muda para badge vermelho "Reprovado" + parecer visível; transições permanecem bloqueadas |
| Transição bloqueada por gate pending | Botão desabilitado + tooltip: "Gate '{nome}' ainda pendente" |
| Transição bloqueada por papel | Botão desabilitado + tooltip: "Seu papel não autoriza esta transição" |
| Papel required sem atribuição | Card do papel com borda vermelha + label "Obrigatório -- não atribuído"; transições desabilitadas |
| Caso ON_HOLD | Banner amarelo "Caso em espera"; transições e gates bloqueados; botão "Retomar" no header |
| Caso COMPLETED | Badge azul "Concluído" + data; painel readonly exceto comentários |
| Caso CANCELLED | Badge cinza "Cancelado"; painel readonly total (sem comentários) |
| Transição com evidence_required | Mini-form inline: textarea (nota) ou upload de arquivo; botão "Confirmar" só habilita após preenchimento |
| Gate waive | Aviso: "Esta ação será auditada. Informe o motivo." + campo min 20 chars |
| Reatribuição | Autocomplete de usuários → modal: "Substituir {atual} por {novo} no papel {papel}?" |
| Erro 422 do motor | Toast com detail da RFC 9457 + correlationId copiável |
| Erro 403 | Toast: "Você não tem permissão para esta ação." |
| Erro 500 | Toast: "Erro inesperado. Tente novamente." + correlationId |

### 2.6 Estados de UI

| Estado | Comportamento |
|---|---|
| **Loading** | Skeleton no header, barra de progresso, e painel principal (skeleton por aba ao trocar) |
| **Empty (gates)** | "Este estágio não possui gates." (quando estágio sem gates) |
| **Empty (assignments)** | "Nenhum responsável atribuído." + botão "Atribuir" se scope `assign` |
| **Empty (timeline)** | "Nenhum evento registrado ainda." |
| **Error (load)** | Toast: "Não foi possível carregar o caso." + correlationId; botão "Tentar novamente" |
| **Error (action)** | Toast inline com detail RFC 9457; estado anterior mantido |
| **Optimistic update** | Comentários e reatribuições: atualização imediata na UI + rollback se erro |

## 3. UX-CASE-002 — Listagem de Casos

### 3.1 Jornada

Como **operador ou gestor com `process:case:read`**, acesso /casos para visualizar todos os casos do meu tenant, filtrar por critérios relevantes, e abrir novos casos se tiver `process:case:write`.

### 3.2 Estrutura

- **Header:** título "Casos" + contador + botão "Novo Caso" (se scope `write`) + campo de busca
- **Filtros:** ciclo (PUBLISHED), status, estágio atual, "Minha responsabilidade", date range
- **Tabela:** código, ciclo, estágio, status, meu papel, gates pendentes, data abertura
- **Paginação:** cursor-based (botão "Carregar mais")
- **Drawer:** abertura de novo caso
- **Badge:** vermelho para gates pendentes com navegação para aba Gates

### 3.3 Ações Selecionadas do Catálogo UX-010

| action_id (UX-010) | kind | scope | label_pt | endpoint_hint | operation_id | domain_event |
|---|---|---|---|---|---|---|
| `view` | query | collection | Visualizar lista de casos | GET /api/v1/cases | `cases_list` | — |
| `filter` | query | collection | Filtrar casos | GET /api/v1/cases?{filtros} | `cases_list` | — |
| `search` | query | collection | Buscar por código/objeto | GET /api/v1/cases?search={q} | `cases_list` | — |
| `paginate` | query | collection | Carregar mais | GET /api/v1/cases?cursor={c} | `cases_list` | — |
| `create` | command | single | Abrir novo caso | POST /api/v1/cases | `cases_open` | `case.opened` |

### 3.4 Happy Path

1. Operador acessa /casos -- skeleton na tabela
2. GET /cases retorna lista paginada -- tabela preenche com casos do tenant
3. Operador aplica filtro "Minha responsabilidade" -- URL atualiza (?assigned_to_me=true)
4. Operador vê badge vermelho "2" na coluna Gates de um caso -- clica -- navega para /casos/:id aba Gates
5. Operador clica "Novo Caso" -- drawer abre -- seleciona ciclo PUBLISHED -- "Abrir caso"
6. POST /cases retorna 201 -- redireciona para /casos/:id do novo caso

### 3.5 Alternativas e Erros

| Cenário | Comportamento |
|---|---|
| Nenhum caso encontrado | Estado vazio: "Nenhum caso encontrado." + CTA "Abrir primeiro caso" (se scope `write`) |
| Ciclo sem PUBLISHED | Drawer com select vazio + mensagem: "Nenhum ciclo publicado disponível." |
| Erro de abertura (422) | Toast: "Apenas ciclos publicados podem ter novos casos abertos." |
| Erro de abertura (409) | Toast: "Caso já foi aberto (Idempotency-Key duplicado)." |
| Filtros na URL (deep link) | Ao carregar, filtros da URL são aplicados automaticamente |
| Busca sem resultado | Tabela vazia + "Nenhum caso corresponde à busca." |

### 3.6 Estados de UI

| Estado | Comportamento |
|---|---|
| **Loading** | Skeleton com N linhas placeholder na tabela |
| **Empty** | "Nenhum caso encontrado." + CTA contextual |
| **Error** | Toast: "Não foi possível carregar os casos." + correlationId; botão "Tentar novamente" |
| **Partial (paginating)** | Spinner inline no botão "Carregar mais"; novos itens adicionados (append) |

### 3.7 Copy (Mensagens)

| Contexto | Mensagem |
|---|---|
| **Sucesso -- abertura** | "Caso aberto com sucesso." (+ redirect para painel) |
| **Erro -- ciclo não publicado** | "Apenas ciclos publicados podem ter novos casos abertos." |
| **Empty** | "Nenhum caso encontrado." |
| **Empty com filtros** | "Nenhum caso corresponde aos filtros aplicados." |
| **Busca vazia** | "Nenhum caso corresponde à busca '{termo}'." |

## 4. Telemetria (UI Action Envelope)

Ambas as telas DEVEM emitir telemetria conforme DOC-ARC-003, utilizando o pacote `@easycf/ui-telemetry`:

| Campo | Valor |
|---|---|

| `entity_type` | `case_instance`, `gate_instance`, `case_assignment`, `case_event` |
| `entity_id` | ID do caso (ou gate, assignment) |
| `tenant_id` | Do JWT |
| `operation_id` | operationId do OpenAPI (ex: `cases_transition`) |
| `correlation_id` | UUID v4 propagado via X-Correlation-ID |
| `status` | `requested` / `succeeded` / `failed` |
| `duration_ms` | Tempo da ação |
| `http_status` | Código HTTP da resposta |
| `problem_type` | Tipo de erro RFC 9457 (se erro) |
