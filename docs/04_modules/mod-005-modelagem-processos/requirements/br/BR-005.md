# BR-005 — Regras de Negócio da Modelagem de Processos

---

## BR-001 — Ciclo PUBLISHED é Imutável

- **Regra:** Um ciclo com `status=PUBLISHED` **NÃO PODE** ser editado via PATCH em nenhum campo. Qualquer tentativa de edição DEVE retornar 422. O único caminho para modificar um ciclo publicado é criar um fork (nova versão DRAFT).
- **Exemplo:** Analista tenta `PATCH /admin/cycles/:id` com `{ nome: "Novo Nome" }` em ciclo PUBLISHED → 422: "Ciclos publicados são imutáveis. Use o fork para criar uma nova versão."
- **Exceções:** Nenhuma. Regra inegociável de integridade arquitetural.
- **Impacto:** STATE, COMPLIANCE

```gherkin
Funcionalidade: Imutabilidade de Ciclo Publicado

Cenário: Ciclo PUBLISHED rejeita edição direta
  Dado que o ciclo tem status=PUBLISHED
  Quando PATCH /admin/cycles/:id é chamado com qualquer campo
  Então 422: "Ciclos publicados são imutáveis. Use o fork para criar uma nova versão."

Cenário: Imutabilidade se estende a sub-elementos
  Dado que o ciclo tem status=PUBLISHED
  Quando POST /admin/stages/:sid/gates é chamado para adicionar gate
  Então 422: "Ciclo publicado é imutável."
```

## BR-002 — Estágio Inicial Único por Ciclo

- **Regra:** Cada ciclo DEVE ter **no máximo** um estágio com `is_initial=true`. A tentativa de marcar um segundo estágio como inicial DEVE retornar 409. A unicidade é garantida por partial unique index: `UNIQUE(cycle_id) WHERE is_initial=true AND deleted_at IS NULL`.
- **Exemplo:** Ciclo "Comercial v1" já tem estágio "Rascunho" como `is_initial=true`. Analista tenta criar estágio "Entrada" com `is_initial=true` → 409: "Este ciclo já possui um estágio inicial. Desmarque o atual antes de definir outro."
- **Exceções:** Nenhuma.
- **Impacto:** STATE, DATA

```gherkin

Cenário: Rejeitar segundo estágio inicial no mesmo ciclo
  Dado que o ciclo já tem 1 estágio com is_initial=true
```

## BR-003 — Publicação Exige Estágio Inicial

- **Regra:** Um ciclo DRAFT só pode ser promovido a PUBLISHED se possuir ao menos 1 macroetapa e 1 estágio com `is_initial=true`. A publicação sem estágio inicial DEVE retornar 422.
- **Exemplo:** Analista tenta publicar ciclo sem nenhum estágio `is_initial=true` → 422: "O ciclo precisa de ao menos um estágio inicial (is_initial=true) para ser publicado."
- **Exceções:** Nenhuma. Gate de negócio no service de publicação.

## BR-004 — Fork Copia Toda a Estrutura

- **Regra:** Ao executar fork de um ciclo PUBLISHED, o sistema DEVE criar um novo ciclo DRAFT com `parent_cycle_id` apontando para o original, `version` incrementada, e cópia completa de **todas** as macroetapas, estágios, gates, role_links e transições com novos UUIDs. A operação DEVE ser atômica (transação).
- **Exemplo:** Fork de "Comercial v1" (PUBLISHED) → cria "Comercial v2" (DRAFT) com toda a estrutura clonada. IDs são novos; referências internas (ex: `from_stage_id` em transições) apontam para os novos IDs.
- **Exceções:** Nenhuma. Fork só é permitido a partir de ciclo PUBLISHED.
- **Impacto:** DATA, STATE

```gherkin
Funcionalidade: Fork Atômico de Ciclo

Cenário: Fork copia estrutura completa
  Dado que existe ciclo v1 PUBLISHED com 3 macroetapas, 8 estágios, 5 gates e 10 transições
  Quando POST /admin/cycles/:id/fork
  Então novo ciclo DRAFT criado com version=2 e parent_cycle_id=ciclo_v1
  E todas as macroetapas, estágios, gates, role_links e transições são copiados com novos UUIDs
  E referências internas (from_stage_id, to_stage_id) apontam para os novos IDs
```

## BR-005 — Deleção Protegida por Dependências

- **Regra:** A deleção (soft delete) de elementos do blueprint é protegida por dependências hierárquicas:
  - **Macroetapa** com estágios não-deletados → 422: "Desative os estágios antes de remover a macroetapa."
  - **Estágio** com instâncias ativas no MOD-006 → 422 com contagem de instâncias: "Este estágio possui N instância(s) ativa(s) em andamento."
  - **Ciclo** com `status=PUBLISHED` → 422: "Somente ciclos em DRAFT podem ser excluídos. Para descontinuar, use DEPRECATED."
- **Exemplo:** Estágio "Em Análise" tem 5 instâncias ativas → `DELETE /admin/stages/:id` retorna 422 com `extensions.active_instances = 5`.
- **Exceções:** Nenhuma.
- **Impacto:** DATA, COMPLIANCE

```gherkin

Cenário: Macroetapa com estágios ativos bloqueia deleção

  Então 422: "Desative os estágios antes de remover a macroetapa."

Cenário: Estágio com instâncias ativas bloqueia deleção
  Dado que o estágio tem 5 instâncias ativas no MOD-006
  Quando DELETE /admin/stages/:id
```

## BR-007 — Gate INFORMATIVE Nunca Bloqueia Transição

- **Regra:** Um gate com `gate_type=INFORMATIVE` **NUNCA** bloqueia o avanço de estágio, independentemente do valor de `required`. Ele apenas registra informação no histórico. A avaliação de gates é sequencial (campo `ordem`), e um gate INFORMATIVE não impede a avaliação dos subsequentes.
- **Exemplo:** Estágio tem gate "Observação do Analista" (INFORMATIVE, required=true). MOD-006 avalia o gate, registra como informativo no histórico, e permite a transição normalmente.
- **Exceções:** Nenhuma. Regra absoluta do tipo INFORMATIVE.
- **Impacto:** FLOW, STATE

## BR-008 — Transição Proibida Entre Estágios de Ciclos Diferentes

- **Regra:** Uma transição (`stage_transitions`) só pode conectar estágios que pertencem ao **mesmo ciclo**. A validação é feita no service derivando o `cycle_id` de cada estágio via FK chain (`stage → macro_stage → cycle`). Auto-transição (`from_stage_id = to_stage_id`) também é proibida (CHECK na DB).
- **Exemplo:** Estágio A pertence ao Ciclo "Comercial" e Estágio B ao Ciclo "Financeiro". Tentativa de criar transição entre eles → 422: "Os estágios de origem e destino devem pertencer ao mesmo ciclo."
- **Exceções:** Nenhuma. Regra de integridade do grafo.
- **Impacto:** DATA, STATE

```gherkin

Cenário: Rejeitar transição cross-ciclo
```

## BR-009 — Papel de Processo ≠ Role RBAC

- **Regra:** O conceito de **Papel** no MOD-005 (ex: Responsável, Aprovador, Apoio, Consulta, Auditor) é uma participação funcional num estágio de processo. Ele **NÃO** substitui nem duplica o **Role** RBAC do MOD-000-F06 (que define escopos como `process:cycle:read`). O vínculo entre um Papel e um usuário/role real é feito na **atribuição de responsáveis** no MOD-006, não no MOD-005.
- **Exemplo:** Papel "APROVADOR" no catálogo global indica que o estágio espera alguém com poder decisório. Qual usuário concreto será aprovador é definido pelo MOD-006 ao instanciar o ciclo.
- **Exceções:** O campo `can_approve` no papel habilita decisão formal em gates APPROVAL — é uma propriedade do papel de processo, não do role RBAC.
- **Impacto:** PERMISSIONS, DATA

```gherkin
Funcionalidade: Papel vs Role

Cenário: Papel é catálogo de participação, não de permissão RBAC
  Dado que papel "APROVADOR" existe no catálogo global (process_roles)
  E role RBAC "Gerente Financeiro" existe no MOD-000
  Então são entidades independentes — papel define tipo de participação, role define permissões do sistema
  E a ligação entre eles ocorre apenas na atribuição de responsáveis (MOD-006)
```

## BR-010 — Máquina de Estados do Ciclo

- **Regra:** O ciclo segue uma máquina de estados estrita com 3 estados e transições unidirecionais:
  - `DRAFT` → `PUBLISHED` (via endpoint `/publish`)
  - `PUBLISHED` → `DEPRECATED` (via PATCH status)
  - `PUBLISHED` → novo `DRAFT` (via fork — cria **novo** ciclo, não altera o original)
  - `DRAFT` pode ser soft-deleted
  - `DEPRECATED` não aceita novas instâncias no MOD-006, mas instâncias ativas continuam até conclusão
  - Reversão `PUBLISHED → DRAFT` é **proibida**
  - Reversão `DEPRECATED → PUBLISHED` é **proibida**
- **Exemplo:** Analista tenta reverter ciclo DEPRECATED para PUBLISHED → 422.
- **Exceções:** Nenhuma. Transições são unidirecionais e irreversíveis.
- **Impacto:** STATE, COMPLIANCE

```gherkin
Funcionalidade: Máquina de Estados do Ciclo

Cenário: Deprecar ciclo PUBLISHED
  Dado que ciclo tem status=PUBLISHED
  Quando PATCH /admin/cycles/:id com { status: "DEPRECATED" }
  Então status=DEPRECATED e novas instâncias bloqueadas

Cenário: Rejeitar depreciação de DRAFT
  Dado que ciclo tem status=DRAFT
  Quando PATCH /admin/cycles/:id com { status: "DEPRECATED" }
  Então 422: "Somente ciclos publicados podem ser depreciados."

Cenário: Soft delete só em DRAFT
  Dado que ciclo tem status=PUBLISHED
  Quando DELETE /admin/cycles/:id
  Então 422: "Somente ciclos em DRAFT podem ser excluídos. Para descontinuar, use DEPRECATED."
```

## BR-011 — Ciclo DEPRECATED Bloqueia Novas Instâncias

- **Regra:** Um ciclo com `status=DEPRECATED` **NÃO PODE** aceitar novas instâncias no MOD-006. A validação de status é feita no ponto de entrada do MOD-006 ao tentar instanciar. Instâncias já ativas em estágios do ciclo depreciado DEVEM continuar normalmente até conclusão ou cancelamento.
- **Exemplo:** Ciclo "Comercial v1" é depreciado. MOD-006 tenta `POST /instances` com `cycle_version_id` apontando para v1 → 422: "Ciclo depreciado não aceita novas instâncias."
- **Exceções:** Nenhuma. Instâncias em andamento não são afetadas pela depreciação.
- **Impacto:** STATE, FLOW

```gherkin
Funcionalidade: Depreciação Bloqueia Instanciação

Cenário: Ciclo DEPRECATED rejeita nova instância

  Então 422: "Ciclo depreciado não aceita novas instâncias."

Cenário: Instâncias ativas continuam após depreciação
  Dado que o ciclo foi depreciado com 3 instâncias ativas
  Quando MOD-006 consulta instâncias ativas do ciclo
  Então as 3 instâncias permanecem ativas e podem avançar normalmente
```

## BR-012 — Reordenação Automática de Elementos

- **Regra:** Ao inserir ou remover macroetapas, estágios ou gates, o sistema DEVE recalcular automaticamente o campo `ordem` dos demais itens da mesma coleção para manter sequência contígua (1, 2, 3...), sem lacunas. A reordenação ocorre via PATCH com novo valor de `ordem`, e todos os itens entre a posição antiga e a nova são deslocados atomicamente (dentro da mesma transação).
- **Exemplo:** Macroetapa com ordem=2 é deletada de um ciclo com 4 macroetapas → as macroetapas 3 e 4 passam a ser 2 e 3 respectivamente, dentro da mesma transação de soft-delete.
- **Exceções:** Nenhuma. A reordenação é obrigatória para garantir consistência do campo `UNIQUE(parent_id, ordem)`.
- **Impacto:** DATA, FLOW

```gherkin
Funcionalidade: Reordenação Contígua

Cenário: Deleção reordena itens subsequentes
  Dado que ciclo tem 4 macroetapas com ordem 1,2,3,4
  Quando DELETE /admin/macro-stages/:id (ordem=2)
  Então macroetapa antiga ordem 3 passa para 2, e antiga 4 para 3

Cenário: Inserção na posição intermediária desloca subsequentes
  Dado que ciclo tem 3 macroetapas com ordem 1,2,3
  Quando POST /admin/cycles/:cid/macro-stages com { ordem: 2 }
  Então a nova macroetapa recebe ordem=2
  E as macroetapas anteriores com ordem >=2 avançam para 3 e 4
```
