- **Motivação:** O formulário UX-ORG-002 (OrgFormPage) não implementa o error handling definido no UX-001 §Alternativas/Erros. Erros 5xx não mostram toast, erros 400/403 são silenciados, e a função `extractFieldErrors` (RFC 9457) existe mas nunca é chamada. Resultado: criação de org units (especialmente N3) falha silenciosamente sem feedback ao usuário.

## Detalhamento

### C01 — Error handler do formulário deve tratar todos os status HTTP

**Requisito violado:** UX-001 §Alternativas/Erros (linhas 233-243) define:
- 409 → Inline campo codigo
- 422 → Inline form
- 403 → EmptyState "Sem permissão"
- 5xx → Toast "Erro inesperado. Tente novamente."

**Estado atual do código:** `OrgFormPage.tsx` linhas 114-126 trata apenas 409 e 422. Qualquer outro status cai em `else` e executa `setFieldErrors({})` — **limpando erros sem feedback**.

**Correção requerida:**

```typescript
// useEffect de activeError — CORRIGIDO
useEffect(() => {
  if (activeError instanceof ApiError) {
    const errors: Record<string, string> = {};

    if (activeError.status === 409) {
      errors.codigo = COPY.validation.codigoDuplicate;
    } else if (activeError.status === 422) {
      // Extrair field-level errors do RFC 9457
      const fieldMap = extractFieldErrors(activeError.problem.extensions);
      for (const [field, msg] of fieldMap) {
        errors[field] = msg;
      }
      if (fieldMap.size === 0) {
        errors._form = activeError.message;
      }
    } else {
      // 400, 403, 5xx — mensagem genérica inline
      errors._form = activeError.message;
    }

    startTransition(() => setFieldErrors(errors));
  } else if (activeError) {
    // Erro de rede (TypeError, AbortError)
    startTransition(() => setFieldErrors({
      _form: COPY.error.networkError,
    }));
  } else {
    startTransition(() => setFieldErrors({}));
  }
}, [activeError]);
```

### C03 — Toast de erro no callback `onError` do submit

O callback `onError` do `createMutation.mutate` (linha 148) e do `updateMutation.mutate` são vazios (`() => {}`). Conforme UX-001, erros 5xx DEVEM mostrar toast.

**Correção:**
```typescript
// create mode
createMutation.mutate(data, {
  onSuccess: (result) => {
    toast.success(COPY.toast.createSuccess(result.codigo, result.nome));
    onSuccess(result.id);
  },
  onError: (error) => {
    const message = error instanceof ApiError
      ? error.message
      : COPY.error.networkError;
    toast.error(message);
  },
});

// edit mode — mesmo padrão
updateMutation.mutate(data, {
  onSuccess: (result) => {
    toast.success(COPY.toast.updateSuccess(result.codigo, result.nome));
    onSuccess(result.id);
  },
  onError: (error) => {
    const message = error instanceof ApiError
      ? error.message
      : COPY.error.networkError;
    toast.error(message);
  },
});
```

### C05 — Incluir campos `filial`, `telefone`, `email_contato` no handleSubmit create

Apesar de o design UX-001 §Modo Criar não listar esses campos no formulário, o `CreateOrgUnitRequest` e o `handleSubmit` DEVEM incluí-los para paridade com a API (que já os aceita). Os campos são opcionais — se não renderizados, seus valores serão string vazia e o spread condicional os ignora.

**Fix no interface:**
```typescript
export interface CreateOrgUnitRequest {
  codigo: string;
  nome: string;
  descricao?: string | null;
  parent_id?: string | null;
  cnpj?: string;
  razao_social?: string;
  filial?: string;
  responsavel?: string;
  telefone?: string;
  email_contato?: string;
}
```

**Fix no handleSubmit:**
```typescript
const data: CreateOrgUnitRequest = {

  ...(cnpj.trim() && { cnpj: cnpj.trim() }),
```
