## INT-001 — Integracao com MOD-006 (Execucao de Casos)

- **Tipo:** Inbound (MOD-006 chama MOD-007)
- **Modo:** api (HTTP sincrono)
- **Auth:** Service-to-service via JWT com scope `param:engine:evaluate` (concedido ao service account do MOD-006)
- **Criticidade:** CRITICA — MOD-006 depende desta integracao para avaliar parametrizacao contextual durante transicoes de estagio

### Endpoint Consumido

| Campo | Valor |
|---|---|
| **Metodo** | POST |
| **URL** | /api/v1/routine-engine/evaluate |
| **operationId** | `routine_engine_evaluate` |
| **Scope** | `param:engine:evaluate` |

### Ponto de Integracao

O motor de transicao do MOD-006 (passo [5] do fluxo de transicao de estagio) chama o motor de avaliacao do MOD-007 para obter campos visiveis, obrigatorios, defaults, dominios, validacoes e bloqueantes para o objeto em transicao no contexto ativo.

### Contrato — Request

```json
{
  "object_type": "PEDIDO_VENDA",
  "object_id": "uuid-do-objeto",
  "context": [
    {
      "framer_id": "uuid-enquadrador-1"
    },
    {
      "framer_id": "uuid-enquadrador-2"
    }
  ],
  "stage_id": "uuid-estagio-destino",
  "dry_run": false
}
```

| Campo | Tipo | Obrigatorio | Descricao |
|---|---|---|---|
| `object_type` | string | SIM | Codigo do target_object (ex: PEDIDO_VENDA) |
| `object_id` | uuid | NAO | ID da instancia do objeto (opcional — para futura personalizacao) |
| `context` | array | SIM | Array de enquadradores ativos (minimo 1 — FR-009) |
| `context[].framer_id` | uuid | SIM | ID do enquadrador ativo |
| `stage_id` | uuid | NAO | Estagio destino no MOD-006 (para filtros futuros) |
| `dry_run` | boolean | NAO | Se `true`, motor executa avaliacao completa (passos 1-5) mas NAO persiste domain_event `routine.applied`. Default: `false`. (FR-009, PEN-007/PEN-006) |

### Contrato — Response (200 OK)

```json
{
  "visible_fields": ["campo_a", "campo_b"],
  "hidden_fields": ["deposito_id"],
  "required_fields": ["projeto_wbs"],
  "optional_fields": ["campo_c"],
  "defaults": {
    "estado": "SP",
    "moeda": "BRL"
  },
  "domain_restrictions": {
    "tipo_op": ["A", "B"]
  },
  "validations": [
    {
      "field": "projeto_wbs",
      "rule": "not_empty",
      "message": "Projeto WBS e obrigatorio para esta operacao"
    }
  ],
  "blocking_validations": [
    {
      "field": "projeto_wbs",
      "rule": "not_empty",
      "message": "Projeto WBS e obrigatorio para esta operacao"
    }
  ],
  "applied_routines": [
    {
      "routine_id": "uuid-rotina",
      "routine_code": "ROT-SERV-01",
      "version": 2,
      "items_applied": 4
    }
  ],
  "dry_run": false
}
```

### Comportamento de Integracao

- Se `blocking_validations` nao vazio: MOD-006 DEVE bloquear a transicao de estagio com 422, listando as validacoes bloqueantes
- Se response retorna arrays vazios (nenhuma rotina ativa): MOD-006 prossegue sem restricoes
- Se o motor retorna erro (500): MOD-006 DEVE bloquear a transicao e retornar erro ao usuario (fail-closed)
- Se `dry_run: true`: response contem resultado identico mas nenhum domain_event e persistido. Util para pre-visualizacao no frontend (UX dry-run preview). MOD-006 NAO deve usar `dry_run: true` em transicoes reais — apenas para preview.

### Failure Behavior

| Parametro | Valor |
|---|---|
| **timeout_ms** | 5000 |
| **retries** | 0 (nao faz retry — call sincrona no fluxo do usuario) |
| **backoff** | n/a |
| **dlq** | false |
| **circuit_breaker** | recomendado — se motor indisponivel, MOD-006 bloqueia transicoes com mensagem amigavel |
| **fallback** | fail-closed (nenhuma transicao permitida se motor indisponivel) |

### Observabilidade

- `X-Correlation-ID` propagado do request MOD-006 para o request ao motor
- Metricas: latencia p95/p99, taxa de sucesso/erro, contagem de rotinas aplicadas
- Logs estruturados: object_type, framers_count, routines_applied_count, blocking_count, duration_ms
- Traces: span dedicado `routine-engine.evaluate` com tags de contexto

## Resolucao: Endpoint FR-010 (link-routine)

**Contexto:** O FR-010 referencia endpoint `POST /admin/incidence-rules/:id/link-routine` que nao estava no catalogo de 24 endpoints do mod.md.

**Decisao:** O vinculo rotina-incidencia sera implementado como **sub-recurso** do endpoint de regras de incidencia:

| Metodo | Path | operationId | Scope |
|---|---|---|---|
| POST | /api/v1/admin/incidence-rules/:id/link-routine | `admin_incidence_rules_link_routine` | `param:framer:write` |
| DELETE | /api/v1/admin/incidence-rules/:id/unlink-routine/:routineId | `admin_incidence_rules_unlink_routine` | `param:framer:write` |

Isso eleva o total de endpoints para **23 + 2 = 25 endpoints**. O mod.md foi atualizado para incluir estes 2 endpoints adicionais.

### Contrato — Link Routine (POST)

```json
// POST /api/v1/admin/incidence-rules/:ruleId/link-routine
// Request:
{
  "routine_id": "uuid-da-rotina-published"
}

// Response 201:
{
  "id": "uuid-do-link",
  "routine_id": "uuid-da-rotina",
  "incidence_rule_id": "uuid-da-regra",
  "created_at": "2026-03-19T10:00:00Z"
}

// Error 422 (rotina nao PUBLISHED):
{
  "type": "https://ecf.api/errors/routine-not-published",
  "title": "Rotina nao publicada",
  "detail": "Apenas rotinas publicadas podem ser vinculadas a regras de incidencia.",
  "status": 422,
  "correlationId": "uuid"
}
```

### Contrato — Unlink Routine (DELETE)

```json
// DELETE /api/v1/admin/incidence-rules/:ruleId/unlink-routine/:routineId
// Response 204: No Content

// Error 404 (vinculo nao encontrado):
{
  "type": "https://ecf.api/errors/link-not-found",
  "title": "Vinculo nao encontrado",
  "detail": "Nao existe vinculo entre esta rotina e regra de incidencia.",
  "status": 404,
  "correlationId": "uuid"
}
```
