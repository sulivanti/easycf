# SEC-006 — Segurança e Compliance da Execução de Casos

---

## 1. Autenticação e Sessão

- Todos os 16 endpoints requerem sessão ativa (Bearer JWT) via MOD-000 Foundation (DOC-FND-000 §1)
- `X-Correlation-ID` (UUID v4) obrigatório em todas as requests e propagado em todos os domain events e logs
- Kill-switch de sessão (DOC-FND-000 §1.2) se aplica — revogação imediata invalida todas as operações em andamento
- Não há fluxo de login, JWT ou RBAC customizado neste módulo — herda integralmente do Foundation

## 2. Autorização (RBAC)

### 2.1 Mapeamento Scope → Endpoint

| Scope | Endpoints Protegidos | Decorator |
|---|---|---|
| `process:case:read` | GET /cases, GET /cases/:id, GET /gates, GET /assignments, GET /events, GET /timeline | `@RequireScope('process:case:read')` |
| `process:case:write` | POST /cases, POST /transition, POST /hold, POST /resume, POST /events | `@RequireScope('process:case:write')` |
| `process:case:cancel` | POST /cases/:id/cancel | `@RequireScope('process:case:cancel')` |
| `process:case:gate_resolve` | POST /gates/:gateId/resolve | `@RequireScope('process:case:gate_resolve')` |
| `process:case:gate_waive` | POST /gates/:gateId/waive | `@RequireScope('process:case:gate_waive')` |
| `process:case:assign` | POST /assignments, PATCH /assignments/:aid | `@RequireScope('process:case:assign')` |
| `process:case:reopen` | POST /cases/:id/events (event_type=REOPENED) | `@RequireScope('process:case:reopen')` |

> Todos os 7 scopes estão registrados no catálogo canônico de DOC-FND-000 §2.2 via Amendments DOC-FND-000-M01 (6 scopes base) e DOC-FND-000-M02 (scope reopen). Ref: PEN-006/PENDENTE-004, PEN-006/PENDENTE-001.

### 2.2 Autorização Complementar (além de RBAC)

| Regra | Contexto | Referência |
|---|---|---|
| `can_approve=true` no papel atribuído ao estágio | Resolução de gate APPROVAL — 403 se papel sem poder decisório | BR-008 |
| `allowed_roles` na transição do blueprint | Motor de transição — 403 se papel do usuário não listado | FR-002 passo 3 |
| Papel required=true com atribuição ativa | Motor de transição — 422 se papel obrigatório sem atribuição | BR-006 |
| Scope `process:case:reopen` para REOPENED | Reabertura de caso COMPLETED — permite reverter status COMPLETED para OPEN via evento REOPENED. Ação excepcional de alto impacto e auditada. Ref: PEN-006/PENDENTE-001 Opção B | BR-016 |

### 2.3 Princípio de Menor Privilégio

- `process:case:gate_waive` DEVE ser concedido apenas a perfis de auditoria/gestão (risco: dispensa de controles obrigatórios)
- `process:case:cancel` DEVE ser separado de `process:case:write` para evitar cancelamentos acidentais
- `process:case:assign` DEVE ser restrito a gestores do caso — operadores comuns NÃO devem reatribuir

## 3. Classificação de Dados

| Campo | Tabela | Classificação | Justificativa |
|---|---|---|---|
| `cancellation_reason` | case_instances | **Interno** | Pode conter contexto de negócio sensível (ex: nome de cliente, motivo comercial) |
| `parecer` | gate_instances | **Interno** | Opinião do aprovador — pode conter referências a pessoas ou valores |
| `descricao` | case_events | **Interno** | Conteúdo livre digitado pelo usuário — risco de PII inadvertida |
| `evidence.url` | stage_history, gate_instances | **Interno** | URLs de documentos — proteger por presigned URL com TTL curto |
| `substitution_reason` | case_assignments | **Interno** | Motivo de reatribuição — pode referenciar pessoas |
| `checklist_items` | gate_instances | **Interno** | Itens de checklist — conteúdo operacional |
| `metadata` | case_events | **Interno** | JSON livre — monitorar para PII inadvertida |
| `codigo` | case_instances | **Público** | Identificador amigável sem informação sensível |
| `status`, `current_stage_id` | case_instances | **Público** | Estado operacional sem dados pessoais |

### 3.1 Campos Mascararáveis (maskable_fields)

Conforme DATA-003 (catálogo de eventos), os seguintes campos DEVEM ser truncados/mascarados nos payloads de domain events:

| Campo | Evento(s) | Regra de Mascaramento |
|---|---|---|
| `cancellation_reason` | `case.cancelled` | Truncar a 200 chars no payload do evento |
| `parecer` | `case.gate_resolved` | Truncar a 200 chars no payload do evento |
| `substitution_reason` | `case.assignment_replaced` | Truncar a 200 chars no payload do evento |
| `descricao` | `case.event_recorded` | Truncar a 200 chars no payload do evento |

> Mascaramento no payload do evento NÃO substitui a classificação ACL — `sensitivity_level` é guard-rail, não regra principal (DOC-FND-000 §3).

## 4. Tenant Isolation (Row-Level Authorization)

- **MUST:** Todas as queries DEVEM filtrar por `tenant_id` no WHERE — sem exceção (DOC-FND-000 §2)
- `case_instances.tenant_id` é a **fonte de verdade** para isolamento de todo o aggregate
- Joins com `stage_history`, `gate_instances`, `case_assignments` e `case_events` DEVEM sempre partir de `case_instances` (garantindo propagação do filtro tenant)
- Consultas à tabela `domain_events` para timeline/auditoria DEVEM incluir `WHERE tenant_id = :tenant_id` (SEC-002)
- Endpoints GET de listagem (cases, gates, assignments, events, timeline) DEVEM **sempre** incluir tenant_id como primeiro predicado no query plan

### 4.1 Validação de Pertencimento

- POST /cases/:id/* — antes de qualquer operação, validar que `case_instances.tenant_id = jwt.tenant_id`
- Se o tenant_id do caso divergir do JWT: retornar 404 (não 403) para evitar information disclosure

## 5. Auditoria

### 5.1 Domain Events como Audit Trail

- Todas as 11 operações de escrita emitem domain events para tabela `domain_events` (DATA-003)
- Cada evento contém: `id`, `tenant_id`, `entity_type`, `entity_id`, `event_type`, `payload`, `created_at`, `created_by`, `correlation_id`, `sensitivity_level`
- `X-Correlation-ID` propagado do request HTTP para o domain event — rastreabilidade end-to-end

### 5.2 Ações Críticas (Auditoria Reforçada)

| Ação | Evento | Registro Adicional | Justificativa |
|---|---|---|---|
| Gate waive | `case.gate_waived` | `case_events` (EXCEPTION) + `domain_events` | Dispensa de controle obrigatório — duplo registro para auditoria interna e domain |
| Cancelamento | `case.cancelled` | `domain_events` com `cancellation_reason` | Ação irreversível (sem UNCANCELLED) |
| Reabertura | `case.event_recorded` (REOPENED) | `case_events` (REOPENED) + `domain_events` | Reverte caso COMPLETED — ação excepcional |
| Reatribuição | `case.assignment_replaced` | `case_events` (REASSIGNED) + `domain_events` | Registro de anterior/novo com motivo |

### 5.3 Causation Chain

- `causation_id` DEVE ser preenchido quando um evento causa outro (DATA-003):
  - Transição para estágio terminal: `case.stage_transitioned` → `case.completed` (causation_id = id do stage_transitioned)
- Permite reconstrução completa da cadeia causal em auditoria

## 6. Gate Waive — Controles Especiais

- Escopo `process:case:gate_waive` DEVE ser concedido apenas a perfis de auditoria/gestão
- Motivo obrigatório: **min 20 caracteres** (BR-014) — validação no backend (não apenas frontend)
- Evento `case.gate_waived` com `sensitivity_level=1` para visibilidade em auditoria
- Duplo registro: `case_events` (EXCEPTION, com gate_id + motivo) + `domain_events`
- UI DEVE exibir aviso explícito: "Esta ação será auditada. Informe o motivo." (UX-006, F03)
- Relatórios de auditoria DEVEM destacar waivers como exceções operacionais

## 7. Retenção de Dados

| Dado | Política de Retenção | Justificativa |
|---|---|---|
| `case_instances` | **Indefinida** (caso pode estar ativo por meses/anos) | Processos de negócio com ciclo longo |
| `stage_history` | **Indefinida** (append-only, vinculada ao caso) | Histórico de estágio é audit trail operacional |
| `gate_instances` | **Indefinida** (vinculada ao caso) | Decisões formais com valor legal/compliance |
| `case_assignments` | **Indefinida** (soft-toggle, nunca deletada) | Histórico de responsabilidade |
| `case_events` | **Indefinida** (append-only) | Timeline operacional |
| `domain_events` | **Padrão Foundation** (retenção configurada no MOD-000) | Auditoria geral |
| Evidence files (URLs) | **Padrão Storage** (DOC-FND-000 §6) | Documentos vinculados a gates |

> **LGPD:** Se campos de texto livre (`descricao`, `parecer`, `cancellation_reason`, `substitution_reason`) contiverem PII inadvertida, o processo de anonimização DEVE ser tratado via pipeline de dados — não há exclusão física de registros append-only.

## 8. LGPD e Proteção de Dados Pessoais

### 8.1 Campos com Risco de PII

| Campo | Risco | Mitigação |
|---|---|---|
| `descricao` (case_events) | Alto — conteúdo livre digitado pelo usuário | Aviso na UI: "Não inclua dados pessoais." + monitoramento por pipeline |
| `parecer` (gate_instances) | Médio — opinião do aprovador pode citar pessoas | Truncamento no payload de eventos (200 chars) |
| `cancellation_reason` | Médio — motivo pode referenciar clientes/funcionários | Truncamento no payload de eventos (200 chars) |
| `substitution_reason` | Baixo — motivo de reatribuição | Truncamento no payload de eventos (200 chars) |
| `evidence.url` | Baixo — URL protegida por presigned URL | TTL curto (1h) nas presigned URLs |

### 8.2 Direitos do Titular (LGPD Art. 18)

- **Acesso:** Dados do caso são acessíveis via GET /cases/:id (scope `process:case:read`)
- **Retificação:** Registros append-only NÃO podem ser editados — retificação via novo evento (COMMENT) com correção
- **Eliminação:** Não aplicável para registros de audit trail (LGPD Art. 16, inciso IV — cumprimento de obrigação legal/regulatória)
- **Portabilidade:** Dados exportáveis via API (GET endpoints)

## 9. Proteções de Input

| Vetor | Mitigação | Referência |
|---|---|---|
| SQL Injection | Queries parametrizadas (ORM) + sanitização do campo `search` na listagem | NFR-006 §8 |
| Path Traversal | `evidence.filename` sanitizado — remover `/`, `\`, `..` | NFR-006 §8 |
| XSS (stored) | `descricao`, `parecer`, `motivo` — sanitização de HTML na renderização (frontend) | DOC-ARC-003 |
| SSRF | `evidence.url` — validação de URL (apenas `https://`, domínios permitidos) | NFR-006 §8 |
| Replay Attack | `Idempotency-Key` com TTL 60s em POST /cases | FR-001, INT-006 §2.1 |
| Brute Force | Rate limiting padrão da plataforma | DOC-GNP-00 |
