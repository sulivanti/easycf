## Detalhamento

> **Contexto:** Este amendment deriva de [FR-000-M01](../fr/FR-000-M01.md).
> Consulte o amendment pai para o contexto completo do campo `role_id` no `CreateUserRequest`.

### Nova regra: SEC-000 §2.3 — Anti-escalação na atribuição de roles

**Situação atual (SEC-000 §2):** Documenta RBAC por scopes (3 segmentos `dominio:recurso:acao`), catálogo de scopes, e cache Redis. NÃO contém regra sobre atribuição de roles na criação de usuários.

**Problema:** FR-000-M01 §3 adiciona `role_id` opcional ao `CreateUserRequest`. Se o backend aceitar qualquer `role_id` sem validação, um usuário com escopo `users:user:write` mas role limitada (ex: "Operador") pode criar usuários com role "Super Admin", obtendo indiretamente todos os scopes do sistema.

### Regra SEC-ANTI-ESC-001 — Validação de anti-escalação

**Definição:** O backend DEVE rejeitar `POST /api/v1/users` com `role_id` quando a role referenciada contém scopes que o usuário autenticado NÃO possui.

**Algoritmo:**

```text
1. Extrair scopes do JWT do requisitante (auth.scopes[])
2. Consultar scopes da role referenciada (role_permissions WHERE role_id = request.role_id)
3. Computar diferença: target_scopes - requester_scopes
4. Se diferença não-vazia → 403 Forbidden
```

**Response de erro (RFC 9457):**

```json
{
  "type": "/problems/privilege-escalation-denied",
  "title": "Privilege escalation denied",
  "status": 403,
  "detail": "Cannot assign role with scopes you do not possess: [system:audit:sensitive, users:user:delete].",
  "extensions": {
    "missing_scopes": ["system:audit:sensitive", "users:user:delete"],
    "correlationId": "{correlation_id}"
  }
}
```

**Exceção:** Usuários com scope `users:role:write` (gestores de roles) podem atribuir qualquer role. Justificativa: quem pode criar/editar roles já tem capacidade implícita de escalar privilégios.

### Gherkin

```gherkin
Cenário: Admin com scopes limitados tenta atribuir role superior
  Dado que o admin tem scopes ["users:user:read", "users:user:write", "users:role:read"]
  E que a role "Super Admin" tem scopes ["users:user:read", "users:user:write", "users:user:delete", "system:audit:sensitive"]
  Quando POST /api/v1/users com role_id = id da role "Super Admin"
  Então status 403 com type "/problems/privilege-escalation-denied"
  E detail contém "system:audit:sensitive, users:user:delete"

Cenário: Admin com todos os scopes atribui qualquer role
  Dado que o admin tem todos os scopes do catálogo canônico
  Quando POST /api/v1/users com role_id = id da role "Super Admin"
  Então status 201 com usuário criado

Cenário: Gestor de roles pode atribuir qualquer role (exceção)
  Dado que o admin tem scope "users:role:write"
  E NÃO possui scope "system:audit:sensitive"
  Quando POST /api/v1/users com role_id = id da role que inclui "system:audit:sensitive"
  Então status 201 com usuário criado (exceção de gestores de roles)

Cenário: Criação sem role_id não requer validação de escalação
  Dado que o admin tem scope "users:user:write"
  Quando POST /api/v1/users sem role_id
  Então status 201 com usuário criado sem vinculação
```

## Impacto nos Pilares

- **FR-000:** Nenhum impacto adicional — FR-000-M01 §3 já documenta a validação condicional (422s). A regra 403 de anti-escalação é complementar.
- **BR-000:** Considerar adicionar BR-015 (ou similar) para formalizar como regra de negócio. Prioridade baixa — a regra SEC é suficiente para o codegen.
- **DATA-000:** Nenhum impacto — validação é em memória (comparação de arrays de scopes)
