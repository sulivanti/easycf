# SEC-002 — Matriz de Autorização de Eventos (Gestão de Usuários)

> Modelo canônico: DOC-FND-000 §3

## Princípios (MUST)

- **Não use "permissão no evento" como fonte de verdade.**
  - **Emit** é controlado pela permissão do **comando** que gera o evento.
  - **View** é controlado pela permissão de leitura da **entity originária** (ACL) + `tenant_id`.
- `visibility_level`/`sensitivity_level` **não substituem** ACL/RBAC/ABAC: servem apenas como **guard-rail/otimizador** (mascarar/filtrar), mantendo o enforcement forte em ACL + tenant.
- **Autorização de Linha (MUST):** toda leitura em `domain_events` e `notifications` MUST filtrar por `tenant_id` e respeitar a ACL do registro originário.
- **Nota Arquitetural (Performance):** Proibido iterar em memória (N+1). O `canRead` de listagens **MUST** ser executado por JOIN ou subquery eficiente contra `tenant_users` ou mapeamento de Roles.

## Glossário

- **Emit**: quem pode disparar o evento (derivado do comando).
- **View**: quem pode ler/visualizar eventos (timeline/auditoria).
- **Notify**: quem recebe notificações (inbox/real-time), resolvido por regra (watchers + papéis + hierarquia).
- **Owner**: o usuário alvo da ação (ex: o usuário que foi criado/desativado).
- **Creator**: o admin que executou a ação.

## Matriz de Eventos — MOD-002

### `user.created`

| Campo | Valor |
|---|---|
| **Ação** | Criar usuário (convite ou senha temporária) |
| **event_type** | `user.created` |
| **Emit (perm do comando)** | `users:user:write` |
| **View (regra)** | `canRead(user) && tenantMatch` |
| **Notify** | creator + admin do módulo |
| **Sensibilidade** | `sensitivity_level=0` |
| **Regras de payload** | snapshot mínimo: `{ user_id, fullName, roleId, mode, status }`. **Sem e-mail** (LGPD). |
| **Mascaramento** | `email` — se presente por erro, mascarar para `sensitivity_level >= 1` |
| **Outbox** | Sim (modo convite) — envio de e-mail de ativação. `dedupe_key`: `user_id + invite` |

### `user.deactivated`

| Campo | Valor |
|---|---|
| **Ação** | Desativar usuário (soft-delete) |
| **event_type** | `user.deactivated` |
| **Emit (perm do comando)** | `users:user:delete` |
| **View (regra)** | `canRead(user) && tenantMatch`; hard delete → auditor/admin |
| **Notify** | owner (usuário desativado) + admin |
| **Sensibilidade** | `sensitivity_level=1` |
| **Regras de payload** | metadados: `{ user_id, deactivated_by, reason_code, timestamp }`. **Sem e-mail, sem nome completo** (LGPD). |
| **Mascaramento** | `email`, `fullName` — não incluir; dados acessíveis via entity originária |
| **Outbox** | Não |

### `user.invite_resent`

| Campo | Valor |
|---|---|
| **Ação** | Reenviar convite de ativação |
| **event_type** | `user.invite_resent` |
| **Emit (perm do comando)** | `users:user:write` |
| **View (regra)** | `canRead(user) && tenantMatch` |
| **Notify** | admin (usuário PENDING recebe e-mail via Outbox, não via notificação in-app) |
| **Sensibilidade** | `sensitivity_level=1` |
| **Regras de payload** | metadados: `{ user_id, resent_by, attempt_number, timestamp }`. **Sem e-mail ou token** (LGPD). |
| **Mascaramento** | `email` — nunca no payload; `token` — nunca exposto |
| **Outbox** | Sim — reenvio de e-mail via Outbox Pattern. `dedupe_key`: `user_id + invite_resend + timestamp`. TTL: 60s. Retries: 3. |

## Regras de Enforcement (MUST)

1. **`tenant_id` obrigatório:** Toda query em `domain_events` MUST incluir `WHERE tenant_id = :current_tenant`
2. **ACL da entity originária:** A leitura de eventos de um `user` requer `canRead(user)` do leitor — não basta ter scope genérico
3. **Mascaramento antes de retornar:** Se `sensitivity_level >= 1`, campos mascaráveis DEVEM ser removidos ou mascarados no payload antes de expor ao frontend
4. **Sem PII em payloads:** Política LGPD (BR-002) — e-mail, CPF e dados sensíveis NUNCA no payload de domain_events. Dados completos acessíveis via entity originária com ACL check.

> **Anti-pattern Foundation (DOC-FND-000 §3):** Não use `visibility_level`/`sensitivity_level` como substituto de ACL/RBAC. São apenas guard-rails.
