# INT-001 — Integrações com MOD-000 Foundation API

- **Sistema externo:** MOD-000 Foundation (Users API — F05 + Roles API — F06)
- **Propósito:** Consumir endpoints REST existentes para gestão de usuários no backoffice. MOD-002 é UX-First e **não cria endpoints** — apenas consome.
- **Modo:** API (REST/JSON sobre HTTPS)
- **Auth:** Bearer Token JWT (sessão ativa via MOD-000 Foundation auth). Token propagado automaticamente pelo HTTP client global.

---

## Contratos por operationId

### `users_list` — Listagem Paginada

- **Método:** `GET /api/v1/users`
- **Usado em:** FR-001 (UX-USR-001) — listagem, busca, filtros, paginação
- **Query params:**
  - `q` (string, opcional) — busca textual por nome/e-mail
  - `status` (enum: ACTIVE|PENDING|BLOCKED|INACTIVE, opcional) — filtro por status
  - `role_id` (uuid, opcional) — filtro por perfil
  - `cursor` (string, opcional) — cursor para paginação
  - `limit` (integer, default: 20) — registros por página
- **Response 200:**

  ```json
  {
    "data": [{ "id": "uuid", "fullName": "string", "email": "string", "status": "enum", "roleId": "uuid", "roleName": "string", "createdAt": "ISO8601" }],
    "meta": { "nextCursor": "string|null", "total": "integer" }
  }
  ```

- **Erros possíveis:** 401, 403, 500

### `users_create` — Criação de Usuário

- **Método:** `POST /api/v1/users`
- **Usado em:** FR-002 (UX-USR-002) — formulário de cadastro
- **Headers adicionais:** `Idempotency-Key: UUID v4` (BR-005)
- **Request body (modo convite):**

  ```json
  { "fullName": "string", "email": "string", "roleId": "uuid", "mode": "invite" }
  ```

- **Request body (modo senha):**

  ```json
  { "fullName": "string", "email": "string", "roleId": "uuid", "password": "string", "forcePasswordReset": true }
  ```

- **Response 201:**

  ```json
  { "data": { "id": "uuid", "fullName": "string", "status": "PENDING|ACTIVE", "createdAt": "ISO8601" } }
  ```

- **Erros possíveis:** 401, 403, 409 (e-mail duplicado), 422 (`extensions.invalid_fields[]`), 500

### `users_get` — Detalhes do Usuário

- **Método:** `GET /api/v1/users/:id`
- **Usado em:** FR-003 (UX-USR-003) — tela de convite, card de status
- **Response 200:**

  ```json
  { "data": { "id": "uuid", "fullName": "string", "status": "enum", "inviteTokenExpired": "boolean", "createdAt": "ISO8601" } }
  ```

- **Erros possíveis:** 401, 403, 404, 500

### `users_delete` — Desativar Usuário (Soft-Delete)

- **Método:** `DELETE /api/v1/users/:id`
- **Usado em:** FR-001 (UX-USR-001) — ação "Desativar" no dropdown
- **Response 200:**

  ```json
  { "data": { "id": "uuid", "status": "INACTIVE" } }
  ```

- **Erros possíveis:** 401, 403, 404, 500

### `users_invite_resend` — Reenviar Convite de Ativação

- **Método:** `POST /api/v1/users/:id/invite/resend`
- **Usado em:** FR-003 (UX-USR-003) — botão "Reenviar convite"
- **Headers adicionais:** `Idempotency-Key: UUID v4` (BR-005, TTL: 60s)
- **Pré-condição:** Apenas `status=PENDING`. Outros status retornam 422.
- **Response 200:**

  ```json
  { "message": "Invite sent" }
  ```

- **Erros possíveis:** 401, 403, 404, 422 (status inválido), 500
- **Nota:** Requer amendment no MOD-000-F05. Contrato mínimo documentado em US-MOD-002-F03 §1.

### `roles_list` — Listagem de Perfis de Acesso

- **Método:** `GET /api/v1/roles`
- **Usado em:** FR-001 (UX-USR-001 — filtro), FR-002 (UX-USR-002 — select de perfil)
- **Response 200:**

  ```json
  {
    "data": [{ "id": "uuid", "name": "string", "description": "string" }]
  }
  ```

- **Erros possíveis:** 401, 403, 500

## Headers Obrigatórios (todos os endpoints)

| Header | Obrigatório | Descrição |
|---|---|---|
| `Authorization` | Sim | Bearer Token JWT (sessão ativa) |
| `X-Correlation-ID` | Sim | UUID propagado em todas as chamadas (DOC-ESC-001 §3.3) |
| `Idempotency-Key` | Condicional | UUID v4 em POST mutáveis: `users_create`, `users_invite_resend` (BR-005) |
| `Content-Type` | Condicional | `application/json` em requisições com body (POST) |

## Comportamento de Falha (Frontend)

| Aspecto | Estratégia |
|---|---|
| **Timeout** | Herda configuração global do HTTP client (recomendado: 15s) |
| **Retry** | Não automático — usuário controla via UI (botão retry, formulário preservado) |
| **Fallback** | Estados de erro com toast + correlationId conforme error_mapping dos Screen Manifests (BR-006) |
| **Dead Letter Queue** | N/A (frontend) |
| **Circuit Breaker** | N/A (frontend) |

## Contrato de Erro (RFC 9457 — DOC-FND-000 §5)

Todas as respostas de erro do MOD-000 seguem o formato **Problem Details for HTTP APIs (RFC 9457)**. O frontend DEVE consumir este formato para alimentar o error_mapping dos Screen Manifests (BR-006).

### Estrutura de Erro Padrao

```json
{
  "type": "https://api.example.com/errors/validation-error",
  "title": "Validation Error",
  "detail": "Um ou mais campos estao invalidos.",
  "status": 422,
  "correlationId": "uuid-propagado-do-X-Correlation-ID",
  "extensions": {
    "invalid_fields": [
      { "field": "email", "message": "Informe um e-mail valido." },
      { "field": "fullName", "message": "Nome completo e obrigatorio." }
    ]
  }
}
```

### Mapeamento de Erros por Status (BR-006)

| HTTP Status | Campo relevante | Uso no frontend |
|---|---|---|
| 401 | `detail` | Modal "Sessao expirada" — ignorar detail, usar copy fixa |
| 403 | `detail` | Toast "Sem permissao" — ignorar detail, usar copy fixa |
| 404 | `detail` | Mensagem inline na tela de convite |
| 409 | `detail` | Mensagem inline no campo e-mail |
| 422 | `extensions.invalid_fields[]` | Inline por campo com `field` + `message` |
| 500 | `correlationId` | Toast generico + correlationId copiavel |

> **MUST:** O interceptor HTTP global DEVE verificar se o payload segue RFC 9457 antes de parsear. Se nao seguir, exibir toast generico com correlationId do header.

## CORS e Seguranca de Transporte

- **CORS:** Gerido pelo MOD-000 Foundation (gateway/middleware). Frontend assume `Access-Control-Allow-Origin` configurado.
- **HTTPS:** Obrigatorio em todos os ambientes (exceto localhost dev).
- **Cookies:** Nenhum cookie gerido pelo MOD-002 — auth via Bearer Token no header.
- **CSP:** Content Security Policy gerida pelo Application Shell (MOD-001).
