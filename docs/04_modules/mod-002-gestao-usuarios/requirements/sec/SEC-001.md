# SEC-001 — Segurança e Compliance (Gestão de Usuários)

## 1. Autenticação

- **Modelo:** Sessão ativa obrigatória via Bearer Token JWT (herda MOD-000 Foundation auth — DOC-FND-000 §1)
- **Anti-pattern Foundation:** MOD-002 NÃO implementa fluxos de login, JWT ou sessão. Assume `@RequireScope` e `tenant_id` como filtro mandatório providos nativamente pelo Foundation.
- **Sessão expirada (401):** Modal "Sessão expirada. Faça login novamente." em todas as telas (error_mapping dos manifests)

## 2. Autorização (RBAC)

- **Modelo:** RBAC por scopes — formato `dominio:entidade:acao` (DOC-FND-000 §2.1)
- **Estratégia de UI:** `hide_if_no_permission` — elementos sem permissão não são renderizados (BR-001)

### Scopes Utilizados (DOC-FND-000 §2.2)

| Scope | Tipo | Onde é usado | Comportamento sem scope |
|---|---|---|---|
| `users:user:read` | Leitura | Acesso a /usuarios, /usuarios/:id/convite | Redirect para /dashboard com toast |
| `users:user:write` | Escrita | Botão "Novo Usuário", /usuarios/novo, "Reenviar convite" | Elemento não renderizado |
| `users:user:delete` | Exclusão | Ação "Desativar" no dropdown | Elemento não renderizado |

### Proteção de Rota (Frontend Guard)

| Rota | Scope mínimo | Redirect sem scope |
|---|---|---|
| `/usuarios` | `users:user:read` | `/dashboard` |
| `/usuarios/novo` | `users:user:write` | `/usuarios` |
| `/usuarios/:id/convite` | `users:user:read` | `/usuarios` |

> **Nota (DOC-FND-000 §2.3):** Guard de frontend é UX, não substitui proteção de rota no backend. O backend DEVE exigir `requireScope()` no `preHandler` (enforcement real).

## 3. Classificação dos Dados

| Dado | Classificação | Sensitivity Level | Presente no MOD-002? |
|---|---|---|---|
| E-mail do usuário | **Restrito (PII)** | 2 | Sim — campo de input no formulário; NUNCA em toasts/modais/erros |
| Nome completo | **Confidencial** | 1 | Sim — exibido em tabela, modal de desativação, título de convite |
| Status do usuário | Interno | 0 | Sim — badge, filtro |
| Perfil de acesso (role) | Interno | 0 | Sim — coluna da tabela, select do formulário |
| Data de criação | Interno | 0 | Sim — coluna da tabela |
| Token de convite | **Restrito** | 2 | NÃO — nunca trafegado para o frontend |

## 4. Política de Retenção

- **Frontend:** Nenhum dado persistido em localStorage/sessionStorage. Estado gerido exclusivamente via React state/query cache.
- **Backend (MOD-000-F05):** Segue política de retenção do Foundation:
  - `domain_events`: manter por 5 anos (auditoria)
  - Registros soft-deleted (`deleted_at`): expurgar após política definida em ADR

## 5. Mascaramento (LGPD — BR-002)

| Campo | Quando mascarar | Como mascarar |
|---|---|---|
| E-mail | Em TODOS os toasts, modais e mensagens de erro | Não exibir — usar texto genérico |
| E-mail | Em payloads de domain_events (DATA-003) | Não incluir no payload |
| Nome completo | Em payloads de `user.deactivated` | Não incluir — acessível via entity originária |

### Regras LGPD Específicas

1. Toast de criação (convite): "Usuário criado com sucesso. Convite enviado." — **sem e-mail**
2. Toast de criação (senha): "Usuário criado com sucesso." — **sem e-mail**
3. Modal de desativação: "O usuário **{nome}** perderá acesso imediatamente." — **nome, nunca e-mail**
4. Toast de reenvio: "Convite reenviado com sucesso." — **sem e-mail**
5. Erro 409: mensagem inline no campo, sem revelar e-mail existente
6. Erro 500: mensagem genérica + correlationId, sem PII

## 6. Auditoria

- **Eventos auditáveis:** `user.created`, `user.deactivated`, `user.invite_resent` (DATA-003)
- **Trilha de mudanças:** Gerida pelo MOD-000-F05 via `domain_events` table
- **Quem pode ver:** `canRead(user) && tenantMatch` + `system:audit:read` para auditores (DOC-FND-000 §2.2)
- **Campos sensíveis em auditoria:** Requerem `system:audit:sensitive` (DOC-FND-000 §2.2)

## 7. Autorização de Eventos (SEC-002)

- **Referência:** `requirements/sec/SEC-002.md` — Matriz de Autorização Emit/View/Notify
- **Princípio MUST:** Não usar "permissão no evento" como fonte de verdade. Usar permissão no **comando** (Emit) e permissão na **entity/ACL** (View).
- **Autorização de Linha MUST:** toda leitura em `domain_events` e `notifications` MUST filtrar por `tenant_id` e respeitar ACL do registro originário.

## 8. Proteções

| Proteção | Aplicação |
|---|---|
| Idempotency-Key | POST /users e POST /invite/resend (BR-005) |
| Rate Limiting | Gerido pelo MOD-000 Foundation (gateway/middleware) |
| Validação de entrada | Validação client-side (onBlur, onSubmit) + validação server-side (422) |
| Logs seguros | correlationId em todas as chamadas; nenhum PII nos logs de erro |
| Secrets | Nenhum secret no frontend — tokens geridos pelo HTTP client |

## 9. Segurança de Transporte e XSS (INT-001, NFR-001 §11)

| Aspecto | Requisito | Referência |
|---|---|---|
| **HTTPS** | Obrigatório em todos os ambientes (exceto localhost dev) | INT-001 §CORS |
| **CORS** | Gerido pelo MOD-000 Foundation (gateway). Frontend assume `Access-Control-Allow-Origin` configurado. | INT-001 §CORS |
| **CSP** | Não utilizar `eval()`, inline scripts ou `unsafe-inline`. CSP gerido pelo Application Shell (MOD-001). | NFR-001 §11 |
| **XSS** | Todos os dados da API renderizados via React (auto-escaped). Nunca `dangerouslySetInnerHTML`. | NFR-001 §11 |
| **Token storage** | Bearer Token gerido pelo HTTP client global. Nunca em localStorage acessível a JS. | NFR-001 §11 |
| **PII em DevTools** | Console logs NÃO devem conter e-mail ou dados sensíveis (DOC-ESC-001 §3.2) | NFR-001 §11 |

## 10. Modelo de Ameaças (Threat Model) — Frontend

| Ameaça | Mitigação | Ref |
|---|---|---|
| **Exfiltração de PII via toast/erro** | Toasts nunca exibem e-mail (BR-002). Erros 500 exibem apenas correlationId. | BR-002, BR-006 |
| **Token hijacking via XSS** | httpOnly cookies (MOD-000). Sem acesso a token via JS. CSP sem `unsafe-inline`. | NFR-001 §11 |
| **Scope escalation via UI** | Guard de rota é UX — enforcement real no backend via `requireScope()` (DOC-FND-000 §2.3) | SEC-001 §2 |
| **CSRF** | Bearer Token via header (não cookie de sessão). Framework mitiga via SameSite. | INT-001 |
| **Replay de requisição** | `Idempotency-Key` em POST mutáveis (BR-005). Cooldown 60s no reenvio (BR-004). | BR-004, BR-005 |
| **Enumeração de e-mail via 409** | Mensagem inline genérica "Este e-mail já está cadastrado no sistema." — sem revelar o e-mail existente | BR-002, INT-001 |
| **Request pendente em navegação** | AbortController cancela requests ao navegar (NFR-001 §10) | NFR-001 §10 |

## 11. Contrato de Erro Seguro (RFC 9457 — INT-001)

- Todas as respostas de erro do MOD-000 seguem **Problem Details for HTTP APIs (RFC 9457)** (INT-001 §Contrato de Erro).
- O interceptor HTTP global DEVE verificar formato RFC 9457 antes de parsear.
- Erros fora do formato: toast genérico + correlationId do header.
- **MUST:** Campos `detail` de erros 409/422 NUNCA devem ser exibidos como-estão se contiverem PII. O frontend usa copy fixa (BR-006).
