## FR-001 — CRUD de Tipos de Enquadrador

- **Goal:** Permitir criar e listar tipos de enquadrador como catálogo base (OPERACAO, CLASSE_PRODUTO, TIPO_DOCUMENTO, CONTEXTO_PROCESSO).
- **Endpoint:** GET/POST /api/v1/admin/framer-types
- **Scope:** `param:framer:read` / `param:framer:write`
- **Dependências:** BR-001 (codigo imutável), DATA-007 (context_framer_types), DATA-003, SEC-007
- **Idempotency:** n/a (CRUD padrão)
- **Timeline/Notifications:** yes — emite `param.framer_type_created` (DATA-003)
- **Done Funcional:**

## FR-002 — CRUD de Enquadradores com Vigência

- **Goal:** Permitir criar, listar, atualizar e inativar enquadradores de contexto com vigência obrigatória, código imutável e expiração automática por background job.
- **Endpoint:** GET/POST/PATCH/DELETE /api/v1/admin/framers
- **Scope:** `param:framer:read` / `param:framer:write` / `param:framer:delete`
- **Dependências:** BR-001 (codigo imutável), BR-002 (vigência), BR-013 (tenant isolation), DATA-007 (context_framers), DATA-003, SEC-007
- **Idempotency:** n/a (CRUD padrão; background job de expiração é idempotente por natureza — re-executar no mesmo enquadrador já INACTIVE não gera side-effect)
- **Timeline/Notifications:** yes — emite `param.framer_created`, `param.framer_updated`, `param.framer_expired` (DATA-003)
- **Done Funcional:**
  1. POST cria enquadrador com valid_from obrigatório, status=ACTIVE
  2. GET lista com paginação cursor-based e filtro por status/framer_type
  3. PATCH atualiza campos (exceto codigo — BR-001)
  4. DELETE soft-delete (status=INACTIVE)
  5. Background job `framer-expiration.job.ts` expira enquadradores com valid_until passado
  6. Emite domain events conforme DATA-003

```gherkin
Funcionalidade: CRUD de Enquadradores com Vigência

Cenário: Criar enquadrador com vigência
  Dado que o usuário tem scope param:framer:write
  Quando POST /admin/framers com { codigo: "SERV-ENG", nome: "Serviço de Engenharia", framer_type_id: "uuid-operacao", valid_from: "2026-04-01" }
  Então 201 com enquadrador em status=ACTIVE
  E domain event param.framer_created emitido

Cenário: Soft-delete de enquadrador
```

## FR-004 — CRUD de Regras de Incidência

- **Goal:** Permitir criar, listar, atualizar e inativar regras de incidência que vinculam enquadradores a objetos-alvo, com UNIQUE constraint e detecção de conflitos em config-time.
- **Endpoint:** GET/POST/PATCH/DELETE /api/v1/admin/incidence-rules
- **Scope:** `param:framer:read` / `param:framer:write` / `param:framer:delete`
- **Dependências:** BR-003 (conflito bloqueado), BR-011 (priority removido), DATA-007 (incidence_rules), DATA-003, SEC-007
- **Idempotency:** n/a (CRUD padrão)
- **Timeline/Notifications:** yes — emite `param.incidence_rule_created`, `param.incidence_rule_updated` (DATA-003)
- **Done Funcional:**
  1. POST cria regra verificando UNIQUE (framer_id, target_object_id) — se conflito retorna 422 (BR-003)
  2. GET lista regras com paginação cursor-based, filtro por framer_id, target_object_id, status
  3. PATCH atualiza vigência (valid_from, valid_until) e status
  4. DELETE soft-delete (status=INACTIVE)
  5. Campo `priority` removido — nenhuma prioridade numérica aceita (BR-011)
  6. Emite domain events conforme DATA-003
  7. **Limite configurável (PEN-007/PENDENTE-005):** POST retorna 422 se enquadrador já atingiu o limite de regras de incidência. Default: 10 regras por enquadrador. Configurável por tenant. Response 422: `{error: "LIMIT_EXCEEDED", current: N, max: M, configurable: true}`

- **Endpoint:** GET/POST/PATCH /api/v1/admin/routines
- **Scope:** `param:routine:read` / `param:routine:write`
- **Dependências:** BR-001 (codigo imutável), BR-005 (imutabilidade), BR-012 (deprecação), DATA-007 (behavior_routines), DATA-003, SEC-007
- **Idempotency:** n/a (CRUD padrão)
- **Timeline/Notifications:** yes — emite `param.routine_created` (DATA-003)
- **Done Funcional:**
  1. POST cria rotina DRAFT com codigo UNIQUE, nome, routine_type=BEHAVIOR
  2. GET lista com paginação cursor-based e filtro por status, routine_type
  3. GET /:id retorna rotina com itens, links de incidência e version history
  4. PATCH atualiza rotina (somente DRAFT — BR-005)
  5. Emite domain event `param.routine_created`

## FR-006 — CRUD de Itens de Rotina

- **Goal:** Permitir adicionar, editar e remover itens de uma rotina DRAFT. 7 tipos de item suportados com formulários adaptativos por tipo.
- **Endpoint:** POST /api/v1/admin/routines/:id/items, PATCH/DELETE /api/v1/admin/routine-items/:id
- **Scope:** `param:routine:write`
- **Dependências:** BR-005 (PUBLISHED imutável), DATA-007 (routine_items), UX-007
- **Idempotency:** n/a (CRUD padrão)
- **Timeline/Notifications:** no
- **Done Funcional:**
  1. POST adiciona item com item_type, action, target_field_id, value (jsonb), condition_expr, ordem
  2. PATCH atualiza item existente (somente se rotina em DRAFT — BR-005)
  3. DELETE remove item (somente se rotina em DRAFT — BR-005)
  4. 7 tipos suportados: FIELD_VISIBILITY, REQUIRED, DEFAULT, DOMAIN, DERIVATION, VALIDATION, EVIDENCE
  5. 8 ações possíveis: SHOW, HIDE, SET_REQUIRED, SET_OPTIONAL, SET_DEFAULT, RESTRICT_DOMAIN, VALIDATE, REQUIRE_EVIDENCE
  6. Campo `is_blocking` (boolean) indica se o item bloqueia transição no MOD-006
  7. Campo `ordem` (integer) define sequência de avaliação no motor
  8. **Limite configurável (PEN-007/PENDENTE-005):** POST retorna 422 se rotina já atingiu o limite de itens. Default: 50 itens por rotina. Configurável por tenant. Response 422: `{error: "LIMIT_EXCEEDED", current: N, max: M, configurable: true}`

## FR-007 — Publicação de Rotina

- **Goal:** Promover rotina DRAFT para PUBLISHED após validação de pré-condições (ao menos 1 item).
- **Endpoint:** POST /api/v1/admin/routines/:id/publish
- **Scope:** `param:routine:publish`
- **Dependências:** BR-005 (imutabilidade), BR-006 (mínimo 1 item), DATA-007, DATA-003, SEC-002
- **Idempotency:** required — publicação duplicada de rotina já PUBLISHED retorna 422 sem side-effect
- **Timeline/Notifications:** yes — emite `param.routine_published` (DATA-003), notifica admin (SEC-002)
- **Done Funcional:**
  1. Valida rotina em status=DRAFT (422 se não DRAFT)
  2. Valida ao menos 1 item vinculado (422 se vazio — BR-006)
  3. Atualiza status para PUBLISHED, published_at=now(), approved_by=current_user
  4. Se `auto_deprecate_previous: true` E rotina tem `parent_routine_id` (é fork): marca versão anterior (parent) como DEPRECATED, emite `param.routine_deprecated` (PEN-007/PENDENTE-004)
     - **Nota (PEN-007/PENDENTE-007):** Links `routine_incidence_links` da versão deprecada permanecem no banco como histórico. O motor ignora (filtra PUBLISHED). A nova versão já possui seus próprios links (copiados pelo fork — ADR-003 ponto 3). UX exibe badge "versão deprecada" nos links inativos (UX-007 §IncidenceMatrix).
  5. Emite domain event `param.routine_published`
- **Request Body (opcional):**

```json
{
  "auto_deprecate_previous": false
}
```

| Campo | Tipo | Default | Descrição |
|---|---|---|---|
| `auto_deprecate_previous` | boolean | `false` | Se `true` e a rotina é um fork (tem `parent_routine_id`), a versão PUBLISHED anterior é automaticamente marcada DEPRECATED. Default conservador (PEN-007/PENDENTE-004). |

```gherkin

  Então 200 com status=PUBLISHED, published_at preenchido
  E domain event param.routine_published emitido

Cenário: Rejeitar publicação de rotina sem itens
  Dado que rotina DRAFT tem 0 itens
  Quando POST /admin/routines/:id/publish
  Então 422: "Rotinas sem itens não podem ser publicadas."

Cenário: Rejeitar publicação de rotina já PUBLISHED
  Dado que rotina tem status=PUBLISHED
  Quando POST /admin/routines/:id/publish

  Quando POST /admin/routines/:id/publish

Cenário: Publicar fork com auto_deprecate_previous=true

Cenário: auto_deprecate_previous ignorado se rotina não é fork

  Quando POST /admin/routines/:id/publish com { "auto_deprecate_previous": true }
  Então 200 com status=PUBLISHED
  E nenhuma rotina é deprecada (flag ignorado silenciosamente)
```

## FR-008 — Fork de Rotina

- **Goal:** Criar nova versão DRAFT a partir de rotina PUBLISHED, copiando itens e links de incidência, registrando histórico de versão.
- **Endpoint:** POST /api/v1/admin/routines/:id/fork
- **Scope:** `param:routine:write`
- **Dependências:** BR-008 (fork copia itens e links), DATA-007 (routine_version_history), DATA-003
- **Idempotency:** n/a (cada chamada cria nova versão — não é idempotente)
- **Timeline/Notifications:** yes — emite `param.routine_forked` (DATA-003)
- **Done Funcional:**
  1. Valida rotina em status=PUBLISHED (422 se DRAFT ou DEPRECATED)
  2. Cria novo DRAFT com version=original.version+1 e parent_routine_id=original.id
  3. Copia todos os itens com novos UUIDs preservando ordem e configuração
  4. Copia links de incidência (routine_incidence_links) para nova versão
  5. Registra routine_version_history com change_reason obrigatório
  6. Emite domain event `param.routine_forked`

```gherkin
Funcionalidade: Fork de Rotina

Cenário: Fork de rotina PUBLISHED com sucesso
  Dado que rotina "ROT-SERV-01" está PUBLISHED com 4 itens e 2 links
  Quando POST /admin/routines/:id/fork com { change_reason: "Ajuste campo WBS" }
  Então 201 com nova rotina em DRAFT, version=original+1
  E 4 itens copiados com novos UUIDs
  E 2 links de incidência copiados
  E routine_version_history criado com change_reason="Ajuste campo WBS"
  E domain event param.routine_forked emitido

Cenário: Rejeitar fork de rotina DEPRECATED
  Dado que rotina tem status=DEPRECATED
  Quando POST /admin/routines/:id/fork com { change_reason: "Teste" }
  Então 422: "Somente rotinas PUBLISHED podem ser forkadas."
```

## FR-009 — Motor de Avaliação (Runtime)

- **Goal:** Endpoint de avaliação que executa 6 passos sequenciais para retornar campos visíveis, obrigatórios, defaults, domínios, validações e bloqueantes para um objeto em determinado contexto. Ponto de integração principal com MOD-006 (motor de transição).
- **Endpoint:** POST /api/v1/routine-engine/evaluate
- **Scope:** `param:engine:evaluate`
- **Dependências:** BR-004 (safety net), BR-009 (sem cache), BR-010 (events com efeito), BR-012 (ignora DEPRECATED), BR-013 (tenant isolation), FR-001, FR-002, FR-003, FR-004, FR-005, FR-006, DATA-007, DATA-003, SEC-002, INT-007
- **Idempotency:** n/a (leitura + side-effect condicional de domain_event; re-chamar com mesmos params gera novo evento se houver efeito)
- **Timeline/Notifications:** yes — persiste `routine.applied` em domain_events quando ao menos 1 rotina é aplicada (DATA-003)
- **Done Funcional:**
  1. Recebe request body: `{ object_type, object_id?, context: [{ framer_id }], stage_id?, dry_run? }`
  2. Valida context array não vazio (422 se vazio)
  3. **Passo 1:** Encontrar regras de incidência ativas para os framers fornecidos no context
  4. **Passo 2:** Encontrar rotinas PUBLISHED vinculadas via routine_incidence_links (ignora DEPRECATED — BR-012)
  5. **Passo 3:** Avaliar itens de cada rotina por ordem crescente
  6. **Passo 4:** Resolver conflitos por restritividade — safety net (BR-004): HIDE > SHOW, SET_REQUIRED > SET_OPTIONAL, domínio menor prevalece
  7. **Passo 5:** Construir response com arrays separados
  8. **Passo 6:** Se applied_routines > 0 **E** `dry_run != true`, persistir domain_event `routine.applied` (BR-010). Se `dry_run: true`, suprimir emissão do domain event (PEN-007/PENDENTE-006)
  9. Retorna: `{ visible_fields, hidden_fields, required_fields, optional_fields, defaults, domain_restrictions, validations, blocking_validations, applied_routines, dry_run? }`
  10. Se nenhuma rotina ativa: 200 com arrays vazios, sem domain_event (BR-010)
  11. Sem cache em nenhum passo — consulta banco ao vivo (BR-009)
  12. Se `dry_run: true`: motor executa avaliação completa (passos 1-5), retorna resultado idêntico com campo `dry_run: true`, mas **NÃO** persiste domain_event `routine.applied` (EVT-012). Sem side-effects. (PEN-007/PENDENTE-006)

## FR-010 — Vínculo de Rotina a Regra de Incidência

- **Goal:** Permitir vincular uma rotina PUBLISHED a uma ou mais regras de incidência, criando a ponte entre contexto (enquadrador + objeto) e comportamento (rotina).
- **Endpoint:** POST /api/v1/admin/incidence-rules/:id/link-routine (ou via payload na criação da regra)
- **Scope:** `param:framer:write`
- **Dependências:** BR-007 (somente PUBLISHED), BR-012 (DEPRECATED bloqueado), DATA-007 (routine_incidence_links)
- **Idempotency:** required — vincular mesma combinação (routine_id, incidence_rule_id) retorna 422 por UNIQUE constraint sem side-effect
- **Timeline/Notifications:** no
- **Done Funcional:**
  1. Valida rotina em status=PUBLISHED (422 se DRAFT — BR-007, 422 se DEPRECATED — BR-012)
  2. Valida UNIQUE (routine_id, incidence_rule_id) — rejeita duplicata
  3. Cria routine_incidence_link
  4. Permite desvincular (DELETE) se necessário
