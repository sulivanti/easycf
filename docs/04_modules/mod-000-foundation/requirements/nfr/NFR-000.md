# NFR-000 — Requisitos Não Funcionais do Foundation

---

## 1. SLOs (Service Level Objectives)

| Métrica | Alvo | Observações |
|---|---|---|
| Latência p95 (login) | ≤ 300ms | Inclui bcrypt compare + session insert |
| Latência p95 (CRUD) | ≤ 200ms | Endpoints de leitura/escrita padrão |
| Latência p95 (SSO callback) | ≤ 1000ms | Inclui roundtrip ao provider externo |
| Disponibilidade | ≥ 99.5% | Medida mensal, exclui manutenção programada |
| Taxa de erro (5xx) | ≤ 0.1% | Excluindo erros de input do cliente |

## 2. Topologia

- **Tipo:** `hybrid` (sync para CRUD, async para e-mail/jobs)
- **Endpoints síncronos:** Login, logout, refresh, CRUD de users/roles/tenants, sessions
- **Processamento assíncrono:** E-mail de boas-vindas, password reset email, purge de uploads efêmeros, re-encoding de avatares

## 3. Healthchecks

| Check | Endpoint/Método | Criticidade |
|---|---|---|
| API alive | `GET /info` | Alta — se falhar, pod reinicia |
| PostgreSQL | Connection pool ping | Alta — se falhar, todas as operações param |
| Redis | `PING` | Média — degradado (fallback DB para RBAC) |
| Storage (MinIO/S3) | `HeadBucket` | Baixa — uploads indisponíveis, CRUD funciona |

## 4. Disaster Recovery

| Métrica | Alvo | Estratégia |
|---|---|---|
| **RPO** (Recovery Point Objective) | ≤ 1 hora | Backups incrementais PostgreSQL + WAL streaming |
| **RTO** (Recovery Time Objective) | ≤ 30 min | Failover automático de DB + re-deploy containers |

## 5. Limites e Throttling

| Recurso | Limite | Enforcement |
|---|---|---|
| Login attempts | 10 req / 15 min / IP | Rate limiter middleware (BR-003) |
| API geral | 100 req / min / usuário | Rate limiter por token |
| Upload (presign) | 10 req / min / usuário | Rate limiter |
| Tamanho de upload | `STORAGE_MAX_ATTACHMENT_BYTES` (env) | Validação no presign |
| Paginação | limit máximo = 100 | Validação Zod na borda |
| Bulk operations | ≤ 1000 items / request | Acima disso → job assíncrono |

## 6. Observabilidade

### 6.1 Correlação E2E (MUST)

- `X-Correlation-ID` propagado em toda requisição (inbound → logs → response → outbound → events)
- Logs estruturados JSON com campos: `timestamp`, `level`, `correlationId`, `userId`, `tenantId`, `method`, `path`, `statusCode`, `duration_ms`
- **Nunca** logar PII/segredos (DOC-ESC-001 §3.2)

### 6.2 Métricas

| Métrica | Tipo | Labels |
|---|---|---|
| `http_request_duration_ms` | Histogram | method, route, status |
| `http_requests_total` | Counter | method, route, status |
| `auth_login_total` | Counter | result (success/failed/blocked/mfa) |
| `redis_cache_hits_total` | Counter | operation (get/set/del) |
| `redis_cache_misses_total` | Counter | — |
| `storage_upload_bytes_total` | Counter | entity_type |

### 6.3 Traces

- Spans básicos em I/O: queries PostgreSQL, chamadas Redis, HTTP outbound (SSO providers)
- Propagação de trace context via headers (W3C Trace Context quando disponível)

## 7. Segurança (complementar a SEC-000)

| Proteção | Implementação |
|---|---|
| CORS | Configurável por ambiente (origin whitelist) |
| Helmet | Headers de segurança (X-Frame-Options, CSP, etc.) |
| Sanitização de input | Zod na borda + parameterized queries |
| Secrets management | Env vars, nunca hardcoded, scanner no CI (EX-CI-003) |
| Dependency scanning | npm audit strict + Trivy + Dependabot (EX-CI-003) |

## 8. Escalabilidade

- **Horizontal:** Stateless API (sessions em DB, não em memória) — escala via réplicas de container
- **Redis:** Single instance para cache RBAC; cluster quando necessário
- **PostgreSQL:** Read replicas para queries pesadas (listagens, auditoria)
- **Storage:** Provider-agnóstico (MinIO local, S3 cloud) — escala independente
