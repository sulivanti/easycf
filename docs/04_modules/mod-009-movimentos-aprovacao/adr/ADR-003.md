# ADR-003 — Outbox Pattern para Domain Events de Movimentos

## Contexto

O MOD-009 emite 13 domain events (DATA-003). Esses eventos servem múltiplos propósitos: auditoria, notificação de aprovadores, timeline de movimentos, e integração com módulos consumidores. A decisão é quais eventos exigem garantia de entrega via outbox pattern e quais podem ser emitidos de forma síncrona simples.

Os eventos se dividem em duas categorias:

- **Eventos 1-3 (administrativos):** `approval.control_rule_created`, `approval.control_rule_updated`, `approval.approval_rule_created`. Operações de configuração, sem side-effects externos críticos.
- **Eventos 4-13 (movimentos):** `movement.created`, `movement.approved`, `movement.rejected`, `movement.executed`, `movement.failed`, `movement.cancelled`, `movement.overridden`, `movement.escalated`, `movement.timeout`, `movement.auto_approved`. Operações com side-effects: notificações a aprovadores, callbacks de execução, timeline de compliance.

Requisitos de contexto:

- Notificações de aprovação (eventos 4, 5, 6, 9, 10, 11) são críticas — aprovador precisa ser notificado para decidir.
- Execução pós-aprovação (evento 7) precisa de garantia at-least-once — operação aprovada deve ser executada.
- DR com RPO=0 (NFR-009 §4) — perda de evento de movimento é inaceitável.

## Decisão

**Outbox pattern habilitado para eventos 4-13 (movimentos); desabilitado para eventos 1-3 (administrativos).**

Eventos de movimentos são escritos na tabela `domain_events` (outbox) dentro da mesma transação que persiste o estado do movimento. Um outbox processor assíncrono consome e publica os eventos, garantindo entrega at-least-once. Cada evento possui `dedupe_key` no formato `movement:{movement_id}:{event_suffix}` para deduplicação por consumidores.

Eventos administrativos (1-3) são emitidos de forma síncrona simples — INSERT em `domain_events` na transação, sem outbox processor. Justificativa: são operações de configuração sem notificações externas, sem callbacks, e sem side-effects que exijam garantia de entrega.

## Alternativas Consideradas

1. **Outbox para todos os 13 eventos:** Overhead desnecessário para eventos administrativos (1-3) que não possuem consumidores assíncronos. O outbox processor teria que processar eventos sem side-effects, aumentando carga sem benefício.

2. **Nenhum outbox — publicação direta em message broker:** Publish diretamente após commit da transação. Problemático: se o publish falhar após o commit, o evento é perdido (dual-write problem). Para eventos de movimento, isso é inaceitável (RPO=0, notificações críticas).

3. **CDC (Change Data Capture) via WAL:** Capturar mudanças nas tabelas de movimentos via WAL e gerar eventos automaticamente. Complexidade operacional elevada; acopla schema de tabelas a contratos de eventos; difícil adicionar `dedupe_key` e `payload_policy` customizado por evento.

4. **Outbox seletivo por evento individual:** Avaliar caso a caso cada evento. A divisão natural já é clara (admin vs. movimentos), e avaliar individualmente adicionaria complexidade de configuração sem benefício — todos os eventos de movimento têm o mesmo perfil de consumo.

## Consequências

- **Positivas:** Garantia at-least-once para notificações e callbacks de execução; transação atômica (estado do movimento + evento no outbox); deduplicação via `dedupe_key`; RPO=0 mantido; eventos administrativos sem overhead desnecessário.
- **Negativas:** Latência entre emit e delivery dos eventos (eventual consistency para notificações — mitigada pelo inbox como fallback); complexidade do outbox processor (healthcheck, monitoramento de lag).
- **Riscos:** Outbox processor parado acumula eventos — alerta configurado para > 15 min sem processamento (NFR-009 §6.5). DLQ crescendo indica problema de consumidores (alerta para DLQ depth > 10 por 10 min).
