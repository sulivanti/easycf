# SEC-002 — Matriz de Autorização de Eventos da Estrutura Organizacional

> Modelo canônico conforme DOC-FND-000 §3.

## Princípios (MUST)

- **Não use "permissão no evento" como fonte de verdade.**
  - **Emit** é controlado pela permissão do **comando** que gera o evento.
  - **View** é controlado pela permissão de leitura da **entity originária** via RBAC (`org:unit:read`).
- `sensitivity_level` **não substitui** ACL/RBAC: serve apenas como **guard-rail**.
- **Cross-tenant (MUST — ADR-003):** `org_units` NÃO possui coluna `tenant_id`. Toda filtragem de leitura em domain events com `entity_type=org_unit` é via RBAC (`org:unit:read`), não via `tenant_id`.

## Glossário

- **Emit**: quem pode disparar o evento (derivado do comando).
- **View**: quem pode ler/visualizar eventos (timeline/auditoria).
- **Notify**: quem recebe notificações (inbox/real-time), resolvido por regra.
- **Owner/Requester/Approver**: papéis típicos definidos por domínio; watchers podem complementar.

## Payload Policy por Evento

| event_type | Campos obrigatórios no payload | Campos opcionais | Payload policy |
|---|---|---|---|
| `org.unit_created` | `id`, `codigo`, `nome`, `nivel`, `parent_id`, `status`, `created_by` | `descricao` | snapshot mínimo |
| `org.unit_updated` | `id`, `codigo`, campos alterados (diff) | — | diff only |
| `org.unit_deleted` | `id`, `codigo`, `deleted_by`, `deleted_at` | `motivo` | metadados |
| `org.unit_restored` | `id`, `codigo`, `restored_by`, `restored_at` | — | metadados |
| `org.tenant_linked` | `org_unit_id`, `tenant_id`, `tenant_codigo`, `created_by` | — | snapshot mínimo |
| `org.tenant_unlinked` | `org_unit_id`, `tenant_id`, `deleted_by` | — | metadados |

> **Regra:** Todos os payloads DEVEM incluir `correlation_id` (propagado do header `X-Correlation-ID`) e `actor_id` (user autenticado). Estes campos são injetados pelo middleware de domain events do MOD-000, não pelo módulo.

## Mascaramento (Masking Policy)

- `maskable_fields`: nenhum em todos os 6 eventos — dados organizacionais não contêm PII
- `sensitivity_level=0` em todos os eventos — guard-rail apenas, não substitui ACL/RBAC

## Regras de Notificação (Notify)

| event_type | Destinatários | Canal | Condição |
|---|---|---|---|
| `org.unit_created` | creator + admin | inbox | Sempre |
| `org.unit_updated` | — | — | Sem notificação |
| `org.unit_deleted` | admin | inbox | Sempre |
| `org.unit_restored` | admin | inbox | Sempre |
| `org.tenant_linked` | admin | inbox | Sempre |
| `org.tenant_unlinked` | admin | inbox | Sempre |

- **MUST:** Destinatários de notify DEVEM possuir scope `org:unit:read` no perfil ativo — caso contrário, notificação não é entregue
- **Resolução de "admin":** Usuários com scope `org:unit:write` ou `org:unit:delete` no tenant/contexto ativo
- **Resolução de "creator":** `actor_id` do evento (user autenticado que executou a operação)

## Regras de Filtragem (MUST)

- **MUST (ADR-003):** `org_units` é cross-tenant — consultas a `domain_events` com `entity_type=org_unit` NÃO filtram por `tenant_id`. O acesso é controlado via RBAC (`org:unit:read`)
- **MUST:** View restrita a usuários com scope `org:unit:read` — verificado via `@RequireScope`, não via RLS
- **MUST:** Notificações (`notify`) DEVEM ser entregues apenas a destinatários com scope `org:unit:read` no perfil ativo
- **MUST:** Timeline de domain events (UX-001 `view_history`) exibida via `GET /api/v1/domain-events?entity_type=org_unit&entity_id=:id` — protegido por `org:unit:read`
- Sensitivity level 0 em todos os eventos — dados organizacionais não contêm PII
