## Detalhamento

### Nova jornada: UX-010 — Sessão Mantida por Atividade (Keep-Alive Transparente)

- **Ator:** Usuário autenticado
- **Pré-condição:** Usuário logado e interagindo com o sistema
- **Fluxo principal:**
  1. Usuário interage normalmente (clicks, digitação, scroll, navegação)
  2. Sistema detecta atividade via eventos DOM (throttle 30s para mousemove)
  3. Quando o access token se aproxima de expirar (< 2min restantes):
     - Se houve atividade nos últimos 30 minutos → refresh proativo em background
     - Usuário **não percebe** nenhuma interrupção
     - Novo token ativo imediatamente
  4. Ciclo se repete enquanto o usuário permanecer ativo
- **Fluxo alternativo — Retorno após inatividade curta (< 30min):**
  1. Usuário para de interagir mas retorna antes de 30 minutos
  2. Se access token ainda válido → request normal, sem interrupção
  3. Se access token expirou → interceptor 401 tenta refresh → sucesso → retry transparente
  4. Usuário não percebe a expiração
- **Fluxo alternativo — Retorno após inatividade longa (> 30min, refresh token válido):**
  1. Usuário retorna após mais de 30 minutos sem interagir
  2. Access token expirou, proactive refresh não ocorreu
  3. Primeiro request → 401 → interceptor tenta refresh
  4. Refresh token ainda válido (< 7 dias) → renovação reativa bem-sucedida
  5. Request original retentado com sucesso
  6. Activity tracker reseta — keep-alive volta ao modo proativo
- **Fluxo alternativo — Sessão expirada (refresh token expirado ou revogado):**
  1. Usuário retorna após longo período ou admin revogou sessão
  2. Primeiro request → 401 → interceptor tenta refresh → 401
  3. Force logout: tokens limpos, cache limpo, redirect `/login`
  4. Complemento de UX-000-C01: toast "Sessão expirada" pode ser exibido na tela de login
- **Pós-condição:** Sessão mantida transparentemente durante uso ativo; encerrada de forma limpa após inatividade prolongada

### Indicadores visuais (recomendações)

| Cenário | Indicador | Prioridade |
|---|---|---|
| Refresh proativo em andamento | Nenhum (totalmente transparente) | Must |
| Refresh reativo (401 → retry) | Nenhum se < 2s; loading spinner se > 2s | Should |
| Force logout | Toast "Sessão expirada" na tela de login | Should |
| Inatividade próxima do limite | Opcional: aviso "Sua sessão expira em 5 minutos" (v2) | Won't (v1) |

### Relação com UX-000-C01

A emenda UX-000-C01 já tratou o cenário de sessão expirada com tabela de erros 401 e jornada "Sessão Expirada por Timeout". Esta emenda **complementa** UX-000-C01:

- **UX-000-C01:** Trata o cenário reativo (401 já aconteceu → redirect login, toast)
- **UX-000-M03 (esta):** Adiciona o cenário proativo (evitar que o 401 aconteça enquanto o usuário está ativo)

```gherkin
Funcionalidade: Experiência de Sessão Keep-Alive

Cenário: Uso contínuo sem interrupção
  Dado que o usuário está preenchendo um formulário complexo há 20 minutos
  Quando o access token se aproxima de expirar
  Então o sistema renova silenciosamente em background
  E o usuário NÃO vê loading, modal, ou qualquer indicador
  E a submissão do formulário funciona normalmente

Cenário: Retorno após café (< 30min)
  Dado que o usuário saiu para tomar café por 18 minutos
  Quando retorna e clica em um botão
  Então o request é processado normalmente (refresh transparente se necessário)
  E o usuário não percebe que a sessão foi renovada

Cenário: Retorno no dia seguinte
  Dado que o usuário fechou o laptop ontem sem fazer logout
  Quando abre o navegador e acessa o sistema
  Então é redirecionado para a tela de login
  E vê toast "Sessão expirada. Faça login novamente."
```
