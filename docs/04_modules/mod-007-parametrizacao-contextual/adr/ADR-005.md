# ADR-005 — Motor de Avaliação Always Fresh (Sem Cache em Nenhuma Camada)

## Contexto

O motor de avaliação executa 6 passos sequenciais: (1) encontrar regras de incidência ativas, (2) encontrar rotinas PUBLISHED vinculadas, (3) avaliar itens por ordem, (4) resolver conflitos por restritividade, (5) construir response, (6) persistir domain_events. A decisão ADR-001 removeu o cache Redis do motor. Este ADR complementa essa decisão formalizando que nenhuma camada de cache (HTTP, CDN, application-level, query cache) deve ser utilizada em qualquer dos 6 passos.

## Decisão

**Motor de avaliação "always fresh":** cada chamada a `POST /routine-engine/evaluate` executa todas as queries ao vivo, sem qualquer cache intermediário.

Justificativas:

1. **Consistência garantida:** O resultado reflete o estado atual do banco no momento da avaliação. Nenhuma decisão de parametrização é baseada em dados potencialmente desatualizados.
2. **Simplicidade operacional:** Sem Redis como dependência; sem lógica de invalidação; sem TTL para calibrar; sem race conditions entre invalidação e escrita.
3. **Reprodutibilidade:** Qualquer chamada pode ser reproduzida com os mesmos dados no banco, facilitando debug e auditoria.
4. **Adequação ao volume:** O volume esperado de chamadas ao motor (proporcional ao número de transições de estágio no MOD-006) não justifica a complexidade de cache nesta fase do produto.

## Alternativas Consideradas

1. **Cache por request (memoize intra-request):** Seria seguro mas desnecessário — cada request faz uma única avaliação.
2. **Read-replica dedicada:** Viável no futuro se latência do banco se tornar gargalo. Não introduz inconsistência se réplica for síncrona.
3. **Materializar resultado em tabela:** Complexidade de manutenção (invalidar materialização a cada mutação) supera o ganho.

> **Nota (PENDENTE-006):** O flag `dry_run: true` no request body não altera o comportamento "always fresh" do motor. A mesma execução completa (6 passos) ocorre; apenas a persistência do domain event `routine.applied` (EVT-012) é suprimida. O response é idêntico ao modo normal.

## Consequências

- **Positivas:** Arquitetura mais simples; zero bugs de cache stale; latência previsível e mensurável; alerta de performance diretamente correlacionado com performance do banco.
- **Negativas:** Se o volume de avaliações crescer significativamente, a latência depende exclusivamente da capacidade do banco. Sem amortecimento de picos via cache.
- **Mitigações:** SLO monitorado (p95 < 200ms); alerta pre-breach em 150ms; indexes otimizados; possibilidade de introduzir read-replica síncrona no futuro sem mudança de código do motor.
