## NFR-001.1 — Performance

### SLOs (Service Level Objectives)

| Métrica | Target | Medição |
|---|---|---|
| Latência p95 — endpoints de leitura (GET) | ≤ 200ms | APM / traces |
| Latência p95 — endpoints de escrita (POST/DELETE) | ≤ 500ms | APM / traces |
| Latência p95 — invalidação Redis (DEL) | ≤ 50ms | Redis metrics |
| Expiração automática — atraso máximo | < 5 min após `valid_until` | Job metrics (`identity.expiration.job.duration_ms`) |
| Cache Redis — hit rate após invalidação | 0% (cache invalidado = miss forçado) | Redis metrics |

### Regras de Performance

- Background job BullMQ executa em lote (batch UPDATE) — sem N+1 (DOC-FND-000 §3 anti-pattern)
- Listagem admin com paginação cursor-based (sem OFFSET — DOC-DEV-001 §3.1)
- Índices com `tenant_id` como prefixo para RLS eficiente (ver DATA-001 — 12 índices definidos)
- Consulta EXISTS para validação N5 inclusa no plano de query (sem round-trip extra)
- Autocomplete de nós org (UX-IDN-001): filtro `nivel <= 4` no backend, não no frontend

### Limites

| Recurso | Limite | Justificativa |
|---|---|---|
| SECONDARY vínculos por usuário | Sem limite explícito | Auditável via domain events; monitorar se > 20/usuário |
| Compartilhamentos ativos por grantee | Sem limite explícito | Monitorar; se > 50 → alerta para revisão |
| Delegações ativas por delegator | Sem limite explícito | Monitorar; se > 10 → alerta para revisão |
| `allowed_actions` (jsonb) max items | 50 scopes | Validar no service para prevenir payloads enormes |
| `delegated_scopes` (jsonb) max items | 50 scopes | Idem |

## NFR-001.2 — Disponibilidade

### Topologia

**Híbrida (sync + async):**
- **Sync:** Endpoints REST (11 endpoints — FR-001) — disponibilidade depende do API server + PostgreSQL
- **Async:** Background job `expire_identity_grants` via BullMQ/Redis — disponibilidade depende do Worker + Redis + PostgreSQL

### Comportamento de Degradação

| Componente | Falha | Impacto | Mitigação |
|---|---|---|---|
| PostgreSQL | Indisponível | Todos os endpoints retornam 503 | Health check; failover automático (infra) |
| Redis | Indisponível | Cache não invalidado; job não executa | Cache: fallback (log + continue); Job: retry 3x → DLQ |
| Worker BullMQ | Indisponível | Registros não expiram automaticamente | Alerting; registros continuam ACTIVE até worker voltar; sem impacto em operações normais |
| API server | Indisponível | Endpoints indisponíveis | Load balancer; health check; auto-scaling |

### Health Checks

| Check | Endpoint/Mecanismo | Frequência |
|---|---|---|
| API liveness | `GET /health/live` (herdado MOD-000) | 10s |
| API readiness | `GET /health/ready` (DB + Redis ping) | 10s |
| Worker liveness | BullMQ `isRunning()` + Redis ping | 30s |
| Job execution | Métrica `identity.expiration.job.last_run_at` | Alerta se > 10min sem execução |

## NFR-001.5 — Observabilidade

### Métricas (Prometheus/OpenTelemetry)

| Métrica | Tipo | Labels | Fonte |
|---|---|---|---|
| `identity.endpoint.latency_ms` | Histogram | `method`, `path`, `status` | API middleware |
| `identity.endpoint.requests_total` | Counter | `method`, `path`, `status` | API middleware |
| `identity.cache.invalidation.count` | Counter | `result` (success/failure) | INT-001.1 |
| `identity.expiration.job.duration_ms` | Histogram | — | INT-001.2 |
| `identity.expiration.records.expired` | Counter | `table` (scopes/shares/delegations) | INT-001.2 |
| `identity.expiration.job.failures` | Counter | — | INT-001.2 |
| `identity.domain_events.emitted` | Counter | `event_type` | Outbox |

### Logs Estruturados

- **Formato:** JSON estruturado (DOC-ESC-001 §3.3)
- **Campos obrigatórios:** `level`, `msg`, `correlation_id`, `tenant_id`, `user_id` (quando disponível), `timestamp`
- **PII:** NUNCA logar PII (nomes, e-mails, CPFs) — apenas UUIDs e status
- **Níveis:**
  - `info`: operações de escrita bem-sucedidas, job completed
  - `warn`: cache invalidation failed, retry attempt
  - `error`: job failed (DLQ), FK violation, validation unexpected

### Traces (OpenTelemetry)

- Spans em: chamadas DB (queries), Redis DEL, BullMQ job processing
- `correlation_id` propagado como `trace_id` ou `baggage`

## NFR-001.6 — Disaster Recovery

| Métrica | Target | Justificativa |
|---|---|---|
| **RPO** (Recovery Point Objective) | 0 (transacional) | Dados em PostgreSQL com WAL; domain events via Outbox (at-least-once) |

| Camada | Foco | Framework | Cobertura mínima |
