## Detalhamento

> **Contexto:** Este amendment deriva de [FR-000-M01](../fr/FR-000-M01.md).
> Consulte o amendment pai para o contexto completo da alteração nos DTOs da Users API.

### Alteração: Tabela `users` — Nova coluna `invite_token_created_at`

**Situação atual (DATA-000 §users):** A tabela `users` tem campos de identidade e autenticação (`id`, `codigo`, `email`, `password_hash`, `mfa_secret`, `status`, `force_pwd_reset`, `created_at`, `updated_at`, `deleted_at`), mas **não armazena** quando o token de convite foi gerado.

**Problema:** O DTO `UserDetail.invite_token_expired` (FR-000-M01 §2) é computado como `invite_token_created_at + TTL > now()`. Sem a coluna, esse cálculo é impossível.

**Alteração no schema:**

| Coluna | Tipo | Nullable | Default | Descrição |
|--------|------|----------|---------|-----------|
| `invite_token_created_at` | `TIMESTAMPTZ` | `true` | `null` | Momento em que o token de convite foi gerado. `null` = usuário nunca recebeu convite (auto-registro, criação por senha). Atualizado a cada reenvio de convite. |

**Regras:**

- Coluna é `nullable: true` — preenchida apenas quando `mode = 'invite'` (FR-000-M01 §3)
- Atualizada a cada `POST /users/:id/invite/resend` (FR-000 §FR-006) para refletir novo TTL
- Valor `null` → `UserDetail.invite_token_expired` retorna `null` (sem convite)
- Valor preenchido → backend computa: `invite_token_created_at + INVITE_TTL_HOURS > now()`

**Migration (Drizzle):**

```typescript
// db/schema/users.ts — adicionar ao schema existente
invite_token_created_at: timestamp('invite_token_created_at', { withTimezone: true }),
```

**Índice:** Não necessário — coluna consultada apenas em `GET /users/:id` (lookup por PK), não em listagens ou filtros.

### Gherkin

```gherkin
Cenário: Coluna invite_token_created_at preenchida na criação por convite
  Dado que o admin cria usuário com mode='invite'
  Quando o backend processa POST /api/v1/users
  Então a coluna invite_token_created_at DEVE ser preenchida com now()
  E o status do usuário DEVE ser PENDING

Cenário: Coluna nula na criação por senha
  Dado que o admin cria usuário com mode='password'
  Quando o backend processa POST /api/v1/users
  Então a coluna invite_token_created_at DEVE ser null

Cenário: Reenvio de convite atualiza timestamp
  Dado que o usuário tem status PENDING e invite_token_created_at = '2026-03-24T10:00:00Z'
  Quando POST /api/v1/users/:id/invite/resend
  Então invite_token_created_at DEVE ser atualizado para now()
```
