# ADR-002 — CTE Recursivo para Tree Query (vs. Materialized Path / Nested Sets)

## Contexto

O MOD-003 precisa de uma tree query eficiente para retornar a hierarquia N1–N5 completa. As abordagens clássicas para representar árvores em SQL são: adjacency list + CTE recursivo, materialized path, e nested sets. A profundidade máxima é fixa (4 níveis de org_units + 1 nível de tenants via JOIN).

## Decisão

**Usar adjacency list com CTE recursivo.** A tabela `org_units` usa `parent_id` (adjacency list) e as tree queries são resolvidas via `WITH RECURSIVE` do PostgreSQL.

## Alternativas Consideradas

1. **Materialized Path (`path` column, ex: `/n1/n2/n3/n4`):** Queries de subtree são rápidas (LIKE), mas mover nós exige reescrever paths de toda a subárvore. Complexidade desnecessária dado que movimentação de nós não está no escopo v1.
2. **Nested Sets (left/right):** Excelente performance de leitura, mas inserts/moves requerem recalcular intervalos de toda a árvore. Over-engineering para profundidade máxima de 4.
3. **Closure Table:** Tabela auxiliar com todas as relações ancestor→descendant. Mantém consultas O(1) mas duplica storage. Desnecessário para volume de até 500 nós/tenant.

## Consequências

- **Positivas:** Simplicidade de implementação; modelo mental claro; PostgreSQL otimiza CTEs recursivos para profundidade baixa; previne loops via CTE de validação (BR-004).
- **Negativas:** Latência pode crescer se volume exceder ~1000 nós (mitigado por índice otimizado em `parent_id` + NFR-001 soft limit de 500). Sem cache — todas as queries executam ao vivo.
- **Evolução futura:** Se movimentação de nós (drag-and-drop) entrar no roadmap v2, considerar migrar para materialized path via ADR dedicada.
