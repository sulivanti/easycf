## BR-000 — Regras de Negócio Core (Foundation — Auth, RBAC, Sessões)

### BR-000-01 — Governança de Geração de Código

- **Regra:** O framework de geração automática de código DEVE reconhecer, aplicar e validar os documentos normativos (DOC-DEV-001, DOC-GNP-00, DOC-ESC-001) como "lei do projeto". Toda spec de módulo gerada via `scaffold-module` deve nascer já conforme: contratos HTTP, observabilidade (correlation-id), tratamento de erros (RFC 9457) e idempotência.
- **Exceções:** Nenhuma fuga dos padrões é permitida sem abertura de ADR.
- **Impacto:** fluxo / segurança / compliance / rastreabilidade.

### BR-000-02 — Sessão Ancorada em Banco (Kill-Switch)

- **Regra:** O JWT carrega um `sessionId` que DEVE ser validado em banco a cada requisição via `verifyActiveSession()`. JWT válido com sessão revogada = 401. Esta não é uma sessão stateless pura — é um desvio intencional justificado pela necessidade de Kill-Switch imediato.
- **Exceções:** Endpoints públicos (`/auth/login`, `GET /info`) não validam sessão.
- **Impacto:** segurança / estado / fluxo.

### BR-000-03 — User Enumeration Prevention

- **Regra:** Respostas de falha de autenticação DEVEM sempre retornar a mesma mensagem genérica (`"E-mail ou senha incorretos."`), `actorId=undefined` e `entityId=UUID_NULO` na auditoria de falha. Nunca revelar se o e-mail existe ou não no sistema.
- **Impacto:** segurança / compliance (LGPD).

### BR-000-04 — RBAC por Escopos com Hierarquia

- **Regra:** Permissões são organizadas por escopos no formato `modulo:recurso:acao` (ex: `users:profile:read`). Scopes são atribuídos a Roles; Roles a usuários por filial/tenant. A regra DENY explícito PREVALECE sobre ALLOW em qualquer nível (filial-pai herda para filha; negação explícita bloqueia independente de outra role que conceda).
- **Impacto:** segurança / fluxo / dados.

### BR-000-05 — Multi-Tenant por Filial

- **Regra:** Toda entidade core DEVE ter `tenant_id` (UUID). Usuários são vinculados a filiais via tabela `tenant_users`. Leitura de dados DEVE filtrar por `tenant_id` do usuário autenticado. Administradores globais com escopo raiz podem ter acesso cross-tenant somente via permissão explícita.
- **Impacto:** dados / segurança / compliance.

### BR-000-06 — Idempotência Obrigatória (Mutações com Efeito Colateral)

- **Regra:** Toda rota de mutação (`POST`, `PUT`, `PATCH`) com efeito colateral DEVE suportar `Idempotency-Key` usando o middleware central `@easycf/core-api/idempotency` (Redis + TTL 30s). Módulos são PROIBIDOS de implementar lógica própria de idempotência.
- **Impacto:** fluxo / dados / segurança.

### BR-000-07 — Soft-Delete Obrigatório

- **Regra:** Hard-deletes diretos são PROIBIDOS em dados faturáveis/auditáveis. Toda exclusão DEVE acionar `deleted_at` (soft-delete). Hard-deletes definitivos ocorrem apenas via política de purging agendada (cron) com prazo de retenção definido.
- **Impacto:** dados / compliance.

### BR-000-08 — Taxonomia Canônica de Entidades

- **Regra:** Toda entidade primária DEVE adotar o campo `codigo` como identificador amigável único (alfanumérico, `NOT NULL`, `UNIQUE`). Status de entidades de negócio DEVEM usar enum textual (`ACTIVE`, `BLOCKED`, `INACTIVE`). Entidades efêmeras (como `user_sessions`) usam booleano (`isRevoked`), nunca enum de status de negócio.
- **Impacto:** dados / fluxo.

    E o actorId na auditoria deve ser undefined

    Dado que um usuário possui Role A (ALLOW: users:list) e Role B (DENY: users:list) na mesma filial
    Quando ele tenta acessar a lista de usuários
    Então o acesso deve ser negado (403)
