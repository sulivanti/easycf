## Detalhamento

### 1. Novo endpoint: POST /api/v1/users/:id/reset-password

Adicionar a FR-006 (Gestão de Usuários):

- **Método:** `POST /api/v1/users/:id/reset-password`
- **Scope:** `users:user:write`
- **Descrição:** Admin inicia reset de senha para o usuário. Gera token de reset e envia e-mail via Outbox Pattern.
- **Request body:** vazio (`{}`)
- **Response 200:** `{ "message": "Password reset email sent." }`
- **Erros:** 401 (sessão), 403 (sem scope), 404 (usuário inexistente), 500
- **Efeito colateral:** Domain event `user.password_reset` emitido
- **Done funcional:**
  - [ ] Endpoint aceita POST com body vazio
  - [ ] Gera token de reset (UUID) e associa ao usuário
  - [ ] Enfileira envio de e-mail de reset via Outbox (dedupe_key: `user_id + password_reset + timestamp`)
  - [ ] Emite domain event `user.password_reset` com payload `{ user_id, reset_by, timestamp }`
  - [ ] Retorna 404 se usuário não encontrado
  - [ ] Retorna 403 se scope ausente

```gherkin
  Cenário: Admin reseta senha de outro usuário
    Dado que o admin com scope "users:user:write" chama POST /users/:id/reset-password
    E o usuário existe com status ACTIVE
    Então resposta 200 com message "Password reset email sent."
    E domain event "user.password_reset" é emitido
    E e-mail de reset é enfileirado via Outbox
```

### 2. Novo endpoint: DELETE /api/v1/users/:id/invite

Adicionar a FR-006 (Gestão de Usuários):

- **Método:** `DELETE /api/v1/users/:id/invite`
- **Scope:** `users:user:delete`
- **Descrição:** Cancela convite pendente de ativação. Altera status do usuário para INACTIVE.
- **Pré-condição:** Status MUST ser PENDING (409 se outro status)
- **Response 200:** `{ "data": { "id": "uuid", "status": "INACTIVE", "updatedAt": "ISO8601" } }`
- **Erros:** 401, 403, 404, 409 (status não é PENDING), 500
- **Efeito colateral:** Domain event `user.invite_cancelled` emitido
- **Done funcional:**
  - [ ] Endpoint aceita DELETE sem body
  - [ ] Verifica status === PENDING (409 ProblemDetails se não)
  - [ ] Altera status para INACTIVE e invalida token de convite
  - [ ] Emite domain event `user.invite_cancelled` com payload `{ user_id, cancelled_by, timestamp }`
  - [ ] Retorna 404 se usuário não encontrado
  - [ ] Retorna 409 se status não é PENDING

```gherkin
  Cenário: Admin cancela convite pendente
    Dado que o admin com scope "users:user:delete" chama DELETE /users/:id/invite
    E o usuário existe com status PENDING
    Então resposta 200 com status alterado para INACTIVE
    E domain event "user.invite_cancelled" é emitido

  Cenário: Tentativa de cancelar convite de usuário não-PENDING
    Dado que o usuário existe com status ACTIVE
    Quando admin chama DELETE /users/:id/invite
    Então resposta 409 ProblemDetails "User is not in PENDING status"
```
