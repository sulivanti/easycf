## 1. SLOs (Service Level Objectives)

| Métrica | Alvo | Endpoint(s) / Contexto | Fonte |
|---|---|---|---|
| **Latência p95 — enqueue** | **< 50ms** | INSERT call_log (status=QUEUED) dentro da transação de negócio | US-MOD-008, OKR-1 |
| **Latência p99 — enqueue** | **< 100ms** | INSERT call_log (status=QUEUED) dentro da transação de negócio | Estimativa — margem para contenção de locks |
| **Latência p95 — worker execution** | **< 35000ms** | HTTP call ao Protheus (timeout_ms + overhead BullMQ) | US-MOD-008, timeout padrão 30s + 5s overhead |
| **Latência p99 — worker execution** | **< 60000ms** | HTTP call com retry (1 retry + backoff) | Estimativa — cenários de retry |
| **Latência p95 — listagem de logs** | **< 500ms** | GET /admin/integration-logs (paginação cursor-based, max 100 registros) | US-MOD-008 |
| **Latência p95 — CRUD admin** | **< 300ms** | POST/PATCH/DELETE de serviços, rotinas, mapeamentos, params | Padrão DOC-GNP-00 |
| **Disponibilidade — motor BullMQ** | **>= 99.9%** | Worker precisa estar online para processar fila | US-MOD-008, OKR-1 |
| **Disponibilidade — API REST** | **>= 99.5%** | Todos os 15 endpoints do módulo | Padrão plataforma |
| **Taxa de erro — execuções** | **< 5%** | Erros 5xx sobre total de chamadas (excluindo erros 4xx do Protheus) | US-MOD-008 |

### 1.1 SLO do Enqueue (Crítico)

O INSERT de `integration_call_logs` com `status=QUEUED` ocorre **dentro da transação de negócio** (Outbox Pattern). O SLA < 50ms é crítico para não degradar a transação principal (ex: transição de estágio no MOD-006).

**Estratégia de otimização:**

- INSERT simples em tabela indexada — evitar queries complexas no caminho crítico
- Index `idx_call_logs_tenant_status` pré-criado para suportar DLQ scan sem impactar INSERT
- Monitorar com métricas de latência p95/p99 do INSERT

## 4. DR (Disaster Recovery)

| Métrica | Alvo | Justificativa |
|---|---|---|
| **RPO** (Recovery Point Objective) | 0 (zero data loss) | Outbox Pattern garante que jobs enfileirados estão no banco. Logs de chamada são auditoria imutável |
| **RTO** (Recovery Time Objective) | < 15 min | Motor de integração é crítico — indisponibilidade gera acúmulo de DLQ |

**Estratégia:**

- Backup PostgreSQL point-in-time recovery (PITR) padrão da plataforma
- Redis persistence (AOF) para garantir que jobs BullMQ não se percam em crash
- Worker auto-restart via process manager (PM2/systemd)
- Jobs em status QUEUED/RUNNING são automaticamente reprocessados quando worker reinicia (BullMQ stalled job detection)

## 7. Concorrência e Consistência

| Cenário | Estratégia | Justificativa |
|---|---|---|
| Outbox INSERT (enqueue) | Transação atômica com a operação de negócio | Outbox Pattern — INSERT call_log QUEUED dentro da mesma transação que originou o disparo |
| Worker concurrent jobs | BullMQ concurrency = env `INTEGRATION_CONCURRENCY` (default 10) | Limita conexões simultâneas ao Protheus |
| Retry re-enqueue | BullMQ `moveToDelayed` com delay calculado | Evita re-enqueue manual — BullMQ gerencia o delay |
| Reprocessamento | INSERT novo call_log + UPDATE original (status=REPROCESSED) na mesma transação | Atomicidade — log original preservado, novo log vinculado via parent_log_id |
| Edição concorrente de serviço | Optimistic locking via `updated_at` | Dois admins editando o mesmo serviço — o segundo recebe 409 Conflict |
| Edição concorrente de rotina | Herda optimistic locking do MOD-007 (behavior_routines) | Controlado na camada do MOD-007 |
| BullMQ stalled jobs | Stalled job interval = 30s, max stalled count = 1 | Jobs travados são automaticamente falhados e re-enfileirados |
| Deduplicação BullMQ | `jobId = integration_call_logs.id` | Previne dupla execução do mesmo call_log |
