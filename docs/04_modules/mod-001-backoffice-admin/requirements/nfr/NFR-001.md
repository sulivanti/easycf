# NFR-001 — Requisitos Não Funcionais do Backoffice Admin

---

## 1. SLOs (Service Level Objectives)

| Métrica | Alvo | Observações |
|---|---|---|
| Latência p95 (login → redirect) | ≤ 500ms | Inclui POST /auth/login + redirect client-side |
| Latência p95 (auth_me) | ≤ 300ms | Inclui GET /auth/me + renderização do Shell |
| Skeleton timeout (Dashboard) | 3000ms | Após 3s sem resposta → estado de erro (FR-005) |
| First Contentful Paint (FCP) | ≤ 1500ms | Tela de login renderizada |
| Time to Interactive (TTI) | ≤ 2500ms | Login form interativo |

## 2. Topologia

- **Tipo:** Frontend SPA (Vite/React) consumindo API REST (MOD-000)
- **Comunicação:** HTTP REST via fetch/axios com interceptor global
- **Autenticação:** JWT em cookies httpOnly (gerenciado pelo browser)
- **Estado:** React state/context (sem Redux/Zustand necessário para MOD-001)

## 3. Observabilidade

### X-Correlation-ID (MUST)

- Toda requisição HTTP não-client_only DEVE incluir `X-Correlation-ID` no header
- O mesmo correlation_id DEVE compor o UIActionEnvelope e o header HTTP
- Cada ação gera um UUID v4 distinto (nunca reutilizar entre ações)

### UIActionEnvelope (MUST)

- Emitido via `@easycf/ui-telemetry` para todas as actions não-client_only
- Campos obrigatórios: correlation_id, screen_id, action_id, status
- `duration_ms` preenchido em succeeded/failed
- `problem_type` derivado de RFC 9457 em failed

### Telemetria de Estados de Erro e Retry (MUST)

- Timeout de skeleton (3s) DEVE emitir UIActionEnvelope `dashboard_skeleton_timeout` (client_only) para rastreabilidade
- Retry manual DEVE emitir UIActionEnvelope `dashboard_retry` com novo `correlation_id` UUID v4
- Cada ciclo retry-succeeded/retry-failed DEVE registrar `duration_ms` e `http_status`
- Referência: DATA-003 (catalogo de UIActionEnvelopes)

### Logs Frontend (SHOULD)

- Erros de rede logados no console com correlation_id
- Sem PII nos logs (apenas IDs, status, duração)

## 4. Segurança (delegada ao MOD-000)

- Rate limits gerenciados pelo backend (MOD-000)
- Sanitização de inputs nos formulários (e-mail, senha)
- Tokens JWT em httpOnly cookies — não acessíveis via JavaScript
- Proteção de rotas: JWT expirado/ausente → redirect /login via interceptor

## 5. Resiliência e Fallback

### Princípio Zero-Blank-Screen (MUST)

O usuário NUNCA deve ver uma tela completamente em branco (white screen). Em qualquer cenário de falha, o sistema DEVE exibir:
- Estado de erro parcial com mensagem informativa
- Toast com `correlationId` copiável
- Ação de recuperação (botão retry ou link de navegação)

### Tabela de Cenários

| Cenário | Comportamento | Referência |
|---|---|---|
| auth_me timeout > 3s | Skeleton → estado de erro parcial (NUNCA tela branca) + Toast retry | BR-009, FR-005, NFR §1 |
| Erro 5xx em auth_me | Toast com correlationId + botão retry, SEM desconectar | BR-010, FR-005 |
| Retry manual (botão "Tentar novamente") | Novo GET /auth/me com novo correlation_id; re-exibe skeleton | BR-009, BR-010, INT-005 |
| Erro 401 em qualquer requisição | Redirect /login via interceptor global | FR-004, SEC-001 |
| Erro de rede no logout | Limpar estado local + redirect /login | FR-004, INT-002 |
| Rate limit (429) no login | Toast + desabilitar botão por retry_after | FR-001 |

### Distinção 401 vs 5xx (MUST)

- **401:** Único cenário que desconecta o usuário (redirect /login via interceptor HTTP global)
- **5xx:** Toast de erro + retry, mantendo o usuário autenticado no Shell (BR-010)
- **Timeout:** Tratado como erro parcial, sem desconexão (BR-009)

## 6. Acessibilidade (SHOULD)

- Navegação por teclado nos formulários de login/forgot/reset
- Foco automático no primeiro campo ao entrar em cada painel
- Labels e aria-labels nos campos de input
- Contraste mínimo WCAG AA nos botões e textos

## 7. Healthchecks

- **N/A — Módulo frontend SPA.** Sem endpoint de healthcheck próprio.
- A disponibilidade do SPA depende do CDN/hosting estático e do MOD-000 (API backend).
- Monitoramento indireto: se `GET /auth/me` falha em série → alerta no backend (MOD-000).

## 8. Disaster Recovery (DR)

- **RPO:** N/A — SPA sem estado persistido no servidor. Assets estáticos no CDN.
- **RTO:** Redeploy do bundle estático (< 5 min via CI/CD pipeline).
- **Dependência crítica:** MOD-000 (Foundation API) — RPO/RTO seguem política do PostgreSQL.

## 9. Limites

| Recurso | Limite | Justificativa |
|---|---|---|
| Tentativas de login | Delegado ao MOD-000 (rate limit) | Backend controla |
| Skeleton timeout | 3000ms fixo | UX: evitar espera infinita |
| Sidebar items | Máx 20 itens (por scopes) | Performance de renderização |
| Module shortcuts | Máx 8 cards | Layout responsivo do grid |
