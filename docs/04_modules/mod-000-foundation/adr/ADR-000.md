## ADR-000 — Decisões Arquiteturais Core do Foundation

---

### ADR-000-01 — Sessão Ancorada em Banco (Desvio de JWT Stateless Clássico)

- **Contexto:** O JWT clássico é stateless — o servidor não valida o token em banco. Isso impede revogação imediata de sessões comprometidas.
- **Decisão:** O JWT carrega um `sessionId` que é validado em banco (`user_sessions`) a cada requisição autenticada via `verifyActiveSession()`. JWT válido + sessão `isRevoked=true` → 401.
- **Alternativas consideradas:**
  - JWT puro stateless: descartado — sem Kill-Switch imediato.
  - Blacklist em Redis: descartado — Redis como banco primário viola `DATA-013`.
- **Consequências:** Maior latência em ~1 query/request, compensada por Redis (cache de RBAC) e índice em `(user_sessions.id, isRevoked)`. Justificado pela necessidade de segurança enterprise.
- **Referência:** US-MOD-000-F01, BR-000-02, DOC-DEV-001 §DATA-013.

---

### ADR-000-02 — Redis como Broker (MUST NOT como Banco Primário)

- **Contexto:** Redis é utilizado em três contextos: Rate Limiting, `Idempotency-Key` cache e fila BullMQ.
- **Decisão:** Redis DEVE ser usado **exclusivamente** como broker de filas (BullMQ) e mecanismos efêmeros de TTL. **NUNCA** como banco primário ou Pub/Sub autônomo. O estado persiste no Postgres.
- **Alternativas consideradas:**
  - Redis como banco de sessão: descartado — viola a regra de Kill-Switch (precisa do DB para validação).
  - Redis Pub/Sub: descartado — Outbox Pattern via Postgres garante durabilidade.
- **Referência:** DOC-DEV-001 §DATA-013, BR-000-06.

---

### ADR-000-03 — Idempotência via Middleware Central (MUST NOT em módulos individuais)

- **Contexto:** Múltiplas rotas de mutação precisam de idempotência. Implementações isoladas por módulo geram inconsistências.
- **Decisão:** O `@easycf/core-api/idempotency` é o middleware centralizado obrigatório. Usa Redis com TTL 30s e armazena a resposta cacheada. Proibida qualquer implementação própria em controllers/services.
- **Referência:** BR-000-06, FR-000-F01, DOC-DEV-001 §0.0.

---

### ADR-000-04 — Soft-Delete Obrigatório + Purging por Política

- **Contexto:** Hard-deletes comprometem auditabilidade e conformidade LGPD/financeira.
- **Decisão:** Toda exclusão DEVE ser soft-delete (`deleted_at`). Hard-deletes definitivos só via cron de purging sob Política de Retenção documentada por entidade. Tabelas com dados faturáveis (`user_sessions`, `tenant_users`) requerem aprovação de ADR específica para purging.
- **Referência:** BR-000-07, DOC-DEV-001 §DATA-010.

---

### ADR-000-05 — Upload via Presigned URL (Sem Binário na API)

- **Contexto:** Receber arquivos binários na API aumenta consumo de memória, adiciona timeout e complica escala horizontal.
- **Decisão:** A API APENAS gera presigned URLs. O cliente faz upload direto ao S3-compatible storage. A API valida a existência do objeto na confirmação (`POST /storage/confirm`). Nunca armazena o binário em memória ou disco.
- **Referência:** FR-000-F16, INT-000-04, DOC-PADRAO-005.
