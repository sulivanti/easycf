# NFR-006 — Requisitos Não Funcionais da Execução de Casos

## 1. SLOs (Service Level Objectives)

| Metrica | Alvo | Endpoint(s) | Fonte |
|---|---|---|---|
| Latencia P95 — Motor de transicao | **< 1000ms** | POST /api/v1/cases/:id/transition | OKR-1, FR-002 (5 validacoes + 4 escritas + emit em transacao) |
| Latencia P95 — Timeline | **< 500ms** | GET /api/v1/cases/:id/timeline | OKR-4, FR-008 (merge de 3 tabelas, ate 100 eventos) |
| Latencia P95 — Listagem | **< 300ms** | GET /api/v1/cases | FR-009 (cursor-based, filtros indexados) |
| Latencia P95 — Abertura | **< 500ms** | POST /api/v1/cases | FR-001 (freeze + create + gate_instances + stage_history) |
| Latencia P95 — CRUD endpoints | **< 500ms** | Demais POST/PATCH/GET | Padrao DOC-GNP-00 |
| Disponibilidade | **99.9%** | Todos os endpoints | SLA Foundation |

### 1.1 SLO do Motor de Transicao (Critico)

O motor de transicao e a operacao mais complexa do modulo — executa 5 validacoes sequenciais (BR-002, BR-004, BR-005, BR-006, BR-012) e 4 escritas atomicas (stage_history, case_instances, gate_instances, case_events) + emit de domain event. O SLA < 1000ms e um OKR do modulo.

**Estrategia de otimizacao:**

- Transacao unica com todas as escritas (atomicidade — rollback completo em falha)
- Validacoes executadas em queries indexadas (idx_gate_instances_case_status, idx_case_assignments_case_active)
- Blueprint lido via FKs diretas (mesmo schema — sem HTTP)
- Se ultrapassar 1000ms: profiling via traces OpenTelemetry para identificar gargalo

### 1.2 SLO da Timeline (Critico)

A timeline intercala 3 historicos (stage_history, gate_instances, case_events) ordenados por timestamp. Para casos com ate 100 eventos, o SLA e < 500ms.

**Estrategia de otimizacao:**

- Tres queries paralelas (uma por tabela) + merge sort em memoria
- Indexes otimizados: `idx_stage_history_case`, `idx_gate_instances_case_status`, `idx_case_events_case`
- Se volume > 100 eventos: considerar paginacao da timeline (cursor-based)

## 4. DR (Disaster Recovery)

| Metrica | Alvo | Justificativa |
|---|---|---|
| **RPO** (Recovery Point Objective) | **0** (zero data loss) | Casos em andamento com historicos de estagio, gates e atribuicoes — perda e inaceitavel |
| **RTO** (Recovery Time Objective) | **< 15 min** | Motor de transicao e operacao critica para processos de negocio em andamento |

**Estrategia:** Backup PostgreSQL point-in-time recovery (PITR) padrao da plataforma. Transacoes atomicas garantem consistencia — nao ha estado parcial em caso de crash.

## 7. Concorrencia e Consistencia

| Cenario | Estrategia | Justificativa |
|---|---|---|
| Transicao concorrente do mesmo caso | Optimistic locking via `updated_at` no case_instances | Dois usuarios tentando transitar o mesmo caso — o segundo recebe 409 Conflict |
| Reatribuicao concorrente do mesmo papel | SELECT FOR UPDATE na atribuicao ativa | BR-007 exige que no maximo UMA atribuicao ativa por (case, stage, role) |
| Resolucao concorrente do mesmo gate | UNIQUE(case_id, gate_id) + status check | Constraint de unicidade + validacao de status=PENDING previne resolucao duplicada |
| Hold + transicao simultaneos | Optimistic locking via `updated_at` | Caso em transicao para ON_HOLD enquanto outro usuario tenta transitar |
| Cancelamento + transicao simultaneos | Optimistic locking via `updated_at` | Primeiro a commitar ganha; segundo recebe 409 ou 422 |

> **Nota:** Optimistic locking usa `WHERE updated_at = :expected_updated_at` no UPDATE. Se o timestamp divergir, retorna 409 com `extensions.conflict_field: "updated_at"`.

### 9.1 Cenarios de Teste Criticos

1. **Motor bloqueia gate pending** — BR-002: gate APPROVAL required=true PENDING impede transicao
2. **Motor permite WAIVED** — gate WAIVED nao bloqueia
3. **INFORMATIVE nunca bloqueia** — BR-005: gate INFORMATIVE required=true PENDING nao impede
4. **Papel required sem atribuicao** — BR-006: bloqueia transicao
5. **Estagio terminal completa** — BR-004: transicao para is_terminal → status=COMPLETED
6. **Freeze cycle_version_id** — BR-001: fork do ciclo nao afeta caso em andamento
7. **Reatribuicao desativa anterior** — BR-007: unicidade de atribuicao ativa
8. **Timeline intercala 3 historicos** — FR-008: ordenacao correta por timestamp
9. **Optimistic locking** — transicao concorrente retorna 409
10. **Idempotency-Key** — POST /cases duplicado retorna mesma response
