# ADR-003 — Rotina PUBLISHED Imutável — Fork como Único Caminho para Nova Versão

## Contexto

Rotinas de comportamento definem parametrizações que afetam diretamente o motor de avaliação e, por consequência, transições de estágio no MOD-006. Permitir edição in-place de rotinas publicadas criaria riscos: (a) alterações não rastreáveis em rotinas que já estão em produção; (b) comportamento do motor muda silenciosamente; (c) auditorias de compliance não conseguem reconstruir o estado da rotina no momento de uma avaliação passada.

## Decisão

**Rotinas com status=PUBLISHED são imutáveis.** Nenhuma operação de edição, adição ou remoção de itens é permitida. O único caminho para criar uma nova versão é o **fork** (`POST /admin/routines/:id/fork`), que:

1. Cria nova rotina em DRAFT com `version = original.version + 1`
2. Copia todos os `routine_items` com novos UUIDs
3. Copia todos os `routine_incidence_links` para a nova versão
4. Exige `change_reason` obrigatório no body
5. Registra entrada em `routine_version_history` com snapshot do motivo
6. Emite domain event `routine.forked` (DATA-003 EVT-009)

O ciclo completo é: DRAFT (editável) → PUBLISHED (imutável, ativo no motor) → DEPRECATED (imutável, ignorado pelo motor). Transição reversa (PUBLISHED → DRAFT) não é permitida.

## Alternativas Consideradas

1. **Edição in-place com changelog automático:** Menor fricção para o usuário, mas perde a garantia de imutabilidade. Auditoria depende de diff entre versões, que é frágil.
2. **Versionamento semântico com branches:** Complexidade elevada para o domínio; rotinas não são código.
3. **Aprovação por fluxo formal antes de edição:** Exige MOD-009 (motor de aprovação), que não está no escopo atual. Pode ser adicionado no futuro como camada sobre o fork.

## Consequências

- **Positivas:** Auditoria total — cada versão é um snapshot completo; comportamento do motor determinístico e rastreável; compliance garantido (qual rotina estava ativa em qual avaliação).
- **Negativas:** Mais clicks para o usuário (fork → editar → publicar em vez de editar diretamente); proliferação de versões se mudanças forem frequentes.
- **Mitigações:** UX de fork pré-popula todos os dados da versão anterior; flag `auto_deprecate_previous: boolean` (default: false) no `POST /admin/routines/:id/publish` permite deprecar a versão anterior automaticamente ao publicar o fork (PEN-004, decidida: Opção 3); botão "Fork e Editar" como ação primária na UI de rotinas PUBLISHED.
