# ADR-003 — Outbox Pattern para Domain Events (Por Que Outbox e Não Publish Direto)

## Contexto

O MOD-010 define 10 domain events (DATA-003) que são emitidos durante operações de CRUD de agentes, vínculos de ações e execuções MCP. Esses eventos alimentam a timeline/auditoria, notificações in-app/e-mail e integrações cross-module. A decisão é o mecanismo de publicação: (a) outbox pattern (write evento na mesma transação do aggregate, consumer separado publica para o event bus), ou (b) publish direto (emitir para o event bus dentro ou imediatamente após a transação de negócio).

Todos os 10 eventos em DATA-003 têm `outbox.enabled: true` com `dedupe_key` e `ttl` definidos.

## Decisão

**Todos os domain events do MOD-010 são publicados via outbox pattern.** O evento é inserido na tabela `domain_events` dentro da mesma transação do comando de negócio. Um consumer assíncrono (poller ou CDC) lê eventos pendentes e publica para o event bus. O `dedupe_key` garante idempotência no consumer (at-least-once delivery com deduplicação).

## Alternativas Consideradas

1. **Publish direto (in-process):** Emitir o evento para o message broker (Redis Streams, RabbitMQ, etc.) dentro do handler do use case, após o commit da transação. Vantagem: latência menor (evento publicado imediatamente). Desvantagem: se o publish falhar após o commit, o evento é perdido — violando a garantia de auditoria. Sem transacionalidade entre a escrita de negócio e a publicação do evento.
2. **Publish direto (dual-write):** Escrever no banco E publicar no broker dentro do mesmo handler. Vantagem: menor latência. Desvantagem: dual-write problem — se um dos dois falhar, o sistema fica inconsistente. Sem suporte a transações distribuídas (2PC) entre PostgreSQL e o broker.
3. **CDC (Change Data Capture) puro:** Usar Debezium ou similar para capturar mudanças nas tabelas de negócio e derivar eventos. Vantagem: zero código de publicação no domínio. Desvantagem: eventos derivados do schema do banco (não do domínio), dificuldade de expressar eventos ricos como `mcp.privilege_escalation_attempt` com payload customizado. Perda de controle sobre o formato do evento.

## Consequências

- **Positivas:** Garantia transacional — se o comando de negócio fez commit, o evento EXISTE na tabela `domain_events`. Deduplicação via `dedupe_key` permite retry seguro do consumer. Auditoria completa (DOC-ARC-003 §1 Dogma 6). Eventos são first-class citizens do domínio, com payload controlado pela `payload_policy` de DATA-003.
- **Negativas:** Latência adicional entre a escrita do evento e sua publicação (depende da frequência do poller, tipicamente 100-500ms). Tabela `domain_events` cresce continuamente (mitigado por política de retenção e particionamento). Necessidade de infraestrutura de consumer/poller.
- **Riscos:** Consumer com backlog pode atrasar notificações. Mitigação: métricas de lag do consumer + alerta quando lag > 5s. Para eventos com sensitivity_level=2 (EVT-010), considerar priorização no consumer.
