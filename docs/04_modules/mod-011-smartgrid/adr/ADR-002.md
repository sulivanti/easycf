# ADR-002 — Sem Persistência Server-Side de Drafts (Export/Import JSON Client-Side)

## Contexto

O SmartGrid permite ao usuário preencher dezenas de linhas na grade de inclusão em massa (UX-SGR-001) antes de salvar. Se o usuário fechar o navegador acidentalmente ou a sessão expirar, todo o trabalho seria perdido. A questão era se MOD-011 deveria implementar auto-save de rascunhos (drafts) no servidor, ou se deveria operar sem persistência intermediária server-side, oferecendo Export/Import JSON como mecanismo de recuperação client-side.

## Decisão

**MOD-011 não persiste drafts no servidor.** O estado da grade é mantido exclusivamente em memória do navegador (React state). A recuperação de trabalho é feita via Export/Import JSON, operação inteiramente client-side. Ao fechar a grade com dados não salvos, um modal de confirmação (CloseConfirmationModal) oferece "Exportar e sair" / "Sair sem exportar" / "Cancelar" (BR-010).

## Alternativas Consideradas

1. **Auto-save de drafts no servidor (tabela de rascunhos):** Exigiria criação de tabela `smartgrid_drafts` no banco, endpoints API de CRUD de rascunhos, lógica de expiração/limpeza. Violaria o princípio de MOD-011 como UX puro (Nível 1 — 0 tabelas). Adicionaria complexidade de persistência e concorrência.
2. **Auto-save em LocalStorage/IndexedDB:** Persistência client-side automática sem servidor. Problema: dados presos em um único browser/dispositivo; sem sincronização entre dispositivos; risco de perda se browser limpar storage; limites de quota.
3. **Export/Import JSON manual (ESCOLHIDA):** Mecanismo explícito controlado pelo usuário. JSON contém estado completo da grade (`operacao_id`, colunas, linhas com dados e status). Zero tabelas adicionais, zero endpoints novos, zero persistência server-side. Modal de confirmação ao fechar previne perda acidental.

## Consequências

- **Positivas:** MOD-011 mantém 0 tabelas e 0 endpoints próprios (Nível 1 UX Consumer); nenhuma complexidade de persistência, concorrência ou expiração de drafts; arquivo JSON portável entre dispositivos e sessões; usuário tem controle explícito sobre quando exportar.
- **Negativas:** Se o usuário esquecer de exportar e o navegador fechar inesperadamente (crash, kill do processo), os dados são perdidos. Modal de confirmação mitiga cenários de fechamento intencional, mas não protege contra crash do browser.
- **Riscos:** Usuários podem reclamar da falta de auto-save. Mitigação: modal de confirmação ao fechar (BR-010), copy clara na UI. Se demanda surgir no futuro, LocalStorage auto-save pode ser adicionado como melhoria incremental sem impacto na arquitetura.
