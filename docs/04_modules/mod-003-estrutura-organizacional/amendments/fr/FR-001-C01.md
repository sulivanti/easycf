## Detalhamento

### Regra Crítica §6 — Adendo: Estratégia de Unicidade de `codigo`

**Contexto:** A DDL define `codigo VARCHAR(50) NOT NULL UNIQUE` — constraint global (tabela cross-tenant, ADR-003). O cenário Gherkin da F01 §5 já especifica o comportamento esperado: "Rejeitar criação com codigo duplicado → 409".

**Decisão:** Usar **catch da constraint violation** do PostgreSQL (código `23505 — unique_violation`) e traduzir para resposta HTTP 409 RFC 9457. **Não usar validação aplicacional prévia (SELECT EXISTS).**

**Justificativa:**

1. **Race condition:** Dois requests simultâneos verificando "código livre?" ambos passariam a validação aplicacional, mas um falharia no INSERT. O catch da constraint é a única forma segura.
2. **Simplicidade:** Evita um SELECT extra antes de cada INSERT — alinhado com Nível 1 (Clean Leve).
3. **Padrão do projeto:** O RFC 9457 já está configurado no middleware global. Basta mapear o erro PostgreSQL para o response padronizado.

**Implementação de referência:**

```typescript
// No use case ou handler de criação de org_unit
try {
  await repo.insert(orgUnit);
} catch (err) {
  if (err.code === '23505' && err.constraint?.includes('codigo')) {
    throw new ConflictError('Código já utilizado.');
  }
  throw err;
}
```

**Resposta HTTP (409):**

```json
{
  "type": "/problems/conflict",
  "title": "Conflict",
  "status": 409,
  "detail": "Código já utilizado.",
  "instance": "/api/v1/org-units",
  "extensions": {
    "correlationId": "uuid",
    "field": "codigo",
    "value": "GC-001"
  }
}
```

> **Nota:** Esta estratégia se aplica a todos os endpoints que fazem INSERT em `org_units`: `POST /api/v1/org-units` (FR-001) e futuros endpoints que criem nós.
