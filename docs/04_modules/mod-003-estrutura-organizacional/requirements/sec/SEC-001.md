# SEC-001 — Segurança e Compliance da Estrutura Organizacional

---

## 1. Autenticação

- **Modelo:** Delegada integralmente ao MOD-000 (Foundation)
- **Sessões:** Gerenciadas pelo MOD-000 (JWT com sessionId ancorado em banco)
- **Tokens:** Cookies httpOnly (accessToken + refreshToken) — não acessíveis via JavaScript

## 2. Autorização

- **Modelo:** RBAC por escopos (MOD-000-F12)
- **Escopos do módulo:**

| Escopo | Descrição | Endpoints protegidos |
|---|---|---|
| `org:unit:read` | Visualizar a estrutura organizacional e árvore | GET /org-units, GET /org-units/tree, GET /org-units/:id |
| `org:unit:write` | Criar, editar e restaurar nós organizacionais e vínculos de tenant | POST /org-units, PATCH /org-units/:id, PATCH /org-units/:id/restore, POST /org-units/:id/tenants |
| `org:unit:delete` | Desativar nós e remover vínculos | DELETE /org-units/:id, DELETE /org-units/:id/tenants/:tenantId |
| `org:dept:read` | Visualizar departamentos | GET /departments, GET /departments/:id |
| `org:dept:write` | Criar, editar e restaurar departamentos | POST /departments, PATCH /departments/:id, PATCH /departments/:id/restore |
| `org:dept:delete` | Desativar departamentos | DELETE /departments/:id |

- **Formato de scopes:** `dominio:entidade:acao` (3 segmentos) — conforme DOC-FND-000 §2.1
- **Registro:** Scopes registrados no catálogo canônico DOC-FND-000 §2.2 (INT-002)
- **Regra:** Requisição sem scope válido retorna 403 RFC 9457 com `type="/problems/forbidden"`
- **Frontend:** Menus/rotas DEVEM ser ocultados se o usuário não possuir o scope correspondente (decisão UX, não substitui proteção de rota backend)

## 3. Classificação dos Dados

- **Classificação:** `Interno` — dados organizacionais (org_units e departamentos) não contêm PII diretamente
- **Sensitivity level:** 0 (dados organizacionais não-sensíveis)

## 4. Política de Retenção

- Domain events: seguir política global de retenção (DOC-FND-000)
- Soft-deleted org_units e departamentos: manter indefinidamente para rastreabilidade
- Idempotency keys: TTL de 60 segundos (INT-005, BR-012) — armazenadas na tabela `idempotency_keys` do MOD-000

## 5. LGPD

- **Minimização:** Campos estritamente necessários (codigo, nome, descricao, nivel)
- **Anonimização:** N/A — dados organizacionais, não pessoais
- `created_by` referencia users.id — aplica-se política de anonimização de users quando aplicável

## 6. Auditoria

- Todas as operações de escrita emitem domain events (DATA-003)
- Events auditáveis: `org.unit_created`, `org.unit_updated`, `org.unit_deleted`, `org.unit_restored`, `org.tenant_linked`, `org.tenant_unlinked`, `org.dept_created`, `org.dept_updated`, `org.dept_deleted`, `org.dept_restored`
- Trilha de mudanças via `domain_events` com `correlation_id`, `created_by`, `entity_type=org_unit`
- **Payload policy:** snapshot minimo sem PII desnecessaria (DATA-003). Campos obrigatorios e opcionais documentados por evento
- **Outbox:** Nao habilitado — todos os eventos sao gravados na mesma transacao DB (consistencia atomica sem outbox pattern)
- **Projecao de volume:** ~60 eventos/dia (NFR-001 §8) — volume baixo, sem preocupacao de storage

## 7. Isolamento e Controle de Acesso

- **Cross-tenant por design:** A tabela `org_units` NÃO possui coluna `tenant_id` — a hierarquia organizacional abrange múltiplos tenants por natureza (ADR-003)
- **MUST:** O acesso a `org_units` é controlado exclusivamente via RBAC (`@RequireScope` com `org:unit:read|write|delete`), não via RLS por tenant
- **MUST:** A coluna `tenant_id` em `org_unit_tenant_links` é FK de vínculo funcional (N4→N5), NÃO coluna de isolamento
- **MUST:** Domain events com `entity_type=org_unit` são visíveis para usuários com scope `org:unit:read` (filtro via RBAC, não tenant_id)
- **Implementação:** `@RequireScope` no preHandler de cada endpoint — nunca depender apenas do frontend

### 7.1 Isolamento de Departamentos (por tenant — SEC-001-M01)

> **Nota:** Departamentos são isolados **por tenant** (com `tenant_id`), diferente de org_units que é cross-tenant (ADR-003). Todas as queries DEVEM filtrar por `tenant_id` do JWT. Não é necessário RLS; o filtro é aplicado na camada de repository.

- **MUST:** Toda query de departamentos DEVE incluir `WHERE tenant_id = :tenantId` (extraído do JWT)
- **MUST:** Departamento de outro tenant → 404 (não 403, para não expor existência)
- **MUST:** `tenant_id` é derivado do JWT — nunca aceito no body da requisição

## 8. Proteções

- **Rate limit:** Herdado do gateway/middleware global (MOD-000)
- **Validação:** Zod schema no Fastify para todos os payloads de entrada
- **Campos imutáveis (BR-003, BR-010):** `codigo` e `parent_id` sao imutaveis apos criacao. PATCH com esses campos retorna 422 — proteção via validação no application layer, não apenas no schema
- **Nivel maximo (BR-011):** Backend rejeita criacao de filho de N4 na tabela `org_units` — nivel > 4 bloqueado com 422
- **Idempotencia (BR-012, INT-005):** Endpoints `POST /org-units` e `POST /org-units/:id/tenants` aceitam `Idempotency-Key`. Fail-open se mecanismo indisponivel
- **Unicidade de codigo (BR-008):** Constraint UNIQUE global; catch PostgreSQL 23505 → 409 Conflict (FR-001-C01)
- **Logs seguros:** Sem PII nos logs; apenas IDs e metadados
- **Sanitização:** Strings sanitizadas contra XSS antes de persistencia (NFR-001 §6)
- **X-Correlation-ID:** Obrigatório em todas as respostas (sucesso e erro RFC 9457)
- **Error handling:** Todos os erros seguem RFC 9457 via middleware global do MOD-000 (INT-006)

## 9. Autorização de Eventos (Emit/View/Notify)

- **MUST:** Não use "permissão no evento" como fonte de verdade. Use permissão no **comando** (Emit) e permissão na **entity/ACL** (View).
- **MUST:** `sensitivity_level=0` para todos os eventos — guard-rail apenas.
- **MUST:** Definir Matriz de Autorização de Eventos em SEC-002.
