# ADR-004 — Optimistic Locking via updated_at no Aggregate CaseInstance

- **status:** proposed
- **timebox_days:** 90

---

## Contexto

O aggregate root `CaseInstance` pode sofrer operações concorrentes de múltiplos usuários: dois operadores tentando transitar o mesmo caso, um gestor cancelando enquanto outro transita, um operador suspendendo (hold) enquanto outro tenta resolver gates que desbloqueiam transição. Sem controle de concorrência, a última escrita ganha (last-write-wins), podendo corromper o estado do caso — por exemplo, transitar de um estágio que já foi transitado por outro usuário.

NFR-006 §7 define a estratégia de concorrência. A decisão impacta a consistência do aggregate e a experiência do usuário.

## Decisão

**Optimistic locking via campo `updated_at` no UPDATE de `case_instances`.** Toda operação que altera o aggregate (transição, hold, resume, cancel, REOPENED) inclui `WHERE updated_at = :expected_updated_at`. Se o timestamp divergir (outra operação commitou primeiro), retorna 409 Conflict com `extensions.conflict_field: "updated_at"`. O frontend exibe mensagem de conflito e recarrega o estado.

## Alternativas Consideradas

| Alternativa | Prós | Contras |
|---|---|---|
| **A) Optimistic locking via updated_at (escolhida)** | Sem lock explícito; sem deadlock; escalável; campo já existe na tabela; latência mínima | Usuário perdedor precisa retentar; sem garantia de ordem (primeiro a commitar ganha) |
| **B) Pessimistic locking (SELECT FOR UPDATE)** | Garantia de serialização; sem retentativas | Lock explícito; risco de deadlock em operações cross-tabela; degradação de performance sob carga; timeout de lock bloqueante |
| **C) Version counter (version field)** | Semântica clara de versão; integração com ORMs | Campo adicional; lógica de incremento; essencialmente equivalente à opção A |
| **D) Fila de comandos por caso (FIFO)** | Serialização perfeita; zero conflitos | Complexidade operacional; latência de fila; overhead desproporcional para o volume esperado |

## Consequências

- `case_instances.updated_at` é atualizado em toda operação que altera o aggregate (via application, não trigger)
- Frontend DEVE recarregar o caso após 409 para obter o estado atualizado e permitir re-tentativa informada
- `SELECT FOR UPDATE` mantido apenas para `case_assignments` (reatribuição — BR-007) onde a unicidade ativa é crítica e a janela de conflito é curta
- Gate resolution usa UNIQUE(case_id, gate_id) como guard — concorrência na resolução é resolvida pela constraint, não pelo locking do aggregate
- Monitorar taxa de 409 para identificar hot spots (casos com muitos operadores simultâneos)
