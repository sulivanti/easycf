#### 2.2.1 `POST /api/v1/movement-engine/evaluate` — Avaliar operação

Este endpoint é chamado por outros módulos (middleware/hook) para avaliar se uma operação deve ser controlada.

**Request:**

```json
{
  "object_type": "pedido_venda",
  "operation_type": "CREATE",
  "origin_type": "API",
  "requested_by": "u1u2u3u4-u5u6-7890-abcd-ef1234567890",
  "operation_payload": {
    "cliente": "ACME Corp",
    "valor_total": 75000,
    "itens": [
      { "produto": "Licença Enterprise", "quantidade": 1, "valor": 75000 }
    ]
  },
  "operation_value": 75000,
  "context": {
    "case_id": "c1c2c3c4-c5c6-7890-abcd-ef1234567890"
  },
  "dry_run": false
}
```

| Campo | Tipo | Required | Default | Descrição |
|---|---|---|---|---|
| `object_type` | string | SIM | — | Tipo de objeto da operação |
| `operation_type` | string | SIM | — | Tipo de operação (CREATE, UPDATE, DELETE) |
| `origin_type` | string | SIM | — | Origem (HUMAN, API, MCP) |
| `requested_by` | uuid | SIM | — | Usuário solicitante |
| `operation_payload` | object | SIM | — | Payload completo para reexecução pós-aprovação |
| `operation_value` | number | NÃO | — | Valor da operação para regras baseadas em threshold |
| `context` | object | NÃO | — | Contexto adicional (case_id, etc.) |
| `dry_run` | boolean | NÃO | `false` | Se `true`, o motor avalia normalmente mas **NÃO persiste** em `controlled_movements`, `approval_instances`, `movement_history`. NÃO emite domain events. Response idêntica com campo adicional `dry_run: true`. (PEN-009-003) |

**Response 200 — Dry-Run (quando `dry_run: true`):**

```json
{
  "controlled": true,
  "dry_run": true,
  "control_rule_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "control_rule_codigo": "VENDAS_ALTO_VALOR",
  "approval_levels": 2,
  "message": "Simulação: operação seria interceptada — 2 níveis de aprovação"
}
```

> **Nota (PEN-009-003):** Em modo dry-run, o response não inclui `movement_id` nem `codigo` (nenhum registro é criado). O campo `dry_run: true` está presente no response para confirmação. Ideal para o botão "Simular motor" (UX-APROV-002 `act-config-simulate`).

**Response 200 — Operação Livre (nenhuma regra aplica):**

```json
{
  "controlled": false
}
```

**Response 202 — Operação Controlada (interceptada):**

```json
{
  "controlled": true,
  "movement_id": "m1m2m3m4-m5m6-7890-abcd-ef1234567890",
  "codigo": "MOV-2026-00042",
  "status": "PENDING_APPROVAL",
  "current_approval_level": 1,
  "control_rule_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "message": "Operação interceptada — aguardando aprovação nível 1"
}
```

**Response 200 — Auto-Aprovação por Scope:**

```json
{
  "controlled": true,
```

#### 2.3.5 `POST /api/v1/movements/:id/retry` — Reprocessar movimento FAILED (PEN-009-006)

> **Decisão (PEN-009-006):** Endpoint dedicado com scope `approval:override`. Operação excepcional com auditoria completa.

**Pré-condições:**

- Movimento DEVE ter status `FAILED` em `movement_executions` (422 se outro status)
- Usuário DEVE ter scope `approval:override`

**Request:**

```json
{
  "reason": "Falha temporária do serviço Protheus resolvida. Reprocessamento autorizado pela coordenação."
}
```

| Campo | Tipo | Required | Descrição |
|---|---|---|---|
| `reason` | string | SIM | Justificativa obrigatória para reprocessamento (min 20 chars) |

**Response 200:**

```json
{
  "movement_id": "m1m2m3m4-m5m6-7890-abcd-ef1234567890",
  "new_execution_id": "exec-uuid-002",
  "retry_of": "exec-uuid-001",
  "status": "PENDING",
  "retried_by": "g1g2g3g4-g5g6-7890-abcd-ef1234567890",
  "retried_at": "2026-03-19T17:00:00Z",
  "message": "Reprocessamento iniciado — nova execução criada"
}
```

**Comportamento:**

1. Cria novo registro em `movement_executions` com `retry_of` FK apontando para a execução original FAILED
2. Re-emite domain event `movement.approved` com `is_final_level: true` para trigger da execução
3. Emite domain event `movement.retried` para auditoria
4. Atualiza `controlled_movements.status` de volta para `APPROVED` (aguardando nova execução)

**Erros:**

- `422` — Movimento não está com status FAILED: `{ rule: "MOVEMENT_NOT_FAILED", current_status: "..." }`
- `422` — Justificativa insuficiente: `{ rule: "JUSTIFICATION_REQUIRED", min_length: 20, actual_length: N }`

## 3. Contratos de Erro (RFC 9457 Problem Details)

Todos os erros retornam `Content-Type: application/problem+json` com `extensions.correlationId`:

### 3.1 Tabela de Erros

| HTTP | type | Quando | extensions |
|---|---|---|---|
| 400 | `validation_error` | Shape ou payload inválido | `invalid_fields[]`, `correlationId` |
| 401 | `unauthorized` | JWT ausente/expirado | `correlationId` |
| 403 | `forbidden` | Scope insuficiente | `required_scope`, `correlationId` |
| 404 | `not_found` | Recurso não existe | `correlationId` |
| 409 | `conflict` | Duplicata (código de regra, nível de alçada) | `conflicting_entity`, `correlationId` |
| 422 | `business_rule` | Regra de negócio violada | Varia por BR, `correlationId` |
| 429 | `rate_limit_exceeded` | Rate limit excedido no motor | `retry_after`, `correlationId` |
| 503 | `service_unavailable` | Motor indisponível | `correlationId` |

### 3.2 Exemplos de Erro

**400 — Validação:**

```json
{
  "type": "about:blank",
  "title": "Validation Error",
  "status": 400,
  "detail": "Um ou mais campos possuem valores inválidos.",
  "instance": "/api/v1/admin/control-rules",
  "extensions": {
    "correlationId": "corr-uuid-123",
    "invalid_fields": [
      { "field": "codigo", "message": "Deve conter 2-50 caracteres (A-Z, 0-9, _)" },
      { "field": "value_threshold", "message": "Deve ser um número positivo" }
    ]
  }
}
```

## 4. Middleware/Hook — Padrão de Integração do Motor

O motor de controle (`POST /movement-engine/evaluate`) é projetado para ser chamado como **middleware/hook** por outros módulos. O padrão de integração é:

### 4.1 Fluxo de Integração (Hook Pattern)

```
Módulo chamador (ex: MOD-003, MOD-006, MOD-008)
  │
  ├─ 1. Recebe request de operação crítica
  ├─ 2. Antes de executar, chama POST /movement-engine/evaluate
  │     ├─ Response { controlled: false } → prossegue normalmente
  │     ├─ Response 202 { controlled: true } → retorna 202 ao cliente
  │     └─ Response 5xx → retorna 503 (never execute without evaluation)
  ├─ 3. Operação aprovada → MOD-009 chama callback/webhook para executar
  └─ 4. Resultado da execução registrado em movement_executions
```

### 4.2 Contrato do Hook (módulo chamador → MOD-009)

O módulo chamador DEVE:

1. **Chamar o motor antes de executar** a operação (middleware pre-execution)
2. **Propagar `X-Correlation-ID`** do request original
3. **Incluir `operation_payload`** com todos os dados necessários para reexecução após aprovação
4. **Respeitar o HTTP status:** se `202`, retornar `202` ao cliente com `movement_id`
5. **Implementar callback de execução:** quando o movimento for aprovado, MOD-009 emite `movement.approved` (nível final) e o módulo chamador deve executar a operação original

### 4.3 Callback de Execução Pós-Aprovação

> **Decisão (PEN-009-001):** Domain event com outbox (assíncrono). Alinhado com ADR-003 e padrão estabelecido para eventos 4-13.

Quando o último nível da cadeia aprova, o MOD-009:

1. Atualiza `controlled_movements.status → APPROVED`
2. Insere registro no outbox com evento `movement.approved` (garantia at-least-once)
3. O módulo chamador consome o evento e executa a operação original usando `operation_payload`
4. O módulo chamador reporta resultado:
   - Sucesso: MOD-009 registra em `movement_executions` com `result: SUCCESS`
   - Falha: MOD-009 registra com `result: FAILED` + `error_message`

#### 4.3.1 Contrato do Evento `movement.approved` (Callback)

**Event type:** `movement.approved`
**Outbox:** Sim — garantia at-least-once via outbox pattern (DATA-003 evento 7)
**Dedupe key:** `movement_id + level`

**Payload:**

```json
{
  "event_type": "movement.approved",
  "movement_id": "m1m2m3m4-m5m6-7890-abcd-ef1234567890",
  "codigo": "MOV-2026-00042",
  "object_type": "pedido_venda",
  "operation_type": "CREATE",
  "origin_type": "API",
  "operation_payload": {
    "cliente": "ACME Corp",
    "valor_total": 75000,
    "itens": [
      { "produto": "Licença Enterprise", "quantidade": 1, "valor": 75000 }
    ]
  },
  "decision": "APPROVED",
  "decided_by": "apr1-uuid",
  "decided_at": "2026-03-19T16:00:00Z",
  "is_final_level": true,
  "approval_level": 2,
  "total_levels": 2,
  "control_rule_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "requested_by": "u1u2u3u4-u5u6-7890-abcd-ef1234567890",
  "context": {
    "case_id": "c1c2c3c4-c5c6-7890-abcd-ef1234567890"

  "correlation_id": "corr-uuid-123",
  "occurred_at": "2026-03-19T16:00:01Z"
}
```

#### 4.3.2 Consumer Contract (Módulo Chamador)

O módulo que submeteu a operação ao motor de controle DEVE:

1. **Subscrever** ao evento `movement.approved` filtrando por `is_final_level: true`
2. **Executar** a operação original usando o `operation_payload` do evento
3. **Reportar resultado** via `POST /api/v1/movements/:movement_id/execution-result`:

**Request (sucesso):**

```json
{
  "result": "SUCCESS",
  "result_payload": { "pedido_id": "ped-uuid-001" }
}
```

**Request (falha):**

```json
{
  "result": "FAILED",
  "error_message": "Erro ao criar pedido: saldo insuficiente",
  "error_code": "INSUFFICIENT_BALANCE"
}
```

1. **Idempotência:** Processamento DEVE ser idempotente (deduplicar por `movement_id` — se já processou, ignorar)
2. **Garantia:** At-least-once via outbox. Consumer DEVE tolerar entregas duplicadas.

> **Módulos consumidores conhecidos:** MOD-006 (Execução de Casos), MOD-010 (MCP — para fluxos CONTROLLED), MOD-008 (Integração Protheus — para operações de integração controladas).

#### 4.3.3 Callback HTTP para Movimentos MCP (origin_type=MCP)

> **Decisão (PEN-010/PENDENTE-005):** Para movimentos com `origin_type=MCP`, além do domain event `movement.approved` (§4.3.1), o MOD-009 DEVE chamar o callback HTTP do MOD-010 para consistência imediata. O domain event coexiste para auditoria e como fallback.

Quando `controlled_movement.origin_type = MCP` e a decisão final é tomada (`is_final_level: true`):

1. MOD-009 chama `POST /api/v1/internal/mcp/executions/{execution_id}/movement-callback` (contrato completo em INT-010 §INT-007)
2. Body inclui: `decision` (APPROVED/REJECTED), `decided_by`, `decided_at`, `movement_id`, `reason`
3. MOD-010 atualiza `mcp_executions.status` de `CONTROLLED_PENDING` para `CONTROLLED_APPROVED` ou `CONTROLLED_REJECTED`
4. Em paralelo, o domain event `movement.approved` é emitido via outbox (§4.3.1) — garantindo auditoria e eventual consistency para outros consumidores

**Configuração:** Ao receber request de evaluate com `origin_type=MCP` (INT-010 §INT-001), o MOD-009 registra o `callback_url` no `controlled_movement`:

```json
{
  "callback_url": "/api/v1/internal/mcp/executions/{object_id}/movement-callback",
  "callback_module": "MOD-010"
}
```

**Failure Behavior:** Se o callback HTTP falhar (timeout/5xx), MOD-009 retenta 3x com backoff (500ms, 1s, 2s). Se esgotar retries, registra em DLQ. O domain event via outbox serve como mecanismo de recuperação — MOD-010 também consome `movement.approved` como fallback.

- **rastreia_para:** INT-010 §INT-007, PEN-010/PENDENTE-005, FR-010

## 5. Async Operations — Failure Behavior

### 5.1 Notificação de Aprovadores

> **Decisão (PEN-009-004):** Canal MVP: **in-app** via badge no SidebarBadge + entrada na lista de inbox. Email como canal adicional configurável por tenant no roadmap (não MVP).

| Propriedade | Valor |
|---|---|
| **trigger** | Evento `movement.created` ou `movement.escalated` |
| **canal MVP** | **In-app** — outbox consumer grava notificação na tabela de notificações in-app. Badge no SidebarBadge atualiza contagem. Entrada adicionada à lista do inbox do aprovador. |
| **canal roadmap** | Email configurável por tenant — enhancement futuro, não MVP. Sem dependência de serviço de email no MVP. |
| **delivery** | Via domain_event → notification queue (assíncrono) → INSERT em tabela de notificações in-app |
| **retry** | 3 tentativas com backoff exponencial (1s, 2s, 4s) |
| **DLQ** | Sim — notificações que falham após retries vão para DLQ |
| **fallback** | Notificação falha não bloqueia o fluxo de aprovação; aprovador pode ver via inbox (polling 60s) |
| **idempotency** | `dedupe_key` no outbox garante entrega at-most-once por consumidor |
| **monitoring** | Alerta se taxa de falha de notificação > 5% em 5 min |

### 5.2 Job de Timeout/Escalada

| Propriedade | Valor |
|---|---|
| **trigger** | Cron job — execução a cada 5 minutos |
| **query** | `SELECT FROM approval_instances WHERE status = 'PENDING' AND timeout_at <= now()` |
| **ação (com escalada)** | Atualiza instância para `ESCALATED`, cria nova instância no nível escalado, emite `movement.escalated` |
| **ação (sem escalada)** | Atualiza instância para `TIMEOUT`, emite `movement.timeout` |
| **concorrência** | Lock distribuído (advisory lock) para evitar dupla execução em múltiplas instâncias |
| **retry** | Job idempotente — re-execução segura (só processa instâncias ainda PENDING) |
| **monitoring** | Alerta se job não executar por > 15 min |

### 5.3 Execução de Operação Aprovada

| Propriedade | Valor |
|---|---|
| **trigger** | Evento `movement.approved` com `is_final_level: true` |
| **retry** | 3 tentativas com backoff exponencial (500ms, 1s, 2s) |
| **DLQ** | Sim — operações que falham após retries registradas como FAILED em `movement_executions` |
| **fallback** | Status `FAILED` em `movement_executions`, `error_message` registrado, evento `movement.failed` emitido |
| **reprocessamento** | Possível via `retry_of` FK em `movement_executions` |
| **monitoring** | Alerta imediato em `movement.failed` |

## 7. Resiliência e Observabilidade

| Aspecto | Valor |
|---|---|
| **Timeout do motor** | 5000ms (síncrono) — se exceder, retorna 503 |
| **Retry (execução)** | 3 tentativas com backoff exponencial (500ms inicial) + jitter |
| **Dead Letter Queue** | Sim — operações FAILED após retries |
| **Idempotência** | `X-Correlation-ID` propagado; `Idempotency-Key` no evaluate e override |
| **Fallback** | Motor indisponível → 503 (nunca executa sem avaliação); notificação falha → inbox disponível |
| **Correlation** | `correlation_id` em todos os registros; `movement_history` como audit trail completo |
| **Structured Logging** | `correlationId` + `operationId` em todos os logs; sem PII |
