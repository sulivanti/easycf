# BR-001 — Visibilidade de Ações por Scope RBAC

- **Regra:** Toda ação de UI no MOD-002 DEVE ser ocultada (`hide_if_no_permission`) quando o usuário autenticado não possuir o scope correspondente. A estratégia é "ocultar", nunca "desabilitar" para ações sem permissão — o elemento simplesmente não é renderizado.
- **Exemplo:**
  - Usuário com `users:user:read` (sem `:write`): vê a listagem mas NÃO vê o botão "Novo Usuário"
  - Usuário com `users:user:read` + `users:user:write` (sem `:delete`): vê botão "Novo Usuário" mas NÃO vê ação "Desativar" no dropdown
  - Usuário sem `users:user:read`: é redirecionado para /dashboard com Toast
- **Exceções:** Nenhuma. A regra é absoluta — sem exceção por role ou contexto.
- **Impacto:** segurança, fluxo, compliance

### Mapeamento de Scopes por Ação

| Scope | Ações visíveis | Telas acessíveis |
|---|---|---|
| `users:user:read` | Listar, filtrar, buscar, paginar, navegar para convite, Editar (navegação) | /usuarios, /usuarios/:id/convite, /usuarios/:id |
| `users:user:write` | Criar usuário, reenviar convite, Bloquear, Desbloquear, Reativar, Resetar senha | /usuarios/novo |
| `users:user:delete` | Desativar usuário, Cancelar convite (modal de confirmação) | — (ação dentro de /usuarios) |

### Critérios de aceite (Gherkin)

```gherkin
Funcionalidade: Visibilidade de ações por scope RBAC

  Cenário: Elemento oculto quando scope ausente
    Dado que o usuário autenticado NÃO possui scope "users:user:write"
    Quando a tela /usuarios é renderizada
    Então o botão "Novo Usuário" NÃO deve existir no DOM
    E a opção "Reenviar convite" NÃO deve existir no dropdown de ações

  Cenário: Redirect quando scope de leitura ausente
    Dado que o usuário autenticado NÃO possui scope "users:user:read"
    Quando ele tenta acessar /usuarios
    Então deve ser redirecionado para /dashboard
    E um Toast deve exibir: "Sem permissão para acessar esta seção."

  Cenário: Ação "Desativar" oculta sem scope de exclusão
    Dado que o usuário possui scopes "users:user:read" e "users:user:write"
    Mas NÃO possui scope "users:user:delete"
    Quando a listagem de usuários é exibida
    Então o dropdown de ações NÃO deve incluir "Desativar"

  Cenário: Ação "Bloquear" oculta sem scope de escrita
    Dado que o usuário possui scope "users:user:read" apenas
    Quando a listagem de usuários ATIVOS é exibida
    Então o dropdown de ações deve incluir apenas "Editar"
    E NÃO deve incluir "Bloquear", "Resetar senha", "Desativar"

  Cenário: Ação "Cancelar convite" oculta sem scope de exclusão
    Dado que o usuário possui scopes "users:user:read" e "users:user:write"
    Mas NÃO possui scope "users:user:delete"
    Quando a listagem de usuários PENDENTES é exibida
    Então o dropdown deve incluir "Editar" e "Bloquear"
    Mas NÃO deve incluir "Cancelar convite"
```

### Matriz de Visibilidade por Status × Scope (BR-001-M01)

| Status | `read` only | `read` + `write` | `read` + `write` + `delete` |
|---|---|---|---|
| **ATIVO** | Editar | Editar, Resetar senha, Bloquear | Editar, Resetar senha, Desativar, Bloquear |
| **INATIVO** | Editar | Editar, Reativar, Bloquear | Editar, Reativar, Bloquear |
| **BLOQUEADO** | Editar | Editar, Desbloquear | Editar, Desbloquear, Desativar |
| **PENDENTE** | Editar | Editar, Bloquear | Editar, Cancelar convite, Bloquear |

> **Nota:** "Reenviar convite" (PENDING) requer `users:user:write` — documentado na Jornada 3 de UX-001.
