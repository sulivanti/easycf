## Detalhamento

### 1. `UserListItem` — Adicionar `role_id` e `role_name`

**Situação atual:** O schema `UserListItem` retorna `id`, `codigo`, `email`, `full_name`, `status`, `created_at`. Não inclui informação de role.

**Problema:** MOD-002 UX-USR-001 (listagem de usuários) precisa exibir o perfil/role do usuário. O frontend hardcoda `roleId: '', roleName: ''` como workaround.

**Alteração no OpenAPI (`v1.yaml` — `UserListItem`):**

```yaml
UserListItem:
  type: object
  properties:
    id: { type: string, format: uuid }
    codigo: { type: string }
    email: { type: string }
    full_name: { type: string }
    status: { type: string }
    role_id: { type: string, format: uuid, nullable: true }    # NOVO
    role_name: { type: string, nullable: true }                 # NOVO
    created_at: { type: string, format: date-time }
```

**Regras:**
- `role_id` e `role_name` são `nullable: true` porque um usuário pode não estar vinculado a nenhuma filial/role (ex: recém-criado por auto-registro)
- O backend DEVE fazer LEFT JOIN com `tenant_users` + `roles` ao montar a listagem
- Se o usuário tem múltiplas vinculações (multi-tenant), retornar a role do **tenant ativo no contexto da sessão** (header `X-Tenant-ID`)
- Se não há vinculação, retornar `null` em ambos os campos

**Gherkin:**

```gherkin
Cenário: Listagem inclui role do tenant ativo
  Dado que o admin está autenticado no tenant "filial-sp"
  E que o usuário "joao@empresa.com" tem role "Operador" nesse tenant
  Quando GET /api/v1/users
  Então cada item da lista inclui role_id e role_name
  E o usuário "joao@empresa.com" aparece com role_name "Operador"

Cenário: Usuário sem vinculação ao tenant
  Dado que o usuário "maria@empresa.com" não tem vinculação ao tenant ativo
  Quando GET /api/v1/users
  Então o item de "maria@empresa.com" tem role_id null e role_name null
```

### 2. `UserDetail` — Adicionar `invite_token_expired`

**Situação atual:** O schema `UserDetail` não inclui informação sobre expiração do token de convite.

**Problema:** MOD-002 UX-USR-003 (fluxo de convite) precisa saber se o token de convite expirou para distinguir entre "Convite Pendente" e "Link Expirado" na UI. O frontend hardcoda `inviteTokenExpired: false`.

**Alteração no OpenAPI (`v1.yaml` — `UserDetail`):**

```yaml
UserDetail:
  type: object
  properties:
    id: { type: string, format: uuid }
    codigo: { type: string }
    email: { type: string }
    full_name: { type: string }
    cpf_cnpj: { type: string, nullable: true }
    avatar_url: { type: string, nullable: true }
    status: { type: string }
    invite_token_expired: { type: boolean, nullable: true }     # NOVO
    created_at: { type: string, format: date-time }
    updated_at: { type: string, format: date-time }
```

**Regras:**
- `invite_token_expired` é `nullable: true` — retorna `null` quando o usuário nunca recebeu convite (ex: auto-registro, status ACTIVE)
- Retorna `true` quando `status = PENDING` e o token de convite ultrapassou o TTL configurado
- Retorna `false` quando `status = PENDING` e o token ainda é válido
- O cálculo é feito no backend comparando `invite_token_created_at + TTL > now()`
- **Não expor o token em si** — apenas o booleano de expiração (LGPD/segurança)

**Pré-requisito de infraestrutura:** A tabela `users` (ou tabela auxiliar `user_invite_tokens`) DEVE armazenar `invite_token_created_at` para viabilizar o cálculo. Se ainda não existe, criar via migration.

**Gherkin:**

```gherkin
Cenário: Token de convite válido
  Dado que o usuário tem status PENDING
  E que o token de convite foi gerado há menos de 48h (TTL padrão)
  Quando GET /api/v1/users/:id
  Então invite_token_expired = false

Cenário: Token de convite expirado
  Dado que o usuário tem status PENDING
  E que o token de convite foi gerado há mais de 48h
  Quando GET /api/v1/users/:id
  Então invite_token_expired = true

Cenário: Usuário ativo (sem convite)
  Dado que o usuário tem status ACTIVE
  Quando GET /api/v1/users/:id
  Então invite_token_expired = null
```

### 3. `CreateUserRequest` — Adicionar `mode` e `role_id`

**Situação atual:** O schema `CreateUserRequest` requer `email`, `password`, `full_name` e aceita `cpf_cnpj` opcional. Não há distinção entre criação por convite e por senha temporária. O backend infere o modo pelo campo interno `createdBy`.

**Problema:** MOD-002 UX-USR-002 (formulário de cadastro) tem dois modos mutuamente exclusivos: "Enviar Convite" (sem senha) e "Senha Temporária" (com senha). O frontend usa discriminated union local, mas a API não aceita `mode` nem `role_id`.

**Alteração no OpenAPI (`v1.yaml` — `CreateUserRequest`):**

```yaml
CreateUserRequest:
  type: object
  required: [email, full_name, mode]
  properties:
    email: { type: string, format: email, maxLength: 255 }
    full_name: { type: string, minLength: 1, maxLength: 255 }
    cpf_cnpj: { type: string, maxLength: 20 }
    mode:
      type: string
      enum: [invite, password]
      description: "Modo de criação: 'invite' envia convite por e-mail, 'password' define senha temporária"
    password:
      type: string
      minLength: 8
      maxLength: 128
      description: "Obrigatório quando mode='password'. Ignorado quando mode='invite'."
    force_pwd_reset:
      type: boolean
      default: true
      description: "Força troca de senha no primeiro login. Apenas quando mode='password'."
    role_id:
      type: string
      format: uuid
      description: "Role a atribuir ao usuário no tenant ativo. Opcional — se omitido, usuário é criado sem vinculação."
```

**Regras:**
- `mode` passa a ser **required** (substitui a inferência por `createdBy`)
- `password` é **required** quando `mode = 'password'`, **ignorado** quando `mode = 'invite'`
- Quando `mode = 'invite'`: backend gera token de ativação, envia e-mail, status = PENDING
- Quando `mode = 'password'`: backend define senha temporária, `force_pwd_reset = true` por padrão, status = PENDING (admin-created)
- `role_id` é **opcional** — quando fornecido, o backend cria a vinculação `tenant_users` atomicamente na mesma transação
- **Backward compatibility:** Para auto-registro público (sem auth), `mode` default = `'password'` e `role_id` não é aceito (403 se fornecido sem auth)

**Validação condicional (backend):**

```text
mode = 'invite' + password fornecido  → 422: "Password must not be sent when mode is 'invite'"
mode = 'password' + password ausente  → 422: "Password is required when mode is 'password'"
role_id fornecido + role não existe   → 422: "Role not found"
role_id fornecido + sem auth          → 403: "Role assignment requires authentication"
```

**Gherkin:**

```gherkin
Cenário: Criação por convite com role
  Dado que o admin tem escopo users:user:write
  Quando POST /api/v1/users com mode='invite', email, full_name, role_id
  Então status 201 com usuário criado (status PENDING)
  E token de convite gerado
  E e-mail de convite enfileirado via Outbox
  E vinculação tenant_users criada com role_id na mesma transação

Cenário: Criação por senha temporária
  Dado que o admin tem escopo users:user:write
  Quando POST /api/v1/users com mode='password', email, full_name, password
  Então status 201 com usuário criado (status PENDING, force_pwd_reset=true)
  E e-mail de boas-vindas enfileirado

Cenário: Modo invite com password (erro)
  Quando POST /api/v1/users com mode='invite' e password preenchido
  Então status 422 com detail "Password must not be sent when mode is 'invite'"

Cenário: Auto-registro público (backward compatible)
  Quando POST /api/v1/users sem auth com email, password, full_name
  Então status 201 com usuário criado (status ACTIVE)
  E mode inferido como 'password' por compatibilidade
```
