# FR-005 — Requisitos Funcionais da Modelagem de Processos

---

## FR-001 — CRUD de Ciclos de Processo

- **Goal:** Permitir criar, listar, visualizar, editar e excluir ciclos de processo com máquina de estados (DRAFT → PUBLISHED → DEPRECATED).
- **Done funcional:**
  - POST /admin/cycles cria ciclo em DRAFT com version=1
  - GET /admin/cycles lista com paginação cursor-based, filtro por status
  - GET /admin/cycles/:id retorna detalhe
  - PATCH /admin/cycles/:id edita (apenas DRAFT — BR-001)
  - DELETE /admin/cycles/:id soft-delete (apenas DRAFT — BR-005)
  - Campo `codigo` imutável após criação (BR-006)
- **Dependências:** BR-001, BR-005, BR-006, BR-010, DATA-005 (process_cycles), INT-005 §1.1
- **Idempotency:** n/a (CRUD padrão)
- **Timeline/Notifications:** yes — emite `process.cycle_created` (DATA-003)

```gherkin
Funcionalidade: CRUD de Ciclos

Cenário: Criar ciclo com sucesso
  Dado que o usuário tem scope process:cycle:write
  Quando POST /admin/cycles com { codigo: "COM", nome: "Comercial" }
  Então 201 com ciclo em status=DRAFT, version=1

Cenário: Listar ciclos com filtro por status
  Dado que existem ciclos em DRAFT e PUBLISHED
  Quando GET /admin/cycles?status=PUBLISHED
  Então retorna apenas ciclos PUBLISHED com paginação cursor-based

Cenário: Rejeitar edição de ciclo PUBLISHED
  Dado que ciclo tem status=PUBLISHED
  Quando PATCH /admin/cycles/:id com { nome: "Novo" }
  Então 422: "Ciclos publicados são imutáveis."
```

- **Nota de implementação (FR-001-C01):** Os handlers de `POST /admin/cycles`, `PATCH /admin/cycles/:id`, `POST /cycles/:id/publish`, `POST /cycles/:id/fork` e `POST /cycles/:id/deprecate` DEVEM enviar no response body os dados reais retornados pelo use case (`result.nome`, `result.descricao`, `result.createdAt.toISOString()`, `result.updatedAt.toISOString()`). Não usar `new Date().toISOString()` nem valores `null` hardcoded para campos não-nullable como `nome`. Ref: spec-fix-cycle-response-schema-mismatch.

## FR-002 — Publicação de Ciclo

- **Goal:** Promover ciclo DRAFT para PUBLISHED após validação de pré-condições.
- **Done funcional:**
  - POST /admin/cycles/:id/publish valida ao menos 1 estágio com is_initial=true (BR-003)
  - Valida ao menos 1 macroetapa com estágios (BR-003)
  - Define published_at=now() e status=PUBLISHED
  - Ciclo torna-se imutável (BR-001)
  - Aceita instâncias no MOD-006
- **Dependências:** BR-001, BR-002, BR-003, DATA-005, INT-005 §1.1
- **Idempotency:** required — publicação duplicada deve retornar 422 se já PUBLISHED (sem side-effect)
- **Timeline/Notifications:** yes — emite `process.cycle_published`, notifica admin (SEC-002)

```gherkin
Funcionalidade: Publicação de Ciclo

Cenário: Publicar ciclo com pré-condições atendidas
  Dado que ciclo DRAFT tem 1 macroetapa e 1 estágio com is_initial=true
  Quando POST /admin/cycles/:id/publish
  Então 200 com status=PUBLISHED e published_at preenchido
  E evento process.cycle_published emitido

Cenário: Rejeitar publicação sem estágio inicial
  Dado que ciclo DRAFT não tem estágio com is_initial=true
  Quando POST /admin/cycles/:id/publish
  Então 422: "O ciclo precisa de ao menos um estágio inicial."

Cenário: Publicação idempotente para ciclo já publicado
  Dado que ciclo já tem status=PUBLISHED
  Quando POST /admin/cycles/:id/publish
  Então 422: "Ciclo já está publicado." (sem side-effect)
```

## FR-003 — Fork de Ciclo Publicado

- **Goal:** Criar nova versão DRAFT a partir de ciclo PUBLISHED, copiando toda a estrutura.
- **Done funcional:**
  - POST /admin/cycles/:id/fork cria novo ciclo DRAFT com parent_cycle_id e version+1
  - Copia atomicamente: macroetapas, estágios, gates, role_links, transições (BR-004)
  - Novos UUIDs para todos os elementos; referências internas remapeadas
  - Retorna 201 com novo ciclo
  - Header `Idempotency-Key` garante deduplicação
- **Dependências:** BR-004, DATA-005 (7 tabelas), INT-005 §1.1
- **Idempotency:** required — fork duplicado (mesmo request) deve gerar apenas 1 ciclo (Idempotency-Key)
- **Timeline/Notifications:** yes — emite `process.cycle_forked` com `causation_id` (DATA-003)

```gherkin
Funcionalidade: Fork de Ciclo

Cenário: Fork copia estrutura completa
  Dado que ciclo v1 PUBLISHED tem 3 macroetapas, 8 estágios e 10 transições
  Quando POST /admin/cycles/:id/fork
  Então 201 com novo ciclo DRAFT, version=2, parent_cycle_id=ciclo_v1.id
  E todos os sub-elementos são clonados com novos UUIDs
  E referências internas (from_stage_id, to_stage_id) apontam para os novos IDs
  E evento process.cycle_forked emitido com causation_id

Cenário: Fork idempotente via Idempotency-Key
  Dado que fork já foi executado com Idempotency-Key=ABC
  Quando POST /admin/cycles/:id/fork com Idempotency-Key=ABC
  Então retorna o mesmo ciclo criado anteriormente sem duplicar
```

## FR-004 — Depreciação de Ciclo

- **Goal:** Marcar ciclo PUBLISHED como DEPRECATED, bloqueando novas instâncias no MOD-006.
- **Done funcional:**
  - PATCH /admin/cycles/:id com { status: "DEPRECATED" } (apenas de PUBLISHED — BR-010)
  - Instâncias ativas existentes continuam até conclusão (BR-011)
  - Novas instâncias bloqueadas no MOD-006 (BR-011)
  - Depreciação de DRAFT retorna 422 (BR-010)
- **Dependências:** BR-010, BR-011, INT-005 §1.1
- **Idempotency:** n/a (PATCH idempotente por natureza)
- **Timeline/Notifications:** yes — emite `process.cycle_deprecated`, notifica admin (SEC-002)


## FR-011 — Endpoint /flow (Grafo Completo)

- **Goal:** Retornar o grafo completo de um ciclo em uma única chamada para consumo pelo editor visual.
- **Done funcional:**

  - Inclui canvas_x/canvas_y para posicionamento visual
  - Query otimizada com JOINs + agrupamento em memória (DATA-005 §5.1)
  - Avaliar CTE/queries separadas se ultrapassar 200ms com grafos >50 nos
- **Dependências:** DATA-005 (todas as tabelas), INT-005 §3, NFR-005
- **Idempotency:** n/a (leitura)
- **Timeline/Notifications:** no

```gherkin
Funcionalidade: Endpoint /flow

Cenário: Retornar grafo completo em <200ms
  Dado que ciclo tem 5 macroetapas, 15 estágios e 20 transições
  Quando GET /admin/cycles/:id/flow
  Então 200 com estrutura aninhada completa
  E response time < 200ms

Cenário: Ciclo inexistente
```

- **Nota de implementação (FR-011-C01):** O handler de `GET /admin/cycles/:id/flow` DEVE mapear o `FlowGraph` retornado pelo domain service (camelCase: `cycleId`, `macroStages`, `isInitial`, `canvasX`, `stageId`, `gateType`, `transitionsOut`, etc.) para o formato do Zod schema `flowResponse` (snake_case: `cycle`, `macro_stages`, `is_initial`, `canvas_x`, `stage_id`, `gate_type`, `transitions_out`). Adicionalmente, o campo `cycle` do response deve conter `{id, codigo, nome, version, status}` obtidos via `getCycleUseCase`. O campo `to_stage_codigo` nas transitions deve ser resolvido via lookup no array de stages. Não enviar `reply.send(result)` diretamente — a conversão camelCase→snake_case é responsabilidade da camada de apresentação. Ref: spec-fix-cycle-response-schema-mismatch.
