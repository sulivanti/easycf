# ADR-004 — Optimistic Locking via `updated_at` para Edicao Concorrente de Ciclos DRAFT

- **status:** accepted
- **timebox_days:** 90

---

## Contexto

Ciclos DRAFT podem ser editados por multiplos analistas simultaneamente (cenario colaborativo). O mod.md define Nivel 2 (DDD-lite + Full Clean) com dominio rico. A NFR-005 §8 ja documenta a estrategia de concorrencia, mas a **decisao arquitetural** de usar `updated_at` como token de versao (em vez de campo `version` dedicado ou pessimistic locking) precisa ser registrada como ADR, pois impacta o contrato de API (todos os PATCH devem enviar `updated_at`) e o tratamento de conflitos no frontend.

## Decisao

**Optimistic locking baseado em `updated_at` (timestamptz) com 409 Conflict (aceita).**

### Justificativa

1. **Campo ja existente:** `updated_at` ja e obrigatorio em todas as entidades com soft delete (DATA-005) — nenhum campo adicional necessario
2. **Precisao adequada:** `timestamptz` com microsegundos e suficiente para o volume de concorrencia esperado (analistas de processo, nao transacoes de alta frequencia)
3. **Simplicidade de contrato:** PATCH envia `updated_at` atual no body; se divergir, 409 com `extensions.conflict_field: "updated_at"` — frontend exibe "Alguem editou este item. Recarregue para ver as mudancas."
4. **Alinhamento com padrao do projeto:** Mesmo padrao usado nos modulos MOD-001, MOD-003 — consistencia cross-modulo
5. **Frontend resiliente:** Auto-save com debounce 800ms (UX-005 §3.2) + 409 → indicador "Erro ao salvar" + retry apos recarregar dados

### Implementacao

```typescript
// Service layer
async updateStage(id: UUID, dto: UpdateStageDto, expectedUpdatedAt: Date): Promise<Stage> {
  const result = await this.repo.updateWhere({
    id,
    updated_at: expectedUpdatedAt,  // WHERE updated_at = :expected
    deleted_at: null,
  }, {
    ...dto,
    updated_at: new Date(),
  });

  if (result.affected === 0) {
    throw new ConflictError('updated_at', 'O registro foi modificado por outro usuario.');
  }
  return result;
}
```

```sql
-- Query gerada
UPDATE process_stages
SET nome = :nome, updated_at = now()
WHERE id = :id
  AND updated_at = :expected_updated_at
  AND deleted_at IS NULL;
-- affected_rows = 0 → 409 Conflict
```

## Alternativas Consideradas

| Alternativa | Pros | Contras |
|---|---|---|
| **A) Optimistic via `updated_at` (aceita)** | Campo existente; padrao cross-modulo; simples | Risco teorico de colisao de timestamp (mitigado por microsegundos) |
| **B) Optimistic via campo `version` (integer)** | Sem risco de colisao; campo semantico | Campo adicional em todas as entidades; increment manual |
| **C) Pessimistic locking (SELECT FOR UPDATE)** | Garante exclusividade | Bloqueia outros usuarios; timeout de lock; UX degradada para edicao colaborativa |
| **D) CRDT / operational transform** | Edicao colaborativa real-time | Complexidade extrema; desnecessario para blueprint administrativo |

## Consequencias

- Todos os PATCH do MOD-005 devem incluir `updated_at` no body ou header (contrato de API)
- 409 Conflict retornado quando `updated_at` diverge — `extensions.conflict_field: "updated_at"`
- Frontend (UX-PROC-001/002) deve tratar 409: recarregar dados + exibir mensagem ao usuario
- Auto-save (debounce 800ms) pode gerar 409 se dois analistas editam o mesmo estagio simultaneamente — comportamento esperado e tratado
- Publicacao (DRAFT → PUBLISHED) usa SELECT FOR UPDATE (NFR-005 §8) como camada adicional — optimistic locking nao e suficiente para operacoes de mudanca de estado
