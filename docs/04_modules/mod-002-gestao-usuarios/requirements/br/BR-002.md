# BR-002 — Proteção de PII em Mensagens de UI (LGPD)

- **Regra:** O e-mail do usuário NUNCA DEVE aparecer em toasts, modais de confirmação ou mensagens de erro. Em todas as telas do MOD-002, o nome do usuário é o único identificador pessoal permitido em mensagens visíveis. Toasts de sucesso devem usar texto genérico.
- **Exemplo:**
  - Toast de criação (convite): "Usuário criado com sucesso. Convite enviado." — sem o e-mail
  - Toast de criação (senha): "Usuário criado com sucesso." — sem o e-mail
  - Modal de desativação: "O usuário **Carlos** perderá acesso imediatamente." — nome, nunca e-mail
  - Toast de reenvio: "Convite reenviado com sucesso." — sem o e-mail
  - Erro 409 (e-mail duplicado): mensagem inline no campo e-mail, nunca em toast
- **Exceções:** O campo de input de e-mail no formulário exibe o e-mail digitado pelo admin — isso é input, não mensagem.
- **Impacto:** compliance, segurança

### Critérios de aceite (Gherkin)

```gherkin
Funcionalidade: Proteção de PII em mensagens de UI

  Cenário: Toast de sucesso sem e-mail
    Dado que o admin criou um usuário com e-mail "carlos@empresa.com"

    Então o Toast de sucesso NÃO deve conter "carlos@empresa.com"
    E NÃO deve conter qualquer substring do e-mail

  Cenário: Modal de desativação usa nome, não e-mail

    Então o modal exibe "O usuário Carlos Silva perderá acesso imediatamente."
    E o e-mail "carlos@empresa.com" NÃO aparece no modal

  Cenário: Toast de reenvio sem e-mail
    Dado que o admin reenviou o convite para um usuário PENDING
    Quando POST /api/v1/users/:id/invite/resend retorna 200
    Então o Toast exibe "Convite reenviado com sucesso."
    E o e-mail do usuário NÃO aparece no Toast
```
