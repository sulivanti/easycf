# ADR-003 — org_units é Cross-Tenant (Sem Coluna tenant_id)

## Contexto

O sistema é multi-tenant. A maioria das tabelas possui `tenant_id` como coluna de RLS (Row-Level Security). A tabela `org_units` define a hierarquia organizacional que **contém** tenants no nível N5 (via `org_unit_tenant_links`). A questão é: org_units precisa de um campo `tenant_id` próprio?

## Decisão

**Não. A tabela `org_units` é cross-tenant por design e NÃO possui coluna `tenant_id`.** A hierarquia organizacional (N1–N4) abrange múltiplos tenants por natureza — são os tenants (N5) que são vinculados à hierarquia, não o contrário. O isolamento de acesso é feito via RBAC (`@RequireScope` com `org:unit:read|write|delete`), não via RLS por tenant.

A coluna `tenant_id` em `org_unit_tenant_links` é uma **FK de vínculo funcional** (N4→N5), NÃO uma coluna de isolamento. Na migration:

```sql
-- NOTA: tenant_id aqui é FK de vínculo N4→N5, NÃO é coluna RLS.
-- A tabela org_units é cross-tenant por design (ADR-003).
```

## Alternativas Consideradas

1. **Adicionar `tenant_id` como RLS em org_units:** Criaria ambiguidade — "de qual tenant é essa árvore?" não tem resposta clara quando a hierarquia serve justamente para organizar múltiplos tenants. Forçaria uma relação 1:1 (tenant:árvore) que contradiz o modelo N4→N5 (1:N).
2. **RLS via PostgreSQL Policies:** Inaplicável — não há coluna `tenant_id` para filtrar.
3. **Derivar acesso via org_unit_tenant_links:** Adicionaria JOINs complexos em toda query de leitura para verificar se o tenant ativo do usuário está vinculado a algum nó N4 da árvore. Over-engineering para Nível 1.

## Consequências

- **Positivas:** Modelo fiel à realidade — uma hierarquia organizacional é inerentemente cross-tenant; sem redundância; `codigo` é UNIQUE globalmente (BR-008); sem ambiguidade de semântica entre colunas homônimas.
- **Negativas:** Foge do padrão Foundation onde toda tabela tem `tenant_id`. Requer ADR (este documento) para justificar a exceção. Queries de domain_events filtram por `entity_type=org_unit` + RBAC, não por `tenant_id`.
- **Mitigação:** O RBAC via `@RequireScope` garante que apenas usuários com permissão acessem os endpoints de org_units. O escopo é gerenciado por perfil, não por tenant.
