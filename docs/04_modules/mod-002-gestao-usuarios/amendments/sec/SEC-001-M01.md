### Alterações neste pilar

#### 1. Mapeamento de Scopes para Novas Ações

Adicionar à tabela "Scopes Utilizados" de SEC-001 §2:

| Scope | Tipo | Onde é usado (adição) | Comportamento sem scope |
|---|---|---|---|
| `users:user:read` | Leitura | Ação "Editar" no dropdown (navegação para /usuarios/:id) | Elemento não renderizado |
| `users:user:write` | Escrita | Ações "Bloquear", "Desbloquear", "Reativar", "Resetar senha" no dropdown | Elementos não renderizados |
| `users:user:delete` | Exclusão | Ação "Cancelar convite" no dropdown | Elemento não renderizado |

> **Nota:** Não são necessários scopes novos. As novas ações reutilizam os 3 scopes existentes:
> - `write` cobre operações de gerenciamento de status (bloquear, desbloquear, reativar) e reset de senha
> - `delete` cobre operações destrutivas (desativar, cancelar convite)
> - `read` cobre navegação (editar é navegação + read)

#### 2. Classificação dos Dados para Novas Ações

Nenhum dado novo é introduzido. As novas ações operam sobre os mesmos dados já classificados (nome, status, e-mail). A regra de PII (e-mail nunca em toasts/modais) se aplica igualmente aos novos modais (bloquear, cancelar convite).

#### 3. Ações Sensíveis (Adição ao Threat Model)

| Ação | Risco | Mitigação |
|---|---|---|
| Resetar senha | Acesso indevido a conta | Scope `users:user:write` + modal de confirmação + notificação ao usuário afetado |
| Bloquear | Negação de serviço a usuário legítimo | Scope `users:user:write` + modal de confirmação + auditoria (DATA-003) |
| Cancelar convite | Impedir ativação de conta válida | Scope `users:user:delete` + modal de confirmação + auditoria (DATA-003) |
