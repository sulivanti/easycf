### 1.1 Latência

| Endpoint/Operação | Métrica | Baseline | Alvo | Fonte |
|---|---|---|---|---|
| Motor de controle (`POST /movement-engine/evaluate`) | P95 latência | — | **< 500ms** | US-MOD-009, OKR-4 |
| Motor de controle (`POST /movement-engine/evaluate`) | P99 latência | — | < 1000ms | Margem para queries complexas (múltiplas regras) |
| Inbox do aprovador (`GET /my/approvals`) | P95 latência | — | **< 300ms** | UX responsividade (inbox é tela principal do aprovador) |

## 4. DR (Disaster Recovery)

| Métrica | Alvo | Justificativa |
|---|---|---|
| **RPO** (Recovery Point Objective) | **0** (zero data loss) | Movimentos controlados são registros de compliance — perda de decisão de aprovação é inaceitável |
| **RTO** (Recovery Time Objective) | **< 15 min** | Módulo crítico para operações — interrupção bloqueia criações, vendas, integrações |

**Estratégia:**

- Backup PostgreSQL point-in-time recovery (PITR) padrão da plataforma
- WAL streaming para réplica síncrona (RPO = 0)
- Failover automático para réplica em caso de falha do primário (RTO < 15 min)
- Tabelas append-only (`movement_history`, `movement_override_log`) naturalmente resilientes — sem risco de corrupção por UPDATE

## 5. Limites e Capacidade

| Recurso | Limite | Justificativa |
|---|---|---|
| Níveis na cadeia de aprovação | até **5** | Cadeias longas degradam SLA de aprovação; limite prático de governança |
| Regras de controle ativas por tenant | até **200** | Motor avalia regras sequencialmente por prioridade; > 200 pode degradar P95 |
| Movimentos pendentes simultâneos por tenant | até **10.000** | Limite de capacidade da inbox; acima requer particionamento |
| Payload da operação (`operation_payload`) | até **256 KB** | Limite para evitar bloat em jsonb; payloads maiores devem usar referência externa |
| Justificativa de override | min **20 chars**, max **2000 chars** | Min: compliance; Max: previne abuse |
| Parecer do aprovador | max **2000 chars** | Limite prático |
| Código da regra (`codigo`) | `^[A-Z0-9_]{2,50}$` | Consistência com padrão de códigos do sistema |
| Timeout de aprovação | min **1 hora**, max **720 horas** (30 dias) | Min: evita timeout instantâneo; Max: evita movimentos eternamente pendentes |
| Paginação — todas as listagens | cursor-based, **50 por página** (max: 100) | Padrão DOC-GNP-00 |

## 7. Concorrência e Consistência

| Cenário | Estratégia | Justificativa |
|---|---|---|
| Aprovação simultânea do mesmo nível | Optimistic locking via `updated_at` na `approval_instances` | Dois aprovadores tentando decidir sobre a mesma instância — o segundo recebe 409 Conflict |
| Dupla avaliação do motor | `Idempotency-Key` + verificação de movimento existente para mesmo `correlation_id` + `object_type` + `object_id` | Evita criação de movimentos duplicados |
| Cancelamento + aprovação simultâneos | SELECT FOR UPDATE no `controlled_movements` | Evita race condition entre cancelamento pelo solicitante e aprovação pelo aprovador |
| Override + aprovação simultâneos | SELECT FOR UPDATE no `controlled_movements` | Override tem precedência se adquirir lock primeiro |
| Timeout job + decisão manual | Job verifica `status = 'PENDING'` com advisory lock | Se aprovador decide antes do timeout, instância já não é PENDING |

## 10. Escalabilidade Futura

| Cenário | Estratégia Prevista | Gatilho |
|---|---|---|
| > 200 regras ativas por tenant | Cache em memória de regras ativas com invalidação por evento | P95 motor > 500ms |
| > 10.000 movimentos pendentes | Particionamento de `controlled_movements` por `status` | Inbox query > 300ms P95 |
| Multi-region | Read replicas para GET endpoints (inbox, movements) | Latência P95 > 500ms em regiões remotas |
| Alta frequência de avaliações | Rate limiting adaptativo + circuit breaker no motor | > 100 req/s sustentado por tenant |
| Auditoria histórica longa | Particionamento de `movement_history` por `created_at` | Volume > 10M registros por tenant |
