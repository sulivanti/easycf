# Emenda: FR-000-C02

- **Documento base:** [FR-000](../../requirements/fr/FR-000.md)
- **estado_item:** MERGED
- **Natureza:** C (Correção)
- **Data:** 2026-03-25
- **owner:** usuário
- **Motivação:** O endpoint `GET /api/v1/auth/me` não existe na API em execução (`apps/api/src/index.ts`) — apenas `/auth/login` foi implementado. Além disso, o contrato de resposta (DTO backend `profileResponse`) diverge do tipo frontend (`AuthMeResponse`): o backend retorna `full_name` + `active_tenant_id` (escalar), enquanto o frontend espera `name` + `tenant: { id, name }` (objeto). Isso causa erro 404 ao carregar o perfil, quebrando sidebar, dashboard e header.
- **rastreia_para:** FR-004, UX-SHELL-001, UX-DASH-001

---

## Detalhamento

### Problema 1 — Rota `/auth/me` ausente no entrypoint

O arquivo `apps/api/src/index.ts` (entrypoint real da API) registra apenas:
- `GET /api/v1/info`
- `POST /api/v1/auth/login`

A rota `GET /api/v1/auth/me` definida em `apps/api/src/modules/foundation/presentation/routes/auth.route.ts` **não está registrada** no app Fastify em execução. O frontend chama `/auth/me`, recebe 404, e todos os componentes que dependem do perfil (AppShell, DashboardPage, ProfileWidget) entram em estado de erro.

**Correção:** Adicionar rota `GET /api/v1/auth/me` ao `index.ts` com verificação de JWT e resposta no formato esperado.

### Problema 2 — Divergência de shape no contrato `/auth/me`

| Campo | Frontend (`AuthMeResponse`) | Backend DTO (`profileResponse`) | Backend UseCase (`ProfileOutput`) |
|-------|----------------------------|--------------------------------|----------------------------------|
| nome | `name: string` | `full_name: string` | `fullName: string` |
| tenant | `tenant: { id, name }` (objeto) | `active_tenant_id: string` (escalar) | `activeTenantId: string` (escalar) |
| código | — | `codigo: string` | `codigo: string` |
| status | — | `status: string` | `status: string` |

O FR-004 especifica: "GET /auth/me retorna: nome, e-mail, avatar, **tenant ativo**, filial ativa, array de scopes". O frontend implementou corretamente com `tenant` como objeto (necessário para exibir o nome do tenant no ProfileWidget e WelcomeWidget). O backend DTO e UseCase não resolvem o nome do tenant — retornam apenas o ID escalar.

**Correção no `index.ts`:** A rota deve:
1. Resolver o JWT do cookie `accessToken`
2. Buscar `content_users` para `fullName` e `avatarUrl`
3. Buscar `tenants` para obter o nome do tenant (`tenant: { id, name }`)
4. Buscar `role_permissions` para resolver os scopes
5. Retornar o shape compatível com `AuthMeResponse`:

```typescript
interface AuthMeResponse {
  id: string;
  name: string;          // mapeado de content_users.fullName
  email: string;
  avatar_url: string | null;
  tenant: {
    id: string;
    name: string;        // mapeado de tenants.name
  };
  scopes: string[];      // mapeado de role_permissions
}
```

### Problema 3 — DTO backend (`profileResponse`) desalinhado

O schema Zod `profileResponse` em `auth.dto.ts` deve ser atualizado para refletir o shape correto quando as rotas modulares forem ativadas:
- `full_name` → `name`
- `active_tenant_id` → `tenant: { id, name }`
- Remover `codigo` e `status` (não consumidos pelo frontend)

Ou, alternativamente, manter o DTO atual e adicionar um mapper na rota que converte `ProfileOutput` para `AuthMeResponse`.

```gherkin
Funcionalidade: Correção do endpoint GET /auth/me

Cenário: Rota /auth/me disponível e funcional
  Dado que o usuário está autenticado com cookie accessToken válido
  Quando GET /api/v1/auth/me
  Então status 200
  E o corpo contém "name" (string, nome completo do usuário)
  E o corpo contém "email" (string)
  E o corpo contém "avatar_url" (string | null)
  E o corpo contém "tenant" (objeto com id e name)
  E o corpo contém "scopes" (array de strings)
  E o corpo NÃO contém "passwordHash" ou "mfaSecret"

Cenário: Rota /auth/me sem autenticação
  Dado que o usuário NÃO possui cookie accessToken
  Quando GET /api/v1/auth/me
  Então status 401
```

## Impacto nos Pilares

- **Pilares afetados:** INT (contrato OpenAPI), DATA (resposta do endpoint), UX (telas que consomem o perfil)
- **Acao requerida:**
  - **INT:** Atualizar schema OpenAPI de `GET /auth/me` para refletir o shape com `tenant` como objeto
  - **DATA:** O DTO `profileResponse` (auth.dto.ts) deve ser alinhado com o shape `AuthMeResponse`
  - **Código:** `apps/api/src/index.ts` deve receber a rota `GET /api/v1/auth/me` com resolução completa de tenant + scopes
