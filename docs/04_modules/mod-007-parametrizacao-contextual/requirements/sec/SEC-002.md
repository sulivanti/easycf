## Princípios (MUST)

- **Não use "permissão no evento" como fonte de verdade.**
  - **Emit** é controlado pela permissão do **comando** que gera o evento. O scope do endpoint origina o direito de emissão.
  - **View** é controlado pela permissão de leitura da **entity originária** (ACL) + `tenant_id`. Nunca confie apenas no scope — o tenant_id é filtro obrigatório.
- `sensitivity_level` **não substitui** ACL/RBAC: serve apenas como **guard-rail** para truncamento/mascaramento no momento da persistência.
- **Autorização de Linha (MUST):** toda leitura em `domain_events` e `notifications` MUST filtrar por `tenant_id` — sem exceção (DOC-FND-000 §2).
- **x-permissions no OpenAPI** é **documentação** — o enforcement real é feito pelo `@RequireScope` no preHandler (DOC-FND-000 §3.3).

---

## Glossário

| Termo | Descrição |
|---|---|
| **Emit** | Quem pode disparar o evento — derivado do scope do comando que o gera. Ex: `param:framer:write` para `framer.created`. |
| **View** | Quem pode ler/visualizar eventos em timeline/auditoria — ACL + tenant. Ex: tenant match + `param:framer:read` para ver eventos de enquadradores. |
| **Notify** | Quem recebe notificações (inbox/real-time) — resolvido por regra dinâmica baseada em perfil/scope. |
| **sensitivity_level** | Guard-rail numérico. 1 = operacional com dados de negócio (nomes de enquadradores, rotinas). Não substitui ACL. |
| **maskable_fields** | Campos truncados/mascarados no payload do evento antes de persistir na tabela `domain_events`. |
| **retenção** | Período de armazenamento do evento. "Padrão Foundation" = configurável por tenant no MOD-000. |

## Regras de Filtragem para Leitura de Eventos

| Camada | Filtro | Obrigatório |
|---|---|---|
| **Row-level** | `WHERE tenant_id = :current_tenant` | MUST — sem exceção |
| **ACL** | Scope de leitura da entity (ex: `param:framer:read` para eventos de enquadradores) | MUST |
| **Sensitivity** | Se `sensitivity_level >= 2`: verificar scope `system:audit:sensitive`. Para MOD-007 todos são nível 1 — sem restrição adicional. | SHOULD |

**Notas — Motor:**
- EVT-012 (`routine.applied`): Emitido condicionalmente — somente quando ao menos 1 rotina é aplicada (BR-010). Se nenhuma rotina é encontrada para o contexto, nenhum evento é persistido.

## Regras Globais de Eventos

| Regra | Descrição |
|---|---|
| **Persistência** | Todos os 14 eventos são persistidos na tabela `domain_events` (Foundation DOC-ARC-003 §1) |
| **Correlation** | Todos os eventos incluem `X-Correlation-ID` do request original |
| **Tenant isolation** | Leitura de eventos filtrada obrigatoriamente por `tenant_id` |
| **Outbox** | Nenhum evento deste módulo usa outbox pattern (todos são intra-sistema) |
| **Retenção** | Padrão Foundation (configurável por tenant) |
| **Emissão condicional** | `routine.applied` só emitido quando `applied_routines > 0` (BR-010) |
| **Supressão dry_run** | Quando `dry_run: true` no request de avaliação, o motor executa passos 1-5 normalmente mas **NÃO** persiste `routine.applied` (EVT-012). Sem side-effects auditáveis (FR-009, PEN-007/PEN-006) |
| **Mascaramento** | `change_reason` truncado a 200 chars em `routine.forked`; `value` jsonb NÃO incluído em `routine.item_added` |
| **Sensitivity** | Todos os 14 eventos têm `sensitivity_level: 1` (operacional com dados de negócio, sem PII) |
