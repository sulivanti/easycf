## BR-002 — Enquadrador com Vigência Obrigatória

- **Regra:** Todo enquadrador DEVE ter `valid_from` preenchido. `valid_until` é nullable (null = sem expiração). Enquadradores com `valid_until` passado são expirados automaticamente por background job (`framer-expiration.job.ts`).
- **Exemplo:** Enquadrador com valid_until=ontem e status=ACTIVE. Background job executa e muda status para INACTIVE. Domain event `param.framer_expired` é emitido.
- **Exceções:** Nenhuma.
- **Impacto:** STATE, COMPLIANCE

```gherkin
Funcionalidade: Vigência Obrigatória de Enquadrador

Cenário: Rejeitar enquadrador sem valid_from
  Dado que o usuário tem scope param:framer:write
  Quando POST /admin/framers com { nome: "Serviço", valid_from: null }
  Então 422 com invalid_fields contendo "valid_from"

Cenário: Expiração automática por background job
  Dado que enquadrador "SERV-ENG" tem valid_until=ontem e status=ACTIVE
  Quando background job framer-expiration executa
  Então enquadrador muda para status=INACTIVE
  E domain event param.framer_expired é emitido

Cenário: Enquadrador sem data de expiração permanece ativo
  Dado que enquadrador "PROD-IMP" tem valid_until=null e status=ACTIVE
  Quando background job framer-expiration executa
  Então enquadrador permanece com status=ACTIVE
```

- **Exemplo:** Já existe regra SERV-ENG + PEDIDO_VENDA. POST com mesma combinação retorna 422.
- **Exceções:** Nenhuma.

## BR-004 — Safety Net Runtime: Regra Mais Restritiva Vence (GAP 2 — Camada 2)

- **Regra:** Se por exceção (dados legados, race condition) dois contextos conflitantes coexistirem, a regra mais RESTRITIVA vence: HIDE > SHOW, SET_REQUIRED > SET_OPTIONAL, domínio menor prevalece. Campos não conflitantes são mesclados (union).
- **Exemplo:** Rotina-A define projeto_wbs como SET_REQUIRED. Rotina-B define como SET_OPTIONAL. Motor retorna SET_REQUIRED (mais restritivo). Rotina-A define deposito_id como HIDE. Rotina-B não menciona deposito_id. Motor retorna deposito_id como HIDE (sem conflito, aplica diretamente).
- **Exceções:** Nenhuma.
- **Impacto:** CONSISTENCY, SAFETY

```gherkin
Funcionalidade: Safety Net Runtime — Restritividade

Cenário: HIDE vence sobre SHOW em conflito de visibilidade
  Dado que Rotina-A define campo "projeto_wbs" como SHOW
  E Rotina-B define campo "projeto_wbs" como HIDE
  Quando motor avalia ambas em runtime
  Então campo "projeto_wbs" retorna como HIDE

Cenário: SET_REQUIRED vence sobre SET_OPTIONAL em conflito de obrigatoriedade
  Dado que Rotina-A define campo "projeto_wbs" como SET_REQUIRED
  E Rotina-B define campo "projeto_wbs" como SET_OPTIONAL
  Quando motor avalia ambas em runtime
  Então campo "projeto_wbs" retorna como SET_REQUIRED

Cenário: Domínio menor prevalece em conflito de restrição
  Dado que Rotina-A define domínio do campo "tipo_op" como ["A","B","C"]
  E Rotina-B define domínio do campo "tipo_op" como ["A","B"]
  Quando motor avalia ambas em runtime
  Então domínio resultante é ["A","B"] (interseção/menor)

Cenário: Campos não conflitantes são mesclados
  Dado que Rotina-A define campo "projeto_wbs" como SET_REQUIRED
  E Rotina-B define campo "deposito_id" como HIDE
  Quando motor avalia ambas em runtime
  Então "projeto_wbs" retorna SET_REQUIRED
  E "deposito_id" retorna HIDE
```

- **Exemplo:** POST /admin/routines/:id/items com rotina PUBLISHED retorna 422: "Rotinas publicadas são imutáveis. Use o fork para criar nova versão."
- **Exceções:** Nenhuma.
- **Impacto:** GOVERNANCE, IMMUTABILITY

```gherkin
Funcionalidade: Imutabilidade de Rotina Publicada

Cenário: Rejeitar adição de item em rotina PUBLISHED
  Dado que rotina tem status=PUBLISHED
```

- **Exemplo:** Fork de rotina com 4 itens e 2 links retorna novo DRAFT com 4 itens (novos UUIDs) + 2 links copiados + routine_version_history registrado com change_reason.
- **Exceções:** Nenhuma.
- **Impacto:** VERSIONING

```gherkin
Funcionalidade: Fork de Rotina com Cópia Completa

Cenário: Fork copia itens com novos IDs
  Dado que rotina PUBLISHED tem 4 itens
  Quando POST /admin/routines/:id/fork com { change_reason: "Ajuste campo WBS" }
  Então nova rotina criada em DRAFT com version=original+1
  E 4 itens copiados com UUIDs novos
  E routine_version_history registrado com change_reason="Ajuste campo WBS"

Cenário: Fork copia links de incidência
```

## BR-012 — Deprecação de Rotina Bloqueia Novos Vínculos

- **Regra:** Rotinas com status=DEPRECATED não podem receber novos vínculos de incidência. Vínculos existentes são mantidos para historicidade, mas nenhum novo `routine_incidence_link` pode ser criado apontando para rotina DEPRECATED. O motor de avaliação ignora rotinas DEPRECATED durante a resolução.
- **Exemplo:** Rotina "ROT-SERV-01" com status=DEPRECATED. POST para vincular a nova incidence_rule retorna 422: "Rotinas depreciadas não aceitam novos vínculos."
- **Exceções:** Nenhuma.
- **Impacto:** GOVERNANCE, STATE

```gherkin
Funcionalidade: Deprecação Bloqueia Novos Vínculos

Cenário: Rejeitar novo vínculo para rotina DEPRECATED
  Dado que rotina tem status=DEPRECATED

  Quando POST /routine-engine/evaluate com context que inclui essa regra
  Então rotina DEPRECATED é ignorada na resolução
  E resultado não contém itens da rotina depreciada
```

## BR-013 — Isolamento por tenant_id Obrigatório em Todas as Queries

- **Regra:** Todas as queries de leitura e escrita do MOD-007 DEVEM filtrar por `tenant_id` do usuário autenticado. Nenhum endpoint retorna dados de tenants diferentes do ativo na sessão. Isso inclui enquadradores, regras de incidência, rotinas, itens e avaliações do motor. O `tenant_id` é injetado automaticamente pelo middleware Foundation (MOD-000) e NUNCA aceito via body/query param.
- **Exemplo:** Usuário do tenant "ABC" lista enquadradores. GET /admin/framers retorna somente enquadradores do tenant "ABC". Enquadradores do tenant "XYZ" nunca aparecem.
- **Exceções:** Nenhuma.
- **Impacto:** SECURITY, COMPLIANCE, DATA_ISOLATION

```gherkin
Funcionalidade: Isolamento por tenant_id

Cenario: Listar enquadradores retorna apenas do tenant ativo
  Dado que usuario autenticado pertence ao tenant "ABC"
  E existem enquadradores nos tenants "ABC" e "XYZ"
  Quando GET /admin/framers
  Entao retorna somente enquadradores do tenant "ABC"

Cenario: Motor de avaliacao respeita tenant_id
  Dado que usuario autenticado pertence ao tenant "ABC"
  E existem regras de incidencia nos tenants "ABC" e "XYZ"
  Quando POST /routine-engine/evaluate
  Entao motor considera apenas regras e rotinas do tenant "ABC"

Cenario: Rejeitar tenant_id no body do request
  Quando POST /admin/framers com { tenant_id: "XYZ", codigo: "SERV", ... }
  Entao campo "tenant_id" no body e ignorado
  E enquadrador e criado com tenant_id da sessao autenticada
```
