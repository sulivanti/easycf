## FR-001 — CRUD de Regras de Controle de Gravação

- **Descrição:** API para criar, listar e atualizar regras de controle de gravação (movement_control_rules). Define quais operações em quais objetos são controladas e por quais origens.
- **Prioridade:** Must
- **Perfis impactados:** Admin (regras)
- **Done funcional:** Admin configura regras de controle com object_type, operation_type, origin_types, prioridade e vigência. Campo `codigo` imutável após criação (BR-009).
- **Efeito colateral?** [ ] Sim [x] Não
- **Dependências (IDs):** BR-009 (BR-004, BR-007, BR-009), DATA-009, SEC-009
- **Idempotency:** n/a (CRUD padrão)
- **Timeline/Notifications:** yes — emite `approval.control_rule_created`, `approval.control_rule_updated` (DATA-003)
- **Endpoints:**
  - `GET /api/v1/admin/control-rules` — `admin_control_rules_list` — scope: `approval:rule:read`
  - `POST /api/v1/admin/control-rules` — `admin_control_rules_create` — scope: `approval:rule:write`
  - `PATCH /api/v1/admin/control-rules/:id` — `admin_control_rules_update` — scope: `approval:rule:write`

```gherkin
Funcionalidade: CRUD de Regras de Controle

Cenário: Criar regra de controle com sucesso
  Dado que admin possui scope approval:rule:write
  Quando POST /admin/control-rules com { codigo: "RCG-PEDIDO-VALOR", object_type: "pedido_venda", operation_type: "CREATE", origin_types: ["API","MCP"], priority: 10, status: "ACTIVE" }
  Então 201 com regra criada e id gerado
  E domain_event approval.control_rule_created é emitido

Cenário: Listar regras de controle com filtro
  Dado que admin possui scope approval:rule:read
  E existem regras ACTIVE e INACTIVE
  Quando GET /admin/control-rules?status=ACTIVE
  Então retorna apenas regras ACTIVE com paginação cursor-based

Cenário: Atualizar regra — campo codigo é imutável
  Dado que regra existe com codigo=RCG-PEDIDO-VALOR
  Quando PATCH /admin/control-rules/:id com { codigo: "RCG-NOVO" }
  Então 422: "O campo codigo é imutável após criação."

Cenário: Atualizar prioridade da regra
  Dado que regra existe com priority=10
  Quando PATCH /admin/control-rules/:id com { priority: 5 }
  Então 200 com priority atualizada para 5
```

## FR-003 — Motor de Controle (Avaliação de Operações)

- **Descrição:** Endpoint síncrono que avalia tentativas de operação crítica contra regras de controle ativas. Decide imediatamente se a operação é livre ou controlada. Se controlada, cria `controlled_movements` e `approval_instances` nível 1. Se auto-aprovação por scope (BR-003), cria e executa imediatamente.
- **Prioridade:** Must
- **Perfis impactados:** Sistema/Módulo chamador
- **Done funcional:** Motor avalia operação, retorna `{ controlled: false }` ou `{ controlled: true, movement_id, status: "PENDING_APPROVAL" }`. HTTP 202 quando controlado. HTTP 200 com `{ controlled: true, status: "AUTO_APPROVED" }` quando auto-aprovado por scope.
- **Efeito colateral?** [x] Sim [ ] Não *(Cria registros em controlled_movements + approval_instances + movement_history)*
- **Dependências (IDs):** BR-009 (BR-001, BR-003, BR-004, BR-007), DATA-009, FR-001, FR-002
- **Idempotency:** required — chamadas duplicadas com mesmo idempotency_key retornam movimento existente sem criar duplicata
- **Timeline/Notifications:** yes — emite `movement.created`, `movement.auto_approved` (DATA-003)
- **Endpoints:**
  - `POST /api/v1/movement-engine/evaluate` — `movement_engine_evaluate` — scope: `approval:engine:evaluate`

```gherkin
Funcionalidade: Motor de Controle

Cenário: Operação controlada — criação de movimento
  Dado que existe regra ACTIVE para object_type=pedido_venda com origin_types=["API"]
  E chamador possui scope approval:engine:evaluate
  Quando POST /movement-engine/evaluate com { object_type: "pedido_venda", operation_type: "CREATE", origin_type: "API", operation_value: 75000 }
  Então 202 com { controlled: true, movement_id: "uuid", status: "PENDING_APPROVAL" }
  E controlled_movement é criado com status=PENDING_APPROVAL
  E approval_instance nível 1 é criada
  E movement_history registra event_type=CREATED
  E domain_event movement.created é emitido

Cenário: Operação livre — nenhuma regra incide
  Dado que não existe regra aplicável
  Quando POST /movement-engine/evaluate com { object_type: "relatorio", operation_type: "READ" }
  Então 200 com { controlled: false }

Cenário: Auto-aprovação por scope
  Dado que existe regra ACTIVE e alçada com allow_self_approve=true e required_scope=approval:decide
  E solicitante possui scope approval:decide
  Quando POST /movement-engine/evaluate
  Então 200 com { controlled: true, status: "AUTO_APPROVED" }
  E movement_history registra event_type=AUTO_APPROVED_BY_SCOPE
  E domain_event movement.auto_approved é emitido

Cenário: Idempotency — chamada duplicada retorna movimento existente
  Dado que chamada anterior com idempotency_key=IK-001 criou movimento MOV-001
  Quando POST /movement-engine/evaluate com mesma idempotency_key=IK-001
  Então 200 com movement_id=MOV-001 sem criar novo movimento
```

- **data_ultima_revisao:** 2026-03-23

## FR-005 — Aprovação e Reprovação de Movimentos

- **Descrição:** API para aprovar ou reprovar movimentos pendentes. Aprovação do nível N avança para nível N+1 (ou executa se último nível). Reprovação encerra a cadeia. Parecer obrigatório. Segregação: solicitante ≠ aprovador (exceto auto-aprovação por scope §3.1).
- **Prioridade:** Must
- **Perfis impactados:** Aprovador
- **Done funcional:** Aprovação/reprovação com parecer, cadeia multinível, segregação validada, domain events emitidos.
- **Efeito colateral?** [x] Sim [ ] Não *(Atualiza approval_instances, controlled_movements, movement_history; pode executar operação)*
- **Dependências (IDs):** BR-009 (BR-002, BR-003, BR-006), DATA-009, SEC-009, FR-003, FR-004
- **Idempotency:** required — aprovação/reprovação duplicada em instância já decidida retorna 422 sem side-effect
- **Timeline/Notifications:** yes — emite `movement.approved`, `movement.rejected`, `movement.level_approved`, `movement.executed` (DATA-003)
- **Endpoints:**
  - `POST /api/v1/my/approvals/:approvalId/approve` — `my_approvals_approve` — scope: `approval:decide`
  - `POST /api/v1/my/approvals/:approvalId/reject` — `my_approvals_reject` — scope: `approval:decide`

```gherkin
Funcionalidade: Aprovação e Reprovação de Movimentos

Cenário: Aprovação nível 1 em cadeia multinível
  Dado que movimento MOV-001 tem cadeia com 2 níveis
  E Maria é aprovadora nível 1 com scope approval:decide
  Quando POST /my/approvals/:approvalId/approve com { parecer: "De acordo com a política comercial" }
  Então 200 com approval_instance nível 1 decision=APPROVED
  E approval_instance nível 2 é criada com status=PENDING
  E movement_history registra LEVEL_APPROVED
  E domain_event movement.level_approved é emitido

Cenário: Aprovação do último nível executa operação
  Dado que movimento MOV-001 tem cadeia com 1 nível
  E Maria é aprovadora com scope approval:decide
  Quando POST /my/approvals/:approvalId/approve com { parecer: "Aprovado" }
  Então 200 com controlled_movement status=APPROVED
  E operação original é executada
  E status transiciona para EXECUTED
  E domain_event movement.approved e movement.executed são emitidos

Cenário: Reprovação encerra cadeia
  Dado que movimento MOV-001 está PENDING_APPROVAL
  E Maria é aprovadora nível 1
  Quando POST /my/approvals/:approvalId/reject com { parecer: "Valor acima do orçamento aprovado" }
  Então 200 com controlled_movement status=REJECTED
  E cadeia é encerrada
  E domain_event movement.rejected é emitido

Cenário: Segregação impede solicitante de aprovar
  Dado que João solicitou movimento MOV-001
  Quando POST /my/approvals/:approvalId/approve é chamado por João
  Então 422: "O solicitante não pode aprovar o próprio movimento (segregação de funções)."

Cenário: Parecer obrigatório ausente
  Dado que Maria é aprovadora
  Quando POST /my/approvals/:approvalId/approve sem campo parecer
  Então 422: "Parecer é obrigatório."
```

## FR-007 — Override com Justificativa

- **Descrição:** API para override de movimento pendente. Requer scope `approval:override` e justificativa obrigatória (min 20 chars). Registra em `movement_override_log` (imutável) e executa a operação imediatamente.
- **Prioridade:** Must
- **Perfis impactados:** Gestor (override)
- **Done funcional:** Override auditado com justificativa, operação executada, domain events emitidos. Override log é imutável.
- **Efeito colateral?** [x] Sim [ ] Não *(Registra em movement_override_log, atualiza controlled_movements, registra movement_history, executa operação)*
- **Dependências (IDs):** BR-009 (BR-005), DATA-009, SEC-009, FR-003
- **Idempotency:** safe — override duplicado em movimento já OVERRIDDEN retorna 422 sem side-effect
- **Timeline/Notifications:** yes — emite `movement.overridden`, `movement.executed` (DATA-003)
- **Endpoints:**
  - `POST /api/v1/movements/:id/override` — `movements_override` — scope: `approval:override`

```gherkin
Funcionalidade: Override com Justificativa

Cenário: Override com justificativa válida executa operação
  Dado que movimento MOV-001 tem status=PENDING_APPROVAL
  E gestor possui scope approval:override
  Quando POST /movements/:id/override com { justificativa: "Urgência operacional aprovada pelo diretor financeiro em reunião extraordinária" }
  Então 200 com status=OVERRIDDEN
  E movement_override_log registra override com justificativa, user_id, timestamp
  E movement_history registra event_type=OVERRIDDEN

Cenário: Override sem scope suficiente
  Dado que usuário NÃO possui scope approval:override
  Quando POST /movements/:id/override
  Então 403: "Escopo insuficiente."

Cenário: Override em movimento não-pendente
  Dado que movimento MOV-001 tem status=EXECUTED
  Quando POST /movements/:id/override com justificativa válida
  Então 422: "Override é permitido apenas em movimentos com status PENDING_APPROVAL."
```
