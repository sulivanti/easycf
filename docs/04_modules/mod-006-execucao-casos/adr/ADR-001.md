# ADR-001 — Motor de Transição Atômico em Transação Única

- **status:** proposed
- **timebox_days:** 90

---

## Contexto

O motor de transição de estágios (FR-002) é a operação mais complexa do MOD-006. Executa 5 validações sequenciais (caso OPEN, transição válida, papel autorizado, gates resolvidos, evidência fornecida) seguidas de 4 escritas (stage_history, case_instances, gate_instances do novo estágio, case_events) + emissão de domain event. Existe o risco de estado parcial se qualquer escrita falhar após as validações passarem — por exemplo, stage_history inserido mas case_instances não atualizado, resultando em inconsistência entre o histórico e o estágio atual do caso.

A decisão impacta diretamente os SLOs (NFR-006 §1.1: P95 < 1000ms) e a confiabilidade do aggregate root CaseInstance.

## Decisão

**Executar todas as validações e escritas em uma única transação PostgreSQL.** As 5 validações são queries indexadas dentro da mesma transação. As 4 escritas (INSERT stage_history, UPDATE case_instances, INSERT gate_instances PENDING, INSERT case_events) + INSERT domain_events ocorrem atomicamente. Se qualquer passo falhar, rollback completo — nenhum estado parcial.

## Alternativas Consideradas

| Alternativa | Prós | Contras |
|---|---|---|
| **A) Transação única (escolhida)** | Atomicidade total; sem estado parcial; rollback automático; modelo mental simples | Lock no aggregate durante a transação; P95 pode ser afetado se transação demorar |
| **B) Saga com compensação** | Permite escritas distribuídas; desacoplamento | Complexidade excessiva para operação intra-schema; compensação de stage_history é problemática (append-only); latência maior |
| **C) Validações fora da transação, escritas dentro** | Reduz tempo de lock | Janela de race condition entre validação e escrita (gate pode ser resolvido/waivado entre check e commit); viola invariantes do aggregate |
| **D) Event sourcing com projeção** | Desacoplamento total; replay possível | Complexidade desproporcional para o domínio atual; curva de aprendizado; projeções adicionam latência |

## Consequências

- Todas as escritas do motor compartilham a mesma transação DB — lock no registro de case_instances durante a transição
- Optimistic locking via `updated_at` (NFR-006 §7) resolve concorrência — segundo request recebe 409 Conflict
- Queries de validação usam indexes otimizados: `idx_gate_instances_case_status`, `idx_case_assignments_case_active` (DATA-006)
- SLO < 1000ms monitorado via OpenTelemetry spans no motor (NFR-006 §6)
- Se no futuro MOD-008 precisar reagir sincronamente a transições, reavaliar para Saga ou Outbox+handler assíncrono
- Domain events emitidos sincronamente na mesma transação — sem outbox por ora (DATA-003)
