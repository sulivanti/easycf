# ADR-002 — PII-Safe UI Pattern: Proteção LGPD na Camada de Apresentação

## Contexto

O MOD-002 exibe dados de usuários (e-mail, nome completo) em 3 telas de backoffice. A LGPD exige que PII não seja exposta inadvertidamente em toasts, modais de confirmação, mensagens de erro ou payloads de domain events. O risco é real: frameworks de UI frequentemente interpolam dados do objeto na mensagem de feedback (ex: `"Usuário ${email} criado com sucesso"`), o que vazaria PII.

A questão é: **como garantir que PII nunca apareça em mensagens de feedback sem depender apenas de disciplina do desenvolvedor?**

## Decisão

**Adotar o padrão PII-Safe Copy: todas as mensagens de feedback usam texto estático (copy fixa), sem interpolação de dados de PII.** O único identificador pessoal permitido em mensagens visuais é o `fullName` (classificação Confidencial, `sensitivity_level=1`), e apenas em contextos onde a identificação é necessária (ex: modal de desativação).

### Regras do pattern

1. **Toasts de sucesso:** Texto genérico fixo — "Usuário criado com sucesso.", "Convite reenviado com sucesso."
2. **Toasts de erro:** Mensagem genérica + `correlationId` copiável. Nunca `detail` cru do RFC 9457.
3. **Modais de confirmação:** Permitido `fullName` para identificação. Nunca e-mail.
4. **Erros inline (422/409):** Mensagem fixa por campo. Erro 409 (e-mail duplicado) usa copy fixa sem revelar o e-mail existente.
5. **Domain events payload:** Sem e-mail, sem PII desnecessária (DATA-003 payload policy).
6. **Console/DevTools:** Logs de debug NÃO incluem e-mail ou dados sensíveis.

### Enforcement

- **Code review:** Checklist PR inclui verificação de ausência de PII em strings de feedback.
- **Testes automatizados:** Gherkin scenarios em BR-002 validam ausência de PII em toasts/modais.
- **Copy centralizada:** Mensagens de feedback definidas em constantes (não inline), facilitando auditoria.

## Alternativas Consideradas

### Alternativa A: Mascaramento automático via utility function

- Criar `maskEmail(email)` que converte `carlos@empresa.com` em `c***@empresa.com`.
- **Rejeitada:** Mascaramento parcial ainda expõe fragmentos de PII. Não atende LGPD estrito. Adiciona complexidade sem resolver o problema raiz — a mensagem não precisa do e-mail.

### Alternativa B: PII-Safe Copy (escolhida)

- Mensagens usam texto fixo, sem dados dinâmicos de PII.
- **Vantagens:** Zero risco de vazamento. Simplicidade. Testável via Gherkin. Auditável.
- **Tradeoffs:** Mensagens menos contextuais (ex: "Usuário criado" vs "Usuário <carlos@empresa.com> criado"). Aceitável para backoffice administrativo.

### Alternativa C: Sanitização no interceptor HTTP

- Interceptor global filtra PII antes de exibir `detail` do erro.
- **Rejeitada:** Frágil — depende de regex/padrões para detectar PII. Falsos negativos são prováveis. Não resolve modais e toasts de sucesso.

### Estrutura de `domain/copy.ts` (PENDENTE-003)

A copy centralizada segue o padrão **object map com plain strings** — sem dependência de lib de i18n (backoffice pt-BR only).

```typescript
// domain/copy.ts — PII-Safe Copy Constants
// Todas as mensagens de feedback (toasts, modais, erros inline) devem ser
// definidas aqui. NUNCA use strings inline em componentes.
// Auditoria LGPD: grep por 'COPY.' para listar todas as mensagens.

export const COPY = {
  // === Toasts de sucesso (texto fixo, sem PII) ===
  toast: {
    userCreated: 'Usuário criado com sucesso.',
    userUpdated: 'Dados do usuário atualizados.',
    userDeactivated: 'Usuário desativado.',
    userActivated: 'Usuário reativado com sucesso.',
    inviteSent: 'Convite enviado com sucesso.',
    inviteResent: 'Convite reenviado com sucesso.',
    inviteCancelled: 'Convite cancelado.',
    copiedToClipboard: 'Copiado para a área de transferência.',
  },

  // === Toasts de erro (genérico + correlationId) ===
  error: {
    generic: 'Ocorreu um erro. Tente novamente.',
    genericWithCorrelation: (correlationId: string) =>
      `Ocorreu um erro. Código de referência: ${correlationId}`,
    networkError: 'Erro de conexão. Verifique sua internet.',
    sessionExpired: 'Sua sessão expirou. Faça login novamente.',
  },

  // === Modais de confirmação (fullName permitido, nunca e-mail) ===
  modal: {
    deactivateTitle: 'Desativar usuário',
    deactivateBody: (fullName: string) =>
      `O usuário ${fullName} perderá acesso imediatamente. Deseja continuar?`,
    deactivateConfirm: 'Sim, desativar',
    deactivateCancel: 'Cancelar',
    cancelInviteTitle: 'Cancelar convite',
    cancelInviteBody: (fullName: string) =>
      `O convite de ${fullName} será cancelado. Deseja continuar?`,
    cancelInviteConfirm: 'Sim, cancelar convite',
    cancelInviteCancel: 'Voltar',
  },

  // === Erros inline (422/409 — copy fixa por campo) ===
  validation: {
    emailRequired: 'E-mail é obrigatório.',
    emailInvalid: 'Formato de e-mail inválido.',
    emailDuplicate: 'Este e-mail já está cadastrado.',
    nameRequired: 'Nome completo é obrigatório.',
    nameMinLength: 'Nome deve ter pelo menos 2 caracteres.',
    profileRequired: 'Selecione um perfil de acesso.',
  },

  // === Labels e textos de UI ===
  label: {
    cooldownResend: 'Aguarde antes de reenviar',
    cooldownTimer: (seconds: number) => `Reenviar em ${seconds}s`,
    noResults: 'Nenhum usuário encontrado.',
    loading: 'Carregando...',
    emptyState: 'Nenhum usuário cadastrado ainda.',
  },
} as const satisfies Record<string, Record<string, string | ((...args: any[]) => string)>>
```

**Regras de uso:**

| Regra | Descrição |
|---|---|
| Import obrigatório | `import { COPY } from '@/domain/copy'` — nunca string literal inline |
| PII permitida | Apenas `fullName` em modais de confirmação (parâmetro explícito) |
| PII proibida | E-mail, CPF, telefone — nunca em nenhum contexto de copy |
| correlationId | Permitido em toasts de erro (não é PII) |
| Extensão | Novos módulos adicionam seções ao mesmo padrão (ou criam `copy.ts` próprio) |
| Auditoria | `grep -r 'COPY\.' src/` lista todas as mensagens — 100% rastreável |
| Migração i18n | Se necessário: extrair values para JSON de locale, manter keys como IDs |

## Consequências

1. **Copy centralizada:** Todas as mensagens de feedback devem ser definidas como constantes em `domain/copy.ts` (estrutura definida em §Estrutura de `domain/copy.ts`), não inline em componentes. Usar `import { COPY } from '@/domain/copy'`.
2. **Code review:** PRs que adicionam toasts ou modais devem ser verificados contra BR-002.
3. **Testes:** Cenários Gherkin de BR-002 são testes de regressão obrigatórios.
4. **Evolução futura:** Se uma feature futura precisar exibir e-mail em mensagem (ex: confirmação de envio de e-mail com preview), será necessário novo ADR para definir exceção controlada.
5. **Impacto em outros módulos:** Este pattern pode ser promovido a padrão global do projeto se validado no MOD-002.
