# ADR-003 — Herança de behavior_routines do MOD-007 via Extensão 1:1

## Contexto

O MOD-008 precisa de rotinas de integração com versionamento (DRAFT/PUBLISHED/DEPRECATED), fork e imutabilidade de versões publicadas. O MOD-007 já implementa esse ciclo de vida na tabela `behavior_routines` com `routine_type` discriminador. A decisão é se o MOD-008 deve herdar essa estrutura via extensão 1:1 (FK `integration_routines.routine_id` -> `behavior_routines.id`) ou criar uma tabela independente com ciclo de vida próprio.

## Decisão

**Herdar `behavior_routines` do MOD-007 via extensão 1:1.** A tabela `integration_routines` é uma extensão de `behavior_routines` com `routine_type='INTEGRATION'`, adicionando apenas campos HTTP (service_id, http_method, endpoint_tpl, retry config, trigger_events). O versionamento, fork e imutabilidade são integralmente controlados pelo MOD-007.

## Alternativas Consideradas

1. **Tabela independente com ciclo de vida próprio:**
   - Prós: Autonomia total do MOD-008; sem dependência do MOD-007; liberdade para evoluir o ciclo de vida independentemente.
   - Contras: Duplicação da lógica de versionamento (DRAFT/PUBLISHED/DEPRECATED), fork atômico e imutabilidade; violação do DRY; divergência potencial entre ciclos de vida MOD-007 e MOD-008; mais código para manter.

2. **Herança via extensão 1:1 (escolhido):**
   - Prós: Reutiliza versionamento, fork e imutabilidade já testados no MOD-007; constraint UNIQUE(routine_id) garante integridade 1:1; filtragem via `routine_type='INTEGRATION'` permite coexistência; fork no MOD-007 copia atomicamente behavior_routines + integration_routines + field_mappings + params (BR-010).
   - Contras: Dependência forte do MOD-007 (FK cross-module); se MOD-007 mudar o schema de behavior_routines, MOD-008 é impactado; queries exigem JOIN entre behavior_routines e integration_routines.

3. **Herança via tabela polimórfica (STI):**
   - Prós: Tabela única com campos discriminados por tipo.
   - Contras: Colunas HTTP nullable para rotinas não-INTEGRATION; tabela behavior_routines ficaria inchada; violaria single responsibility; dificulta indexação específica.

## Consequências

- **Positivas:** Reuso de versionamento já validado; consistência entre rotinas de parametrização (MOD-007) e integração (MOD-008); fork atômico copia toda a estrutura (BR-010); endpoints herdados (GET/POST /admin/routines) funcionam para ambos os tipos.
- **Negativas:** Dependência forte do MOD-007 — breaking changes em behavior_routines impactam MOD-008; queries de listagem exigem JOIN; fork precisa de lógica estendida para copiar integration_routines, field_mappings e params.
- **Riscos:** Se MOD-007 alterar a API de fork, o MOD-008 precisa adaptar. Mitigação: contrato de integração INT-001 documentado; testes de integração cross-module.
