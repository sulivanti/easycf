# ADR-002 — Retry Gerenciado pelo Outbox, Não pelo BullMQ

## Contexto

O motor de execução do MOD-008 precisa implementar retry com backoff exponencial para chamadas HTTP que falham com erros transitórios (5xx, timeout, erros de rede). Tanto o Outbox (re-enfileiramento via UPDATE do call_log) quanto o BullMQ nativo (built-in retry com `backoff` option) oferecem mecanismos de retry. A decisão impacta a observabilidade, o controle de estado e a governança do reprocessamento.

## Decisão

**O retry é gerenciado pelo Outbox (re-enfileiramento via call_log), não pelo BullMQ nativo.** Quando uma chamada falha e ainda há retries restantes, o worker atualiza o `attempt_number` no call_log, calcula o delay (`retry_backoff_ms * 2^(attempt-1)`) e usa `moveToDelayed` do BullMQ para re-agendar. O controle de estado (attempt count, retry exhaustion, DLQ) fica integralmente no PostgreSQL.

## Alternativas Consideradas

1. **BullMQ native retry (`attempts` + `backoff` options):**
   - Prós: Zero código custom de retry; BullMQ gerencia delay, contagem e DLQ automaticamente; menor complexidade no worker.
   - Contras: Estado do retry fica no Redis (volátil); se Redis perder dados, o histórico de tentativas se perde; dificulta auditoria (não há registro persistente de cada attempt); a governança de DLQ fica no BullMQ, não no banco; reprocessamento manual exige consultar Redis + PostgreSQL.

2. **Outbox-managed retry (escolhido):**
   - Prós: Estado completo e persistente no PostgreSQL (`attempt_number`, timestamps de cada tentativa, status transitions); auditoria integrada; reprocessamento governado (novo call_log com `parent_log_id`); métricas de retry deriváveis do banco; DLQ controlada no banco com domain events.
   - Contras: Código custom para cálculo de delay e re-enfileiramento; worker precisa invocar `moveToDelayed` explicitamente; ligeiramente mais complexo que o retry nativo.

3. **Retry duplo (BullMQ + Outbox):**
   - Prós: Redundância.
   - Contras: Dois mecanismos de retry competindo; risco de contagem duplicada de attempts; complexidade para reconciliar estados entre Redis e PostgreSQL. Violaria o princípio de fonte única de verdade.

## Consequências

- **Positivas:** Fonte única de verdade para estado de retry (PostgreSQL); auditoria completa de cada tentativa; DLQ governada com domain events e notificações; reprocessamento com `parent_log_id` forma chain de rastreabilidade; métricas `integration_retry_total` deriváveis diretamente do banco.
- **Negativas:** Worker precisa de lógica explícita de retry (delay calculation, moveToDelayed, DLQ transition); não aproveita o retry nativo do BullMQ (feature built-in desabilitada).
- **Riscos:** Se o worker tiver bug no cálculo de delay, retries podem não ocorrer. Mitigação: testes unitários para cálculo de backoff exponencial + teste de integração BullMQ para retry path completo (NFR-008 §9).
