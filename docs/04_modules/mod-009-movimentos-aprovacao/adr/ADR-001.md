# ADR-001 — Motor de Controle Síncrono

## Contexto

O MOD-009 precisa interceptar operações críticas no momento em que são solicitadas, avaliando regras de controle e decidindo se a operação é livre ou controlada. A decisão arquitetural central é o modo de avaliação do motor: síncrono (o chamador aguarda a resposta do motor antes de prosseguir) ou assíncrono (a operação é enfileirada e avaliada posteriormente).

O motor avalia regras ACTIVE ordenadas por `priority`, combinando até 4 critérios (VALUE, HIERARCHY, ORIGIN, OBJECT+OPERATION). O resultado determina se a operação prossegue imediatamente (`controlled: false`), se é interceptada (`controlled: true`, HTTP 202), ou se é auto-aprovada por suficiência de escopo (`AUTO_APPROVED`, HTTP 200).

Requisitos de contexto:
- O módulo chamador precisa saber **imediatamente** se a operação foi interceptada para retornar o status correto ao cliente (200 vs 202).
- A avaliação de regras envolve queries simples em tabelas indexadas (`idx_mcr_tenant_eval`), com SLO de P95 < 500ms.
- O princípio "nunca executar sem avaliação" (BR-001) exige que nenhuma operação crítica prossiga sem resposta do motor.

## Decisão

**O motor de controle opera de forma síncrona.** O endpoint `POST /movement-engine/evaluate` retorna a decisão na mesma request (200 ou 202). O módulo chamador aguarda a resposta antes de prosseguir ou interceptar.

Fluxo: (1) Módulo chamador envia request ao motor, (2) motor busca regras ativas, avalia critérios, decide, (3) se controlado, cria `controlled_movements` + `approval_instances` nível 1 + `movement_history` em transação atômica, (4) retorna resposta síncrona.

O SLO de P95 < 500ms (NFR-009 §1.1) garante que o overhead no fluxo do chamador é aceitável. O índice `idx_mcr_tenant_eval` otimiza a hot query do motor.

## Alternativas Consideradas

1. **Fila assíncrona (message queue):** A operação seria enfileirada e o motor consumiria da fila. Problemático porque o chamador não teria resposta imediata — não saberia se deve retornar 200 ou 202 ao cliente. Quebraria o contrato HTTP do middleware/hook pattern (INT-009 §4.1). Introduziria eventual consistency onde não é aceitável (BR-001: "nunca executar sem avaliação").

2. **Avaliação assíncrona com polling:** O chamador enfileira a operação e faz polling para obter o resultado. Aumentaria latência percebida (múltiplas requests), complexidade de implementação, e não resolveria o problema de bloqueio — o chamador ainda precisaria esperar.

3. **Cache local de regras + avaliação no chamador:** Cada módulo chamador avaliaria regras localmente usando cache. Distribuiria a lógica do motor por múltiplos módulos, violando DDD (o domínio de aprovações ficaria fragmentado). Risco de inconsistência entre caches de diferentes módulos. Complexidade de invalidação de cache.

## Consequências

- **Positivas:** Resposta determinística para o chamador; contrato HTTP claro (200/202); transação atômica garante consistência entre movimento e instância de aprovação; simplicidade de integração via hook pattern; sem eventual consistency no fluxo crítico.
- **Negativas:** Módulo chamador tem dependência runtime do motor — se o motor estiver indisponível, retorna 503 (degradação graceful, nunca executa sem avaliação). Latência adicionada ao fluxo do chamador (mitigada pelo SLO < 500ms).
- **Riscos:** Se o volume de regras ativas por tenant crescer além de 200 (NFR-009 §5), a avaliação sequencial pode degradar o P95. Mitigação prevista: cache em memória com invalidação por evento (NFR-009 §10).
