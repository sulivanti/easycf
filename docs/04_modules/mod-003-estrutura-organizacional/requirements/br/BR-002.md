# BR-002 — Regras de Negócio de Departamentos

---

## BR-013 — Unicidade de `codigo` por Tenant

- **Regra:** O campo `codigo` DEVE ser único dentro de cada tenant (UNIQUE(tenant_id, codigo)). Tentativas de inserção com codigo duplicado no mesmo tenant devem retornar HTTP 409 (Conflict) com body RFC 9457. A unicidade é garantida pela constraint do banco (catch PostgreSQL 23505 → 409).
- **Exemplo:** Tenant A tem departamento com codigo="DEPT-DIR". Criar outro departamento com codigo="DEPT-DIR" no Tenant A → 409. Criar com mesmo codigo no Tenant B → permitido.
- **Exceções:** Departamentos soft-deleted (deleted_at IS NOT NULL) são excluídos da constraint de unicidade pelo índice parcial.
- **Impacto:** DATA, STATE
- **Diferença de org_units:** org_units usa unicidade global (BR-008); departamentos usam unicidade por tenant.

### Critérios de Aceitação (Gherkin)

```gherkin
Funcionalidade: Unicidade de codigo por tenant

Cenário: Codigo duplicado no mesmo tenant retorna 409
  Dado que existe um departamento com codigo="DEPT-DIR" no tenant "T1"
  Quando POST /api/v1/departments com { codigo: "DEPT-DIR", nome: "Diretoria 2" } no tenant "T1"
  Então deve retornar 409 com type="/problems/conflict"
  E deve conter detail "Já existe um departamento com o código 'DEPT-DIR' neste tenant."

Cenário: Mesmo codigo em tenants diferentes é permitido
  Dado que existe um departamento com codigo="DEPT-DIR" no tenant "T1"
  Quando POST /api/v1/departments com { codigo: "DEPT-DIR", nome: "Diretoria" } no tenant "T2"
  Então deve retornar 201

Cenário: Codigo de departamento soft-deleted pode ser reutilizado
  Dado que existe um departamento soft-deleted com codigo="DEPT-OLD" no tenant "T1"
  Quando POST /api/v1/departments com { codigo: "DEPT-OLD", nome: "Novo Departamento" } no tenant "T1"
  Então deve retornar 201
```

## BR-014 — `codigo` Imutável Após Criação

- **Regra:** O campo `codigo` é identificador de negócio estável e DEVE ser imutável após a criação. Tentativas de alteração via PATCH devem retornar 422.
- **Exemplo:** PATCH /departments/:id com { codigo: "NOVO-COD" } → 422 "O campo 'codigo' é imutável após criação."
- **Exceções:** Nenhuma. Regra inegociável (consistente com BR-003 de org_units).
- **Impacto:** DATA, COMPLIANCE

### Critérios de Aceitação (Gherkin)

```gherkin
Funcionalidade: Imutabilidade do codigo de departamento

Cenário: Tentativa de alterar codigo retorna 422
  Dado que existe um departamento com codigo="DEPT-DIR"
  Quando PATCH /api/v1/departments/:id com { codigo: "DEPT-NOVO" }
  Então deve retornar 422 com detail "O campo 'codigo' é imutável após criação."

Cenário: Update de outros campos sem codigo é permitido
  Dado que existe um departamento com codigo="DEPT-DIR"
  Quando PATCH /api/v1/departments/:id com { nome: "Novo Nome" }
  Então deve retornar 200
```

## BR-015 — Soft Delete de Departamento

- **Regra:** O DELETE DEVE executar soft delete: setar `status='INACTIVE'` + `deleted_at=now()`. O departamento continua existindo no banco para rastreabilidade. Sem restrição de dependências (entidade flat, sem filhos).
- **Exemplo:** DELETE /departments/:id → status='INACTIVE', deleted_at=now(). O departamento não aparece mais nas listagens padrão.
- **Exceções:** Departamento já inativo (deleted_at IS NOT NULL) → 422 "Departamento já está desativado."
- **Impacto:** DATA, STATE

### Critérios de Aceitação (Gherkin)

```gherkin
Funcionalidade: Soft delete de departamento

Cenário: Soft delete de departamento ativo
  Dado que existe um departamento ativo com id=":id"
  Quando DELETE /api/v1/departments/:id
  Então deve retornar 204
  E o departamento deve ter status="INACTIVE" e deleted_at preenchido

Cenário: Soft delete de departamento já inativo retorna 422
  Dado que existe um departamento já desativado com id=":id"
  Quando DELETE /api/v1/departments/:id
  Então deve retornar 422 com detail "Departamento já está desativado."
```

## BR-016 — Restore de Departamento Sem Restrições

- **Regra:** O restore de departamento (PATCH /departments/:id/restore) DEVE setar `status='ACTIVE'` + `deleted_at=NULL`. Sem restrições adicionais — a entidade é flat, sem dependências hierárquicas (diferente de org_units BR-009 que exige pai ativo).
- **Exemplo:** PATCH /departments/:id/restore → status='ACTIVE', deleted_at=NULL.
- **Exceções:** Departamento já ativo → 422 "Departamento já está ativo."
- **Impacto:** DATA, STATE

### Critérios de Aceitação (Gherkin)

```gherkin
Funcionalidade: Restore de departamento

Cenário: Restore de departamento inativo
  Dado que existe um departamento inativo com id=":id"
  Quando PATCH /api/v1/departments/:id/restore
  Então deve retornar 200
  E o departamento deve ter status="ACTIVE" e deleted_at=NULL

Cenário: Restore de departamento já ativo retorna 422
  Dado que existe um departamento ativo com id=":id"
  Quando PATCH /api/v1/departments/:id/restore
  Então deve retornar 422 com detail "Departamento já está ativo."
```

## BR-017 — Cor Hex Válida ou Null

- **Regra:** O campo `cor` aceita apenas valores no formato `#RRGGBB` (7 caracteres: # + 6 hex digits) ou null. Validação ocorre em duas camadas: Zod no handler (resposta rápida 422) e CHECK constraint no banco (guard-rail).
- **Exemplo:** { cor: "#2E86C1" } → válido. { cor: "#FFF" } → 422. { cor: null } → válido. { cor: "vermelho" } → 422.
- **Exceções:** Nenhuma.
- **Impacto:** DATA, UX

### Critérios de Aceitação (Gherkin)

```gherkin
Funcionalidade: Validação de cor hex

Cenário: Cor hex válida é aceita
  Quando POST /api/v1/departments com { codigo: "D1", nome: "Dept", cor: "#2E86C1" }
  Então deve retornar 201

Cenário: Cor hex inválida retorna 422
  Quando POST /api/v1/departments com { codigo: "D1", nome: "Dept", cor: "#FFF" }
  Então deve retornar 422 com detail contendo "cor"

Cenário: Cor null é aceita
  Quando POST /api/v1/departments com { codigo: "D1", nome: "Dept", cor: null }
  Então deve retornar 201

Cenário: Cor string não-hex retorna 422
  Quando POST /api/v1/departments com { codigo: "D1", nome: "Dept", cor: "vermelho" }
  Então deve retornar 422 com detail contendo "cor"
```

## BR-018 — Soft Limit de 100 Departamentos por Tenant

- **Regra:** O sistema DEVE emitir warning no response header `X-Limit-Warning` ao criar departamento quando `count > 80` (80% do soft limit de 100) para o tenant. O limite NÃO é hard block — criação é sempre permitida. Formato do header: `departments_count=82, soft_limit=100, threshold=80%`.
- **Exemplo:** Tenant com 82 departamentos ativos → POST retorna 201 + header `X-Limit-Warning: departments_count=82, soft_limit=100, threshold=80%`.
- **Exceções:** Nenhuma. Warning informativo, não bloqueia operação (consistente com PENDENTE-004 de org_units).
- **Impacto:** NFR, CAPACITY

### Critérios de Aceitação (Gherkin)

```gherkin
Funcionalidade: Soft limit de departamentos por tenant

Cenário: Criação sem warning abaixo de 80%
  Dado que o tenant "T1" possui 50 departamentos ativos
  Quando POST /api/v1/departments com dados válidos no tenant "T1"
  Então deve retornar 201
  E o response NÃO deve conter header X-Limit-Warning

Cenário: Warning ao ultrapassar 80% do soft limit
  Dado que o tenant "T1" possui 81 departamentos ativos
  Quando POST /api/v1/departments com dados válidos no tenant "T1"
  Então deve retornar 201
  E o response header X-Limit-Warning deve conter "departments_count=82, soft_limit=100, threshold=80%"

Cenário: Criação permitida mesmo acima do soft limit de 100
  Dado que o tenant "T1" possui 100 departamentos ativos
  Quando POST /api/v1/departments com dados válidos no tenant "T1"
  Então deve retornar 201
  E o response header X-Limit-Warning deve conter "exceeded=true"
```
