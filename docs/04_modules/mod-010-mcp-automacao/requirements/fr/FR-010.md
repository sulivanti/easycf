## FR-001 — CRUD de Agentes MCP

- **Goal:** Permitir criar, listar e editar agentes MCP com identidade técnica governada, gerando API key segura na criação.
- **Done funcional:**
  - POST /admin/mcp-agents cria agente com `codigo` (imutável), `nome`, `owner_user_id`, `allowed_scopes` (validados contra blocklist Phase 1 — BR-002)
  - API key 256 bits base64url gerada, retornada UMA VEZ no body da resposta, armazenada como hash bcrypt (rounds >= 12) — BR-004
  - GET /admin/mcp-agents lista com paginação cursor-based, filtros por status, owner_user_id
  - GET /admin/mcp-agents/:id retorna detalhe (campo `api_key` AUSENTE — BR-004)
  - PATCH /admin/mcp-agents/:id edita nome, allowed_scopes (re-validados contra blocklist), status (ACTIVE↔INACTIVE, nunca para REVOKED — BR-006)
  - Campo `codigo` ignorado no PATCH (BR-005)
  - Campo `last_used_at` atualizado automaticamente a cada execução (não editável via API)
- **Dependências:** BR-001, BR-002, BR-004, BR-005, DATA-010 (mcp_agents), SEC-010
- **Idempotency:** POST não idempotente (nova key a cada criação). PATCH idempotente.
- **Timeline/Notifications:** yes — emite `mcp.agent.created`, `mcp.agent.updated`, `mcp.agent.scopes_updated` (DATA-003)

```gherkin

Cenário: Criar agente com sucesso
```

## FR-002 — Revogação de Agentes MCP

- **Goal:** Permitir revogar agentes MCP de forma imediata e irreversível, com registro de motivo para auditoria.
- **Done funcional:**
  - POST /admin/mcp-agents/:id/revoke com `{ reason: "texto" }` (reason obrigatório — BR-015)
  - Status muda para REVOKED, `revoked_at` preenchido com timestamp UTC
  - Tentativa de revogar agente já REVOKED retorna 409
  - Tentativa de reativar agente REVOKED via PATCH retorna 422 (BR-006)
  - Execuções futuras do agente retornam 403
- **Dependências:** BR-006, BR-015, DATA-010 (mcp_agents), SEC-010
- **Idempotency:** n/a (ação irreversível, POST não idempotente — segunda chamada retorna 409)
- **Timeline/Notifications:** yes — emite `mcp.agent.revoked` (DATA-003, sensitivity_level=1)

## FR-007 — Gateway MCP (Execução)

- **Goal:** Prover gateway de entrada para execuções MCP com autenticação por API key, validação de 8 passos sequenciais, e despacho por política.
- **Done funcional:**
  - POST /mcp/execute com header `X-MCP-Agent-Key` e body `{ action_code: "string", payload: {} }`
  - Header `X-Correlation-ID` aceito se presente, gerado se ausente
  - Algoritmo de 8 passos (BR-010):
    1. Autenticar agente via API key + bcrypt compare → 401 se inválido
    2. Verificar status=ACTIVE → 403 se INACTIVE/REVOKED
    3. Buscar ação pelo action_code → 404 se não encontrada
    4. Verificar vínculo agente↔ação + valid_until → 403 se expirado/inexistente
    5. Verificar required_scopes ⊆ allowed_scopes → 403 se insuficiente
    6. Dupla verificação blocklist Phase 1 → 403 + privilege_escalation_attempt se violado (BR-012)
    7. INSERT mcp_executions com status=RECEIVED
    8. Avaliar execution_policy e despachar
  - Despacho DIRECT (lógica detalhada — PEN-010/PENDENTE-003):
    1. `McpDispatcher` recebe ação com `dispatch_type=DIRECT`
    2. Resolve `ActionExecutor` via DI registry por `action_code` (strategy pattern)
    3. Se `linked_routine_id` presente, chama `RoutineEvaluator` port (MOD-007 via INT-002) ANTES da execução — parametriza a ação com resultado da rotina
    4. `ActionExecutor` executa lógica específica da ação e produz `result_payload`
    5. Se `linked_integration_id` presente, enfileira job `IntegrationQueue` port (MOD-008 via INT-003) APÓS execução (post-processing assíncrono)
    6. Atualiza `mcp_executions.status` → `DIRECT_SUCCESS` (se OK) ou `DIRECT_FAILED` (se erro)
    7. Retorna 200 com `result_payload`
  - Despacho CONTROLLED: chama MOD-009 motor, atualiza status→CONTROLLED_PENDING, retorna 202 com movement_id
  - Despacho EVENT_ONLY: emite domain_event, atualiza status→EVENT_EMITTED, retorna 200
  - Campo `last_used_at` do agente atualizado a cada execução
- **Dependências:** BR-001, BR-002, BR-008, BR-009, BR-010, BR-012, DATA-010 (mcp_executions), INT-010 (MOD-009), SEC-010
- **Idempotency:** n/a (cada execução é única, registrada com ID próprio)
- **Timeline/Notifications:** yes — emite `mcp.execution.completed`, `mcp.execution.blocked`, `mcp.privilege_escalation_attempt` (DATA-003)

```gherkin
Cenário: Execução DIRECT com sucesso

  Quando POST /mcp/execute com { action_code: "CONSULTAR-ESTOQUE", payload: {} }
  Então 200 com { status: "DIRECT_SUCCESS", result: {...} }
```

## FR-010 — Habilitação de Phase 2 Create Per-Agent

- **Goal:** Prover endpoint dedicado para habilitar Phase 2 (`phase2_create_enabled`) em um agente MCP individual, com confirmação explícita, motivo obrigatório e auditoria dedicada.
- **Done funcional:**
  - POST /admin/mcp-agents/:id/enable-phase2 com body `{ "reason": "motivo" }` (reason obrigatório, min 10 chars)
  - Scopes requeridos: `mcp:agent:write` + `mcp:agent:phase2-enable` (ambos obrigatórios — dupla autorização)
  - Validação: admin DEVE ser o `owner_user_id` do agente OU ter scope `mcp:agent:phase2-enable`
