## 2. Contratos de Request/Response

### 2.1 POST /api/v1/cases — Abertura de Caso

**Request:**

```json
{
  "cycle_id": "uuid",
  "tenant_id": "uuid",
  "org_unit_id": "uuid | null",
  "object_type": "string | null",
  "object_id": "uuid | null"
}
```

**Response 201:**

```json
{
  "id": "uuid",
  "codigo": "CASO-2026-00042",
  "cycle_id": "uuid",
  "cycle_version_id": "uuid",
  "current_stage_id": "uuid",
  "status": "OPEN",
  "opened_by": "uuid",
  "opened_at": "2026-03-19T10:00:00Z",
  "gate_instances": [
    { "id": "uuid", "gate_id": "uuid", "status": "PENDING", "gate_type": "APPROVAL" }
  ]
}
```

**Erros:**

- 422: `"Apenas ciclos publicados podem ter novos casos abertos."` (BR-009)
- 409: Idempotency-Key duplicado (TTL 60s)

**Headers obrigatórios:**

- Request: `Authorization: Bearer {jwt}`, `X-Correlation-ID: {uuid}`, `Idempotency-Key: {uuid}`
- Response: `X-Correlation-ID: {uuid}`

### 2.2 POST /api/v1/cases/:id/transition — Motor de Transição

**Request:**

```json
{
  "transition_id": "uuid",
  "evidence": { "type": "note|file", "content": "string", "url": "string | null" },
  "motivo": "string | null"
}
```

**Response 200:**

```json
{
  "id": "uuid",
  "codigo": "CASO-2026-00042",
  "current_stage_id": "uuid",
  "status": "OPEN | COMPLETED",
  "stage_history_entry": { "id": "uuid", "from_stage_id": "uuid", "to_stage_id": "uuid" },
  "new_gate_instances": [
    { "id": "uuid", "gate_id": "uuid", "status": "PENDING" }
  ]
}
```

**Erros (em ordem de validação do motor — FR-002):**

- 422: `"Caso não está aberto."` (passo 1)
- 422: `"Transição inválida para o estágio atual."` (passo 2)
- 403: `"Seu papel ({role}) não autoriza esta transição."` (passo 3)
- 422: `"Gate '{nome}' não foi resolvido."` com `extensions.pending_gates[]` (passo 4)
- 422: `"O papel '{nome}' é obrigatório neste estágio e não foi atribuído."` (passo 5)
- 422: `"Esta transição requer evidência (nota ou arquivo)."` (passo 6)

### 2.7 GET /api/v1/cases — Listagem com Filtros

**Query Params:**

- `cycle_id` (uuid, opcional) — filtrar por ciclo
- `status` (string, opcional) — OPEN|COMPLETED|CANCELLED|ON_HOLD
- `stage_id` (uuid, opcional) — filtrar por estágio atual
- `assigned_to_me` (boolean, opcional) — apenas casos onde o usuário tem atribuição ativa
- `opened_after` (ISO datetime, opcional) — date range
- `opened_before` (ISO datetime, opcional) — date range
- `search` (string, opcional) — busca textual por codigo
- `object_id` (uuid, opcional) — exact match em case_instances.object_id via idx_case_instances_object (B-tree parcial). Mutuamente exclusivo com `search` — se ambos enviados, `object_id` prevalece. Consumidores: MOD-007, MOD-008. Ref: PEN-006/PENDENTE-003 Opção A
- `cursor` (string, opcional) — cursor para paginação
- `limit` (integer, opcional, default 20, max 50) — itens por página

**Response 200:**

```json
{
  "items": [
    {
      "id": "uuid",
      "codigo": "CASO-2026-00042",
      "cycle_name": "Comercial v1",
      "current_stage_name": "Em Análise",
      "status": "OPEN",
      "pending_gates_count": 2,
      "my_role": "RESPONSAVEL | null",
      "opened_at": "2026-03-19T10:00:00Z"
    }
```

## 3. Integrações Consumidas

### 3.1 MOD-005 — Blueprint de Processos (Leitura)

| Recurso Consumido | Modo | Finalidade | Failure Behavior |
|---|---|---|---|
| `process_cycles` (status, version) | DB read (mesmo schema) | Validar ciclo PUBLISHED na abertura; freeze cycle_version_id | Falha → 500 (dependência crítica) |
| `process_stages` (is_initial, is_terminal) | DB read | Determinar estágio inicial e terminal | Falha → 500 |
| `process_gates` (gate_type, required) | DB read | Criar gate_instances e validar no motor | Falha → 500 |
| `stage_transitions` (allowed_roles, evidence_required) | DB read | Validar transições no motor | Falha → 500 |
| `process_roles` | DB read | Validar papéis em atribuições | Falha → 500 |
| `stage_role_links` (required) | DB read | Validar papéis obrigatórios no motor | Falha → 500 |

> **Modo de integração:** Leitura direta via DB (mesmo schema PostgreSQL). Não há chamada HTTP entre MOD-006 e MOD-005. A integridade referencial é garantida por FKs.

### 3.2 MOD-004 — Identidade Avançada (Leitura)

| Recurso Consumido | Modo | Finalidade | Failure Behavior |
|---|---|---|---|
| `access_delegations` (status, scopes, expires_at) | DB read | Validar delegação na atribuição; verificar expiração | Falha → 422 "Delegação inválida" |

> **Modo de integração:** Leitura direta via DB. Verificação de expiração via background job (ADR-005) usando `DelegationCheckerPort`.

#### Contrato: DelegationCheckerPort.getExpiredDelegations (PEN-006/PENDENTE-002 Opção B)

| Campo | Valor |
|---|---|
| **Método** | `getExpiredDelegations(): Promise<ExpiredDelegation[]>` |
| **Input** | Nenhum (job escaneia todas as atribuições ativas com delegation_id) |
| **Output** | `Array<{ assignment_id: UUID, delegation_id: UUID, expired_at: ISO timestamp }>` |
| **Timeout** | 3s |
| **Retry** | 1 tentativa |
| **Fallback** | Log error + skip iteração (fail-safe — assignments não desativadas por engano) |
| **Dependência** | MOD-004 `access_delegations` via port (nunca JOIN direto) |
| **Chamador** | Background job cron 5 min (ADR-005) |

## 5. Padrões de Erro (RFC 9457)

Todos os erros DEVEM seguir o formato Problem Details (RFC 9457):

```json
{
  "type": "https://api.example.com/problems/gate-pending",
  "title": "Gate não foi resolvido",
  "status": 422,
  "detail": "Gate 'Aprovação Gerencial' não foi resolvido.",
  "instance": "/api/v1/cases/uuid/transition",
  "extensions": {
    "correlationId": "uuid",
    "pending_gates": [{ "id": "uuid", "nome": "Aprovação Gerencial", "gate_type": "APPROVAL" }]
  }
}
```

### Catálogo de Problem Types

| Problem Type | Status | Cenário |
|---|---|---|
| `gate-pending` | 422 | Motor bloqueado por gate não resolvido |
| `role-not-authorized` | 403 | Papel do usuário não autoriza transição |
| `evidence-required` | 422 | Transição exige evidência não fornecida |
| `role-required-unassigned` | 422 | Papel obrigatório sem atribuição ativa |
| `case-not-open` | 422 | Caso não está com status OPEN |
| `invalid-transition` | 422 | Transição não existe no blueprint para estágio atual |
| `cycle-not-published` | 422 | Ciclo não está PUBLISHED |
| `cancel-requires-reason` | 422 | Cancelamento sem motivo |
| `waive-requires-scope` | 403 | Sem scope para dispensar gate |
| `waive-requires-reason` | 422 | Motivo de dispensa com menos de 20 chars |
| `checklist-incomplete` | 422 | Checklist com itens não marcados |
| `gate-approval-requires-role` | 403 | Papel sem can_approve tentando resolver gate APPROVAL |

## 6. Paginação (DOC-ARC-001)

- **Tipo:** Cursor-based (sem offset) — evita degradação com volume (FR-009)
- **Parâmetros:** `cursor` (opaque string) + `limit` (default 20, max 50)
- **Response:** `nextCursor` (null se última página) + `totalEstimate` (estimativa, não count exato)
- **Ordenação padrão:** `opened_at DESC`

## 7. Email Queue — Convenções BullMQ/Redis (DOC-PADRAO-002 §3.4)

- **Sistema:** MOD-006 Execução de Casos (interno — infraestrutura Redis/BullMQ)
- **Modo:** Fila assíncrona (BullMQ + ioredis)
- **Referência normativa:** DOC-PADRAO-002 §3.4.1–3.4.6

### Requisitos

1. **Conexão BullMQ (§3.4.2):** Email queue MUST reutilizar conexão ioredis singleton (opção `connection`). MUST NOT criar conexão Redis por fila.
2. **Nomenclatura de fila (§3.4.2 + §3.4.3):** Fila MUST ser nomeada `mod-006:email`. MUST usar `:` como separador.
3. **Job cleanup (§3.4.2):** SHOULD configurar `defaultJobOptions.removeOnComplete` para evitar crescimento infinito.
4. **Database (§3.4.5):** Filas BullMQ MUST usar db1 (`noeviction`). Conexão ioredis para BullMQ MUST especificar `db: 1`.
5. **Health check (§3.4.6):** MUST expor Redis health check no endpoint `/health`.
