# Emenda: FR-000-M03

- **Documento base:** [FR-000](../../requirements/fr/FR-000.md)
- **estado_item:** MERGED
- **Natureza:** M (Melhoria)
- **Data:** 2026-03-30
- **owner:** arquitetura
- **Motivação:** O frontend não utiliza o endpoint `/auth/refresh` (FR-003) automaticamente. O access token de 15min expira de forma rígida e o usuário perde o trabalho em formulários longos. Esta melhoria implementa: (1) refresh proativo baseado em atividade do usuário, (2) interceptor 401 com retry automático, (3) idle timeout configurável de 30 minutos.
- **rastreia_para:** spec-smart-session-keepalive, FR-003, FR-001, FR-002, UX-000-C01

## Detalhamento

### Novo requisito: FR-020 — Smart Session Keep-Alive

- **Descrição:** O sistema DEVE renovar automaticamente o access token enquanto o usuário estiver ativamente interagindo com a aplicação. Quando o usuário para de interagir por mais de `VITE_SESSION_IDLE_TIMEOUT_MS` (padrão: 30 minutos), o sistema para de renovar e a sessão expira naturalmente no próximo request.
- **Prioridade:** Must
- **Perfis impactados:** Todos os usuários autenticados
- **Done funcional:**
  - Activity Tracker detecta interações via eventos DOM (click, keydown, scroll, touchstart, mousemove com throttle 30s)
  - Timestamp da última atividade persistido em `sessionStorage` (chave: `ecf_last_activity`) — sobrevive a page reload
  - Timer de check a cada 30s: se `isActive && tokenExpiresIn < 120s` → chama `POST /auth/refresh` proativamente
  - Se inativo por mais de 30min (configurável) → para de renovar proativamente
  - Interceptor 401 em ambos HTTP clients (`http-client.ts` e `api-client.ts`):
    - Ao receber 401 → tenta `POST /auth/refresh` uma vez
    - Se refresh OK → retenta a request original automaticamente
    - Se refresh FAIL → force logout (`/login` + limpeza tokens + queryClient.clear)
    - Mutex/queue: apenas 1 refresh simultâneo; requests concorrentes 401 aguardam o mesmo refresh
  - Variável de ambiente: `VITE_SESSION_IDLE_TIMEOUT_MS=1800000` (30min, build-time Vite)
- **Efeito colateral?** [ ] Não — utiliza o endpoint `/auth/refresh` existente sem alteração
- **Timeline/Notifications?** [ ] Não
- **Dependências:** FR-003 (refresh token rotation), FR-001 (login/logout), FR-002 (kill-switch), BR-002

### Alteração em FR-003 — Anotação de consumo automático

Adicionar ao "Done funcional" do FR-003 existente:

> **Consumo automático (FR-020):** O frontend invoca `POST /auth/refresh` de duas formas: (a) proativamente via `useSessionKeepAlive` quando faltam ≤2min para expiração do access token e há atividade recente; (b) reativamente via interceptor 401 nos HTTP clients. Ambas as formas respeitam a rotação obrigatória e o mutex de refresh.

### Novos componentes frontend

| Componente | Caminho | Responsabilidade |
|---|---|---|
| `useActivityTracker` | `apps/web/src/modules/foundation/hooks/useActivityTracker.ts` | Detecção de atividade DOM, throttle, persistência sessionStorage, cálculo `isActive` |
| `useSessionKeepAlive` | `apps/web/src/modules/foundation/hooks/useSessionKeepAlive.ts` | Timer 30s, decisão de refresh proativo, orquestração com activity tracker |
| Interceptor 401 | Modificação em `http-client.ts` e `api-client.ts` | Retry automático após refresh, mutex queue, force logout |

### Restrições de segurança (preservadas da spec)

- **SEC-001**: Refresh token transmitido APENAS via cookie httpOnly — nunca exposto ao JS
- **SEC-002**: Token rotation preservada — cada refresh invalida o anterior
- **SEC-003**: Detecção de reuso (`handleReuseDetected`) permanece ativa
- **SEC-004**: Idle timeout respeitado mesmo com access token válido
- **SEC-005**: Mousemove throttled (1 registro / 30s)

```gherkin
Funcionalidade: Smart Session Keep-Alive

Cenário: Refresh proativo durante uso ativo
  Dado que o usuário está logado e interagindo ativamente
  E faltam menos de 2 minutos para o access token expirar
  Quando o timer de check executa
  Então POST /auth/refresh é chamado proativamente
  E o usuário NÃO percebe interrupção
  E o novo access token substitui o anterior

Cenário: Sessão expira por inatividade
  Dado que o usuário está logado
  E não interage com o sistema por mais de 30 minutos
  Quando o access token expira e o usuário tenta uma ação
  Então o interceptor tenta POST /auth/refresh
  E se o refresh token ainda for válido, a sessão é renovada
  E se o refresh token expirou ou sessão foi revogada, redirect para /login

Cenário: Múltiplos requests 401 simultâneos
  Dado que 5 requests são enviados e todos recebem 401
  Quando o interceptor processa os erros
  Então apenas 1 POST /auth/refresh é enviado
  E os 5 requests originais são retentados após sucesso do refresh

Cenário: Refresh falha por sessão revogada
  Dado que o admin revogou a sessão do usuário via kill-switch (FR-002)
  Quando qualquer request recebe 401 e o interceptor tenta refresh
  Então POST /auth/refresh retorna 401
  E o usuário é redirecionado para /login
  E localStorage e queryClient são limpos

Cenário: Page reload preserva atividade
  Dado que o usuário recarrega a página (F5)
  Quando a página carrega
  Então o lastActivity é recuperado do sessionStorage
  E o keep-alive continua normalmente sem perder contexto de atividade
```
