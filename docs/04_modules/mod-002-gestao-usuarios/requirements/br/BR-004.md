# BR-004 — Anti-Spam: Cooldown 60s no Reenvio de Convite

- **Regra:** Após um reenvio de convite bem-sucedido, o botão "Reenviar convite" DEVE ficar desabilitado por 60 segundos com countdown visível em tempo real. O cooldown é client-side (sem polling à API). Se o reenvio falhar (erro 500), o cooldown NÃO é iniciado — o admin pode tentar novamente imediatamente.
- **Exemplo:**
  - Admin clica "Reenviar convite" → 200 OK → botão desabilitado, exibe "Aguarde 58 segundos..." → countdown decrementa → ao chegar a 0, botão reabilita
  - Admin clica "Reenviar convite" → 500 → Toast de erro → botão permanece habilitado (sem cooldown)
- **Exceções:** Nenhuma. O cooldown é fixo (60s) e não configurável. Não há bypass por role.
- **Impacto:** fluxo, segurança

### Known Limitation v1 (PENDENTE-002)

> **Limitação:** O cooldown de 60s é client-side por aba. Abrir a mesma tela em múltiplas abas permite contornar o cooldown (cada aba tem timer independente). O rate limiting genérico do gateway (SEC-001 §8) protege contra abuso sistemático, mas não há rate limiting backend específico por `user_id + invite_resend`.
>
> **Justificativa:** Backoffice interno com usuários confiáveis — cenário de abuso via múltiplas abas é improvável. Complexidade de BroadcastChannel (polyfill Safari < 15.4) ou amendment no MOD-000 não se justifica para v1.
>
> **Reavaliar quando:** Telemetria indicar abuso (múltiplos `invite_resend` do mesmo admin em janela < 60s). Opções futuras: BroadcastChannel API (Opção B) ou rate limiting backend (Opção C).
