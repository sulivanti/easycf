# ADR-005 — Background Job para Expiração Automática de Atribuições

- **status:** accepted
- **timebox_days:** 90

---

## Contexto

BR-015 define que atribuições vinculadas a delegações MOD-004 devem ser desativadas quando a delegação expira. BR-017 define que atribuições com `valid_until` preenchido devem auto-expirar. Em ambos os casos, a desativação precisa ocorrer sem intervenção do usuário, registrando evento REASSIGNED (auto-expirado) em case_events.

A questão é: qual mecanismo técnico executa essa expiração? On-demand (lazy, no próximo acesso) ou proativo (background job)?

## Decisão

**Background job periódico (cron) que varre atribuições expiradas e as desativa proativamente.** O job executa a cada 5 minutos, consultando `case_assignments WHERE is_active=true AND (valid_until < now() OR delegation expirada)`, desativando e registrando case_events + domain events.

## Alternativas Consideradas

| Alternativa | Prós | Contras |
|---|---|---|
| **A) Background job periódico (escolhida)** | Proativo; expiração previsível; independente de acesso do usuário; auditável (case_event + domain_event registrados no momento exato) | Latência de até 5 min entre expiração real e desativação; custo do job rodando periodicamente; precisa de healthcheck |
| **B) Lazy evaluation (on-demand)** | Zero custo quando ninguém acessa; sem job | Expiração não registrada até próximo acesso; pode gerar surpresas na transição (papel "expirou" durante tentativa de transitar); case_event com timestamp atrasado |
| **C) Database trigger / scheduled function** | Expiração exata no timestamp; sem application code | Lógica de negócio no banco; difícil testar; sem domain events; cross-concern com MOD-004 |
| **D) Event-driven via MOD-004 (delegation.expired)** | Reação imediata; desacoplamento | MOD-004 não emite evento de expiração hoje (missing_info); apenas cobre delegação, não valid_until; dependência de MOD-004 implementar evento |

## Consequências

- Job roda a cada 5 minutos — granularidade aceitável para processos de negócio (não é real-time)
- Indexes otimizados: `idx_case_assignments_valid_until` e `idx_case_assignments_delegation` (DATA-006)
- Cada desativação registra `case_events` (event_type=REASSIGNED, metadata.reason="auto-expirado") e emite domain event `case.assignment_replaced`
- Healthcheck `/health/jobs` monitora se o job está rodando (NFR-006 §3)
- Se no futuro MOD-004 emitir `delegation.expired`, o job pode ser complementado por handler reativo (Alternativa D como melhoria, não substituição)

## Mecanismo de Verificação — DelegationCheckerPort (PEN-006/PENDENTE-002 Opção B)

O job utiliza `DelegationCheckerPort` (port já definido em mod.md §3) para verificar expiração de delegações, **sem acesso direto** ao schema do MOD-004.

### Assinatura do Port

```typescript
interface DelegationCheckerPort {
  // Método existente
  checkDelegation(delegationId: string): Promise<DelegationStatus>;

  // Novo método — adicionado para expiração automática
  getExpiredDelegations(): Promise<Array<{
    assignment_id: string;   // UUID do case_assignment vinculado
    delegation_id: string;   // UUID da access_delegation expirada
    expired_at: string;      // ISO timestamp da expiração
  }>>;
}
```

### Comportamento do Job

1. **Trigger:** Cron a cada 5 minutos
2. **Ação:** Chama `DelegationCheckerPort.getExpiredDelegations()` para delegações expiradas + query direta em `case_assignments WHERE valid_until < now() AND is_active = true` para atribuições temporárias
3. **Side-effect:** Desativa cada `case_assignment` (is_active=false), registra `case_events` (event_type=REASSIGNED, metadata.reason="auto-expirado"), emite domain event `case.assignment_replaced`
4. **Tolerância:** Até 5 minutos entre expiração real e desativação (intervalo do cron)
5. **Fallback:** Se `getExpiredDelegations()` falhar: log error + skip iteração (fail-safe). Job retenta no próximo ciclo. Assignments não são desativadas por engano.
6. **Timeout:** 3s para chamada ao port. 1 retry.
7. **Healthcheck:** `/health/jobs` monitora last_run_at e status do job (NFR-006 §3)

### Evolução Futura

Quando MOD-004 implementar domain event `delegation.expired`, o job pode ser complementado por handler reativo (Alternativa D como melhoria, não substituição). O port garante que a mudança é transparente para o job.

### Justificativa

Port/adapter `DelegationCheckerPort` já existe na arquitetura do módulo (mod.md §3). Basta adicionar `getExpiredDelegations`. Desacoplamento via port garante que mudanças no schema do MOD-004 não quebram o job. Ref: PEN-006/PENDENTE-002 Opção B.
