### Alteracoes neste pilar

#### A1 — FR-004: CRUD de Regras de Incidencia — campo `incidence_type`

**POST /api/v1/admin/incidence-rules:**

Aceitar campo `incidence_type` no request body:

```json
{
  "framer_id": "uuid",
  "target_object_id": "uuid",
  "valid_from": "2026-04-01",
  "incidence_type": "OBR"
}
```

- Valores aceitos: `OBR`, `OPC`, `AUTO`
- Default: `OBR` (se nao informado)
- Validacao: 422 se valor invalido

**PATCH /api/v1/admin/incidence-rules/:id:**

Aceitar `incidence_type` como campo atualizavel:

```json
{
  "incidence_type": "OPC"
}
```

**GET /api/v1/admin/incidence-rules:**

Retornar `incidence_type` em cada regra no response. Adicionar filtro opcional `?incidence_type=OBR`:

```json
{
  "data": [
    {
      "id": "uuid",
      "framer_id": "uuid",
      "target_object_id": "uuid",
      "incidence_type": "OBR",
      "status": "ACTIVE",
      "valid_from": "2026-04-01",
      "valid_until": null
    }
  ]
}
```

```gherkin
Funcionalidade: CRUD de Regras de Incidencia com incidence_type

Cenario: Criar regra com incidence_type OBR (default)
  Dado que enquadrador e objeto-alvo existem
  Quando POST /admin/incidence-rules com { framer_id, target_object_id, valid_from }
  Entao 201 com incidence_type="OBR" (default)

Cenario: Criar regra com incidence_type OPC
  Quando POST /admin/incidence-rules com { framer_id, target_object_id, valid_from, incidence_type: "OPC" }

  Quando POST /admin/incidence-rules com { incidence_type: "AUTO", ... }
  Entao 201 com incidence_type="AUTO"

Cenario: Rejeitar incidence_type invalido
  Quando POST /admin/incidence-rules com { incidence_type: "INVALID", ... }
  Entao 422 com invalid_fields contendo "incidence_type"

Cenario: Atualizar incidence_type de regra existente
  Dado que regra existe com incidence_type="OBR"

  Dado que existem regras OBR e OPC
  Quando GET /admin/incidence-rules?incidence_type=OBR
```

#### A2 — FR-009: Motor de Avaliacao — retornar `incidence_type` nos resultados

O response do `POST /api/v1/routine-engine/evaluate` DEVE incluir `incidence_type` em cada rotina aplicada, para que o frontend possa renderizar resultados color-coded no dry-run modal.

**Response atualizado:**

```json
{
  "visible_fields": [...],
  "hidden_fields": [...],
  "required_fields": [...],
  "optional_fields": [...],
  "defaults": [...],
  "domain_restrictions": [...],
  "validations": [...],
  "blocking_validations": [...],
  "applied_routines": [
    {
      "routine_id": "uuid",
      "routine_name": "Rot. Validacao",
      "routine_version": 3,
      "incidence_type": "OBR",
      "items_applied": [
        {
          "item_id": "uuid",
          "item_type": "VALIDATION",
          "action": "VALIDATE",
          "incidence_type": "OBR"
        }
      ]
    }
  ],
  "dry_run": true
}
```

**Nota:** O `incidence_type` vem da `incidence_rules` que vincula a rotina ao contexto avaliado. Cada `applied_routine` herda o `incidence_type` da regra que a ativou.

```gherkin
Funcionalidade: Motor de Avaliacao retorna incidence_type

Cenario: Evaluate retorna incidence_type por rotina aplicada
  Dado que regra de incidencia tem incidence_type="OBR"
  E rotina PUBLISHED esta vinculada a essa regra
  Quando POST /routine-engine/evaluate com context valido
  Entao applied_routines[0].incidence_type = "OBR"

Cenario: Dry-run retorna incidence_type para color-coding
  Dado que existem regras com tipos OBR, OPC e AUTO
  Quando POST /routine-engine/evaluate com { dry_run: true, ... }
  Entao cada applied_routine tem incidence_type correspondente
  E frontend pode renderizar OBR em roxo, OPC em teal, AUTO em ambar
```
