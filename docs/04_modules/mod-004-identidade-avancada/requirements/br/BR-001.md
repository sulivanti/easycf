# BR-001 — Regras de Negócio da Identidade Avançada

## BR-001.1 — Dimensão Ortogonal ao tenant_users

- **Regra:** `user_org_scopes` (vínculo usuário ↔ área org N1–N4) é uma dimensão ortogonal ao `tenant_users` (vínculo a filial N5). Criar um não implica criar o outro.
- **Exemplo:** Usuário "Ana" tem `tenant_users` em 3 filiais (SP, RJ, BH). Simultaneamente, tem `user_org_scopes` apontando para a Macroárea de Tecnologia (N3) que contém essas filiais. DELETE no vínculo org NÃO altera `tenant_users`.
- **Exceções:** Nenhuma. As duas dimensões são sempre independentes.
- **Impacto:** `DATA`, `FLOW`, `PERMISSIONS` — Processos (MOD-005/006/007/008) que filtrem por área organizacional usam `user_org_scopes`; operações por filial usam `tenant_users`. APIs de listagem (GET /admin/users/:id) retornam `tenants[]` e `org_scopes[]` em arrays separados.

## BR-001.2 — Máximo 1 PRIMARY por Usuário

- **Regra:** Um usuário pode ter no máximo 1 vínculo `user_org_scopes` com `scope_type=PRIMARY` ativo. SECONDARY não tem limite.
- **Exemplo:** Usuário "Carlos" tem PRIMARY em N3-Tecnologia. Tentativa de POST com `scope_type=PRIMARY` para N3-Financeiro retorna 409. Deve-se revogar o PRIMARY atual antes.
- **Exceções:** Nenhuma. Constraint de unicidade validada no service (não via UNIQUE no banco, pois a constraint é `UNIQUE(user_id, org_unit_id)`, não por `scope_type`).
- **Impacto:** `STATE`, `FLOW` — Define a área principal para contexto padrão em processos. Sidebar e dashboards podem usar PRIMARY como filtro default.

## BR-001.3 — Somente N1–N4 em user_org_scopes

- **Regra:** Vínculos organizacionais só são permitidos em nós N1–N4 da estrutura org. Nós N5 (tenant/filial) são gerenciados via `tenant_users` (MOD-000-F09).
- **Exemplo:** POST com `org_unit_id` apontando para "Filial Centro-SP" (N5 via `org_unit_tenant_links`) retorna 422. O admin deve usar a API de `tenant_users` para vincular a filiais.
- **Exceções:** Nenhuma. A validação verifica se `org_unit_id` possui link em `org_unit_tenant_links` (presença indica N5).
- **Impacto:** `DATA`, `PERMISSIONS` — Evita duplicação de dimensões e mantém separação semântica clara entre área org e filial.

## BR-001.4 — Delegação Não Transfere Poder Decisório

- **Regra:** Delegações (`access_delegations`) NUNCA podem conter escopos com sufixo `:approve`, `:execute` ou `:sign`. A validação é feita no service via regex pattern `*:(approve|execute|sign)`.
- **Exemplo:** User-A tenta delegar `["finance:invoice:read", "finance:invoice:approve"]`. O service retorna 422 indicando que `finance:invoice:approve` é um escopo proibido em delegação.
- **Exceções:** Nenhuma. Regra inegociável — garante segregação de funções mesmo em delegações temporárias.
- **Impacto:** `PERMISSIONS`, `COMPLIANCE` — Impede que delegatee aprove, execute ou assine em nome do delegator. Escopo de ação fica limitado a leitura, edição de rascunhos e consultas.

## BR-001.5 — Delegação Somente de Escopos Próprios

- **Regra:** O delegator só pode delegar escopos que ele próprio possui no token JWT atual. Escopos que vieram de outra delegação são excluídos (ver BR-001.6).
- **Exemplo:** User-A possui `["finance:invoice:read", "hr:salary:read"]`. Ao tentar delegar `["hr:salary:read", "procurement:order:read"]`, o service retorna 422 para `procurement:order:read` (não possuído).
- **Exceções:** Nenhuma.
- **Impacto:** `PERMISSIONS` — O delegator não pode conceder mais do que tem.

## BR-001.6 — Sem Re-delegação em Cadeia

- **Regra:** Escopos obtidos por delegação (`access_delegations` onde caller = delegatee) não podem ser re-delegados a um terceiro.
- **Exemplo:** User-A delega `finance:invoice:read` para User-B. User-B tenta delegar esse mesmo escopo para User-C. O service verifica que o escopo de User-B veio de delegação e retorna 422.
- **Exceções:** Nenhuma. Previne cascata de delegações sem rastreabilidade.
- **Impacto:** `PERMISSIONS`, `COMPLIANCE` — Cadeia máxima: 1 nível. Auditoria pode rastrear delegações diretamente (delegator → delegatee, sem intermediários).

## BR-001.7 — Validação de Autorização por Scope (Compartilhamento)

- **Regra:** Em `access_shares`, a validação `authorized_by ≠ grantor_id` é condicional ao scope `identity:share:authorize`:
  - **Com scope:** auto-autorização é permitida (grantor = authorized_by).
  - **Sem scope:** `authorized_by` deve ser diferente de `grantor_id`.
- **Exemplo (com scope):** Admin "João" possui `identity:share:authorize`. Ele cria share com `grantor_id=João, authorized_by=João`. Service aceita (201).
- **Exemplo (sem scope):** Usuário "Maria" NÃO possui `identity:share:authorize`. Ela tenta `grantor_id=Maria, authorized_by=Maria`. Service retorna 422.
- **Exceções:** Validação no service (sem CHECK constraint no banco). Decisão técnica 2026-03-15 (US-MOD-004 v1.1.0).
- **Impacto:** `PERMISSIONS`, `COMPLIANCE` — Flexibiliza segregação para usuários com permissão explícita, mantendo controle auditável.

## BR-001.8 — Vigência Obrigatória em Shares e Delegations

- **Regra:** `valid_until` é obrigatório em `access_shares` e `access_delegations`. Não existe compartilhamento ou delegação permanente via estes mecanismos.
- **Exemplo:** POST `/access-shares` sem `valid_until` no body retorna 422: "A data de expiração é obrigatória para compartilhamentos."
- **Exceções:** `user_org_scopes` pode ter `valid_until=null` (vínculo org permanente é permitido).
- **Impacto:** `STATE`, `COMPLIANCE` — Garante que todos os acessos expandidos têm prazo. Background job expira registros automaticamente.

## BR-001.9 — Motivo Obrigatório em Compartilhamentos

- **Regra:** O campo `reason` é obrigatório em `access_shares` para rastreabilidade e auditoria. Campo vazio ou ausente é rejeitado.
- **Exemplo:** POST `/access-shares` com `reason: ""` retorna 422: "O motivo do compartilhamento é obrigatório."
- **Exceções:** `access_delegations` também exige `reason` (campo NOT NULL na tabela), mas a mensagem de erro é diferente.
- **Impacto:** `COMPLIANCE` — Auditores podem consultar por que cada compartilhamento foi criado, quem autorizou e qual o motivo.
