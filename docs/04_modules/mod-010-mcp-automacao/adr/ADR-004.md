# ADR-004 — Blocklist como Wildcard Pattern Matching (Por Que Regex-like e Não Lista Exata)

## Contexto

O `ScopeBlocklistValidator` do MOD-010 precisa bloquear permanentemente escopos de aprovação, exclusão, assinatura e execução de gates para agentes MCP (BR-002, Phase 1). A blocklist Phase 1 contém 6 padrões: `*:delete`, `*:approve`, `approval:decide`, `approval:override`, `*:sign`, `*:execute`. A decisão é se a blocklist deve usar: (a) matching por padrão wildcard (`*:delete` bloqueia qualquer scope terminado em `:delete`), ou (b) lista exata de escopos bloqueados (enumerar todos os scopes como `org:unit:delete`, `process:case:delete`, etc.).

O catálogo de escopos do sistema cresce com cada módulo novo. Novos módulos podem introduzir escopos como `inventory:item:delete` que devem ser automaticamente bloqueados para agentes MCP sem precisar atualizar a blocklist.

## Decisão

**A blocklist usa wildcard pattern matching.** O padrão `*:delete` bloqueia qualquer scope cujo segmento final seja `:delete`, independentemente do prefixo. Padrões exatos como `approval:decide` e `approval:override` bloqueiam por match exato. O `ScopeBlocklistValidator` aplica os 6 padrões em ordem, retornando o primeiro match encontrado com o padrão que causou o bloqueio (ex: `"org:unit:delete corresponde ao padrão *:delete"`).

## Alternativas Consideradas

1. **Lista exata (enumeração completa):** Listar todos os escopos bloqueados explicitamente (ex: `org:unit:delete`, `process:case:delete`, `mcp:agent:delete`). Vantagem: determinístico, sem surpresas. Desvantagem: cada módulo novo precisa atualizar a blocklist do MOD-010 — forte acoplamento e risco de esquecer um scope novo. A blocklist ficaria desatualizada silenciosamente.
2. **Regex completo:** Usar expressões regulares para os padrões (ex: `^.*:delete$`). Vantagem: flexibilidade máxima. Desvantagem: regex é mais complexo de auditar, mais lento para avaliar, e introduce riscos de ReDoS (Regular Expression Denial of Service) se um padrão malformado entrar na blocklist.
3. **Tag-based:** Marcar scopes com tags (`dangerous`, `approval`, `destructive`) no catálogo Foundation e bloquear por tag. Vantagem: semântico e extensível. Desvantagem: requer mudança no schema de scopes do Foundation (MOD-000), adicionando complexidade cross-module. Tag system não existe atualmente.

## Consequências

- **Positivas:** Blocklist é forward-compatible — novos módulos que criem escopos com `:delete`, `:approve`, `:sign` ou `:execute` são automaticamente bloqueados. Simplicidade de implementação (string matching com wildcard). Mensagem de erro informativa inclui o padrão que causou o bloqueio.
- **Negativas:** Possibilidade de false positives — um scope legítimo com sufixo `:delete` seria bloqueado mesmo que não seja uma operação de exclusão destrutiva (mitigado pela convenção de nomenclatura do projeto onde `:delete` SEMPRE significa exclusão). Wildcard é limitado a sufixo (`*:suffix`) — se futuramente for necessário bloquear por prefixo ou infixo, a implementação precisará ser estendida.
- **Riscos:** Se a convenção de nomenclatura de scopes mudar (ex: `remove` em vez de `delete`), a blocklist precisará ser atualizada. Mitigação: documentar a convenção no DOC-FND-000 e garantir que novos módulos sigam o padrão.
