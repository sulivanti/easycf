# FR-000 — Requisitos Funcionais do Foundation

---

## FR-001 — Autenticação Nativa (Login/Logout)

- **Descrição:** O sistema DEVE prover login nativo com e-mail e senha, emitindo JWT com sessionId ancorado em banco. Logout revoga sessão imediatamente.
- **Prioridade:** Must
- **Perfis impactados:** Todos os usuários
- **Done funcional:**
  - POST /auth/login aceita `{email, password, remember_me?, device_fp?}`
  - Retorna `{access_token, refresh_token, token_type, expires_in, user}`
  - Seta cookies httpOnly (accessToken + refreshToken)
  - Cria registro em `user_sessions` com isRevoked=false
  - Emite evento `auth.login_success` com correlation_id
  - POST /auth/logout marca sessão como isRevoked=true
- **Efeito colateral?** [x] Sim — Idempotency-Key obrigatório (TTL 30s)
- **Dependências:** BR-001, BR-002, BR-003, BR-008, BR-010, BR-012, DATA-000, SEC-000
- **TTL de Sessão:** Normal=12h, Estendida (remember_me)=30d

```gherkin
Funcionalidade: Autenticação Nativa

Cenário: Login bem-sucedido sem MFA
  Dado que existe usuário com email "joao@empresa.com" e status ACTIVE
  Quando POST /auth/login com credenciais válidas
  Então status 200 com access_token, refresh_token, user
  E sessão criada em user_sessions
  E evento auth.login_success emitido

Cenário: Logout
  Dado que o usuário está autenticado
  Quando POST /auth/logout
  Então sessão marcada isRevoked=true
  E evento auth.logout emitido
```

## FR-003 — Renovação de Token (Refresh) com Rotação

- **Descrição:** O sistema DEVE permitir renovação de access_token via refresh_token com **rotação obrigatória** (PENDENTE-002, OAuth2 BCP). A cada refresh, novo refresh_token é emitido e o anterior é invalidado. Reuse detection invalida toda a família de tokens.
- **Prioridade:** Must
- **Perfis impactados:** Todos os usuários autenticados
- **Done funcional:**
  - POST /auth/refresh aceita cookie refreshToken
  - Valida sessão ativa em banco (isRevoked=false)
  - **Rotação:** Emite novo `refresh_token`, invalida o anterior, retorna novo `access_token` + novo `refresh_token` (200)
  - **Token families:** Cada refresh_token pertence a uma família (raiz = login original). Armazenar `token_family_id` em `user_sessions` ou tabela dedicada
  - **Reuse detection:** Se refresh_token já utilizado for apresentado → invalida TODA a família (kill-switch automático) → emite evento `auth.token_reuse_detected` (sensitivity_level=2) → retorna 401
  - **Grace period:** Token antigo aceito por 10s após rotação (mitiga race conditions em múltiplas abas)
  - Sessão revogada → retorna 401
- **Efeito colateral?** [x] Sim — emite novo refresh_token, invalida anterior
- **Timeline/Notifications?** [x] Sim — evento `auth.token_reuse_detected` → DATA-003, SEC-002
- **Dependências:** BR-002, FR-001, DATA-003, SEC-000, SEC-002

```gherkin
Funcionalidade: Refresh Token com Rotação

Cenário: Renovação com sessão válida — rotação
  Dado que o access_token expirou mas a sessão é válida
  Quando POST /auth/refresh com cookie refreshToken
  Então status 200 com novo access_token E novo refresh_token
  E refresh_token anterior é invalidado

Cenário: Renovação com sessão revogada
  Dado que a sessão foi revogada via Kill-Switch
  Quando POST /auth/refresh
  Então status 401

Cenário: Reuse detection — token reutilizado
  Dado que o refresh_token já foi utilizado em uma rotação anterior
  E o grace period de 10s expirou
  Quando POST /auth/refresh com o token antigo
  Então status 401
  E TODA a família de tokens é invalidada (kill-switch)
  E evento auth.token_reuse_detected emitido (sensitivity_level=2)

Cenário: Grace period — múltiplas abas
  Dado que o refresh_token foi rotacionado há menos de 10s
  Quando POST /auth/refresh com o token antigo (dentro do grace period)
  Então status 200 com novo access_token E novo refresh_token
```

## FR-006 — Gestão de Usuários (CRUD + Soft Delete)

- **Descrição:** O sistema DEVE prover CRUD completo de usuários com auto-registro público, separação de tabelas (users + content_users), soft delete para LGPD e reenvio de convite de ativação.
- **Prioridade:** Must
- **Perfis impactados:** Usuário anônimo (registro), Administrador (gestão)
- **Done funcional:**
  - POST /api/v1/users — auto-registro público (201)
  - GET /api/v1/users — listagem paginada cursor-based (requer auth)
  - GET /api/v1/users/:id — detalhe (requer auth)
  - PATCH /api/v1/users/:id — edição (requer users:user:write)
  - DELETE /api/v1/users/:id — soft delete (requer users:user:delete)
  - POST /api/v1/users/:id/invite/resend — reenvio de convite de ativação (operationId: `users_invite_resend`, requer `users:user:write`, apenas status=PENDING, gera novo token invalidando o anterior, evento `user.invite_resent` com sensitivity_level=1, e-mail via Outbox Pattern, retorna 200 `{ message: "Invite sent" }` sem revelar e-mail/token, Idempotency-Key TTL 60s)
  - Inserção atômica em users + content_users via transação Drizzle
  - E-mail de boas-vindas assíncrono (fire-and-forget)
  - Eventos: `user.created`, `user.updated`, `user.deleted`, `user.invite_resent`
- **Efeito colateral?** [x] Sim — Idempotency-Key em POST (TTL 60s)
- **Dependências:** BR-004, BR-009, BR-012, DATA-000, DATA-003, INT-000, SEC-000, SEC-002, UX-000

```gherkin
Funcionalidade: Gestão de Usuários

Cenário: Auto-registro bem-sucedido
  Quando POST /api/v1/users com dados válidos
  Então status 201 com dados criados (sem passwordHash)
  E inserção atômica em users + content_users
  E evento user.created emitido

Cenário: Soft Delete
  Dado que admin tem escopo users:user:delete
  Quando DELETE /api/v1/users/:id
  Então deletedAt preenchido, status INACTIVE
  E evento user.deleted emitido

Cenário: Reenvio de convite bem-sucedido
  Dado que admin tem escopo users:user:write
  E que o usuário alvo tem status PENDING
  Quando POST /api/v1/users/:id/invite/resend com Idempotency-Key
  Então status 200 com { message: "Invite sent" }
  E token anterior invalidado, novo token gerado
  E evento user.invite_resent emitido via Outbox
  E e-mail de ativação enfileirado (falha não retorna erro)

Cenário: Reenvio para usuário não-PENDING
  Dado que o usuário alvo tem status ACTIVE
  Quando POST /api/v1/users/:id/invite/resend
  Então status 422 (RFC 9457) com detail explicando restrição de status
```

## FR-015 — MFA / TOTP (RFC 6238)

- **Descrição:** O sistema DEVE prover fluxo completo de MFA via TOTP (RFC 6238): setup com geração de secret/QR, verificação de código e desativação.
- **Prioridade:** Must
- **Perfis impactados:** Todos os usuários autenticados
- **Done funcional:**

## FR-016 — SSO OAuth2 (Google + Microsoft)

- **Descrição:** O sistema DEVE prover login via SSO OAuth2 com Google e Microsoft/Azure AD usando Authorization Code Flow. Quando o e-mail do provider já existe com conta nativa, o sistema DEVE exigir confirmação via senha nativa antes de vincular o provider SSO (ADR-004).
- **Prioridade:** Must
- **Perfis impactados:** Todos os usuários
- **Done funcional:**
  - GET /auth/google — redirect para Google OAuth2
  - GET /auth/google/callback?code=... — troca code por token, cria/reutiliza usuário, emite JWT
  - GET /auth/microsoft — redirect para Microsoft/Azure AD
  - GET /auth/microsoft/callback?code=... — troca code por token, cria/reutiliza usuário
  - Redirect final: `{FRONTEND_URL}/sso-success?token=JWT&provider={provider}&is_new_user={bool}`
  - **Fluxo de Identity Linking (ADR-004):** Se e-mail do provider já existe com conta nativa:
    1. Callback SSO detecta e-mail existente sem `sso_provider_id` vinculado
    2. Gera `temp_sso_token` (escopo `sso-link-only`, TTL 10min) com claims: `email`, `provider`, `provider_id`
    3. Redirect para `{FRONTEND_URL}/sso-link-confirm?token={temp_sso_token}&provider={provider}`
    4. Frontend exibe tela de confirmação solicitando senha nativa
    5. POST /auth/sso/link-confirm aceita `{temp_sso_token, password}`:
       - Valida `temp_sso_token` (escopo, TTL, integridade)
       - bcrypt compare da senha nativa
       - Vincula `sso_provider` + `sso_provider_id` à conta existente
       - Cria sessão em `user_sessions`
       - Emite evento `auth.sso_linked` (sensitivity_level=1)
       - Retorna `{access_token, refresh_token, token_type, expires_in, user}`
    6. Se senha incorreta: retorna 400 (RFC 9457) — máx. 5 tentativas, depois invalida `temp_sso_token`
  - Se Microsoft não retorna `mail`: redirect `/login?error=no_email_provided&provider=microsoft`
  - Eventos: `auth.sso_login`, `auth.sso_linked`
- **Efeito colateral?** [x] Sim (criação de usuário quando novo, vinculação de provider)
- **Timeline/Notifications?** [x] Sim — eventos auth.sso_login, auth.sso_linked → DATA-003, SEC-002
- **Dependências:** INT-001, INT-002, FR-001, DATA-000, DATA-003, SEC-000, SEC-002, ADR-004

```gherkin
Funcionalidade: SSO OAuth2

Cenário: Login Google — usuário novo
  Quando GET /auth/google/callback com code válido
  E o e-mail não existe no sistema
  Então cria usuário + content_users em transação
  E emite evento auth.sso_login
  E redirect para /sso-success com is_new_user=true

Cenário: Login Google — usuário existente com SSO já vinculado
  Quando GET /auth/google/callback com code válido
  E o e-mail já existe no sistema com sso_provider_id vinculado
  Então reutiliza registro existente
  E emite evento auth.sso_login
  E redirect para /sso-success com is_new_user=false

Cenário: Login SSO — e-mail existente com conta nativa (Identity Linking)
  Dado que o e-mail do provider SSO já existe com conta nativa sem vínculo SSO
  Quando GET /auth/google/callback com code válido
  Então gera temp_sso_token com escopo sso-link-only (TTL 10min)
  E redirect para /sso-link-confirm com token e provider

Cenário: Confirmação de Identity Linking — senha correta
  Dado que o usuário recebeu temp_sso_token após SSO callback
  Quando POST /auth/sso/link-confirm com temp_sso_token e senha nativa correta
  Então status 200 com access_token, refresh_token, user
  E sso_provider e sso_provider_id vinculados à conta existente
  E evento auth.sso_linked emitido
  E sessão criada em user_sessions

Cenário: Confirmação de Identity Linking — senha incorreta
  Dado que o usuário recebeu temp_sso_token
  Quando POST /auth/sso/link-confirm com senha incorreta
  Então status 400 (RFC 9457) com detail "Senha incorreta"
  E tentativas restantes decrementadas

Cenário: Identity Linking — tentativas esgotadas
  Dado que o usuário errou a senha 5 vezes
  Quando POST /auth/sso/link-confirm
  Então status 429 com detail "Token invalidado por excesso de tentativas"
  E temp_sso_token é invalidado

Cenário: Falha no SSO
  Quando GET /auth/google/callback e o token exchange falha
  Então redirect para /login?error=sso_failed&provider=google
```

## FR-017 — Recuperação de Senha (Forgot / Reset)

- **Descrição:** O sistema DEVE prover fluxo completo de recuperação de senha via token UUID com TTL de 1 hora.
- **Prioridade:** Must
- **Perfis impactados:** Todos os usuários (público)
- **Done funcional:**
  - POST /auth/forgot-password aceita `{email}` — gera token UUID, TTL 1h, envia e-mail
  - POST /auth/reset-password aceita `{token, new_password}` — valida token, redefine senha
  - Resposta de forgot-password DEVE ser sempre genérica (BR-001 — User Enumeration Prevention)
  - Token expirado → 422 com mensagem "Link expirado"
  - Eventos: `auth.forgot_password_requested`, `auth.password_reset`
- **Efeito colateral?** [x] Sim — Idempotency-Key recomendado
- **Dependências:** BR-001, INT-005, FR-001, DATA-000, DATA-003, SEC-000

```gherkin
Funcionalidade: Recuperação de Senha

Cenário: Solicitação com e-mail existente
  Dado que o e-mail está cadastrado
  Quando POST /auth/forgot-password
  Então status 200 com mensagem genérica
  E e-mail de reset enviado com token UUID
  E evento auth.forgot_password_requested emitido

Cenário: Solicitação com e-mail inexistente
  Dado que o e-mail NÃO está cadastrado
  Quando POST /auth/forgot-password
  Então status 200 com mensagem genérica idêntica (BR-001)
  E NENHUM e-mail é enviado

Cenário: Reset com token válido
  Dado que o token é válido e não expirou
  Quando POST /auth/reset-password com nova senha
  Então senha atualizada com sucesso
  E evento auth.password_reset emitido

Cenário: Reset com token expirado
  Dado que o token tem mais de 1 hora
  Quando POST /auth/reset-password
  Então status 422 com mensagem "Link expirado"
```
