# FR-000 — Requisitos Funcionais do Foundation

---

## FR-001 — Autenticação Nativa (Login/Logout)

- **Descrição:** O sistema DEVE prover login nativo com e-mail e senha, emitindo JWT com sessionId ancorado em banco. Logout revoga sessão imediatamente.
- **Prioridade:** Must
- **Perfis impactados:** Todos os usuários
- **Done funcional:**
  - POST /auth/login aceita `{email, password, remember_me?, device_fp?}`
  - Retorna `{access_token, refresh_token, token_type, expires_in, user}`
  - Seta cookies httpOnly (accessToken + refreshToken)
  - Cria registro em `user_sessions` com isRevoked=false
  - Emite evento `auth.login_success` com correlation_id
  - POST /auth/logout marca sessão como isRevoked=true
- **Efeito colateral?** [x] Sim — Idempotency-Key obrigatório (TTL 30s)
- **Dependências:** BR-001, BR-002, BR-003, BR-008, BR-010, BR-012, DATA-000, SEC-000
- **TTL de Sessão:** Normal=12h, Estendida (remember_me)=30d

```gherkin
Funcionalidade: Autenticação Nativa

Cenário: Login bem-sucedido sem MFA
  Dado que existe usuário com email "joao@empresa.com" e status ACTIVE
  Quando POST /auth/login com credenciais válidas
  Então status 200 com access_token, refresh_token, user
  E sessão criada em user_sessions
  E evento auth.login_success emitido

Cenário: Logout
  Dado que o usuário está autenticado
  Quando POST /auth/logout
  Então sessão marcada isRevoked=true
  E evento auth.logout emitido
```

## FR-002 — Gerenciamento de Sessões (Kill-Switch)

- **Descrição:** O sistema DEVE permitir listar sessões ativas, revogar sessão individual e revogar todas as sessões (kill-switch global).
- **Prioridade:** Must
- **Perfis impactados:** Usuário autenticado, Administrador
- **Done funcional:**
  - GET /auth/sessions — lista sessões ativas do usuário
  - DELETE /auth/sessions/:id — revoga sessão específica (204)
  - DELETE /auth/sessions — revoga TODAS as sessões do usuário
  - Eventos: `session.revoked`, `session.revoked_all`
- **Efeito colateral?** [x] Sim
- **Dependências:** BR-002, FR-001, DATA-000, SEC-000

```gherkin
Funcionalidade: Kill-Switch de Sessões

Cenário: Revogar sessão específica
  Dado que o usuário tem múltiplas sessões
  Quando DELETE /auth/sessions/:sessionId
  Então apenas essa sessão deve ser revogada (204)

Cenário: Revogar todas as sessões
  Quando DELETE /auth/sessions
  Então TODAS as sessões do usuário são revogadas
  E evento session.revoked_all emitido
```

## FR-003 — Renovação de Token (Refresh)

- **Descrição:** O sistema DEVE permitir renovação de access_token via refresh_token quando a sessão em banco ainda for válida.
- **Prioridade:** Must
- **Perfis impactados:** Todos os usuários autenticados
- **Done funcional:**
  - POST /auth/refresh aceita cookie refreshToken
  - Valida sessão ativa em banco (isRevoked=false)
  - Retorna novo access_token (200)
  - Sessão revogada → retorna 401
- **Efeito colateral?** [ ] Não
- **Dependências:** BR-002, FR-001

```gherkin
Funcionalidade: Refresh Token

Cenário: Renovação com sessão válida
  Dado que o access_token expirou mas a sessão é válida
  Quando POST /auth/refresh com cookie refreshToken
  Então status 200 com novo access_token

Cenário: Renovação com sessão revogada
  Dado que a sessão foi revogada via Kill-Switch
  Quando POST /auth/refresh
  Então status 401
```

## FR-006 — Gestão de Usuários (CRUD + Soft Delete)

- **Descrição:** O sistema DEVE prover CRUD completo de usuários com auto-registro público, separação de tabelas (users + content_users) e soft delete para LGPD.
- **Prioridade:** Must
- **Perfis impactados:** Usuário anônimo (registro), Administrador (gestão)
- **Done funcional:**
  - POST /api/v1/users — auto-registro público (201)
  - GET /api/v1/users — listagem paginada cursor-based (requer auth)
  - GET /api/v1/users/:id — detalhe (requer auth)
  - PATCH /api/v1/users/:id — edição (requer users:write)
  - DELETE /api/v1/users/:id — soft delete (requer users:delete)
  - Inserção atômica em users + content_users via transação Drizzle
  - E-mail de boas-vindas assíncrono (fire-and-forget)
  - Eventos: `user.created`, `user.updated`, `user.deleted`
- **Efeito colateral?** [x] Sim — Idempotency-Key em POST (TTL 60s)
- **Dependências:** BR-004, BR-009, BR-012, DATA-000, INT-000, SEC-000, UX-000

```gherkin
Funcionalidade: Gestão de Usuários

Cenário: Auto-registro bem-sucedido
  Quando POST /api/v1/users com dados válidos
  Então status 201 com dados criados (sem passwordHash)
  E inserção atômica em users + content_users
  E evento user.created emitido

Cenário: Soft Delete
  Dado que admin tem escopo users:delete
  Quando DELETE /api/v1/users/:id
  Então deletedAt preenchido, status INACTIVE
  E evento user.deleted emitido
```

## FR-007 — Gestão de Roles/RBAC

- **Descrição:** O sistema DEVE prover CRUD de Roles com escopos de permissão, cache Redis e soft delete.
- **Prioridade:** Must
- **Perfis impactados:** Administrador
- **Done funcional:**
  - POST /api/v1/roles — criar role com escopos (requer roles:write)
  - GET /api/v1/roles — listar roles (requer roles:read)
  - PUT /api/v1/roles/:id — substituição completa de escopos (requer roles:write)
  - DELETE /api/v1/roles/:id — soft delete (requer roles:write)
  - Cache Redis `auth:scopes:role:{id}` com invalidação automática
  - Eventos: `role.created`, `role.updated`, `role.deleted`
- **Efeito colateral?** [x] Sim
- **Dependências:** BR-005, BR-006, BR-011, DATA-000, SEC-000

```gherkin
Funcionalidade: Gestão de Roles

Cenário: Criação de role
  Dado que o usuário tem escopo roles:write
  Quando POST /api/v1/roles com name, scopes
  Então status 201 com role criada
  E evento role.created emitido

Cenário: Sem permissão
  Dado que o usuário NÃO tem escopo roles:write
  Quando qualquer mutação em /roles
  Então status 403 (RFC 9457)
```
