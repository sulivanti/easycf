# Emenda: FR-000-C08

- **Documento base:** [FR-000](../../requirements/fr/FR-000.md)
- **estado_item:** MERGED
- **Natureza:** C (Correção)
- **Data:** 2026-03-30
- **owner:** ECF Core
- **Motivação:** RefreshTokenUseCase gera novo access token sem scopes e sem tenantId porque o refresh token JWT não contém esses campos. Após ~15min (expiração do access token), todas as rotas protegidas retornam `403 — Permissão insuficiente. Scope requerido: users:role:read` / `process:cycle:read`.
- **rastreia_para:** spec-refresh-token-scopes-fix, FR-003, BR-002, BR-011, SEC-000

## Detalhamento

### Requisito Afetado: FR-003 — Renovação de Token (Refresh) com Rotação

**Bug:** O `RefreshTokenUseCase` decodifica o refresh token para extrair `scopes` e `tenantId`, mas o refresh token nunca os conteve:

```typescript
// services-impl.ts:54-57 — refresh token gerado SEM scopes e SEM tid
const refreshToken = this.app.jwt.sign(
  { sub: payload.userId, sid: payload.sessionId, type: 'refresh' },
  { expiresIn: this.refreshExpiresIn },
);
```

```typescript
// refresh-token.use-case.ts:69-73 — scopes = [], tenantId = undefined
const tokenPair = await this.tokenService.generatePair({
  userId: payload.userId,
  sessionId: session.id,
  tenantId: payload.tenantId,  // undefined
  scopes: payload.scopes,       // []
});
```

**Correção:** O `RefreshTokenUseCase` DEVE re-buscar scopes e tenantId do banco de dados (via `TenantUserRepository` + `RoleRepository` + `CacheService`) em vez de extraí-los do payload do refresh token.

### Alterações no Done Funcional de FR-003

Adicionar ao Done Funcional:

- **Re-resolução de scopes:** A cada refresh, os scopes DEVEM ser re-buscados da role atual do usuário no banco de dados (não copiados do token anterior). Isso garante que alterações de role/permissões tomem efeito no próximo refresh sem necessidade de logout.
- **Re-resolução de tenantId:** A cada refresh, o tenantId ativo DEVE ser re-resolvido via `TenantUserRepository.findByUserId()`.
- **Sem tenant ativo:** Se o usuário não tiver mais um `tenantUser` com `status === 'ACTIVE'`, o refresh DEVE falhar com `SessionRevokedError` (401).
- **Cache de scopes (GUD-001):** Utilizar `CacheService.getOrSet('auth:scopes:role:{roleId}', ..., 300)` para evitar queries desnecessárias (mesmo padrão do `getProfileUseCase`).
- **Compatibilidade:** Refresh tokens emitidos antes do fix (sem `tid`/`scopes` no payload) DEVEM continuar funcionando — o payload mínimo `sub`+`sid` é suficiente pois scopes/tenant são re-buscados do banco.

### Novos Cenários Gherkin (adição a FR-003)

```gherkin
Cenário: Refresh re-busca scopes do banco (C08)
  Dado que o access_token expirou
  E o usuário tem role com scopes [users:role:read, process:cycle:read]
  Quando POST /auth/refresh com cookie refreshToken
  Então status 200 com novo access_token
  E o novo access_token contém scopes [users:role:read, process:cycle:read]
  E o novo access_token contém tid do tenant ativo

Cenário: Refresh reflete mudança de role (C08)
  Dado que um admin removeu scope process:cycle:read da role do usuário
  Quando POST /auth/refresh
  Então o novo access_token NÃO contém process:cycle:read

Cenário: Refresh sem tenant ativo (C08)
  Dado que o tenantUser do usuário foi desativado
  Quando POST /auth/refresh
  Então status 401 (SessionRevokedError)

Cenário: Compatibilidade com tokens pré-fix (C08)
  Dado que o refresh_token foi emitido antes do fix (sem scopes/tid no payload)
  Quando POST /auth/refresh
  Então status 200 — scopes e tenantId re-buscados do banco normalmente
```
