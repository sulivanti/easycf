# ADR-002 — API Key via bcrypt (Por Que bcrypt e Não HMAC/JWT para Agentes)

## Contexto

Agentes MCP precisam de um mecanismo de autenticação no endpoint `POST /mcp/execute`. O módulo precisa decidir entre: (a) API key armazenada como hash bcrypt com compare em cada requisição, (b) HMAC com segredo compartilhado e assinatura por requisição, (c) JWT de curta duração emitido para o agente, ou (d) mTLS com certificados por agente.

O requisito BR-004 exige que a API key seja retornada UMA VEZ (criação ou rotação) e NUNCA exposta novamente. O agente MCP opera programaticamente — não tem sessão humana interativa.

## Decisão

**Autenticação de agentes MCP via API key (256 bits base64url) armazenada como hash bcrypt (rounds >= 12).** Cada requisição ao gateway envia a key no header `X-MCP-Agent-Key` e o gateway executa `bcrypt.compare()` contra o hash armazenado em `mcp_agents.api_key_hash`.

## Alternativas Consideradas

1. **HMAC com segredo compartilhado:** Agente assina cada requisição com HMAC-SHA256 usando segredo compartilhado. Vantagem: compare é O(1) sem custo computacional de bcrypt. Desvantagem: segredo precisa ser armazenado em plaintext (ou com envelope encryption) no servidor para validar a assinatura — pior postura de segurança em caso de vazamento de banco. Além disso, adiciona complexidade ao cliente (calcular assinatura com timestamp/nonce para evitar replay).
2. **JWT de curta duração:** Agente obtém JWT via `client_credentials` grant e envia no header `Authorization`. Vantagem: validação em O(1) via verificação de assinatura JWT, sem hit no banco. Desvantagem: adiciona fluxo OAuth2 que não existe para agentes MCP (eles não são clientes OAuth2). A gestão de refresh tokens para agentes automatizados seria complexa e adicionaria dependência do identity provider.
3. **mTLS:** Certificado X.509 por agente com validação no TLS handshake. Vantagem: autenticação no nível de transporte, sem custo por request. Desvantagem: complexidade operacional elevada (PKI, rotação de certificados, CRL/OCSP). Desproporcional para o escopo do MOD-010.

## Consequências

- **Positivas:** Modelo simples — um segredo (API key) gerado pelo servidor, retornado uma vez, armazenado pelo cliente. Em caso de vazamento de banco, atacante obtém apenas hashes bcrypt (custo de brute-force proporcional a 2^256 * custo_bcrypt). Rotação trivial via `POST /rotate-key`. Sem dependência de identity provider externo. Sem estado de sessão.
- **Negativas:** bcrypt compare é computacionalmente caro (~60-80ms por requisição). Isso limita throughput do gateway e adiciona latência fixa a cada execução. Para volumes altos (>100 req/s), pode exigir pool dedicado ou hardware com aceleração.
- **Riscos:** Se o volume de agentes e requisições crescer, o custo de bcrypt pode se tornar gargalo. Mitigação: monitorar `mcp_bcrypt_duration_seconds` e considerar migração para Argon2id (mais moderno, configurável em memória/paralelismo) ou offloading para worker pool.
